    AI 협업 · 투명성

    AI는 어디에 어떻게 쓰였는가

    1. 사용 정의

    AI는 생성·탐색·리팩토링의 속도를 올리는 보조 도구로 썼다. 설계의 근거와 평가, 가설 갱신은 본인이 수행하였다. 아래 표는 각 도구의 실제 역할이다.

    2. 개발 AI 분업

    도구주 용도구체 사례
    Codex리서치·스켈레톤초기 폴더 구조, 탐지 규칙 의사코드
    Claude Code리팩토링·테스트 설계402개 pytest 전수, SSE 스트림 재작성
    Cursor실시간 IDE 보조타입 힌트 보정, 오탈자 수정
    Claude + Obsidian설계 의사결정 기록Daily 노트 · 가설 수정 히스토리

    3. 런타임 AI 분업

    의도적으로 3개 공급자를 동시에 운영한다. Anthropic이 primary, Gemini는 영미권 뉴스 요약에 강점, Groq는 초저 레이턴시 fallback.

    • Primary — Anthropic (Claude) · 한국어 뉘앙스 · 규범 준수
    • Fallback 1 — Google Gemini · 비용 · 영미 컨텍스트
    • Fallback 2 — Groq · 50ms 단위 응답 · 장애 시 최후 안전망
    장애 상황 재현 테스트에서 3단 라우터가 평균 2.4초 이내에 결과를 반환하는 것을 확인했다. primary 전용 구성은 같은 조건에서 타임아웃 10초 이상이었다.

    4. AI가 하지 않는 것

    • 추천 코인/포지션 제안 — 가격 예측을 생성하지 않는다
    • 가설 수립·KPI 결정 — 지원자 본인이 한다
    • 최종 배포 승인 — 테스트 green + 리뷰 후 사람이 푸시

    5. 재현 가능성

    설계 · 리팩토링 세션 로그는 Obsidian에 날짜별로 남아 있다. 파이프라인의 deterministic 구간(규칙 · 집계)은 재실행 시 동일 입력에 동일 출력이며, LLM 구간은 프롬프트 버저닝과 seed 고정으로 변동 폭을 줄였다.

    판단 가능한 서비스인지 수치로 증명한다

    1. 북극성 & Kill 기준

    북극성은 Daily Brief 열람률 60%, Kill 기준은 2주 기준 30/10/30.

    허위 시그널 비율< 10%> 30%

    2. 테스트 커버리지

    pytest 402개 전체 green. smoke fallback(keyless 빌드) 경로 포함. 단위 테스트(수집·집계·라우터 fallback)와 시나리오 테스트(전체 파이프라인 E2E mock)가 같이 돈다.

    • 수집기 단위 — chain별 fixture 기반 parser 테스트
    • 규칙 엔진 — 8개 규칙 × edge case per-rule
    • LLM 라우터 — primary/fallback 3가지 장애 시나리오
    • SSE stream — Redis L2 TTL(45/60초) 경계 테스트

    4. 하이브리드 브리핑 — 비용 절감 근거

    KST 09 · 15 · 21시에만 전체 브리핑 생성, 나머지 시간은 증분(incremental)으로만 업데이트한다. LLM 호출 수 감소로 $21 → $9/month.

    "하루 3회의 진짜 레터 + 그 사이의 얇은 증분" — 독서감 훼손 없이 비용 절반 이하로 내렸다.

    5. 비추적 지표

    의도적으로 추적하지 않는 지표가 있다. 일일 발송 수량 · DAU 절대값 · 시그널 수. 이것을 KPI로 두면 양을 늘리기 위해 품질을 희생하게 된다.

    Decision Log · Day 1 → Day 10

    가설은 어떻게 갱신되었는가

    Day 1–3 · 리서치 → IA → 스택

    • 가설 ① Whale Alert API를 구독해 빠르게 붙인다 → 기각. 체인 커버리지 부족 + 월 비용이 MVP 예산 초과. self-collect 방향으로 선회.
    • 가설 ② 규칙 엔진만으로 충분하다 → 부분 기각. "왜 중요한가"에 답할 컨텍스트가 없어 한국어 해석 레이어 필요. LLM 라우터 설계 도입.

    Day 4–6 · 구현

    • 가설 ③ Streamlit이 7일에 적합하다 → 기각. 공유 URL · SEO · 반응성이 나오지 않는다. Next.js 15 App Router로 재작성 결정. 첫 2일 속도 손해는 이후 운영 · 확장에서 회수됨.
    • 가설 ④ Anthropic 단일 공급자로 충분 → 기각. 장애 재현 테스트에서 타임아웃 10초 이상. 3단 라우터 도입.

    Day 7 · MVP 제출

    제출 시점 기준 체인 6개 상시 + 4개 카나리 + 규칙 8개 + 라우터 3공급자 + pytest 402. 지원서와 함께 이 /about 페이지의 초안을 동봉.

    Day 8–10 · 제출 후 이터레이션

    • 번들 분석 · 코드 스플리팅 → First Load JS −34%
    • 하이브리드 브리핑 도입 → Runtime cost −57%
