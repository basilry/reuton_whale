## 결론

현재 개발분은 Whale Alert 의존형 MVP에서 자체 수집, 자체 LLM 라우터, 규칙 기반 SignalEngine, 다중 Sheets 탭, Telegram 개인화 브리핑으로 이동한 구조적 개선이다. 방향성은 타당하고 테스트 기반도 전보다 훨씬 좋아졌지만, 운영 데이터 흐름의 몇 지점은 아직 "기능은 구현됐으나 프로덕션에서 학습/기록/전달이 누락될 수 있는" 상태다.

Critical 수준의 즉시 중단 결함은 확인되지 않았다. 다만 아래 Major 항목 4개는 런타임 품질, 시그널 신뢰도, 비용/프롬프트 추적성에 직접 영향을 주므로 main 후속 정상화의 우선순위로 잡는 것이 맞다.

## 코드리뷰 Findings

### Major 1. `address_activity`가 일일 파이프라인에서 저장되지 않아 baseline 기반 시그널이 장기적으로 비어 있을 수 있음

위치:

- `src/main.py:276`
- `src/main.py:370`
- `src/signals/baseline.py:50`
- `src/storage/sheets_client.py:440`

내용:

`build_chain_baselines(sheets, now)`는 `storage.list_address_activity(since=since)`를 읽어 체인/토큰/방향별 baseline을 만든다. 그런데 `src/main.py`의 Stage 9 저장 경로는 `sheets.append_transactions(transactions)`만 호출하고, `append_address_activity()`는 호출하지 않는다. 즉, normal daily pipeline만 돌릴 경우 `address_activity` 탭이 자동으로 누적되지 않아 `cex_outflow_spike`, `cex_inflow_spike` 같은 baseline 기반 규칙이 충분한 과거 데이터를 얻지 못한다.

영향:

- baseline 기반 이상 탐지가 운영 환경에서 계속 약하게 동작하거나 비활성 상태에 가까워질 수 있다.
- 테스트/스모크에서는 fixture와 dry-run이 통과하므로 프로덕션에서 늦게 발견될 가능성이 있다.
- `transactions` 탭과 `address_activity` 탭의 역할이 분리됐지만, 파이프라인이 두 저장소를 모두 채우지 않는다.

권장 수정:

- `Event`를 `ADDRESS_ACTIVITY_HEADERS`로 변환하는 단일 adapter를 추가한다.
- Stage 9에서 `append_transactions(transactions)`와 함께 `append_address_activity(raw_events)`를 호출한다.
- 중복 방지는 `tx_hash + watched_address + direction + token` 조합으로 정의한다.
- `main.run_pipeline` 통합 테스트에 `append_address_activity` 호출 검증을 추가한다.

### Major 2. Signal 기반 `daily_brief.top_transactions`가 aggregate/TG-only 신호에서 빈 심볼, 0 USD로 저장될 수 있음

위치:

- `src/main.py:95`
- `src/main.py:100`
- `src/main.py:107`
- `src/main.py:109`
- `src/main.py:381`

내용:

`_signals_to_top5()`는 `Signal.evidence_tx_hashes`를 기준으로 `raw_events`를 역참조해 `symbol`, `amount_usd`, `hash`를 만든다. 하지만 aggregate signal이나 Telegram 기반 이벤트는 `evidence_tx_hashes`가 비어 있거나 `events_by_hash`에 매칭되지 않을 수 있다. 이 경우 `hash=""`, `symbol=""`, `amount_usd=0`이 되고, 이 값이 `daily_brief.top_transactions`에 저장된다.

영향:

- 저장된 daily brief와 Streamlit 대시보드에서 의미 있는 시그널이 빈 거래처럼 보일 수 있다.
- `total_volume_usd`가 실제 신호 규모보다 작게 계산된다.
- LLM 브리핑 본문은 좋아도 구조화 데이터 품질이 떨어져 후속 분석과 UI 신뢰도가 낮아진다.

권장 수정:

- `top_transactions`라는 이름을 유지하더라도 signal-first schema로 저장한다.
- 필드에 `signal_id`, `rule`, `severity`, `score`, `summary`, `source`, `evidence_count`, `window_start`, `window_end`를 포함한다.
- exact tx hash가 없을 때는 `Signal.extra`의 집계값이나 evidence window aggregate를 사용한다.
- 금액을 알 수 없는 경우 0이 아니라 `null` 또는 별도 `amount_usd_known=false`로 표현한다.

### Major 3. Telegram 개인화 전송 경로가 LLM daily brief를 우회함

위치:

- `src/main.py:410`
- `src/distributor/telegram_bot.py:110`
- `src/distributor/telegram_bot.py:115`

내용:

`main`은 `analyzer.generate_daily_brief(signals)`로 LLM 브리핑을 생성한 뒤 `bot.send_daily_brief(brief_text, signals=signals)`를 호출한다. 그런데 `TelegramBot._brief_for_subscriber()`는 `signals`와 `personalize_fn`이 있으면 `brief_text`를 사용하지 않고 `_format_signal_brief(personalized)`를 반환한다. 결과적으로 signal이 하나라도 있는 운영일에는 개인화 경로를 가진 구독자가 LLM이 작성한 daily brief 대신 rule summary 템플릿만 받을 수 있다.

영향:

- README와 아키텍처에서 강조한 LLM narrative 가치가 실제 사용자 전송 경로에서 사라질 수 있다.
- 개인화 관심사가 없는 구독자도 동일한 template-only 요약을 받을 가능성이 있다.
- 저장된 `daily_brief.summary`와 Telegram 수신 메시지가 서로 달라져 운영 디버깅이 어려워진다.

권장 수정:

- 기본값은 항상 `brief_text`를 전송한다.
- 실제 user interest가 있고 필터링 결과가 달라지는 경우에만 personalized section을 추가한다.
- 개인화가 필요하면 `brief_text + "\n\n개인 관심 시그널"` 구조로 보강하고, 원문 LLM 브리핑을 버리지 않는다.
- 테스트에서 "signals가 있어도 관심사가 없으면 brief_text가 유지된다"는 케이스를 추가한다.

### Major 4. 신규 LLM telemetry 필드가 `analysis_log` 실제 Sheets schema에 저장되지 않음

위치:

- `src/analyzer/claude_analyzer.py:203`
- `src/analyzer/claude_analyzer.py:220`
- `src/main.py:347`
- `src/storage/schema.py:51`
- `src/storage/sheets_client.py:361`
- `src/storage/sheets_client.py:572`

내용:

신규 `LLMAnalyzer`는 `task`, `model_id`, `prompt_version`, `tokens_in`, `tokens_out`, `cost_usd`, `latency_ms`를 `save_analysis_log()`에 넘긴다. 하지만 `ANALYSIS_LOG_HEADERS`는 여전히 `prompt_hash`, `prompt`, `response`, `model`, `tokens_used`, `created_at`만 정의한다. `save_analysis_log()`는 `save_analysis()` alias이고, `dict_to_row(entry, ANALYSIS_LOG_HEADERS)`는 header에 없는 신규 telemetry 필드를 버린다.

영향:

- 실제 Sheets의 `analysis_log`에는 신규 모델/프롬프트/비용/latency 정보가 남지 않는다.
- README의 "LLM 호출, prompt version, 캐시 추적" 설명과 운영 데이터가 불일치한다.
- 비용 최적화, prompt regression 추적, 모델 fallback 분석이 어려워진다.
- 기존 worksheet는 `_ensure_worksheets()`가 header를 갱신하지 않으므로 schema만 바꿔도 마이그레이션이 필요하다.

권장 수정:

- `ANALYSIS_LOG_HEADERS`를 신규 telemetry 중심으로 확장한다.
- cache 저장용 필드와 call telemetry 필드를 한 탭에 둘지, `analysis_cache`와 `llm_usage_log`로 분리할지 결정한다.
- 기존 Google Sheet의 header migration 스크립트를 제공한다.
- `save_analysis_log()`가 telemetry entry를 저장했을 때 실제 row에 `model_id`, `prompt_version`, `cost_usd`가 남는 테스트를 추가한다.

## 다관점 리뷰

### 제품/사용자 가치

좋아진 점:

- 단순 고래 거래 알림에서 "시그널 기반 브리핑"으로 제품 가치가 선명해졌다.
- Whale Alert API 의존도를 낮추고 Telegram, 자체 주소 감시, rule engine으로 차별화 방향이 생겼다.
- 사용자 관심사 기반 개인화 구조가 들어가 Telegram bot의 반복 사용성을 높일 수 있다.

리스크:

- 현재는 "시그널이 얼마나 맞았는지"에 대한 outcome loop가 부족하다.
- 사용자에게 보이는 브리핑과 저장되는 구조화 데이터가 일부 경로에서 다르게 생성될 수 있다.
- 대시보드/Telegram/Sheets가 같은 truth를 보고 있다는 보장이 아직 약하다.

권장:

- 매일 발송된 signal에 대해 24h/72h 후 가격 변화와 volatility outcome을 기록한다.
- 사용자 피드백 명령어를 추가해 "유용함/노이즈" 라벨을 수집한다.
- 데모에서는 단순 거래 목록보다 "왜 중요한지, 과거 대비 얼마나 이례적인지"를 전면에 둔다.

### 아키텍처/엔지니어링

좋아진 점:

- `StorageProtocol`과 다중 Sheets tab으로 이전보다 책임 분리가 좋아졌다.

리스크:

- Google Sheets가 운영 DB 역할을 계속 맡기에는 중복 제어, migration, 대량 row read 비용이 커질 수 있다.
- 신규 탭이 추가됐지만 기존 worksheet header migration 전략이 없다.

권장:

- "schema 변경 시 migration required" 원칙을 문서화하고 스크립트화한다.
- `transactions`, `address_activity`, `signals`, `daily_brief`, `analysis_log` 각각의 owner와 write path를 README에 명시한다.
- 중복 방지 key를 탭별로 명확히 정의한다.

### 운영/SRE

좋아진 점:

- 전체 테스트 238개와 smoke pipeline이 통과한다.
- retry/backoff, Telegram blocked 처리, dry-run, smoke script가 있어 운영 점검 기반이 있다.

리스크:

### 데이터/시그널 품질

좋아진 점:

- 규칙 기반 시그널은 설명 가능성이 높고 디버깅하기 쉽다.
- `cex_outflow`, `cex_inflow`, Telegram burst, baseline spike 등 제품 스토리와 연결되는 rule이 들어갔다.
- LLM을 raw decision maker가 아니라 narrative layer로 쓰는 방향은 비용과 안정성 측면에서 좋다.

리스크:

- baseline source가 비어 있으면 이상치 탐지 품질이 실제보다 과대평가될 수 있다.
- aggregate signal의 evidence와 amount가 구조화 저장에서 손실될 수 있다.
- false positive/false negative를 추적하는 데이터셋이 아직 없다.

권장:

- rule별 precision proxy를 만든다. 예: 24h return, volume spike, liquidation proximity.
- signal row에 `outcome_24h`, `outcome_72h`, `review_label` 같은 사후평가 필드를 별도 탭으로 저장한다.
- LLM 브리핑 prompt에는 signal evidence와 confidence를 분리해서 넣는다.

### 보안/프라이버시

좋아진 점:

- paid Whale Alert API key 의존을 제거하거나 낮춘 것은 운영 보안과 비용 면에서 유리하다.
- `.env.example` 중심으로 secret 설정 경로가 정리됐다.
- Telegram subscriber 정보는 최소 필드 중심이다.

리스크:

- Telegram channel raw text와 subscriber interest는 운영상 개인정보 또는 민감 관심사로 취급해야 한다.
- LLM provider로 전달되는 prompt에 raw event와 사용자 관심사가 섞이면 데이터 처리 정책이 필요하다.
- Google Sheets 권한이 넓게 열리면 운영 데이터 전체가 노출될 수 있다.

권장:

- service account 권한을 단일 spreadsheet로 제한한다.
- Telegram raw text 보존 기간을 정한다.
- LLM prompt에는 subscriber 식별자를 넣지 않는 원칙을 유지한다.
- README에 "전송되는 데이터 범위"를 짧게 명시한다.

### 비용/FinOps

좋아진 점:

- 자체 `LLMRouter`는 OpenRouter 의존 제거, provider fallback, task별 모델 선택의 기반이 된다.
- prompt cache와 analysis log 구조가 있으면 비용 추적이 가능하다.
- signal-first로 바꾸면서 모든 거래를 LLM 분석하지 않아도 되는 구조가 됐다.

리스크:

- `analysis_log` schema mismatch 때문에 실제 비용 필드가 저장되지 않는다.
- 개인화 브리핑을 사용자별 LLM 생성으로 확장하면 비용이 급격히 늘 수 있다.

권장:

- 우선은 공통 LLM brief 1개 + rule-based personalized appendix 방식을 유지한다.
- `daily_brief`, `weekly_trend`, `nl_intent` task별 token/cost budget을 README에 적는다.
- `cost_usd`가 하루 임계치를 넘으면 fallback model 또는 no-LLM summary로 전환하는 guardrail을 둔다.

### 포트폴리오/사업화

좋아진 점:

- "API wrapper"가 아니라 ingestion, rule engine, LLM orchestration, delivery, dashboard를 갖춘 제품형 프로젝트로 보인다.
- README 업데이트로 실행/운영 설명이 강화됐다.
- 자체 LLM router와 signal engine은 기술 인터뷰나 포트폴리오 설명에서 강점이 된다.

리스크:

- 실제 운영 스크린샷, 발송 샘플, signal outcome이 없으면 완성도 증거가 약하다.
- 시그널 품질이 검증되지 않으면 "알림 봇" 이상으로 설득하기 어렵다.

## 정상화 우선순위

### P0. 운영 데이터 흐름 정합성

- `raw_events`를 `address_activity`에 저장하도록 main pipeline을 수정한다.
- `analysis_log` schema를 신규 telemetry에 맞게 갱신하고 existing Sheet migration을 제공한다.
- `_signals_to_top5()`가 hash 없는 signal도 의미 있는 구조화 데이터로 저장하도록 수정한다.

### P1. 사용자 수신 경험 정합성

- Telegram 전송에서 LLM `brief_text`를 기본 본문으로 유지한다.
- 개인화는 본문 대체가 아니라 관심 시그널 섹션 추가로 처리한다.
- daily_brief 저장본과 Telegram 발송본의 차이를 run log에 남긴다.

### P2. 운영 품질과 신뢰도 강화

- stale TRACK 주석을 제거한다.
- staging Sheet 기반 canary run을 추가한다.

## 리뷰 신뢰도와 잔여 리스크

신뢰도:

- 정적 코드 검토, 변경 범위 확인, 전체 테스트, smoke test까지 수행했으므로 코드 수준의 주요 연결 문제는 상당 부분 확인했다.
- 외부 API 실계정, 실제 Google Sheet, 실제 Telegram channel/listener는 이번 리뷰에서 직접 실행하지 않았다.

잔여 리스크:

- 실제 Sheets header가 현재 코드 schema와 다른 경우 운영 결과가 로컬 테스트와 다를 수 있다.
- Telethon listener의 장시간 실행 안정성은 별도 soak test가 필요하다.
- signal 품질은 코드 통과 여부가 아니라 운영 후 outcome 데이터로 검증해야 한다.

## 최종 판단

현재 개발 방향은 유지해도 된다. 다만 다음 개발 사이클은 새 기능 추가보다 "운영 데이터 경로 정상화"에 집중해야 한다. 특히 `address_activity` 누적, `analysis_log` telemetry 저장, signal summary 구조화, Telegram 브리핑 경로 정합성을 먼저 해결하면 WhaleScope는 데모 수준을 넘어 운영 가능한 실험 제품으로 올라갈 수 있다.
