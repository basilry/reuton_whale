# WhaleScope 개선안 적용내역

## 1. 요약

2026-04-16 리뷰 기반 개선안의 주요 항목을 코드에 반영했다.

핵심 적용 범위는 다음과 같다.

- Telegram listener 수집 데이터가 daily pipeline으로 다시 들어가도록 `tg_whale_events` read path와 Event adapter를 추가했다.
- 운영자 API는 `DASHBOARD_PASSWORD` 기반 인증과 generic error response를 적용했다.
- 사용자용 `/insights` 화면은 내부 env, Google API, stack trace성 오류를 노출하지 않게 정리했다.
- Telethon listener는 synchronous LLM/Sheets I/O가 event loop를 막지 않도록 `asyncio.to_thread` 기반으로 분리했다.
- listener heartbeat를 `system_log`에 기록하고, 운영 대시보드는 Sheets 연결 여부가 아니라 `run_type=telethon_listener` heartbeat를 기준으로 상태를 표시하게 했다.
- Next.js dashboard를 운영자용 `/`와 사용자용 `/insights`로 분리하고, 디자인 레퍼런스 기반의 ocean/whale tone UI를 적용했다.
- README와 dashboard README에 실제 실행 단위, Render/Vercel 배포 경계, Telethon 인증, listener health 기준을 반영했다.

현재 상태:

- 변경사항은 아직 커밋되지 않았다.
- `main` 작업트리에 수정/신규 파일이 남아 있다.
- `.dmux-hooks/`도 untracked로 보이나, 이번 개선 적용 범위로 확인한 항목은 아니므로 커밋 전 포함 여부를 별도로 판단해야 한다.

## 2. 리뷰 Finding별 반영 상태

| Finding | 상태 | 반영 내용 |
|---|---|---|
| Listener data is never fed back into the signal pipeline | 반영 | `SheetsClient.list_tg_whale_events(since)` 추가, TG row를 `Event(source="tg")`로 변환, `run_daily_pipeline()` Stage 3에서 chain events와 병합 |
| Public API exposes operational data and raw error messages | 반영 | 운영 API route에 `requireDashboardAuth()` 적용, raw error message 대신 generic response 반환 |
| User page leaks internal load errors | 반영 | `/insights`에서 내부 error string을 UI에 렌더링하지 않고 서버 로그로만 남김 |
| Telethon handler blocks the event loop on LLM and Sheets I/O | 반영 | LLM fallback과 Sheets append를 `asyncio.to_thread()`로 실행 |
| Listener health card is inferred from Sheets connectivity | 반영 | `system_log`의 `telethon_listener` heartbeat 기반 `listenerHealth` 모델 추가 |
| React duplicate key warning in user story list | 반영 | `/insights` story key에 `hash`와 `index`를 포함해 중복 key 경고 제거 |

## 3. Backend Pipeline 변경

### 3-1. TG 이벤트 read path

변경 파일:

- `src/storage/protocol.py`
- `src/storage/sheets_client.py`
- `src/main.py`
- `tests/test_storage_new_tabs.py`
- `tests/test_main.py`

적용 내용:

- `Storage` Protocol에 `list_tg_whale_events(since=None)`를 추가했다.
- `SheetsClient.list_tg_whale_events()`는 `tg_date` 또는 `collected_at` 기준으로 since 필터링한다.
- `src/main.py`에 TG row adapter를 추가했다.
- `run_daily_pipeline(dry_run=False)` Stage 3에서 최근 24시간 `tg_whale_events`를 읽고 `raw_events`에 병합한다.
- TG 이벤트는 signal pipeline에는 들어가지만 `transactions`, `address_activity`에는 저장하지 않는다. 이유는 TG row에는 실제 `tx_hash`가 없을 수 있고, owner label이 address가 아닐 수 있기 때문이다.

설계 판단:

- TG 이벤트의 `tx_hash`는 기본적으로 `None`이다.
- owner label은 address처럼 표시하지 않도록 label 그대로 안전하게 사용한다.
- TG 이벤트는 `source="tg"`로 유지해 규칙과 dashboard에서 chain 이벤트와 구분 가능하게 했다.

## 4. SignalEngine / Rules 보정

변경 파일:

- `src/signals/rules.py`
- `tests/test_corroboration.py`

적용 내용:

- TG 이벤트가 `raw_events`에 들어오면서 chain-only rule을 오염시킬 수 있는 위험을 차단했다.
- `cex_outflow_spike`, `cex_inflow_spike`, `cold_to_hot_transfer`, `smart_money_accumulation`, `weekly_net_accumulation`에서 chain 전용 이벤트만 처리하도록 `e.source == "chain"` 조건을 추가했다.
- `tests/test_corroboration.py`에 TG 단독 이벤트가 chain-only `cex_inflow_spike`를 만들지 않는 회귀 테스트를 추가했다.

중요성:

- 이 보정이 없으면 Telegram row가 `cex_inflow_spike` 같은 chain rule을 `source="chain"`처럼 발생시킬 수 있다.
- 교차검증은 `corroborated_move`와 SignalEngine의 `source="both"` 승격 경로에서만 의미 있게 처리되어야 한다.

## 5. Telethon Listener 안정화

변경 파일:

- `src/ingestion/telethon_listener.py`
- `scripts/run_listener.py`
- `scripts/create_telethon_session.py`
- `src/config.py`
- `tests/test_config.py`
- `tests/test_telethon_parser.py`
- `.env.example`
- `.gitignore`

적용 내용:

- `TELETHON_PHONE`과 `TELETHON_SESSION_STRING`을 config에 추가했다.
- 최초 로컬 로그인은 `TELETHON_PHONE=+82...` 형식으로 가능하게 했다.
- Render 같은 non-interactive 환경에서는 `TELETHON_SESSION_STRING`을 사용하도록 했다.
- `scripts/create_telethon_session.py`로 StringSession 생성 유틸을 추가했다.
- `.gitignore`에 `*.session`, `*.session-journal`을 추가해 Telegram session 파일이 커밋되지 않게 했다.
- LLM fallback과 Sheets append는 `asyncio.to_thread()`로 event loop 밖에서 실행한다.
- storage append 실패와 LLM fallback 실패는 listener 프로세스를 죽이지 않고 warning/error로 기록한다.

주의:

- `TELETHON_SESSION_STRING`은 Telegram 계정 접근 권한을 가진 secret이다.
- 전화번호는 `010...`이 아니라 국가번호 포함 E.164 형식이어야 한다.
- 문서에는 예시 형식만 남기고 실제 개인 번호는 기록하지 않는다.

## 6. Listener Heartbeat와 Dashboard Health

변경 파일:

- `src/ingestion/telethon_listener.py`
- `apps/dashboard/lib/metrics.ts`
- `apps/dashboard/app/page.tsx`
- `README.md`
- `apps/dashboard/README.md`

적용 내용:

- listener 시작 시 `system_log`에 `event=listener_start` 기록
- 메시지 처리 성공 시 `event=message_processed` 기록
- 인증 오류 시 `event=auth_error` 기록
- 저장/처리 오류 시 `event=message_error` 기록
- dashboard `DashboardData`에 `listenerHealth` 모델 추가
- 운영 대시보드의 Telegram listener card는 `sourceState` 또는 최신 daily pipeline run status가 아니라 `run_type=telethon_listener` 로그를 기준으로 상태를 산출한다.

상태 산출:

| 상태 | 의미 |
|---|---|
| `정상` | 최근 `message_processed` heartbeat가 있음 |
| `대기 중` | listener가 실행 중이나 최근 처리 메시지는 없음 |
| `인증 필요` | Telethon session 미인증 또는 phone/auth 오류 |
| `확인 필요` | listener heartbeat가 오래됐거나 처리 오류가 기록됨 |

해결한 문제:

- 기존에는 Google Sheets가 연결되어 있으면 listener도 사용 가능한 것처럼 보일 수 있었다.
- 이제 Sheets 연결 성공과 listener 실행/인증 상태가 분리된다.

## 13. 배포 전 체크리스트

- listener에는 `TELETHON_API_ID`, `TELETHON_API_HASH`, `TELETHON_SESSION` 또는 `TELETHON_SESSION_STRING`, `TG_CHANNEL` 필요
- 비대화형 환경에서는 `TELETHON_SESSION_STRING` 사용 권장

Google Sheets:

- 서비스 계정 이메일에 Spreadsheet 편집자 권한 부여
- `python -m scripts.init_sheets` 실행
- `python scripts/import_watched_addresses.py` 실행
- `system_log`에서 `telethon_listener` heartbeat 확인

## 14. 남은 리스크

- `DASHBOARD_PASSWORD`를 Vercel Production에 설정하지 않으면 운영 API 인증은 비활성화된다.
- listener heartbeat는 Google Sheets에 의존하므로 Sheets 자체가 장애 상태면 heartbeat도 기록되지 않는다.
- TG 이벤트는 tx hash가 없을 수 있어 교차검증은 시간/금액 tolerance 중심으로 동작한다.
- Render에서 Telethon 인증을 안정화하려면 `TELETHON_SESSION_STRING` 또는 persistent disk 중 하나를 확정해야 한다.
- `.dmux-hooks/` untracked 항목은 커밋 범위 판단이 필요하다.
