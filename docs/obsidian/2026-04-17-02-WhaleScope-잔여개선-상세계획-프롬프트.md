## Phase B — Dashboard UI 심화 (SystemLogPanel client 전환 + 모달 + MetricCard anchor)

### B.1 목표

2026-04-17 진행 노트 "미완료 항목"에서 남긴 Dashboard UI 심화 작업 4건을 해소한다.

3. `apps/dashboard/app/page.tsx`의 `SERVICE_ACTIONS` 상수에 `href` 필드 추가하고 MetricCard에 실제 전달
4. 모달 오버레이 CSS(`.log-detail-modal`, 백드롭) 추가

### B.2 파일:라인:함수 수준 상세

#### B.2.a `apps/dashboard/components/system-log-panel.tsx`
- 파일 최상단에 `"use client"` directive 추가
- `import { useState } from "react"` 추가
- 컴포넌트 내부에 `const [selectedLog, setSelectedLog] = useState<SystemLog | null>(null)` 도입
- 각 로그 row를 `<button type="button" onClick={() => setSelectedLog(row)}>`로 감싸거나 `<div role="button" tabIndex={0}>` 패턴 사용(접근성 고려)
- 모달 JSX: `selectedLog && <div className="log-detail-modal">...<pre>{JSON.stringify(selectedLog, null, 2)}</pre><button onClick={() => setSelectedLog(null)}>닫기</button></div>`
- ESC 키로 닫기 지원(`useEffect`에서 keydown 리스너)

#### B.2.b `apps/dashboard/components/metric-card.tsx`
- `MetricCardAction` 타입을 `{ label: string; href?: string; onClick?: () => void }`로 확장
- 렌더링 시 `href` 있으면 `<a>`, 없으면 `<button>` — 서버 컴포넌트 호환 위해 기본은 `<a>` 우선

```typescript
{action.href ? (
  <a href={action.href} className="metric-card__action-btn">{action.label}</a>
) : (
  <button type="button" onClick={action.onClick} className="metric-card__action-btn">
    {action.label}
  </button>
)}
```

- 주의: `<a>`로 렌더링하면 client 컴포넌트가 아니어도 동작. 모달 유형 액션은 `SystemLogPanel` 내부에서만 허용하고 MetricCard는 링크 액션만 받는 쪽으로 격리.

#### B.2.c `apps/dashboard/app/page.tsx`

import { timingSafeEqual } from "node:crypto";

export function verifyDashboardPassword(provided: string | null | undefined): boolean {
  const expected = process.env.DASHBOARD_PASSWORD;
  if (!expected || !provided) return false;
  if (provided.length !== expected.length) return false;
  const a = Buffer.from(provided);
  const b = Buffer.from(expected);
  return timingSafeEqual(a, b);
}

## Phase D — P2 Event loop blocking 해소 (`asyncio.to_thread` 도입)

### D.1 문제

- `src/ingestion/telethon_listener.py:99-110` — `_router.call_task()` 동기 호출(LLM API 2~3s 블로킹)
- `src/storage/sheets_client.py:548-563` — `append_tg_whale_event()` 동기 gspread 호출(200~500ms 블로킹)
- listener가 메시지 burst를 받으면 event loop이 막혀 heartbeat 카운트가 밀리고 Telegram 연결이 disconnect 될 수 있음.

### D.2 수정 전략

```python
# src/ingestion/telethon_listener.py

async def _handle_message(self, msg_id, msg_date, text):
    parsed = parse_tg_message(text)
    confidence = "high"

    if parsed is None and self._router is not None:
        try:
            result = await asyncio.to_thread(
                self._router.call_task,
                "nl_intent",
                system="Extract whale transaction details as JSON.",
                user=text,
            )
            parsed = json.loads(result.text)
            confidence = "low"
        except Exception as exc:
            self._error_count += 1
            logger.warning("LLM fallback failed msg_id=%s: %s", msg_id, exc)
            return

    if parsed is None:
        return

    event_row = { ... }

    try:
        await asyncio.to_thread(self._storage.append_tg_whale_event, event_row)
    except Exception as exc:
        self._error_count += 1
        logger.error("Storage failed msg_id=%s: %s", msg_id, exc)
        return

    self._last_message_at = datetime.now(timezone.utc)
    self._message_count += 1
```

- router, storage가 스레드 안전하지 않을 가능성 → gspread의 `Worksheet`는 단일 클라이언트 기준 스레드 안전하지 않음. 동시 호출 방지를 위해 listener 내부에 `asyncio.Lock`을 두고 `to_thread` 호출 구간을 감싼다.

### D.3 리스크

| 리스크 | 완화책 |
|---|---|
| gspread 스레드 안전성 | listener 내부 `asyncio.Lock`으로 직렬화. 성능 저하는 있지만 event loop unblock이 우선 |
| `to_thread`로 내보낸 예외 전파 | try/except로 감싸고 `_error_count` 증가 + 로그 |
| anthropic/gemini/groq 클라이언트가 내부적으로 async-safe하지 않은 경우 | 이미 동기 `call_task`로 고정되어 있음. 스레드로 내보내는 것은 blocking 제거 목적 |

## 병렬 실행 가이드 (TeamCreate / git worktree)

Phase B, C, E는 서로 파일 겹침이 없다(확인: B는 `apps/dashboard/components|app|globals.css`, C는 `src/ingestion/*`, `apps/dashboard/lib/auth.ts`, `src/storage/protocol.py`, E는 `apps/dashboard/lib/format.ts`, `src/ingestion/normalizer.py`, `tests/test_storage_tg_events.py`).

C와 E가 `apps/dashboard/lib/` 하위에서 다른 파일을 만진다는 점만 주의하면 병렬 worktree 실행 가능.

**worktree 권장 구성**

```bash
cd /Users/basilry/Projects/02015_reuton_whale
git worktree add .claude/worktrees/phase-b main
git worktree add .claude/worktrees/phase-c main
git worktree add .claude/worktrees/phase-e main
```

각 worktree에서 위 Phase별 프롬프트를 독립 세션으로 실행. 완료 후 각 worktree에서 직접 main으로 push하지 말고 별도 브랜치로 push해 PR 세 개를 만들면 리뷰가 쉬움.

**이전 세션 교훈 반영** — 2026-04-17 진행 노트에 따르면 worktree race condition과 에이전트 권한 거부 이슈가 있었음. 이번에는 다음을 추가:
- worktree 생성 순차 실행(병렬 금지). `git worktree add` 3번을 직렬로.
- 각 worktree 세션에 Write/Edit/Bash 권한을 명시적으로 허용한 상태에서 시작.
- 병렬 실행 중 한 세션 실패가 다른 세션에 영향 주지 않도록, 각 세션은 자기 브랜치만 push.
