# WhaleScope 페이지 정보구조 및 운영/사용자 분리 보고서

## 0. 결론

현재 대시보드는 다음 기준으로 정리했다.

| 경로 | 역할 | 상태 |
|---|---|---|
| `/` | 일반 사용자용 인사이트 홈 | 기존 `/insights` 화면을 루트로 이동 |
| `/admin` | 운영자용 대시보드 | 기존 루트 운영 화면을 `/admin`으로 이동 |
| `/insights` | 레거시 호환 경로 | Next config에서 `/`로 308 redirect |

이 구조가 더 맞는 이유는 명확하다.

- 평가자나 실제 사용자가 처음 들어오는 루트는 제품 경험이어야 한다.
- 운영 상태, worker heartbeat, system log, Sheets 연결 상태는 일반 사용자에게 1차 노출될 정보가 아니다.
- `시그널`, `리포트`는 현재 독립 제품 페이지라기보다 사용자 홈 안의 섹션 또는 향후 확장 후보에 가깝다.

## 4. 시그널 페이지와 리포트 페이지의 의도 검토

### 4.1 현재 코드 기준

현재 독립된 `/signals`, `/report`, `/reports` 페이지는 없다.

- `시그널`은 사용자 홈의 `#signals` 섹션과 운영 화면의 `#signals` 패널이다.
- `리포트`는 사용자 홈의 고래 브리핑 또는 운영 화면의 Daily Brief에 가까운 개념이다.
- 기존 TopNavbar의 `시그널`, `리포트`는 실제 독립 페이지가 아니라 `/insights#signals`, `/insights#transactions` 같은 앵커를 최상위 탭처럼 올린 것이었다.

### 4.2 왜 혼재가 생겼나

혼재의 원인은 디자인 프롬프트와 실제 제품 로드맵의 층위가 달랐기 때문이다.

- `docs/05_ClaudeCode_Prompt_DesignRefactor.md`는 디자인 시스템을 빠르게 확장하기 위해 `대시보드 / 인사이트 / 시그널 / 리포트`를 top navigation sample처럼 사용했다.
- `docs/06_DesignCheck_ImprovementProposal.md`도 이 문제를 이미 지적했다. 해당 문서는 `시그널`과 `리포트`가 `/insights` 하위 앵커인데 최상위 탭과 같은 층위로 배치되어 정보 계층 혼동이 생긴다고 봤다.
- 장기로드맵 기준으로도 제출 전 핵심은 "수집 -> 선별 -> 요약 -> 전달 -> 확인"이며, 독립 시그널/리포트 페이지는 제출 후 Phase F 확장에 가깝다.

### 4.3 로드맵 정합성 판단

`2026-04-15-WhaleScope-장기로드맵-개선순서` 기준으로 보면, 제출 전 우선순위는 다음이다.

- Phase A: 과제 서사와 One Pager 고정
- Phase B: 실제 LLM API 기반 제출 demo path 정리
- Phase C: 제출 전 필수 코드 정상화
- Phase D: Product Engineer 관점 강화
- Phase E: 제출 패키징
- Phase F: 제출 이후 장기 제품 로드맵

따라서 지금 시점에서 독립 `시그널 페이지`, `리포트 페이지`를 먼저 만드는 것은 우선순위가 낮다. 현재는 사용자 홈 안에서 주요 시그널과 데일리 브리핑을 보여주는 것이 과제 평가에 더 맞다.

### 4.4 결정

현재 결정은 다음이 타당하다.

- 최상위 페이지는 `/` 사용자 홈과 `/admin` 운영 화면만 둔다.
- `시그널`은 `/` 내부 섹션으로 유지한다.
- `리포트`는 `/`의 데일리 브리핑/고래 브리핑으로 통합한다.
- 독립 리포트 히스토리, 시그널 상세, 룰 성능 페이지는 제출 이후 로드맵으로 둔다.

## 5. Telegram 연결 UX 검토

### 5.1 현재 상태

현재 사용자 가입 진입점은 Telegram bot의 `/start` 명령이다.

- `src/distributor/telegram_bot.py`의 `handle_start()`가 `subscribers` 시트에 `chat_id`를 등록한다.
- `/watchlist`, `/pause`, `/status`, `/language`는 가입 이후 보조 명령이다.
- 사용자 웹 화면의 Telegram CTA는 아직 실제 bot link가 아니라 `#watchlist`로 연결되어 있다.

### 5.2 QR 코드 vs 초대주소 복사

추천은 "복사 가능한 초대주소 + QR 코드" 조합이다.

| 방식 | 장점 | 한계 | 판단 |
|---|---|---|---|
| 초대주소 복사 | 모바일/데스크톱 모두 직관적, 구현 단순 | bot username 설정 필요 | 필수 |
| QR 코드 | 데스크톱에서 모바일 Telegram으로 넘어가기 좋음 | QR 생성 라이브러리 또는 이미지 생성 필요 | 있으면 좋음 |
| Telegram channel invite | 공개 채널 listen과 혼동될 수 있음 | 사용자가 bot을 구독하는 흐름과 다름 | 사용자 CTA로는 비추천 |

### 5.3 필요한 설정

웹 UI에서 bot 연결 링크를 만들려면 다음 설정이 필요하다.

```env
TELEGRAM_BOT_USERNAME=whalescope_demo_bot
NEXT_PUBLIC_TELEGRAM_BOT_USERNAME=whalescope_demo_bot
```

현재 WhaleScope 데모 봇: **`whalescope_demo_bot`** → <https://t.me/whalescope_demo_bot>

권장 링크:

```text
https://t.me/whalescope_demo_bot?start=whalescope
```

주의:

- `TELEGRAM_BOT_TOKEN`은 server-side secret이다. 절대 `NEXT_PUBLIC_`로 노출하지 않는다.
- `TELEGRAM_BOT_USERNAME`은 공개 가능한 display/link 값이라 `NEXT_PUBLIC_`로 둘 수 있다.
- Telethon listener의 `TG_CHANNEL=@whale_alert_io`는 운영 수집용 채널 설정이지, 사용자 가입 UX가 아니다.

## 6. 고래 브리핑은 어떤 데이터를 기반으로 해야 하는가

### 6.1 현재 실제 기반

현재 WhaleScope의 고래 브리핑은 원시 트랜잭션 전체를 LLM에 그대로 넣는 구조가 아니다.

현재 흐름:

1. `EtherscanCollector`가 Ethereum 감시 주소 이벤트를 수집한다.
2. `SolscanCollector`가 Solana 감시 주소 이벤트를 수집한다.
3. Telethon listener가 공개 Telegram whale alert 이벤트를 `tg_whale_events`에 저장한다.
4. `src/main.py`가 최근 TG 이벤트를 읽어 `raw_events`에 병합한다.
5. `SignalEngine`이 `signals.yaml` 규칙과 baseline을 기준으로 중요 이벤트를 선별한다.
6. LLM은 선별된 signal 목록을 한국어 daily brief로 변환한다.
7. 결과는 `daily_brief`, `signals`, `transactions`, `system_log`에 저장된다.

즉, 브리핑의 핵심 입력은 "원시 데이터 전체"가 아니라 "규칙 기반으로 선별된 시그널"이다.

### 6.2 좋은 제품 관점의 데이터 계층

고래 브리핑은 최소 네 계층의 근거를 가져야 한다.

| 계층 | 현재 여부 | 설명 |
|---|---|---|
| 온체인 이동 | 있음 | Etherscan/Solscan 기반 감시 주소 이동 |
| 외부 알림 교차검증 | 있음 | Telegram whale alert channel 이벤트 |
| 과거 baseline | 일부 있음 | `address_activity` 기반 최근 CEX in/out 평균과 표준편차 |
| 시장 반응 | 부족 | Binance/Upbit orderbook, trades, ticker 같은 venue-level 데이터가 아직 없음 |

### 6.3 현재 한계

- `daily_brief`의 `highlights`, `signalThemes`는 dashboard 정규화에서 빈 배열 fallback이 많다. 브리핑 스키마 자체를 강화해야 사용자 화면이 더 설득력 있어진다.
- 일부 시그널 규칙은 필요한 context가 엔진에서 충분히 주입되지 않아 사실상 비활성에 가깝다.
- Binance/Upbit 같은 거래소 시장 데이터가 없어 "고래가 움직였다" 이후 "시장이 어떻게 반응 중인가"까지는 설명하지 못한다.

### 6.4 권장 브리핑 구조

사용자용 고래 브리핑은 다음 형태가 좋다.

```text
오늘의 결론
- 어떤 자산에서 어떤 방향의 고래 흐름이 있었는가

왜 중요한가
- CEX 유입/유출, cold/hot wallet 이동, TG 교차확인, baseline 대비 이상치 여부

지금 확인할 것
- 가격 반응, 거래소 유동성, 최근 체결 방향, 다음 관찰 조건

주의사항
- 투자 조언 아님, 데이터 지연/누락 가능성
```

## 7. 남은 개선안 순서

### P0. 현재 라우팅 구조 확정

- README와 Vercel 배포 문서를 이 기준으로 계속 맞춘다.

### P1. Telegram 연결 CTA 실제화

- `TELEGRAM_BOT_USERNAME=whalescope_demo_bot`, `NEXT_PUBLIC_TELEGRAM_BOT_USERNAME=whalescope_demo_bot` 설정 (두 값 동일 유지).
- 사용자 홈 Telegram CTA를 `https://t.me/whalescope_demo_bot?start=whalescope`로 연결.

## 8. 최종 판단

현재 `/`와 `/insights`의 역할 혼재는 라우팅 재배치로 큰 틀에서 해소됐다. 남은 핵심은 독립 페이지를 늘리는 것이 아니라, 사용자 홈의 브리핑 품질과 Telegram 연결 행동을 실제화하는 것이다.

뤼튼 Product Engineer 과제 관점에서도 이 방향이 더 맞다. 평가자가 처음 보는 화면은 운영 콘솔보다 "사용자가 왜 이 서비스를 쓸지"를 보여줘야 하며, 운영 콘솔은 `/admin`에서 구현 완성도와 시스템 신뢰성을 보여주는 보조 화면으로 두는 것이 좋다.
