# 감시 지갑 큐레이션 & 고래 스토리 엔진 기획

## 0. 요약

유저홈 `감시 목록` 카드의 **"나의 관심목록"은 하드코딩**되어 있고 실제 텔레그램 사용자 개인화가 연동되지 않음(이슈 #4). 같은 섹션의 **"고래 스토리"는 `recentTransactions` 의 최근 3건을 템플릿으로 찍어내는 것**(이슈 #5)이라 "스토리"라고 부를 수 없는 수준이다.

두 문제의 본질은 같다: **우리는 지갑 주소를 의미 있게 분류하지 않고 있다.** 지갑의 "정체(identity)" 와 "행동(activity)" 을 함께 가진 데이터 모델로 재정비하면, 감시 목록은 **큐레이션된 흥미로운 주소**로, 고래 스토리는 **그 주소들의 행동을 해석한 서사**로 동시에 개선된다.

이 문서는 요구 #4 + #5 를 **하나의 데이터 모델(§A)과 하나의 스토리 엔진(§B)** 으로 엮어 기획한다.

---

## 1. 현재 상태 감사

### 1.1 감시 목록 (`#watchlist`)

```ts
// apps/dashboard/app/page.tsx:333
function buildWatchlist(data) {
  const baseSymbols = ["BTC", "ETH", "USDC"];  // 하드코딩
  const fromBrief = data?.latestBrief?.topTransactions?.map(i => i.symbol.toUpperCase()) ?? [];
  const symbols = Array.from(new Set([...baseSymbols, ...fromBrief])).slice(0, 4);
  return symbols.map(symbol => ({
    symbol,
    title: symbol,
    note: getWatchlistReason(symbol, data?.recentSignals ?? [], data?.recentTransactions ?? []),
    badge: symbol === "ETH" ? `${formatCompactNumber(metrics.signalCount)}개 신호` : tone === "positive" ? "관심" : "기본",
  }));
}
```

진단:
- 엔티티가 **심볼(코인)** 이다. "감시 목록" 이라면서 **지갑을 감시하지 않음**.
- `baseSymbols` 하드코딩. 개인화 없음.
- 텔레그램 연결 여부가 UI 에 반영되지 않음.
- `getWatchlistReason` 은 심볼 기준 최근 시그널을 나열 — 의미 있는 서사 아님.

### 1.2 고래 스토리 (`#watchlist` 카드 하단 또는 별도 카드)

```ts
// apps/dashboard/app/page.tsx:360
function buildStories(data) {
  const transactions = data?.recentTransactions ?? [];
  if (!transactions.length) return [{ fallback }];
  return transactions.slice(0, 3).map((item, index) => ({
    title: `${symbol} ${formatCurrency(amountUsd) || "이동"}`,
    body: `${from}에서 ${to}로 이동했습니다. 사용자는 이 흐름을 단기 수급 변화의 단서로 볼 수 있습니다.`,
    meta: `${chain} · ${formatDateTime(...)}`,
  }));
}
```

진단:
- 그냥 최근 트랜잭션 3건을 `<from>에서 <to>로` 템플릿으로 찍음.
- 동일 주소의 **연속 행동** 은 포착 못함.
- "스토리" 라는 이름은 과장. 사실상 이벤트 로그.

---

## 2. 전체 전환 방향

1. **감시 목록의 엔티티를 `심볼` → `지갑` 으로 전환**.
2. **지갑 레지스트리(curated_wallets)** 를 새 1급 테이블로 두고, 정체(identity) 를 붙인다.
3. **스토리 엔진** 은 레지스트리 ∧ 최근 트랜잭션 ∧ 시그널을 조인하여 **주소별 서사** 를 만든다.
4. **유저홈 감시 목록 카드에서 "나의 관심목록" 섹션 제거**. 대신 **큐레이션 목록**을 확장.
5. 개인화는 **별도 기능**으로 미뤄둔다 (텔레그램 연결 시 개인 즐겨찾기 — Phase 2).

# §A — 감시 지갑 큐레이션 (요구 #4)

- `btc-og`, `eth-founder`, `nft-whale`, `defi-farmer`, `arbitrage-bot`, `accumulator`, `distributor` ...

### A.1.3 라벨 품질 등급

| Grade | 의미 | UI 표현 |
|---|---|---|
| A | 공식/본인 확인 | 라벨 그대로 표시 |
| B | 공개 자료 교차 검증 | 라벨 + "(공개자료 기반)" |
| C | 클러스터링/휴리스틱 추정 | 라벨 + "(추정)" 배지 |
| D | 저신뢰 | 내부 운영용만, UI 노출 금지 |

## A.2 데이터 모델

### A.2.1 curated_wallets (새 테이블)

```ts
type CuratedWallet = {

  address: string;                   // 체크섬 normalize
  chain: "bitcoin" | "ethereum" | "tron" | "solana" | ...;
  category: WalletCategory;          // A.1.1

  source_refs: Array<{ kind: "tweet" | "article" | "onchain-cluster" | "etherscan-tag"; url?: string; note?: string }>;
```

## A.3 큐레이션 운영 플로우

### A.3.1 시드 리스트 (초기 구축)

1. **Etherscan/Arkham 공개 라벨** 에서 A~B 등급 ~200건 시드.
2. **거래소 공식 cold wallet 공개분** (Binance, Coinbase, Kraken 등 공식 블로그/프루프오브리저브).
3. **역사적 기록**: Vitalik, Satoshi 추정, Mt.Gox compensation wallet, FTX estate wallet.
4. **DAO treasury** (Uniswap, Arbitrum, Optimism, Aave) — 공식 문서.
5. **ETF 보관주소** (Blackrock/Fidelity 공개 리포트).

→ 최초 200~500개를 Google Sheets `curated_wallets` 탭에 수동 입력 (Phase 0).

### A.3.2 지속 확장

| 채널 | 방식 |
|---|---|
| Arkham "Entity" 페이지 주기적 스크레이핑 | 법적 리스크 검토 필요 |
| Twitter 공개 자랑 포스팅 (누가 어느 주소라 선언) | 수동 큐레이션 |
| 온체인 클러스터링으로 신규 후보 도출 | 자동 → C 등급 |
| 사용자 제보 (Phase 2) | 운영자 승인 후 등재 |

### A.3.3 품질 관리

- 주기적으로 `last_activity_at` 이 1년 이상 지나면 `is_active = false`.
- 라벨 오탈자/오정보 리포트 창 (Phase 2).
- 중복 주소 병합 절차.

## A.4 유저홈 `감시 목록` 카드 재설계

### A.4.1 새 정보 계층

1. **상단 필터 탭**: 전체 / 거래소 / 개인 고래 / 기관·펀드 / DAO / ETF / 스마트머니
2. **리스트**: 큐레이션 지갑 카드 N개 (우선순위 priority DESC, 최근 활동 있는 것 우선)
3. **각 카드 구성**:
   ```
   ┌─────────────────────────────────────────┐
   │ 🏦 Binance Cold #1        [기관·거래소]  │
   │ 0x...A3F · Ethereum                     │
   │ 최근 24h: +$82M 유입 · 12건             │
   │ "오늘 오전 대규모 유입 → 거래소 이동"     │
   │                         [트랜잭션 보기]  │
   └─────────────────────────────────────────┘
   ```
4. **Empty state**: "큐레이션 지갑이 아직 없습니다" (Phase 0 이후 발생 불가)

### A.4.2 "나의 관심목록" 제거

- 유저홈의 `buildWatchlist` 함수, 하드코딩 `baseSymbols`, 심볼 카드 레이아웃 전부 제거.
- `getWatchlistReason` 헬퍼도 제거 대상.
- **개인 관심 기능을 원하는 사용자를 위해** 별도 경로(`/me` 등)를 Phase 2 로 분리 설계.

### A.4.3 정보 밀도 vs 모바일 반응형

데스크탑에서 4~6 카드, 모바일(393px) 에서 2 카드 + "더 보기" 버튼. 카드 내부 텍스트는 `max-w-prose`(~65ch) 유지.

# §B — 고래 스토리 엔진 (요구 #5)

## B.1 "스토리" 의 정의

단일 트랜잭션이 아니라 **큐레이션 지갑의 최근 의미 있는 움직임을 한두 문장으로 요약한 서사**. 특징:

- **주체가 명시됨**: "Binance Hot #1 이…", "Vitalik 의 지갑에서…"
- **맥락이 포함됨**: 최근 며칠간의 행동 패턴 + 이번 움직임의 위치.
- **해석이 있음**: "이는 ~한 신호로 볼 수 있다" (단, 투자 조언은 금지 카피).
- **출처가 추적 가능**: 클릭 시 트랜잭션 해시, 블록 탐색기 링크.

## B.2 스토리 종류 (Template 라이브러리)

| Story Type | 트리거 | 예시 |
|---|---|---|
| `activation` | 30일+ 비활성 지갑이 움직임 | "2년간 움직이지 않던 ETH 고래 Wallet #7 이 오늘 1,200 ETH 를 이동했습니다." |
| `accumulation` | N 기간 순매수 강세 | "Binance 가 지난 7일간 BTC를 4,500 개 순흡수했습니다." |
| `distribution` | N 기간 순매도 강세 | "BlackRock IBIT 보관 지갑이 이번주 800 BTC 를 외부로 이동했습니다." |
| `exchange_in` | 외부 → 거래소 | "Whale #22 가 BTC 1,245개 를 Binance 로 이동했습니다. 매도 준비로 해석될 수 있습니다." |
| `exchange_out` | 거래소 → 외부 | "Kraken 에서 ETH 3,800 개가 장기 보관용 지갑으로 이동했습니다." |
| `rotation` | 자산 → 자산 | "Multicoin 이 ETH 를 매도하고 SOL 을 매입한 것으로 보입니다." |
| `stablecoin_surge` | 스테이블 대량 수신/전송 | "Upbit 로 USDT 12M 이 유입되었습니다 (USD 프록시)." |
| `cluster_sync` | 같은 entity 의 여러 주소가 동시 행동 | "Jump Crypto 추정 클러스터 5개 지갑이 30분 사이 동시 움직였습니다." |

## B.3 생성 파이프라인

```
[입력]
  - curated_wallets 중 last_activity_at 이 최근 N시간 내
  - 해당 지갑의 최근 트랜잭션 (transactions 테이블)
  - 최근 7일 wallet_activity_snapshots

[Stage 1] 템플릿 매칭
  - 각 지갑에 대해 B.2 의 어느 Story Type 이 해당하는지 평가 (임계치 기반)
  - 동시에 여러 템플릿이 걸리면 우선순위로 하나 선택

[Stage 2] LLM 서사화
  - 매칭된 템플릿 + 컨텍스트(지갑 identity, 최근 활동, 연관 시그널) 를 LLM 에 투입
  - 시스템 프롬프트:
    "당신은 중립적인 온체인 분석가다. 투자 조언을 하지 말고, 사실 + 해석 가능성을 2~3문장으로 서술."
  - 온도 낮게 (0.3), max 120자 이내 출력

[Stage 3] 가드레일
  - "매수해야 합니다", "투자 추천" 같은 위험 카피 필터링
  - 수치 오기 방지 (LLM 이 숫자를 환각하면 원본으로 덮어쓰기)

[Stage 4] 저장
  - whale_stories 테이블에 적재
  - 동일 지갑의 24h 내 중복 스토리는 merge
```

```
  "interpretation": "매도 준비 가능성이 있으나 확정은 아닙니다."
```

## B.4 whale_stories 스키마

```ts
type WhaleStory = {
  story_id: string;
  wallet_id: string;                 // curated_wallets.wallet_id
  story_type: StoryType;             // B.2
  detected_at: string;
  window: "1h" | "24h" | "7d" | "30d";

  // 사실
  assets: string[];
  amount_usd: number | null;
  counterparty_entity: string | null;
  related_transaction_hashes: string[];
  related_signal_ids: string[];

  // 사람 카피
  title: string;
  body: string;
  interpretation: string | null;
  tone: "neutral" | "watch" | "critical" | "positive";

  // LLM 메타
  llm_model: string;
  llm_token_usage: { prompt: number; completion: number };

  // 운영
  is_published: boolean;
```

## E. 오픈 이슈

- **Google Sheets 로 `curated_wallets` 를 300개 이상 운영할 수 있는가**: 수동 편집은 가능하지만 여러 팀원이 편집 시 충돌/버전 관리 약함. Phase 2 이전에 Postgres 전환 추천.
- **제재 목록 지갑(mixer/sanctioned)의 UI 노출 범위**: 정책적·법적 리뷰 필요. 기본은 **모니터링만** 하고 유저홈 노출 금지.
- **LLM guardrail 실패 샘플 사후 학습**: 위반 케이스를 로그로 축적하고 프롬프트를 업데이트하는 루프 필요.
- **"개인으로 추정" 지갑의 오탐 책임**: grade C 를 잘못 A 로 올리면 명예훼손 소지. grade 가 C 인 것을 UI 에서 **반드시 "(추정)" 배지로 명시**.
