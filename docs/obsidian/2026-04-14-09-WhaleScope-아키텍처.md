# WhaleScope System Architecture

Clean-slate 설계 문서. 기존 구현과의 호환을 제약으로 두지 않고, 지금까지의 리뷰·결정을 반영한 **목표 아키텍처**를 정의한다. 현 코드에서 이 문서로 가는 마이그레이션은 §10에서 단계별로 기술.

## 1. Context

- WhaleScope는 대형 암호화폐 지갑(고래)의 온체인 움직임을 자체 수집하고, 규칙 기반 시그널로 정제한 뒤 LLM이 한국어 내러티브를 덧붙여 Telegram으로 배포하는 개인 프로젝트다.
- 목표는 유료 API 의존을 제거하고, 공개 데이터·공급자 무료 티어만으로 월 운영비 $0~3을 유지하는 것.
- 과제·포트폴리오 맥락이 강하므로 설계 품질 자체가 산출물이다(운영 확장성보다 "설명 가능한 의사결정"의 밀도가 중요).

## 2. Requirements

### Functional
1. 감시 지갑 리스트를 주기 폴링해 거래를 수집한다.
2. Whale Alert 공개 Telegram 채널을 User API로 수신한다.
3. 두 소스를 교차검증해 시그널을 생성한다.
4. 일간 브리프와 주간 트렌드 코멘터리를 생성한다.
5. Telegram Bot으로 사용자 구독·배포·자연어 개인화를 제공한다.
6. Streamlit 대시보드로 타임라인·아카이브를 제공한다.

### Non-functional
- **비용**: 월 $0~3 상한 (Claude/Anthropic API 비용 제외 시 $0).
- **가용성**: 일간 브리프 지연 < 30분 이내 (99% 달성 목표, best-effort).
- **Latency**: 자연어 개인화 응답 < 3초 (p95).
- **스케일**: 구독자 1~1K, 감시 주소 100~300, 시그널 일 5~30건.
- **관측성**: 호출·시그널·브리프 전 과정이 단일 로그 체계로 추적 가능.
- **재현성**: 같은 날짜 입력으로 브리프 재생성 가능(프롬프트·모델·데이터 버전 추적).

### Constraints
- 1인 개발·유지. 운영 인력 없음.
- 실행 환경은 GitHub Actions(예약 cron) + 상시 구동 1개 프로세스(텔레그램 수신 + 봇).
- 저장소 초기는 Google Sheets. 필요 시 SQLite/Postgres 이행.
- 유료 API 금지(Anthropic API는 허용하되 최소화).

## 4. Component Breakdown

### 4.1 Ingestion Layer
- **TelethonListener** (상시 프로세스). `@whale_alert_io` 수신 → 정규식 1차 파싱 → 필요 시 LLM 폴백 → `tg_whale_events` 적재.
- **ChainCollector** (Actions cron 15분). watched_addresses 전체를 순회하며 Etherscan/Solscan API 호출 → 정규화 → `address_activity` 적재.
- 두 컴포넌트는 상호 독립적이며 한쪽 장애가 다른 쪽에 전파되지 않는다.

### 4.2 Normalization Layer
- 두 source의 상이한 스키마를 하나의 `Event` dataclass로 수렴.
- `source ∈ {tg, chain, both}` — `both`는 Corroboration 단계에서 마킹.
- USD 환산은 `PriceService`(CoinGecko free) 사용, 캐시 1분 TTL.

### 4.3 Signal Layer
- **RuleSet**은 `config/signals.yaml`. 코드 변경 없이 임계값·윈도우 튜닝.
- **Corroboration**: tx_hash 일치 우선, 없으면 (시간 ±3분, USD ±5%) 매칭. 일치 시 `source=both`, `confidence += 1 tier`.
- **Personalization**: 구독자별 `user_interests`의 weight를 score에 곱해(0.7~1.5 범위 클램프) 개인화된 랭킹 구성.
- 산출: `Signal(id, rule, severity, score, evidence_tx, window_start, window_end)`.

### 4.4 Narration Layer (LLM)
- 자체 구축 라우터(참고: [[2026-04-14-08-WhaleScope-LLM라우터-자체구축-다관점리뷰]] 종합 7.8).
- 3 공급자 adapter + `config/llm_routing.yaml`로 작업별 preferred/fallback 모델 매핑.
- 책임은 **해석과 자연어 변환**에만 한정. 스코어·랭킹은 Signal Layer에서 이미 끝난 상태.
- `analysis_log`에 `(prompt_hash, prompt_version, model_id, tokens, cost, latency)` 기록.

### 4.5 Distribution Layer
- **TelegramBot**: `/start`, `/preferences`, `/pause`, `/status` + 자유 메시지(LLM 의도 추출).
- **Streamlit**: 비번 보호 + 타임라인·주간 아카이브·관심사 편집.

### 4.6 Storage
- 단계적 전환: Sheets (MVP) → SQLite (구독자 500+) → Postgres (팀·멀티테넌트).
- 추상 인터페이스 `Storage`(Protocol) 뒤에 구현체 교체. 현 Sheets 의존 코드는 `SheetsStorage`로 격리.

### 4.7 Observability
- 모든 계층이 `StructuredLogger`로 JSON 이벤트 방출.
- 중요 이벤트는 `system_log` 테이블에 영속 저장(run_id, stage, status, timing, errors, details).
- 주 1회 자동 `digest` 리포트: 수집량·시그널·브리프 품질 지표.

## 6. Key Decisions (Embedded ADRs)

### ADR-001: Hybrid Data Collection (TG Channel + On-chain Polling)
**Status**: Accepted

**Context**: 유료 Whale Alert API 제거 목표. 순수 on-chain polling만으로는 커버리지(전 세계 유의미한 거래 망라)가 약하고, TG 채널만으로는 교차검증이 불가.

**Decision**: 두 소스를 동등 계층으로 운영하고 Signal Layer에서 corroboration으로 결합.

**Alternatives Considered**:
| 대안 | 채택 안 한 이유 |
|---|---|
| On-chain polling only | 공개 고래 이벤트 누락. 신뢰도는 높지만 coverage 부족. |
| TG channel only | 단순 래퍼 문제. 자체 신호 창출 어려움. |
| Whale Alert 유료 API | 비용·의존·차별화 전부 미해결. |

**Consequences**:
- (+) 커버리지 × 신뢰도 양면 강화, `corroborated_move` 시그널로 confidence tier 공식화.
- (−) 파싱 로직 이중화, TG 계정 운영 리스크.

### ADR-002: LLM Router — Self-Build over OpenRouter
**Status**: Accepted (조건부)

**Context**: 다중 LLM 접근이 필요. OpenRouter 게이트웨이(종합 7.5) vs 자체 구축(종합 7.8) 비교 결과 자체 구축이 근소 우위.

**Decision**: Anthropic·Gemini·Groq 3개 공급자 직접 통합. 스코프 봉쇄 3원칙(3개 공급자 고정, 단발성 prompt→text만, usage는 응답 필드에서만).

**Alternatives Considered**:
| 대안 | 점수 | 채택 안 한 이유 |
|---|---|---|
| OpenRouter 게이트웨이 | 7.5 | 무료 티어 축소(20~200/일) + 공급자 고유기능 접근 지연 |
| 단일 공급자(Anthropic) | — | 무료 티어 부재, 장애 시 폴백 없음 |
| LangChain 사용 | — | 규모 대비 과잉 설계, 학습·유지보수 비용 과다 |

**Consequences**:
- (+) 무료 티어 최대 활용, prompt caching·long context 등 공급자 최적화 접근, 강한 포트폴리오 시그널.
- (−) Adapter 3개 유지, SDK 파손 대응 책임.
- **Reject trigger**: 공급자 5개 필요, SDK 파손 월 4시간 초과, 사용자 1K+ 도달.

### ADR-003: Storage Ladder — Sheets → SQLite → Postgres
**Status**: Accepted

**Context**: 초기엔 접근성·수정 편의성이 우선. 확장 시점엔 쿼리 성능·트랜잭션이 필요.

**Decision**: `Storage` Protocol 추상화 뒤에서 MVP는 Sheets, 트리거 도달 시 SQLite(1인용), 팀/멀티테넌트 시 Postgres로 단계 전환.

**Trigger**:
- Sheets → SQLite: 구독자 500+ 또는 Sheets API 쿼터 경고 3회/월
- SQLite → Postgres: 2명 이상 협업, 실시간 대시보드 필요성

**Consequences**:
- (+) 현시점 설계는 단순. 마이그레이션 경로가 선명.
- (−) 추상화 유지 비용(Protocol + 구현체 분리) 필수.

### ADR-004: Signal Engine as Primary Value
**Status**: Accepted

**Context**: Claude 점수화에만 의존하면 블랙박스·비용·재현성 문제가 있음. 자체 규칙을 가진 SignalEngine이 서비스의 핵심 자산이 되어야 한다.

**Decision**: 규칙을 YAML로 외부화, Claude는 "서술자"로만 활용. 스코어·랭킹·개인화는 SignalEngine이 책임.

**Consequences**:
- (+) 블랙박스 최소화, 품질을 precision/recall로 측정 가능.
- (+) 시그널 수정이 코드 없이 가능 → 튜닝 속도 향상.
- (−) 초기 규칙 5~8개 설계·튜닝 비용.

### ADR-005: Execution Model — Cron for batch, Long-running for realtime
**Status**: Accepted

**Context**: 일간·주간 집계는 Actions cron으로 충분. TG 수신은 상시 연결 필요.

**Decision**: **GitHub Actions cron**으로 배치 파이프라인, **단일 long-running 프로세스**(초기엔 개인 서버 또는 무료 VPS / Fly.io free tier 등)로 TG listener + bot polling.

**Alternatives**:
- 전부 Actions: TG listener 구동 불가(세션·상시 연결 불가).
- 전부 상시 프로세스: 배치 스케줄 관리가 부담.

**Consequences**:
- (+) 배치는 재현성 + 로그 자동 수집. 상시 프로세스는 얇게 유지.
- (−) 두 실행 환경 secrets 동기화 필요.

## 7. Cross-Cutting Concerns

### Configuration
- `config/signals.yaml` — 시그널 규칙
- `config/llm_routing.yaml` — 작업별 모델 선호
- `config/watched_addresses.csv` — 감시 지갑 시드
- `.env` — secrets(API keys, Telegram credentials)

### Secrets
- GitHub Actions Secrets + 상시 프로세스의 로컬 `.env`.
- 회전 주기: 분기 1회 권장.
- PII/거래 데이터는 공개 성격이나 사용자 발화(개인화 메시지)는 LLM 공급자 정책에 따름 — README에 명시.

### Prompt Versioning
- `prompts/` 디렉터리에 텍스트 파일로 분리. 파일 sha1 앞 8자리가 `prompt_version`.
- 산출물(분석 로그, 브리프 메타)에 항상 기록.

### Cost Guardrails
- `analysis_log`에서 일일 비용 집계.
- 월 예산 초과 70% 시 Telegram 경고, 100% 시 fallback 모델로 강제 다운그레이드.

### Testing Strategy
- Unit: 각 adapter, normalizer, 규칙 로직.
- Integration: 가짜 이벤트 시퀀스로 파이프라인 end-to-end.
- Replay: `system_log` 기반으로 특정 날짜 재현성 테스트.

## 9. Quality Attributes — 검증 방법

| 속성 | 목표 | 검증 방법 |
|---|---|---|
| Availability | 일간 브리프 지연 < 30분 (99%) | `system_log` 기준 월별 집계 |
| Latency | NL 응답 p95 < 3s | Bot handler timer + metrics |
| Cost | 월 $3 이하 | `analysis_log` 일일 집계 |

| Accuracy (NL parse) | 80%+ | 수동 라벨 50건 기준 평가 |

## 10. Migration from Current Codebase (현 구현 → 목표)

현 레포는 이미 80% 부품이 있음. 재작성보다 **재배열**이 효율적.

1. **Interface 추출** — `LLMProvider`, `Storage`, `SignalRule` Protocol 정의.
2. **`ClaudeAnalyzer` 리네이밍** → `LLMAnalyzer` + `AnthropicProvider` 추출. 기존 호출 경로 유지.
3. **`scorer.py` → `signal_engine.py`로 승격** + 규칙 YAML화.
4. **`whale_alert.py` 삭제** + `EtherscanCollector`/`SolscanCollector` 신설.
5. **`TelethonListener` 별도 프로세스로 분리.**
6. **Sheets 탭 추가** (watched_addresses, address_activity, signals, tg_whale_events, user_interests, weekly_trend, analysis_log은 기존).
7. **관측성** — `system_log`에 run_id + stage 기록 보강.
8. **Fly.io 또는 자체 서버 배포** — TG listener + bot 구동.

## 11. Open Questions

- `user_interests`의 weight 범위를 0.7~1.5로 제한했는데, 사용자가 "X 코인은 아예 보여주지 마"라고 할 때 exclude 플래그를 별도로 둘지 아니면 weight 0.0을 허용할지.
- Streamlit 대시보드의 인증: 단일 비번 vs Supabase/Cloudflare Access. 사용자 수 증가 시 재설계.
- SQLite 이행 시점에서 Sheets의 "사람이 직접 편집 가능한 UX"를 어떻게 대체할지(Streamlit 편집 화면으로 이전 권장).
- `corroborated_move` 규칙에서 tx_hash 없는 TG 이벤트 매칭 휴리스틱의 precision — 실측 후 임계값 조정.
