- **같은 파일을 두 이슈가 수정**하면 반드시 같은 티켓으로 묶는다. Worktree로 분리해도 PR 머지 단계에서 rebase/merge 충돌이 확정적으로 발생하기 때문.
- **병합한 티켓 내부**는 논리 단위로 **커밋만 분리**해서 history 가독성을 유지한다. 예: T2는 `refactor(utils): ...` + `chore(main,baseline): drop _parse_dt alias` 두 커밋.

## 2. Worktree 운영 규약

사용자가 우려한 "워크트리 겹쳐서 깃 장애"의 실제 원인은 다음 세 가지가 대부분이다. 아래 수칙을 지키면 회피된다.

### 2.1 사고 유형과 차단 수칙

| 사고 유형 | 원인 | 차단 수칙 |
|---|---|---|
| (A) `fatal: '<branch>' is already checked out at '<path>'` | 같은 브랜치를 두 worktree에 체크아웃 | 브랜치명 **고유 접두사**(`feat/h0-*`, `refactor/m1-*` 등) + 생성 시 `-b <new-branch>` 강제 |
| (B) 머지 단계 rebase 충돌 | 두 브랜치가 같은 라인을 수정 | §1의 **파일 겹침 매트릭스대로 병합 티켓**(T2, T3)으로 통합 |
| (C) `index.lock` 충돌 | 같은 repo에 두 git 프로세스 동시 실행 | 각 worktree는 **독립 CWD**를 유지하고, Claude Code 세션도 그 CWD로만 작업 (Bash는 상대 경로 금지, 절대 경로 사용) |
| (D) worktree 경로 꼬임 | main repo와 worktree 경로가 같은 디렉토리 하위 | worktree 디렉토리를 **repo 바깥**에 둔다: `../wt-whale-<ticket>/` |
| (E) 분기점 드리프트 | 티켓별로 다른 시점에서 분기 | 모든 티켓을 **같은 시점**(= 티켓 시작 시점의 `main` HEAD)에서 분기 |

### 2.2 Worktree 배치 표준

**경로**: repo 루트를 `repo=/sessions/sharp-eloquent-euler/mnt/02015_reuton_whale`이라 할 때,
```
$repo/..                          ← repo 바깥에 worktree 디렉토리 둠
  ├── 02015_reuton_whale/         (main, 기준 repo)
  ├── wt-whale-T1-panel/          (T1 worktree)
  ├── wt-whale-T2-safe-float/     (T2)
  ├── wt-whale-T3-sheets-lock/    (T3)
  ├── wt-whale-T4-rules-docs/     (T4)
  ├── wt-whale-T5-changelog/      (T5)
  └── wt-whale-T6-env-staleness/  (T6)
```

**생성 커맨드 (main repo 안에서 한 번씩 실행)**:
```bash
# 사전 확인
cd "$repo"
git status                                # clean working tree 보장
git checkout main && git pull --ff-only   # main 최신화

# 기준 커밋 고정 (모든 worktree가 여기서 분기)
BASE=$(git rev-parse main)
echo "$BASE" > /tmp/whale-base-sha        # 로그용

# T1
git worktree add -b feat/h0-system-log-panel-mount            ../wt-whale-T1-panel          "$BASE"
# T2
git worktree add -b refactor/m1-safe-float-and-parse-dt-cleanup ../wt-whale-T2-safe-float    "$BASE"
# T3
git worktree add -b refactor/m2-sheets-client-lock-and-docs   ../wt-whale-T3-sheets-lock    "$BASE"
# T4
git worktree add -b docs/m3-cex-rules-intent                  ../wt-whale-T4-rules-docs     "$BASE"
# T5
git worktree add -b docs/pm-2026-04-17-changelog              ../wt-whale-T5-changelog      "$BASE"
# T6
git worktree add -b refactor/l3-listener-staleness-env        ../wt-whale-T6-env-staleness  "$BASE"

git worktree list   # 확인: 6개 + main = 7줄
```

### 2.3 각 Worktree 내부 규칙

각 Claude Code 세션은 **자신의 worktree CWD 안에서만** 동작한다. 즉:
- Bash 명령은 모두 `cd ../wt-whale-<ticket>` 혹은 **절대 경로 사용**.
- 다른 worktree 브랜치를 checkout 하지 않는다 (유형 A 방지).
- `git add -A`로 다른 디렉토리를 실수로 스테이징하지 않는다 (항상 파일 명시).
- 커밋 후 `git push origin <branch-name>`으로 원격 push (있다면).
- 작업 끝난 worktree는 **본인이 삭제하지 않는다** — 머지 단계에서 일괄 정리.

### 2.4 병합(머지) 순서

머지는 main repo(`$repo`)에서 수행한다. 로컬 fast-forward 순서:
```bash
cd "$repo"
git checkout main
# 각 브랜치가 BASE에서 파생됐고 파일 겹침이 없으므로 순차 FF 머지로 충돌 없음
for br in \
  feat/h0-system-log-panel-mount \
  refactor/m1-safe-float-and-parse-dt-cleanup \
  refactor/m2-sheets-client-lock-and-docs \
  docs/m3-cex-rules-intent \
  docs/pm-2026-04-17-changelog \
  refactor/l3-listener-staleness-env; do
    git merge --no-ff -m "merge: $br" "$br" || { echo "merge failed: $br"; break; }
done
```

리뷰용 PR을 쓸 거라면 각 브랜치별 PR을 순차 머지. GitHub 리뷰 순서도 위 목록과 같게 유지(CI 안정성).

### 2.5 정리
```bash
cd "$repo"
for p in T1-panel T2-safe-float T3-sheets-lock T4-rules-docs T5-changelog T6-env-staleness; do
  git worktree remove --force "../wt-whale-$p"
done
git worktree prune
git branch -D \
  feat/h0-system-log-panel-mount \
  refactor/m1-safe-float-and-parse-dt-cleanup \
  refactor/m2-sheets-client-lock-and-docs \
  docs/m3-cex-rules-intent \
  docs/pm-2026-04-17-changelog \
  refactor/l3-listener-staleness-env 2>/dev/null || true
```

**(2-1) 공용 유틸 신설** — `src/utils/number_utils.py`:
```python
"""Numeric parsing helpers shared across pipeline modules."""
from __future__ import annotations

import logging
from typing import Any

from src.utils.logger import get_logger

logger = get_logger("utils.number_utils")


def safe_float(
    value: Any,
    *,
    field_name: str | None = None,
    strip_commas: bool = False,
    default: float = 0.0,
    log_level: int = logging.DEBUG,
) -> float:
    """Parse ``value`` as float with forgiving fallbacks.

    Args:
        value: Raw input (str, int, float, None, etc.).
        field_name: Optional label used in failure log for diagnostics.
        strip_commas: When True, remove ``,`` from string inputs before
            parsing (e.g. ``"1,234.5"``).
        default: Returned when parsing fails or input is None/empty.
        log_level: ``logging`` level used when a parse fails.

    Returns:
        Parsed float, or ``default`` on failure.
    """
    if value is None or value == "":
        return default
    try:
        if strip_commas:
            return float(str(value).replace(",", ""))
        return float(value)
    except (TypeError, ValueError):
        logger.log(
            log_level,
            "safe_float failed field=%s value=%r; defaulting to %s",
            field_name,
            value,
            default,
        )
        return default
```

def test_safe_float_parses_int_str_float():

def test_safe_float_none_and_empty_return_default():
    assert safe_float(None) == 0.0
    assert safe_float("") == 0.0
    assert safe_float(None, default=-1.0) == -1.0


def test_safe_float_invalid_returns_default():
    assert safe_float("abc") == 0.0
    assert safe_float("abc", default=9.9) == 9.9
