## 4. 제시된 Review Findings 반영 상태

### Finding 1 — Listener data is never fed back into signal pipeline

상태: **해결됨**

현재 `src/main.py`는 `sheets.list_tg_whale_events(since=since_dt)`를 호출하고, `tg_row_to_event(row)`로 변환한 TG 이벤트를 `raw_events`에 병합한다. 이후 `SignalEngine`은 `engine.run(raw_events, now, baselines=baselines)`로 온체인 이벤트와 TG 이벤트를 함께 처리한다.

핵심 근거:

- `src/main.py`: `list_tg_whale_events`
- `src/ingestion/tg_normalizer.py`: `tg_row_to_event`
- `src/storage/sheets_client.py`: `list_tg_whale_events`

### Finding 2 — Public API exposes operational data and raw error messages

상태: **부분 해결 / 운영 설정 의존**

현재 dashboard API route들은 `requireDashboardAuth(request)`를 거치고, 오류는 `createGenericErrorResponse`로 일반 메시지를 반환한다. `error.message`를 그대로 client에 넘기지 않는다.

다만 `DASHBOARD_PASSWORD`가 비어 있으면 인증이 비활성화된다. 따라서 Vercel 운영 환경에서는 `DASHBOARD_PASSWORD`를 반드시 설정해야 한다.

핵심 근거:

- `apps/dashboard/lib/auth.ts`
- `apps/dashboard/app/api/dashboard/route.ts`
- `apps/dashboard/app/api/transactions/route.ts`
- `apps/dashboard/app/api/signals/route.ts`
- `apps/dashboard/app/api/system-log/route.ts`

### Finding 3 — User page leaks internal load errors

상태: **해결됨**

`/insights`는 내부 error message를 직접 노출하지 않고, 데이터 연결 실패 시 “데이터 연결을 확인 중입니다. 잠시 후 다시 시도하세요.” 수준의 일반 문구만 표시한다.

핵심 근거:

- `apps/dashboard/app/insights/page.tsx`

### Finding 4 — Telethon handler blocks the event loop on LLM and Sheets I/O

상태: **해결됨**

Telethon listener의 LLM fallback과 Sheets 저장 경로가 `asyncio.to_thread`로 감싸져 있다.

핵심 근거:

- `src/ingestion/telethon_listener.py`: `router.call_task`
- `src/ingestion/telethon_listener.py`: `append_tg_whale_event`
- `src/ingestion/telethon_listener.py`: `append_system_log`

### Finding 5 — Listener health card is inferred from Sheets connectivity

상태: **해결됨**

운영 대시보드는 `sourceState`만으로 listener 상태를 판단하지 않는다. `apps/dashboard/lib/metrics.ts`에서 `system_log`의 `run_type=telethon_listener` heartbeat를 우선 보고, heartbeat가 없으면 `tg_whale_events` 최신 기록을 fallback으로 사용한다.

핵심 근거:

- `apps/dashboard/lib/metrics.ts`: `normalizeListenerHealth`
- `apps/dashboard/app/page.tsx`: `data.listenerHealth`

## 5. 현재 디자인 시스템 상태

`docs/06_DesignCheck_ImprovementProposal.md` 기준으로 dashboard 디자인 시스템은 큰 틀에서 잘 수렴했다.

확인된 강점:

- `design-tokens.css`를 중심으로 색상, spacing, radius, elevation, motion이 토큰화됨.
- `globals.css`에서 컴포넌트성 스타일을 제거하고 대부분 CSS Module로 이동.
- `/`, `/insights`, `/preview` 3개 화면이 빌드 대상에 포함됨.
- TopNavbar, InsightsSidebar, ThemeToggle, 공유 카드/테이블 컴포넌트가 분리됨.
- `transition: all`, gradient text, purple/pink slop 패턴은 현재 주요 CSS에서 발견되지 않음.

현재 남은 디자인 리스크:

- `apps/dashboard/app/insights/insights.module.css`의 `.riskBanner`에 `border-left: 4px solid var(--bad-soft)`가 남아 있음. design-engineering 문서가 정의한 side-stripe anti-pattern 위반이다.
- `top-navbar`가 preview의 editorial-calm 기준 대비 더 조밀하고 operational한 톤으로 남아 있음.
- `TopNavbar`와 `InsightsSidebar`의 정보구조가 일부 중복된다. 예: `#signals`가 양쪽 네비게이션에서 모두 등장.
- `rg` 기준 `rgba`/`rgb` 히트는 실행 CSS 선언이 아니라 `system-log-panel.module.css`의 주석에서 발생했다. 다만 자동 검사 기준을 엄격히 0-hit로 둘 경우 주석도 정리하는 편이 낫다.

### P1 — 운영 보안

- Vercel에 `DASHBOARD_PASSWORD` 설정 필수.
- 설정이 없으면 dashboard API는 공개 접근 가능 상태가 된다.

### P2 — 운영성 개선

- dashboard의 signal action/watchlist override가 인메모리라 재시작 시 사라진다. 운영 수준으로 보려면 Google Sheets 또는 별도 저장소에 영속화 필요.
- API cache/rate-limit도 인메모리라 다중 인스턴스에서는 instance별로 분리된다. 과제 데모에는 충분하지만 운영 확장 전 Redis 등 외부 store 검토 필요.

### P2 — 시각 QA

- `npm run dashboard:dev`로 로컬 서버를 띄운 뒤 `/`, `/insights`, `/preview`를 desktop/mobile, light/dark로 확인해야 한다.
- `docs/06_DesignCheck_ImprovementProposal.md`에는 localhost 미실행으로 시각 QA가 blocked였다고 기록되어 있다.

## 8. 현재 프로젝트 판단

현재 WhaleScope는 “아이디어 스케치”나 “미완성 MVP”라기보다, **과제 제출 가능한 수준의 기능/문서/테스트 기반에 디자인 시스템을 입히는 후반 단계**로 보는 것이 맞다.

단, 제출/배포 전에는 아래 세 가지를 정리해야 한다.

1. 워킹트리의 미추적 파일 정리와 커밋 범위 결정.
2. `riskBanner`, TopNavbar IA, 로고/favicon 교체 등 디자인 폴리싱 마무리.
3. Vercel 운영 보안 변수(`DASHBOARD_PASSWORD`) 설정과 실제 화면 QA.

가장 먼저 할 일은 **미추적 파일 정리 → 디자인 P0 1건 수정 → 시각 QA → 커밋/푸시** 순서다.
