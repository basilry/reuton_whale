# WhaleScope 4차 개선 프롬프트 (병렬 실행용)

> 3차 개선 프롬프트 실행 중 서브에이전트 간 스코프 침범으로 3쌍의 중복 커밋 발생(`prompts.py` inline, Etherscan backoff 테스트, corroboration 리팩터). 이 문서는 **teamCreate 병렬 실행 시 충돌을 구조적으로 차단**하기 위해 파일 단위 소유권(STRICT_SCOPE)과 웨이브 기반 머지 순서를 명시한다.

## 0. 실행 환경 전제

- **Baseline**: `feature/arch-migration` @ `49c38d7` (2차 리뷰 이후 머지된 정리 브랜치).
- **이미 해결된 이슈** (머지됨): B2(Storage Protocol), B4(corroboration 리팩터), M1(prompts.py 제거), M3(Etherscan backoff), M5(prompt_loader 캐시).
- **남은 이슈**: B1, B3, B5, B6, M2, M4, M6.
- **실행 방식**: Claude Code `teamCreate`로 각 트랙을 별도 에이전트에 위임. 각 에이전트는 독립된 브랜치와 독립된 파일 세트만 수정.

## 1. 전역 규칙 (모든 트랙 공통)

### 1.1 STRICT_SCOPE 원칙

**각 트랙은 "허용 파일 목록"에 명시된 파일만 생성·수정·삭제할 수 있다.** 허용 외 파일을 수정해야 할 필요가 발생하면 **작업을 중단하고 사용자에게 보고**한다. 다른 트랙이 해결할 문제라고 판단되면 그 트랙의 스코프임을 기록하고 자기 스코프로 돌아온다.

**금지**: "acceptance criteria를 맞추려면 어차피 필요하니 건드린다" 식의 자발적 확장. 그것이 지난 번 충돌의 근본 원인이었다.

### 1.2 공통 운영 규칙

- **분기**: 각 트랙은 `feature/arch-migration`에서 분기. 브랜치명은 트랙 ID 고정(아래 명시).
- **커밋 범위**: 한 커밋은 하나의 논리 단위. 머지 시 리버트 가능하게.
- **테스트**: 변경한 파일과 직접 연관된 테스트만 실행해 그린 확인. 전체 테스트 green은 Wave 종료 후 게이트에서 확인.
- **시크릿**: 절대 커밋 금지. 신규 env var 도입 시 `.env.example`만 갱신.
- **Co-Author**: 모든 커밋 trailer에 `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>`.
- **포스트앰블 금지**: 에이전트는 작업 완료 후 브랜치명·커밋 해시·수정 파일 목록만 출력. 설명·자축·제안 금지.

### 1.3 PREFLIGHT (에이전트 시작 직후 수행)

```
1. git fetch && git checkout feature/arch-migration && git pull
2. git checkout -b <TRACK_BRANCH>
3. 본 문서 §3 해당 트랙의 STRICT_SCOPE 허용 파일 목록을 복창(메모)
4. pytest tests/<scope> -q 로 현재 상태 기록 (baseline)
5. 착수
```

### 🔵 W1-A : 테스트 그린 복구 (B1)

**브랜치**: `feature/fix-b1-tests`
**STRICT_SCOPE (허용 파일)**:
```
tests/test_main.py
tests/test_telethon_parser.py
tests/conftest.py      # 신규 fixture 필요 시에만
```
**절대 금지**: `src/**` 수정, 다른 `tests/test_*.py` 수정.

**문제 정의**:
- `pytest tests/test_main.py tests/test_telethon_parser.py` 실행 시 10+ 건 실패.
- 실패 원인은 리팩터 이후 프로덕션 코드의 시그니처/의존성이 바뀌었으나 테스트의 mock·호출 경로가 따라오지 못함.
- **프로덕션 코드는 정상 동작한다는 가정**에서 출발. 테스트를 현실에 맞춘다.

**순서**:
1. 각 실패 케이스별로 traceback을 읽고 (mock 시그니처 mismatch / 사라진 모듈 import / 비동기 처리 누락) 중 어느 범주인지 분류한다.
2. `test_main.py`: `run_daily_pipeline`이 `SignalEngine`, `LLMRouter`, `PriceService` 인스턴스를 내부에서 생성하므로, 테스트에서는 `unittest.mock.patch`로 생성 지점을 가로챈다. 기존 `ClaudeAnalyzer` 알리어스는 `src.main`에 남아있으므로 테스트 호환.
3. `test_telethon_parser.py`: Telethon 이벤트 핸들러가 `LLMResult.text`로 변경됨(`471b26b` 커밋). 테스트 목 객체의 속성을 `text`로 통일.
4. 매 수정 후 해당 파일만 pytest로 돌려 그린을 확인하고 다음 실패로 이동.
5. **프로덕션 코드가 틀린 것으로 의심되면 작업 중단하고 사용자에게 보고**. 임의로 `src/`를 수정하지 않는다.

**수용 기준**:
- `pytest tests/test_main.py tests/test_telethon_parser.py -q` 100% 그린.
- `pytest -q` 전체에서 **더 늘어난 실패 0건**(베이스라인 대비 회귀 금지).
- 신규 테스트 0건(기존 테스트만 수정). 필요하다고 판단되는 신규 테스트는 코멘트로 TODO 표기만.

**커밋 메시지 (예시)**:
```
test(main): align mocks with LLMRouter/SignalEngine injection
test(telethon): switch LLMResult.content to .text
```

### 🔵 W1-B : CoinGecko 미지원 심볼 리포트 (M4)

**브랜치**: `feature/fix-m4-price`
**STRICT_SCOPE (허용 파일)**:
```
src/analyzer/price_service.py
src/storage/sheets_client.py      # 신규 메서드 append만 허용, 기존 메서드 수정 금지
src/storage/storage_protocol.py   # Protocol에 새 메서드 시그니처만 추가
tests/test_price_service.py
tests/test_sheets_client.py       # 신규 테스트 append만
```
**절대 금지**: `src/main.py`, `src/signals/**`, `src/distributor/**`.

**문제 정의**:
현재 `price_service.get_usd(symbol)`은 `_COINGECKO_IDS`에 없는 심볼에 대해 조용히 `None`을 반환. 결과적으로 `amount_usd=0`으로 전파되어 스레시홀드 기반 룰을 통과하지 못함 → 신규 토큰의 고래 거래가 누락.

**구현**:
1. `PriceService` 생성자에 `self._unknown_counts: Counter[str] = Counter()` 추가.
2. `get_usd` 내부 "`coin_id` 없음" 분기에서 `self._unknown_counts[sym] += 1` 후 기존처럼 fallback.
3. `PriceService`에 `def drain_unknown_report(self) -> list[tuple[str, int]]` 추가. 내부 카운터를 반환하고 초기화.
4. `Storage` Protocol(`src/storage/storage_protocol.py`)에 `def append_system_log(self, level: str, category: str, payload: dict) -> None` 시그니처 추가.
5. `SheetsClient`에 동명 메서드 구현 — `system_log` 시트(기존) 또는 새 시트에 한 행 append. 기존 시트 스키마가 있으면 그대로 활용, 없으면 `timestamp, level, category, payload_json` 헤더로 생성.
6. **`main.py`나 파이프라인 호출부는 수정 금지.** 본 트랙은 "API 제공"까지만. 실제 배선은 **W2-D** 트랙에서 한 줄 추가하도록 TODO 주석만 `price_service.py` 상단에 남긴다: `# TODO(M4-wire): main.py finally 블록에서 drain_unknown_report() → storage.append_system_log 호출.`

**테스트**:
- `tests/test_price_service.py`: 미지원 심볼 호출 3회 → `drain_unknown_report()`가 `[("XYZ", 3)]` 반환 후 재호출 시 `[]`.
- `tests/test_sheets_client.py`: `append_system_log` 호출 시 row 포맷 검증.

**수용 기준**:
- 신규/변경 테스트 그린.
- `main.py` diff 0바이트.
- Protocol 추가 메서드가 모든 구현체(`SheetsClient` 외 기타 모의 구현)에 반영됐는지 확인. 없으면 `NotImplementedError` 기본 구현을 Protocol에 둔다.

**커밋 메시지**:
```
feat(price): track unknown-symbol lookups with drain API
feat(storage): add append_system_log to Storage Protocol
test(price): cover unknown-symbol counter and drain reset
```

### 🔵 W1-C : Telegram 재전송 큐 (M6)

**브랜치**: `feature/fix-m6-telegram`
**STRICT_SCOPE (허용 파일)**:
```
src/distributor/telegram_bot.py
src/utils/retry.py              # 기존 retry util이 있으면 재사용, 없으면 이 파일 신규
tests/test_distributor.py
```
**절대 금지**: `src/main.py`, `src/signals/**`, `src/analyzer/**`, `src/ingestion/**`.

**문제 정의**:
`telegram_bot.send_daily_brief`는 HTTP 429·5xx·timeout에 대해 재시도 없음. blocked/deactivated만 문자열 매칭으로 감지. 라이브 운영 시 일시적 장애로 구독자 메시지가 유실.

**구현**:
1. `send_daily_brief` 내부 각 subscriber 전송 부분을 `_send_with_retry(chat_id, text)`로 추출.
2. `_send_with_retry`는 `telegram.error.RetryAfter`·`TimedOut`·`NetworkError`에 한해 최대 3회 재시도(지수 백오프 base 2s, cap 30s, jitter ±20%).
3. `Forbidden`(blocked)·`ChatNotFound`(deactivated)는 즉시 `blocked` 카운트로 분류(기존 로직 유지).
4. `RetryAfter.retry_after` 값을 우선 존중(429 헤더와 동일 의미).
5. 재시도 3회 실패 시 `failed` 카운트 증가 + 로거 warning. 예외를 상위로 전파하지 않는다(다른 구독자 영향 방지).
6. `src/utils/retry.py`에 기존 `@retry` 데코레이터가 있으면 재사용. 단, 비동기 함수 지원 필요 — 없으면 `async_retry` 헬퍼 추가.

**테스트** (`tests/test_distributor.py`):
- 429(`RetryAfter`) 1회 후 성공 → 전송 카운트 1, 재시도 1회 확인.
- 연속 NetworkError 3회 → `failed` 카운트 1, 예외 미전파.
- `Forbidden` 즉시 → `blocked` 카운트 1, 재시도 0.
- 다중 subscriber에서 한 명 실패가 다른 전송에 영향 주지 않음.

**수용 기준**:
- 신규/수정 테스트 그린.
- `send_daily_brief` 반환 dict 스키마(`sent`/`failed`/`blocked`)는 **변경 금지**. 파이프라인이 의존.

**커밋 메시지**:
```
feat(distributor): retry Telegram send on transient errors
test(distributor): cover RetryAfter, NetworkError, Forbidden paths
```

### 🟠 W2-D : 이중 스코어링 경로 정리 (B3 + M2)

**브랜치**: `feature/fix-b3-dual-path`
**선행 조건**: W1-A 머지 완료(테스트 그린).
**STRICT_SCOPE (허용 파일)**:
```
src/main.py
src/analyzer/scoring.py         # deprecation 주석만, 삭제 금지
tests/test_main.py              # 분기 회귀 케이스 추가
```
**절대 금지**: `src/signals/**`, `src/distributor/**`, `src/storage/**`, `src/analyzer/claude_analyzer.py`.

**문제 정의**:
현재 파이프라인이 Stage 5(`SignalEngine.run`)와 Stage 6-7(`TransactionScorer.pre_filter` + `analyze_batch`)을 무조건 모두 실행한다. Stage 8의 brief는 SignalEngine 결과만 쓰지만, Stage 9의 `daily_brief` 시트에 저장되는 `top5`는 레거시 scorer 결과. 사용자 브리프와 감사 시트의 근거가 다르다.

또한 M2: `_event_to_dict`의 `hash`/`from_address`/`to_address` 필드명 매핑은 레거시 scorer만을 위한 호환 계층 — 레거시를 fallback으로만 쓴다면 이 변환기도 조건부 실행.

**구현**:
1. Stage 5에서 `signals` 리스트 크기 측정. **`len(signals) > 0`이면 Stage 6-7(pre_filter + analyze_batch) 스킵**. signals에서 `top5`를 추출하는 함수 `_signals_to_top5(signals, events)` 신규 작성.
2. `_signals_to_top5`: `signals`를 `score` 내림차순 정렬 → 상위 5 → 각각에 대응하는 `Event`를 `evidence_tx_hashes`로 매핑 → 기존 top5 dict 스키마(`hash`, `symbol`, `amount_usd`, `importance_score`, `interpretation`, `type`)와 동일 형태로 반환.
3. `signals`가 비었을 때만 기존 레거시 경로(scorer + analyze_batch)를 **fallback**으로 실행. 이때만 `_event_to_dict` 변환 수행.
4. 로그에 어떤 경로가 탔는지 명시: `logger.info("[%s] Stage path: signals=%d, using %s", run_id, len(signals), "signals" if signals else "legacy_fallback")`.
5. `scoring.py` 모듈 상단에 deprecation 주석:
   ```python
   # DEPRECATED: TransactionScorer is fallback-only. Remove after baselines
   # stabilize (>= 14 days of address_activity data). Tracked: B3 second phase.
   ```
6. Stage 9 `save_daily_brief`는 항상 `top5`를 저장하지만, 출처가 signals인지 legacy인지 `alert_count` 필드로 구분 가능 — 필드 추가 대신 `details`에 `path=signals|legacy` 포함.

**테스트** (`tests/test_main.py`):
- signals 존재 시: scorer.pre_filter·analyzer.analyze_batch이 **호출되지 않음**을 assert.
- signals 0건 시: 기존 레거시 경로가 실행되는지 확인(기존 테스트 유지).
- `_signals_to_top5` 유닛 테스트: 6개 signals → top5 반환, score 내림차순, 스키마 일치.

**수용 기준**:
- 전체 `pytest -q` 그린.
- 로그에 "Stage path" 라인이 나타남.
- `save_daily_brief` 저장 스키마 하위 호환.

**커밋 메시지**:
```
refactor(pipeline): make legacy scorer fallback-only when signals empty
feat(pipeline): extract top5 from signals when available
test(main): assert scorer skipped when signals present

### 🟠 W2-E : 베이스라인 주입 (B5)

**브랜치**: `feature/fix-b5-baseline`
**선행 조건**: W2-D 머지 완료.
**STRICT_SCOPE (허용 파일)**:
```
src/signals/baseline.py         # 신규
src/main.py                     # 단일 호출부만 추가
src/signals/models.py           # RuleContext docstring만 수정 가능, 필드 변경 금지
tests/test_baseline.py          # 신규
tests/test_main.py              # baseline 주입 한 케이스만 추가
```
**절대 금지**: 기존 rule 함수(`src/signals/rules*.py`) 수정, `SignalEngine` 시그니처 변경.

**문제 정의**:
`RuleContext.chain_baselines`가 빈 dict로 초기화되어 `cex_outflow_spike`·`cex_inflow_spike`·`weekly_net_accumulation` 룰의 3σ 임계가 작동하지 않는다. 룰은 시그널을 만들지 않고 대기 중.

**구현** (`src/signals/baseline.py` 신규):
```
def build_chain_baselines(
    storage: Storage,
    as_of: datetime,
    lookback_days: int = 14,
) -> dict[str, dict[str, float]]:
    """Read address_activity for the last N days, compute per-chain mean+std of daily net flow.
    Returns {chain: {"mean": float, "std": float, "n": int}}.
    Empty dict if lookback data < 7 days (3σ unreliable)."""
```

- 구현은 `storage.list_address_activity(since=as_of - timedelta(days=lookback_days))`에 의존. 해당 메서드가 Storage Protocol에 없다면 **추가는 이 트랙에서 한다** (Protocol + SheetsClient 구현 + 테스트). 단, 기존 메서드 수정은 여전히 금지.
- 수식: 날짜별 net flow 시퀀스(≥7 길이) → `statistics.mean` / `statistics.stdev` → 체인별 dict.
- 데이터 < 7일이면 해당 체인 key를 omit → 룰이 자연스럽게 skip.

`main.py` 변경:
- Stage 5 진입 전 `baselines = build_chain_baselines(sheets, now)` 한 줄.
- `engine.run(raw_events, now, baselines=baselines)` — 단, 현재 `engine.run(events, now)` 시그니처에 `baselines`가 없으므로 **이 트랙에서 시그니처 확장** 허용. 내부적으로 `RuleContext(now=now, chain_baselines=baselines or {})`로 주입.
- `SignalEngine.run` 시그니처 변경은 허용하되 **기본값 `None`으로 후방 호환**.

**테스트** (`tests/test_baseline.py`):
- 7일치 픽스쳐(`ETH` chain, 일일 net flow 시퀀스) → `build_chain_baselines` 반환 mean/std 검증.
- 3일치만 있는 체인 → omit.
- 빈 storage → 빈 dict.

`tests/test_main.py`:
- `build_chain_baselines`가 호출되는지 patch로 검증 1케이스.

**수용 기준**:
- baselines 주입 시 `cex_outflow_spike` 등 3σ 룰이 실제 시그널을 만들 수 있음을 픽스쳐 기반 E2E 테스트 1개로 증명.
- 기본값 `None` 보장으로 기존 테스트 무변경.

**커밋 메시지**:
```
feat(signals): build_chain_baselines helper from address_activity
feat(pipeline): inject baselines into SignalEngine context
test(baseline): cover mean/std computation and low-sample skip
```

### 🟣 W3-F : 멀티유저 personalize 이동 (B6)

**브랜치**: `feature/fix-b6-personalize`
**선행 조건**: W1-C, W2-E 머지 완료.
**STRICT_SCOPE (허용 파일)**:

tests/test_main.py
```
**절대 금지**: `SignalEngine.personalize` 로직 변경, 룰 파일 수정.

**문제 정의**:
`main.py:232`의 `signals = engine.personalize(signals, [])`는 전역 1회 호출로 모든 사용자에 동일 결과 전달. `user_interests` 시트가 있어도 개인화 미작동. 멀티유저 전제에서 시그널이 사용자별로 다르게 필터/가중되어야 한다.

**구현**:
1. `main.py`에서 `engine.personalize(signals, [])` 줄 **제거**. 시그널 리스트 그대로 `bot.send_daily_brief(signals, brief_text)`로 전달.
2. `WhaleScopeBot.send_daily_brief`의 시그니처를 `(brief_text: str, signals: list[Signal] | None = None)`로 확장(하위 호환).
3. 내부 subscriber 루프에서:
   - `interests = self.storage.list_user_interests(chat_id)` 호출
   - `personalized = engine.personalize(signals, interests)` — 단, `bot`이 `engine`을 모르므로 개인화 함수만 주입 받는다. `WhaleScopeBot.__init__(..., personalize_fn: Callable[[list[Signal], list[dict]], list[Signal]] | None = None)` 파라미터 추가.
   - `main.py`에서 봇 생성 시 `personalize_fn=engine.personalize` 주입.
4. 개인화된 `signals`가 0건이면 `"오늘은 관심 기준에 부합하는 시그널이 없습니다."` 메시지로 대체(기존 brief_text 대신).
5. `personalize_fn=None`이면 `brief_text`를 그대로 사용(현재 동작 유지).

**테스트** (`tests/test_distributor.py`):
- interests 데이터가 있고 personalize_fn 주입됐을 때, subscriber별로 다른 시그널 리스트가 렌더링되는지 확인.
- interests 빈 경우 fallback 메시지 렌더링.

`tests/test_main.py`:

**수용 기준**:
- `send_daily_brief` 호출부 전체 그린.
- 기존 단일 메시지 사용자 케이스 동작 유지.

**커밋 메시지**:

test: cover per-user signal personalization in distributor
