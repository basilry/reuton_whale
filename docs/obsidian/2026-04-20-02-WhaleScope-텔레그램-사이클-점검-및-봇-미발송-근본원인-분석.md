# WhaleScope 텔레그램 사이클 점검 및 봇 미발송 근본 원인 분석

## 0. 결론 (TL;DR)

봇이 채널에 **관리자로 정상 등록되어 있음에도 메시지가 가지 않는** 증상의 가장 개연성 높은 원인은 Telegram API 레벨이 아니라 **운영 환경변수 레벨**이다.

1. 채널 관리자 등록 여부는 `getChatMemberCount` 경로(=`channel_health` 잡)에만 영향을 준다. 이 경로는 정상 동작 중이어서 `channel_health` 시트에 `status=ok`가 기록된다. 그래서 "봇이 살아는 있다"라고 착각할 수 있다.
2. 실제 메시지를 보내는 `sendMessage` 경로는 `TelegramBroadcastAdapter.broadcast_text` 안의 **4단 게이트**를 통과해야 비로소 호출된다. 이 게이트가 두 환경변수에 의해 차단되고 있을 가능성이 가장 크다:
   - `TELEGRAM_BROADCAST_ENABLED`
   - `TELEGRAM_BROADCAST_DRY_RUN`
3. `render.yaml`에는 이 두 키가 **아예 선언되어 있지 않다**. 선언이 없으면 `config.py`의 기본값(`ENABLED=false`, `DRY_RUN=true`)이 적용되어, 어떤 슬롯에서도 실제 Telegram API 호출이 발생하지 않는다.
4. 결과적으로 `broadcast_log` 시트에는 `status=skipped_disabled` 또는 `status=dry_run`만 반복 기록되고, 실제 채널에는 메시지가 뜨지 않는다.

검증과 조치 방법은 §5~§7에 정리했다.

## 1. WhaleScope 텔레그램 통합 전체 사이클

### 1.1 상위 구조

```
┌──────────────────────────────────────────────────────────────┐
│  Render cron  (*/15 * * * *)                                 │
│  python -m src.pipeline.run_all                              │
└────────────────┬─────────────────────────────────────────────┘
                 │
                 ▼
    ┌──────────────────────────┐
    │  run_all.py               │
    │  - due_job_names(slot)    │   슬롯별 실행 잡 결정
    │  - slot-duplicate guard   │
    │  - job heartbeat          │
    └──────────────────────────┘
                 │
     ┌───────────┼────────────────┬──────────────┬────────────────┐
     ▼           ▼                ▼              ▼                ▼
  signals   broadcast_periodic  brief  broadcast_daily  channel_health
  news_rss   (매 15분)         (매시   (KST 09:00)     (KST 09:15)
  curated                      정각)
  stories
  (0/6/12/18)
```

### 1.2 텔레그램 계층은 두 레인으로 나뉜다

**Lane A — 채널 브로드캐스트 (`TelegramBroadcastAdapter`)**
- 대상: `@whalescope_alertz` 공개/비공개 채널
- 전송 주체: 채널 관리자 봇 (`TELEGRAM_BROADCAST_BOT_TOKEN` 또는 폴백으로 `TELEGRAM_BOT_TOKEN`)
- 경로: `requests.post({api}/bot{token}/sendMessage)`
- 호출 지점: `broadcast_daily`, `broadcast_periodic`

**Lane B — 구독자 직접 발송 (`WhaleScopeBot.send_daily_brief`)**
- 대상: Google Sheets `subscribers` 시트에 등록된 사용자 chat_id
- 전송 주체: 대화봇 (`TELEGRAM_BOT_TOKEN`)
- 경로: `python-telegram-bot` Application → `bot.send_message(chat_id, ...)`
- 호출 지점: `broadcast_daily` 안에서만 호출 (periodic에는 구독자 레인 없음)

**Lane C — 채널 헬스 프로브 (`channel_health`)**
- 대상: 동일 채널
- 경로: `{api}/bot{token}/getChat`, `getChatMemberCount`
- 용도: "봇이 채널에 여전히 붙어 있는지" 확인용. 메시지 발송과는 무관.

### 1.3 스케줄 매트릭스

`run_all.due_job_names(slot)`은 슬롯별로 실행 대상 잡을 결정한다.

| 잡 | 실행 조건 (KST) | 비고 |
|---|---|---|
| `signals` | 모든 15분 슬롯 | |
| `curated_balance` | 모든 15분 슬롯 | |
| `news_rss` | 모든 15분 슬롯 | |
| `broadcast_periodic` | 모든 15분 슬롯 | 채널 레인 A |
| `stories` | minute==0, hour∈{0,6,12,18} | |
| `brief` | minute==0 (매시 정각) | daily brief 원본 생성 |
| `broadcast_daily` | minute==0 AND hour==9 | 채널 레인 A + 구독자 레인 B |
| `channel_health` | minute==15 AND hour==9 | 레인 C |
| `weekly_trend` | minute==0, hour==8, weekday==1 (화요일) | |

이 스케줄 상 `broadcast_daily`는 **하루 단 한 번 (KST 09:00)** 실행된다. `broadcast_periodic`은 15분마다 실행되지만 §3.3 조건을 만족해야 실제로 발송된다.

## 2. `TelegramBroadcastAdapter.broadcast_text` 게이트 체인

채널 브로드캐스트의 실질적 결정권자. 모든 채널 발송은 이 함수를 거치며, 아래 순서로 조기 반환한다. 함수 위치: `src/notify/telegram_broadcast.py:117-239`.

```
broadcast_text(text, kind, dedup_key, metadata)
 ├─ 1. normalized_text = text.strip()
 │     └─ 빈 문자열 → status="skipped_empty"                   ① 조기반환
 ├─ 2. not self._enabled
 │     └─ ENABLED=false  → status="skipped_disabled"           ② 조기반환
 │         error="TELEGRAM_BROADCAST_ENABLED is false"
 ├─ 3. not self._chat_id OR not self._token
 │     └─ 설정 누락      → status="skipped_unconfigured"       ③ 조기반환
 │         error="missing TELEGRAM_BROADCAST_CHAT | ..._BOT_TOKEN"
 ├─ 4. self._dry_run
 │     └─ DRY_RUN=true   → status="dry_run"                    ④ 조기반환
 │         error="TELEGRAM_BROADCAST_DRY_RUN is true"
 │
 ▼ (여기 도달해야 실제 API 호출 발생)
 5. requests.post(f"{_TELEGRAM_API_BASE}/bot{self._token}/sendMessage", ...)
     ├─ HTTP 200 + ok    → status="sent", message_id 기록
     ├─ HTTP 비정상      → status="failed", error=description
     └─ 네트워크 예외    → status="failed", error=exc
```

모든 결과는 `BroadcastAttempt` → `append_broadcast_log`로 Sheets에 기록되므로, **`broadcast_log.status` 열만 봐도 어느 게이트에서 막혔는지 정확히 판별할 수 있다.**

### 2.1 `state_label()`로 한 줄 진단

같은 파일 `state_label()`은 위 조건을 압축해 라벨 하나로 반환한다:

```
disabled        ← not self._enabled
missing_chat_id ← chat_id 비어있음
missing_token   ← token 비어있음
dry_run         ← self._dry_run
ready           ← 위 모두 통과
```

운영 중 이 값이 `"ready"`가 아니면 채널에 아무 것도 도달하지 않는다.

## 3. 각 파이프라인의 추가 게이트

`broadcast_text` 상위에도 여러 단계의 skip 조건이 있다. 순서가 중요하다.

### 3.1 `broadcast_daily.py` (KST 09:00)

| # | 게이트 | status | 비고 |
|---|---|---|---|
| 1 | `force=False` && `current.hour != 9` (KST) | `skipped_window` | 가장 자주 히트. GitHub Actions workflow_dispatch면 `_should_force_broadcast()`=True로 우회 |
| 2 | `has_logged_run_in_window`(09:00~09:15 슬롯) | `skipped_duplicate` | 이미 오늘 브로드캐스트 끝남 |
| 3 | 최신 `daily_brief.summary` 비어있음 | `completed_empty` | brief 잡이 먼저 돌았어야 함 |
| 4 | `TelegramBroadcastAdapter.broadcast_daily_brief` → §2 게이트 체인 | (위 상태 중 하나) | 채널 레인 A |
| 5 | `WhaleScopeBot.send_daily_brief` | result에 `sent/failed/blocked` 집계 | 구독자 레인 B, 별도 토큰·별도 게이트 없음 |

중요 포인트: **구독자 레인(B)은 `ENABLED/DRY_RUN` 플래그를 거치지 않는다.** 즉, 구독자에게는 DM이 가는데 채널에만 안 가는 상태가 발생할 수 있다. 이 경우 "봇은 살아 있는 것처럼 보이지만 채널은 죽어 보이는" 설명이 된다.

### 3.2 `broadcast_periodic.py` (모든 15분 슬롯)

| # | 게이트 | status |
|---|---|---|
| 1 | slot-window 중복 실행 | `skipped_duplicate` |
| 2 | `signals=0 AND transactions=0` | `skipped_empty` (broadcast_log에도 기록) |
| 3 | `content_hash`가 최근 1시간 내 live/dry_run으로 기록됨 | `skipped_duplicate_content` |
| 4 | `TelegramBroadcastAdapter.broadcast_text` → §2 게이트 체인 | (위 상태 중 하나) |

periodic은 구독자 레인 없음. **채널에만** 메시지를 보낸다. 따라서 §2 게이트에서 막히면 이 슬롯은 완전 침묵.

### 3.3 `channel_health.py` (KST 09:15)

이 잡은 `sendMessage`를 호출하지 않는다. `getChat` + `getChatMemberCount`만 호출해서 채널 메타데이터를 `channel_health` 시트에 적재한다. 결과:

- `status=ok` + `member_count`가 들어오면 → 토큰·채널 ID·봇 관리자 권한은 **모두 정상**
- `status=missing_config` → 토큰/채널 ID 환경변수가 비었음
- `status=error` + error 메시지 → 토큰 무효, 채널 삭제/교체, 봇 kick 등 Telegram 레벨 이슈

**이 시트 값이 `ok`인데 채널에 메시지가 안 온다면, 원인은 무조건 §2 게이트(ENABLED/DRY_RUN/CHAT/TOKEN)다.**

## 4. 환경변수 매트릭스 (현재 설계 vs 현재 배포)

`src/config.py` + `src/pipeline/common.py`의 기본값과 `render.yaml`, `.env.example`, GitHub Actions workflow의 선언을 교차 비교.

| 변수 | 기본값 | `.env.example` | `render.yaml` 선언 | GitHub Actions vars | 역할 |
|---|---|---|---|---|---|
| `TELEGRAM_BOT_TOKEN` | "" (required) | 빈칸 | ✅ sync:false | ✅ secrets | 대화봇 토큰, 구독자 레인 + 폴백 채널 발송 |
| `TELEGRAM_BROADCAST_BOT_TOKEN` | "" | 빈칸 | ✅ sync:false | ✅ secrets | 채널 관리자 봇 토큰 (우선). 없으면 `TELEGRAM_BOT_TOKEN`으로 폴백 |
| `TELEGRAM_BROADCAST_CHAT` | `@whalescope_alertz` | `@whalescope_alertz` | ✅ sync:false | ✅ secrets | 채널 식별자 |
| **`TELEGRAM_BROADCAST_ENABLED`** | **`false`** | `false` | **❌ 미선언** | ✅ vars 기본값 `'false'` | **채널 발송 마스터 스위치** |
| **`TELEGRAM_BROADCAST_DRY_RUN`** | **`true`** | `true` | **❌ 미선언** | ✅ vars 기본값 `'true'` | **건식 실행 모드 (API 호출 없음)** |
| `FORCE_BROADCAST_DAILY` | `false` | `false` | ❌ 미선언 | 간접 (workflow_dispatch) | 시간창 밖에서도 daily 브로드캐스트 강제 |

`render.yaml`에 볼드 표시한 두 변수가 빠져 있기 때문에, Render 배포본에서 Render 대시보드 UI로 수동 추가하지 않았다면 기본값 `ENABLED=false`, `DRY_RUN=true`가 그대로 적용된다. 즉 Render cron의 `broadcast_periodic`·`broadcast_daily` 모두 §2 게이트 ②(`skipped_disabled`) 또는 ④(`dry_run`)에서 조기반환된다.

## 5. "봇이 채널 관리자인데 알림 안 옴" 근본 원인 랭킹

증상을 그대로 유발할 수 있는 원인을 가능성 순으로 나열한다.

### ★ 최우선 가설 — `TELEGRAM_BROADCAST_ENABLED=false`

- 유발 흐름: `broadcast_text` → §2 게이트 ②에서 즉시 `skipped_disabled`. API 호출 아예 없음.
- 관리자 등록과 완전 무관(Telegram 서버에 요청 자체가 가지 않으므로).
- 증상: `channel_health`는 `ok`, `broadcast_log.status`는 `skipped_disabled`로 가득.
- 근거:
  - `render.yaml` 미선언 → 기본값 `false`
  - GitHub Actions workflow도 `vars.TELEGRAM_BROADCAST_ENABLED || 'false'`로 폴백 → Organization/Repository Variables에 `TELEGRAM_BROADCAST_ENABLED=true`가 등록돼 있지 않으면 `'false'`

### ★ 유력 가설 — `TELEGRAM_BROADCAST_DRY_RUN=true`

- 유발 흐름: ENABLED는 켜졌지만 DRY_RUN이 켜져 있으면 §2 게이트 ④에서 `dry_run`. 로그만 쌓이고 실제 발송 없음.
- 증상: `broadcast_log.status=dry_run`, `delivery_mode=dry_run`. 채널 관리자 본인이 보면 봇은 `last_active` 갱신도 없고 메시지도 없음.

### 차순위 — `TELEGRAM_BROADCAST_CHAT` 또는 `TELEGRAM_BROADCAST_BOT_TOKEN` 누락

- 유발 흐름: §2 게이트 ③에서 `skipped_unconfigured` + `error="missing ..."`.
- 주의: `TELEGRAM_BROADCAST_BOT_TOKEN`이 비어도 `TELEGRAM_BOT_TOKEN`이 존재하면 폴백되어 통과한다(broadcast_daily·periodic 둘 다 `env.telegram_broadcast_token or env.telegram_token`). 그래서 이 경로에서 막힐 가능성은 CHAT 값 누락 쪽이 더 크다.

### 가능성 낮음 — `broadcast_periodic`이 `skipped_empty` 반복

- 유발 흐름: 15분 창 안에 signals/transactions가 0건이면 §3.2 게이트 ②.
- 근거 약함: 체인 커버리지가 ETH/SOL/XRP/TRX/BTC/DOGE로 확장된 지금(2026-04-20 기준 이미 적용 완료) 15분 동안 0건이 지속될 가능성은 낮다. 다만 `signals` 잡이 `skipped_inactive`/`completed_empty`로 계속 돌면 이 조건이 재현될 수 있다.

### 가능성 낮음 — `broadcast_periodic`이 `skipped_duplicate_content`

- 유발 흐름: 메시지 content_hash가 1시간 내 live/dry_run과 동일.
- 증상: 시그널이 바뀌지 않는 한 채널에는 1시간에 한 번만 뜸. 완전 침묵은 아님.

### 가능성 낮음 — `broadcast_daily`가 `skipped_window`

- 유발 흐름: KST 09시가 아닌 슬롯에서 실행되면 `skipped_window`. Render cron 자체는 매 15분 `run_all`을 돌리므로 KST 09:00 정각은 한 번 맞음.
- 주의: 만약 해당 슬롯에 Render 인스턴스가 콜드스타트 실패하거나 quota 초과로 run_all이 실행되지 않으면 그날은 완전 skip. 이건 `run_log` 시트에서 `broadcast_daily` 엔트리가 아예 없는 날로 드러난다.

### 외부 (Telegram 레벨) — 사실상 배제

- 봇 토큰 리볼브/무효화 → `channel_health`에서 `error` 기록.
- 봇 kick / 관리자 권한 박탈 → 같은 경로에서 `getChat` 실패로 포착.
- Flood wait (`RetryAfter`) → `failed`로 기록, 최근 24h에 여러 번 반복되면 눈에 띔.

**요약**: 현 코드베이스와 `render.yaml`만 놓고 보면 원인의 90% 이상이 §5의 최우선/유력 가설(ENABLED/DRY_RUN) 하나로 설명된다.

## 6. 구독자 DM은 가는데 채널만 안 오는 경우

사용자가 "알림을 안 준다"라고 했을 때 실제로 의미할 수 있는 두 가지 케이스를 분리해야 한다.

| 시나리오 | 채널 | 구독자 DM | 진단 |
|---|---|---|---|
| ENABLED=false | X | O (09시) | `TELEGRAM_BROADCAST_ENABLED=true`만 켜면 됨 |
| DRY_RUN=true | X | O (09시) | `TELEGRAM_BROADCAST_DRY_RUN=false` 추가 |
| BROADCAST_BOT_TOKEN 잘못된 토큰 | X (401/403) | O | `broadcast_log.error` 확인 후 토큰 재발급 |
| 봇이 채널 kick됨 | X (403) | O | Telegram 쪽 재초대, channel_health도 error |
| 구독자 0명 | O | (없음) | 구독자 증가 전까지는 정상 |
| `TELEGRAM_BOT_TOKEN` 없음 | O (BROADCAST_TOKEN으로 발송 성공 시) | X | 구독자 레인 죽음, 채널은 살아있음 |

현재 사용자 증상 ("채널 관리자인데 알림 안 옴")은 **첫 번째 또는 두 번째 줄**에 해당할 확률이 가장 높다. 확인은 §7로.

## 7. 검증 체크리스트 (10분 안에 근본 원인 확정)

아래 순서대로 수행하면 어디서 막혔는지 반드시 드러난다.

### 7.1 Google Sheets `broadcast_log` 시트 직독

1. 시트 열기 → `kind` 열을 `broadcast_periodic`으로 필터
2. 최근 2시간 (8행) 기준으로 `status` 열만 본다:
   - 전부 `skipped_disabled` → §5 최우선 가설 확정
   - 전부 `dry_run` → §5 유력 가설 확정
   - 전부 `skipped_unconfigured` → CHAT 또는 TOKEN 누락
   - 전부 `skipped_empty` → 시그널 파이프라인 쪽 문제
   - 전부 `skipped_duplicate_content` → 메시지가 너무 동일, 시그널 다양성 부족
   - `sent`는 있는데 채널에서 안 보임 → 브라우저·네이티브 앱 새로고침, 올바른 채널 구독 여부 확인
   - `failed` + error=`...Bad Request...` → 토큰/채널 미스매치

### 7.2 `channel_health` 시트 직독

- `status=ok` + `member_count` 수치 존재 → 토큰/채널/관리자 권한은 정상. 원인은 §2 게이트.
- `status=error` + Telegram API 설명 → 토큰 또는 봇 권한 자체가 깨짐.

### 7.3 Render 대시보드 환경변수 확인

Render Dashboard → `whalescope-pipeline` 서비스 → Environment 탭에서 아래 키 존재·값 확인:

- `TELEGRAM_BROADCAST_ENABLED` → 반드시 `true`
- `TELEGRAM_BROADCAST_DRY_RUN` → 반드시 `false`
- `TELEGRAM_BROADCAST_CHAT` → `@whalescope_alertz` 또는 채널 ID (`-100xxxxxxxxxx`)
- `TELEGRAM_BROADCAST_BOT_TOKEN` → 채널 관리자 봇의 실제 토큰 (또는 비우고 `TELEGRAM_BOT_TOKEN`에 의존)

키 자체가 없다면 추가하고 Deploy. 값이 있는데 위와 다르면 즉시 수정.

### 7.4 `/admin` rollout 진단 화면 (이미 구현됨)

설계 문서 #26 및 #01에 따라 `/admin`에 broadcast observability가 포함되어 있다. 거기서 최근 24시간 `broadcast_log`의 status 분포를 바로 볼 수 있다. 위 시트 직독과 등가 정보지만 UI가 더 빠르다.

### 7.5 GitHub Actions 수동 트리거 (선택)

- Actions → `Whale Broadcast Daily` → Run workflow
- 이 워크플로우는 Repository/Organization Variables에서 `TELEGRAM_BROADCAST_ENABLED`, `TELEGRAM_BROADCAST_DRY_RUN`을 읽는다. 둘 다 `true`/`false`로 설정해 두고 수동 실행하면 정각이 아니어도 `FORCE_BROADCAST_DAILY`나 `GITHUB_EVENT_NAME=workflow_dispatch` 덕분에 window 가드를 우회해 end-to-end를 테스트할 수 있다.

## 8. 권장 수정안

### 8.1 즉시 조치 (단일 변경으로 증상 해소)

Render 대시보드에서 서비스 `whalescope-pipeline` 환경변수에 다음 두 개를 추가한 뒤 수동 Deploy 또는 다음 cron 슬롯을 기다린다.

```
TELEGRAM_BROADCAST_ENABLED=true
TELEGRAM_BROADCAST_DRY_RUN=false
```

이후 다음 15분 슬롯의 `broadcast_log`에서 `status=sent` 엔트리가 나오는지 확인한다.

### 8.2 선언적 보강 (`render.yaml`에 영구 기록)

배포마다 환경변수가 휘발되거나, 새 서비스 인스턴스에서 실수로 누락되는 사고를 막기 위해 `render.yaml`에 두 키를 명시적으로 선언한다. 값 자체는 secret이 아니므로 `value`로 직접 박아도 되고, 운영 정책상 sync:false로 대시보드 관리에 맡겨도 된다.

```yaml
# render.yaml envVars에 추가
- key: TELEGRAM_BROADCAST_ENABLED
  value: "true"
- key: TELEGRAM_BROADCAST_DRY_RUN
  value: "false"
- key: FORCE_BROADCAST_DAILY
  value: "false"
```

이렇게 해두면 `render.yaml`만 봐도 "브로드캐스트는 켜진 상태"임이 감사 가능해진다.

### 8.3 GitHub Actions Variables도 동기화

`Whale Broadcast Daily` 워크플로우의 vars 폴백 기본값이 `'false'`/`'true'`이므로, Repository Settings → Secrets and variables → Actions → Variables에 다음을 등록한다.

```
TELEGRAM_BROADCAST_ENABLED=true
TELEGRAM_BROADCAST_DRY_RUN=false
```

이걸 안 해두면 workflow_dispatch로 수동 트리거했을 때 여전히 `skipped_disabled`/`dry_run`으로 끝난다.

### 8.4 (선택) "미설정 시 경보" 보강

운영 실수로 같은 증상이 재발하는 걸 막기 위한 소프트웨어 수준의 가드를 고려할 만하다.

- `load_pipeline_env()` 또는 `TelegramBroadcastAdapter.__init__`에서 `enabled=False` 또는 `dry_run=True`인 상태를 `observability.service_heartbeat`에 `service="telegram.broadcast" status="degraded"`로 매 run 기록.
- 또는 `broadcast_log`에 `skipped_disabled`가 6슬롯 연속 쌓이면 `system_log`에 warning.
- 이 가드는 채널 메시지가 끊겼을 때 "Telegram이 문제인가" 하고 먼 곳부터 뒤지는 시간을 아낀다.

## 9. 부가 관측 — 문서 #01과의 정합성

문서 `2026-04-20-01-WhaleScope-체인-커버리지-적용완료-QA-종합보고서`는 TG mirror observability(Whale Alert 등 외부 채널 이벤트 수집 관측) 관점을 다룬다. 이 경로는 `src/ingestion/telethon_listener.py` + `src/ingestion/tg_normalizer.py`를 통해 **수신(inbound)** 되는 데이터이고, 본 문서에서 다룬 `TelegramBroadcastAdapter`/`WhaleScopeBot`은 **발신(outbound)** 이다. 두 경로는 토큰·채널·게이트가 서로 독립이며, TG mirror가 정상이어도 outbound 브로드캐스트가 죽을 수 있다.

즉 `/admin`에서 TG mirror 24h 건수가 건강하게 올라오는 중이라 해도, 같은 화면의 broadcast observability가 `skipped_disabled` 일색이면 본 문서의 진단이 그대로 적용된다.

## 10. 남은 후속 과제

1. `render.yaml` 환경변수 선언 누락 정정 (§8.2).
2. GitHub Actions Variables 동기화 (§8.3).
3. `TelegramBroadcastAdapter`가 `state_label() != "ready"` 상태로 진입할 때 별도 alert를 쏘는 경량 가드 도입 검토 (§8.4).
4. 구독자 레인(Lane B)도 `ENABLED`/`DRY_RUN` 플래그를 따를 것인지 정책 결정. 현재는 채널만 게이트되고 구독자 DM은 무조건 나간다. MVP 기간에는 현재 정책이 안전하지만, 유료 전환 이후에는 둘을 같은 플래그 아래 묶는 것이 일관성에 유리.
