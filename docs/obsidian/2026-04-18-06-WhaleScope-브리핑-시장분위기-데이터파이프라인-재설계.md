# 유저홈 브리핑·시장분위기·거래소 시그널 데이터 파이프라인 재설계

## 0. 요약

유저홈 상단의 **"고래브리핑"** 과 **"시장 분위기"** 가 업데이트되지 않는 문제(이슈 #2), 그리고 **"거래소 유입/유출"** 시그널의 의미가 모호한 문제(이슈 #3)를 함께 다룬다. 두 문제의 공통 원인은 **데이터 파이프라인의 출력 계약(output contract)이 UI 요구 수준보다 낮다**는 점이다.

- **#2 원인 가설**: `buildBriefCopy()`·`buildMarketMood()` 가 참조하는 `latestBrief`, `recentSignals` 가 (a) 만들어지지 않거나, (b) 만들어졌지만 최신 시점이 반영되지 않거나, (c) 만들어졌지만 `cleanGeneratedBrief()` 에서 빈 문자열로 귀결되는 경로가 있다.
- **#3 원인 가설**: 현재 시그널은 `{rule, severity, score, body}` 정도의 얕은 스키마라서, "거래소 유입/유출" 이 **어느 거래소인지, 어느 코인인지, 어느 법정통화인지, 방향은 무엇인지**를 알 수 없다.

본 문서는 이 두 문제를 **파이프라인·스키마·UI 3 레이어** 에서 동시에 재설계한다.

## 1. 이슈 #2 — 고래브리핑 & 시장 분위기가 업데이트되지 않는 문제

### 1.1 현재 코드 경로

#### 1.1.1 서버 진입

```tsx
// apps/dashboard/app/page.tsx
async function loadInsightState(): Promise<InsightState> {
  try {
    const data = await getDashboardData({
      transactionLimit: 6,
      signalLimit: 6,
      systemLogLimit: 4,
    });
    return { data, sourceConnected: true };
  } catch (error) {
    console.error("[insights/page]", error);
    return { data: null, sourceConnected: false };
  }
}
```

`data.latestBrief`, `data.recentSignals`, `data.recentTransactions`, `data.metrics` 를 소비.

#### 1.1.2 브리핑 카피 빌더

```tsx
function buildBriefCopy(data: DashboardData | null) {
  const brief = data?.latestBrief;
  if (brief) {
    return {
      title: "오늘의 고래 브리핑",
      summary: brief.summary ? cleanGeneratedBrief(brief.summary) : "오늘 수집된 브리핑이 없습니다.",
      highlights: brief.highlights.length > 0 ? brief.highlights : [...fallback],
      note: brief.alertCount > 0 ? `${brief.alertCount}개의 경고성 포인트가 함께 기록되었습니다.` : "경고성 포인트는 제한적입니다.",
    };
  }
  // Fallback: "아직 연결된 브리핑이 없습니다..."
}
```

#### 1.1.3 시장 분위기 빌더

```tsx
function buildMarketMood(data: DashboardData | null) {
  const recentSignals = data?.recentSignals ?? [];
  const criticalCount = recentSignals.filter(s => getSignalTone(s.severity, s.score) === "critical").length;
  const watchCount = recentSignals.filter(s => getSignalTone(s.severity, s.score) === "watch").length;
  // 3단계 분기: "주의" (critical > 0) / "관찰" (watch > 0) / "안정"
}
```

### 1.2 업데이트 안 되는 원인 후보

이 함수들은 구현 자체에 버그는 없어 보인다. **데이터 소스 쪽에서 업데이트가 멈췄을 가능성**이 가장 높다.

| 후보 | 진단 방법 | 확률 |
|---|---|---|
| A. 브리핑 생성 워커가 실행 안 됨 | `data.metrics.latestRunStatus` 가 `failed`/`pending` 인지 | ★★★★ |
| B. Google Sheets `brief_runs` 시트에 데이터는 있으나 최신행이 아닌 과거 행을 읽음 | `getDashboardData` 구현 내 `ORDER BY` 확인 | ★★★ |
| C. `brief.summary` 가 존재하지만 `cleanGeneratedBrief()` 가 공백 문자열로 귀결 | 유닛 테스트로 확인 | ★★ |
| D. LLM 호출 실패로 `summary` 가 비어 있음 | 워커 로그 확인 | ★★★ |
| E. `recentSignals` 가 7일 이상 과거 데이터만 반환 | `getDashboardData` 의 기간 필터 확인 | ★★★ |
| F. Next.js ISR/fetch 캐시 때문에 갱신이 늦음 | `revalidate: 0` 또는 `cache: "no-store"` 확인 | ★★ |

**1차 액션**: 위 A~F 를 순차로 진단하는 디버깅 플레이북을 만들어서 먼저 실제 원인을 특정.

### 1.3 근본 처방 — 브리핑 파이프라인 재설계

단기 원인 제거로는 같은 문제가 재발한다. **브리핑을 "항상 최신"으로 유지하는 파이프라인** 으로 재설계한다.

#### 1.3.1 요구

- 매 N분마다 (또는 이벤트 트리거 시) 브리핑이 자동 생성/갱신.
- 새 브리핑이 없어도 "최근 M시간 이내의 마지막 브리핑" 을 보여주거나 **스테일 배지**(예: "30분 전 생성") 를 표시.
- 워커 실패 시 **명시적으로** 사용자에게 연결 문제를 표현 (현재는 fallback 카피가 애매).

#### 1.3.2 브리핑 스키마 (v2)

```ts
type BriefRecord = {
  brief_id: string;                  // 고유 ID (run_id 기반)
  run_started_at: string;            // ISO8601
  run_finished_at: string | null;
  status: "success" | "partial" | "failed" | "pending";
  // 사람 읽기 영역
  summary: string;                   // 3~5문장 LLM 요약
  highlights: string[];              // 3~5개 핵심 포인트
  note: string;                      // 경고성 부연 설명
  // 메타
  signal_themes: string[];           // 본 브리핑 생성에 반영된 시그널 테마 라벨
  alert_count: number;
  total_volume_usd: number;
  transaction_count: number;
  top_transactions: Array<{ hash: string; symbol: string; amount_usd: number; ... }>;
  // 품질
  llm_model: string;                 // e.g. "gpt-4o-mini-2024-07-18"
  llm_latency_ms: number;
  llm_token_usage: { prompt: number; completion: number };
  llm_guardrail_flags: string[];     // 생성 시 트리거된 경고
};
```

#### 1.3.3 파이프라인 단계

```
[Stage 0] 스케줄러 tick (예: Render Cron 15분마다)
  ↓
[Stage 1] 원자료 수집 — WhaleAlert/Arkham/자체 수집
  ↓ raw_events 테이블에 upsert
[Stage 2] 정규화 — transactions 테이블에 normalize
  ↓
[Stage 3] Enrichment — 지갑 라벨, 코인 메타, 거래소 식별 (→ 감시지갑 문서 참조)
  ↓
[Stage 4] 집계 — 최근 1h/4h/24h 의 volume, count, top_tx 재계산 (brief_inputs 테이블)
  ↓
[Stage 5] 시그널 엔진 — 규칙별 트리거, severity/score 계산 (signals 테이블)
  ↓
[Stage 6] 브리핑 생성 — Stage 4+5 를 LLM 에 투입, 문장 생성 (brief_runs 테이블)
  ↓
[Stage 7] 후처리 — cleanGeneratedBrief 적용, 길이 제한, guardrail
  ↓
[Stage 8] 게시 — latestBrief = (오늘의 최신 success 행)
```

**핵심**: Stage 6 이 실패해도 Stage 4+5 는 살아 있음 → UI 는 "브리핑 생성 실패, 시그널과 지표는 정상" 같은 부분 성공을 표현할 수 있어야 함.

#### 1.3.4 dashboard 의 데이터 로더 개선

`getDashboardData` 를 다음과 같이 분해:

```ts
type DashboardData = {
  generatedAt: string;
  meta: {
    latestRunStatus: RunStatus;
    latestRunAgeMinutes: number;     // 몇 분 전에 생성됐는지
    pipelineStages: Array<{ name: string; status: RunStatus; lastSuccessAt: string | null }>;
  };
  latestBrief: BriefRecord | null;   // Stage 6 의 가장 최신 success
  fallbackBrief: BriefRecord | null; // 최근 success 가 오래됐을 때 참조용
  recentSignals: SignalRecord[];     // Stage 5
  recentTransactions: TransactionRecord[]; // Stage 2
  metrics: MetricsSnapshot;          // Stage 4
};
```

`buildBriefCopy` 는 다음 우선순위로 고름:
1. `latestBrief.status === "success"` 이고 `latestRunAgeMinutes ≤ 60` → 그대로 표시
2. `latestRunAgeMinutes > 60` 이지만 `fallbackBrief` 존재 → 표시 + 스테일 배지 "1시간 이상 전 브리핑"
3. 둘 다 없으면 → "브리핑 생성 대기" + 파이프라인 상태로 원인 안내

### 1.4 시장 분위기 재설계

현재 `buildMarketMood` 는 **시그널 개수만** 기준. 실제 시장 분위기는 시그널 외에도:

| 입력 | 반영 방법 |
|---|---|
| 순유입/순유출(= exchange flow net) | 순 플로우가 강한 유입이면 "매수 우세" 편향 |
| 변동성 지표 | 최근 4h 거래량 Z-score > 2 면 "고변동" |
| 시그널 커버리지 | 여러 규칙이 동시에 트리거되면 분위기 강화 |
| 외부 레퍼런스 | 예를 들어 BTC 가격의 1h 변동폭 | 

**새 출력**:

```ts
type MarketMood = {
  label: "안정" | "관찰" | "주의" | "과열" | "연결 대기";
  score: number;         // -100 ~ +100 (음수 매도 편향, 양수 매수 편향, 절대값이 크면 변동성)
  drivers: Array<{ label: string; delta: number; source: "signal" | "flow" | "volatility" }>;
  copy: string;          // 사람이 읽는 요약
  detail: string;        // 숫자 포함 요약
  confidence: "low" | "medium" | "high"; // 데이터 커버리지 기반
};
```

UI 는 `score` 를 작은 게이지/바/스파크라인으로 시각화해도 좋음 (선택).

### 1.5 프론트 캐시 처리

Next.js App Router 특성상 ISR/fetch cache 가 **문제가 될 수 있다**. `loadInsightState` 내부에서:

```ts
const data = await getDashboardData({...}, { cache: "no-store" }); // 또는 revalidate: 0
```

혹은 페이지에 `export const revalidate = 60;` 를 명시하여 최소 1분 단위 재생성 보장.

### 1.6 관찰성 (Observability)

- 브리핑 파이프라인 각 Stage 는 성공/실패 + 소요 시간을 **운영 페이지**(`/admin`) 에서 확인 가능해야 한다.
- 실패 시 Sentry/로그에 담기고, 헤더 칩이 `데이터 연결됨` 이어도 브리핑 카드 안에 "브리핑 생성 실패" 배지를 따로 표현.
- Slack 웹훅으로 실패 알림 (단, 조용한 실패는 금지).

### 1.7 검증 체크리스트

- [ ] 브리핑 카드가 직전 성공 이후 **60분 이내** 면 항상 표시.
- [ ] 60분 초과 시 `오래된 브리핑` 배지와 생성 시각 병기.
- [ ] 24시간 초과 시 fallback 카피로 전환.
- [ ] 브리핑 파이프라인의 Stage 6 실패 후에도 Stage 5 시그널은 최신으로 유지.
- [ ] `cache: "no-store"` 또는 `revalidate ≤ 60s` 보장.
- [ ] 시장 분위기 드라이버가 최소 2개 이상 표시.

## 2. 이슈 #3 — "거래소 유입/유출" 시그널 모호성 해결

### 2.1 사용자 피드백 (원문)

> 유저홈 시그널 섹션에서, 거래소 유입/유출에 대한 것이 너무 모호해. 정확하게, 특정 코인으로의 유입/유출인지 혹은 특정 화폐(원/달러/유로/엔/위안 등)의 유입인지, 아니면 특정 거래소(업비트/크라켄/빗썸/바이낸스 등)로의 현금 유입인지가 모호해.

### 2.2 왜 모호했는가

현재 시그널 스키마는 `{rule, severity, score, body}` 중심이다. `body` 에 자연어로 "거래소로 유입" 이라 쓰면, **어느 거래소, 어느 코인, 어느 법정통화**가 사라진다. 거래소 플로우 시그널은 **최소 4차원**이 필요하다:

```
<exchange> × <asset> × <direction> × <quote_basis>
```

### 2.3 4차원 정의

| 차원 | 값 예시 | 의미 |
|---|---|---|
| `exchange` | "upbit", "binance", "bithumb", "kraken", "coinbase", "okx" ... | 표준 거래소 슬러그. UNKNOWN 허용. |
| `asset` | "BTC", "ETH", "USDT" ... | 블록체인/자산 심볼. |
| `direction` | "inflow" / "outflow" | 거래소 지갑 기준 **외부 → 거래소** 이면 inflow, **거래소 → 외부** 이면 outflow. |
| `quote_basis` | "coin" / "usd" / "krw" / "eur" / "jpy" / "cny" | 표시 금액의 기준 통화 (Fiat 로 환산한 경우 그 통화). |

### 2.4 용어 계층 정리 (중요)

온체인에서는 **법정통화의 직접적인 이동은 없음**. "원화가 업비트로 들어갔다" 는 말은 온체인 수준에서는 틀린 서술이다. 우리가 말할 수 있는 것은:

1. **온체인 자산 이동** (USDT/USDC 스테이블 포함 암호자산) 이 거래소 지갑으로 들어갔다 / 나갔다.
2. **거래소 내부 장부상 통화 잔고 변화** — 이것은 온체인에서 보이지 않음. 거래소 API 로만 간접 추정 가능.

따라서 **시그널은 "암호자산 온체인 이동" 을 1급 엔티티로** 서술해야 한다. "법정통화 유입"이라는 서술이 필요할 때는 **스테이블코인 이동을 법정통화 프록시로 사용**한다는 정책을 명시해야 한다.

#### 프록시 매핑 정책

| 스테이블 | 법정 프록시 | 근거 |
|---|---|---|
| USDT / USDC / DAI / FDUSD | USD | 주요 페그 |
| EURC / EURT | EUR | |
| (예정) KRW 페그 | KRW | 현재 없음. 국내 거래소 분석은 별도 로직 |

**정책 주의**: 한국 거래소(Upbit/Bithumb) 의 "원화 유입" 은 프록시 불가. 오직 **거래소 Hot wallet 주소로 들어오는 암호자산 수량**만 본다.

### 2.5 거래소 식별 방법

| 방법 | 정확도 | 비용 | 채택 |
|---|---|---|---|
| A. 알려진 거래소 지갑 주소 레이블 사전(Arkham/Nansen/Etherscan 태그) | 높음 | 데이터 확보 필요 | ✅ |
| B. 거래소 출금 패턴 휴리스틱(주소 클러스터링) | 중간 | 복잡 | 보조 |
| C. 온체인 분석 업체 API 유료 | 매우 높음 | 비용 | Phase 2 |

→ A 를 1차로. 자체 `exchange_wallets` 테이블에 `{address, chain, exchange, wallet_type: hot|cold|deposit}` 축적.

### 2.6 새 스키마 — 시그널 (v2)

```ts
type SignalRecord = {
  signal_id: string;
  detected_at: string;
  rule: SignalRule;                   // 표준 rule 슬러그 (아래 2.8 카탈로그)
  severity: "low" | "medium" | "high" | "critical";
  score: number;                      // 0~100

  // 🔷 구체적 사실 차원
  asset: string | null;               // e.g. "BTC"
  amount_native: number | null;       // 네이티브 수량
  amount_usd: number | null;          // USD 환산
  exchange: string | null;            // 예: "binance"
  direction: "inflow" | "outflow" | null;
  quote_basis: "coin" | "usd" | "krw" | "eur" | "jpy" | "cny";

  // 🔷 트랜잭션 참조
  transaction_hashes: string[];       // 이 시그널을 구성하는 tx 해시 리스트
  wallet_addresses: string[];

  // 🔷 표시 카피
  title: string;                      // "Binance로 BTC 1,245 유입 (~$82M)"
  body: string;                       // 부연 (LLM 또는 템플릿)
  narrative: string;                  // buildSignalNarrative 대체

  // 🔷 메타
  window_start: string;               // 집계 기간 시작
  window_end: string;
  confidence: "low" | "medium" | "high";
};
```

### 2.7 거래소 플로우 시그널 타이틀 템플릿

```
<Exchange>로 <Asset> <Amount Native> 유입 (~$<Amount USD>)
<Exchange>에서 <Asset> <Amount Native> 유출 (~$<Amount USD>)
```

예:
- "Binance로 BTC 1,245 유입 (~$82M)"
- "Upbit에서 ETH 3,800 유출 (~$13M)"
- "Bithumb로 USDT 45M 유입 (~$45M · USD 프록시)"

**법정통화 프록시를 쓴 경우 반드시 `· USD 프록시` 같은 주석** 을 보여준다. 사용자의 오해(원화가 이동한 것으로 착각) 를 원천 차단.

### 2.8 시그널 룰 카탈로그 (v2)

| 룰 슬러그 | 분류 | 정의 |
|---|---|---|
| `exchange.inflow.large` | 거래소 플로우 | 특정 거래소 Hot wallet 으로 Z-score ≥ N 의 대량 유입 |
| `exchange.outflow.large` | 거래소 플로우 | 〃 유출 |
| `exchange.inflow.stable` | 거래소 플로우 | 스테이블코인 유입 ≥ 임계액 (USD 프록시) |
| `exchange.outflow.stable` | 거래소 플로우 | 〃 유출 |
| `whale.wallet.move` | 지갑 이동 | 라벨된 고래 지갑이 대량 이동 |
| `whale.wallet.cluster.activation` | 지갑 이동 | 장기 sleep 지갑 클러스터가 동시 활동 |
| `price.spike.positive` | 시장 | 특정 거래소의 1h 가격 스파이크 양 (실시간 티커 문서와 연동) |
| `price.spike.negative` | 시장 | 〃 음 |
| `funding.rate.extreme` | 시장 | 선물 펀딩레이트 극단 (Phase 2) |
| `stablecoin.issuance` | 매크로 | 발행/소각 대량 감지 |

각 룰은 **트리거 조건, 입력 데이터 소스, 집계 윈도우, 임계치** 를 문서화한다. → 별도 `signals.md` 레퍼런스 필요.

### 2.9 UI 변경 — 시그널 카드

Before:
```
거래소 유입 감지
severity: high
```

After:
```
Binance로 BTC 1,245 유입
~$82M · 지난 1시간 · severity: high · 점수 72
[지갑 2개 · 트랜잭션 3건 보기]
```

구성 요소:
- **1행**: `{title}` (exchange + asset + direction + amount 이 한 문장에 전부 드러남)
- **2행**: `amount_usd` · window · severity 뱃지 · score
- **액션**: 트랜잭션 상세 드로어 (감시지갑 문서와 연결)

### 2.10 법정통화 질의 대응

사용자가 "원화로 얼마가 업비트에 들어갔나" 같은 질문을 했을 때의 정책:

- **온체인만 가지고는 답할 수 없음** 을 명시하는 안내 카피 + 스테이블 프록시의 최근 이동량을 표시.
- 향후 거래소 공개 API 에서 fiat deposit volume 을 제공하는 거래소가 있다면 별도 파이프라인 추가 (Phase 3).

### 2.11 검증 체크리스트

- [ ] 모든 `exchange.*.large` 시그널이 `exchange`, `asset`, `direction`, `amount_native`, `amount_usd`, `quote_basis` 6 필드 충족.
- [ ] 프록시 사용 시 UI 에 "USD 프록시" 등 주석.
- [ ] UI 에서 "거래소 유입" 같은 모호한 타이틀이 등장하지 않음 (타이틀에 반드시 거래소 이름이 포함).
- [ ] `exchange_wallets` 사전이 최소 20 개 주요 거래소 × 3 체인 이상 커버.
- [ ] UNKNOWN 거래소는 `exchange = null` 로 표시하고 룰은 트리거하지 않는다 (데이터 품질 보호).

## 3. 스키마 마이그레이션

### 3.1 기존 → 신규 대조

| 기존 컬럼 | 신규 컬럼 | 변환 |
|---|---|---|
| `signals.body` (자유 문자열) | `signals.title` + `signals.body` 분리 | 룰별 템플릿 재생성 |
| `signals.rule` | 유지 but 카탈로그 고정 | 기존 값 매핑 표 필요 |
| 없음 | `signals.exchange`, `signals.asset`, `signals.direction`, `signals.amount_*` | 새 파이프라인으로 채움 |
| `brief_runs.summary` | 유지 | 없음 |
| 없음 | `brief_runs.status`, `brief_runs.pipeline_stages[]` | 상태 추적 추가 |

### 3.2 Rollout

1. **Phase A**: 새 필드를 **nullable 로 추가**하고 신규 파이프라인이 채우도록.
2. **Phase B**: 프론트가 새 필드 존재 시 새 렌더링, 없으면 fallback.
3. **Phase C**: 90% 이상 커버리지 도달 시 fallback 제거.
4. **Phase D**: 기존 `body` 자유서술 컬럼은 readonly 아카이브로 격리.

## 5. 오픈 이슈

- **거래소 지갑 사전의 출처**: Arkham/Nansen 은 유료. Etherscan 태그는 커버리지 낮음. 자체 수집 채널 필요.
- **한국 원화 프록시 부재**: 원화 스테이블이 대중적이지 않음. Upbit/Bithumb 의 원화 입금 대용 지표를 어떻게 제시할지 미해결.
- **LLM 비용**: 브리핑이 15분마다 생성되면 월 비용이 늘어남. 캐시 전략, 유의미한 이벤트가 있을 때만 재생성하는 이벤트 드리븐 전환도 고려.
- **NoSQL vs SQL**: 현재 Google Sheets. `brief_runs.pipeline_stages[]` 같은 중첩 구조는 Sheets 로는 불편. Postgres 전환 가속 근거가 될 수 있음.
