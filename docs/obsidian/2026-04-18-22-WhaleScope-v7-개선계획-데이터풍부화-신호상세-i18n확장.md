# WhaleScope v7 개선계획 — 데이터 풍부화 · 신호 상세 · i18n 확장

## 0. 배경

v6까지로 운영 파이프라인(`run_all` slot-snap, duplicate guard, `service_health` heartbeat, `/admin` 5-카드 상태 모델, light 테마, `ko/en` 셀렉터 뼈대, Telegram 공개 UX, 뉴스 stale 경고, 스토리 `generatedAt`)은 반영됐다. 시트에는 최신 데이터가 쌓이고 있다.

그러나 유저 홈 관점에서 여전히 다음 문제가 관찰된다.

1. 일일 브리핑이 "주목할 고래 움직임이 없다"로 고정된다(실제 $539M USDT 이동이 있는데도).
2. 뉴스 RSS 스크래핑이 대부분의 슬롯에서 실행되지 않는다(로그에 `['signals', 'curated_balance']`만 찍힘, 동시에 CoinGecko 429).
3. 시장 분위기 로직이 모호하고, 브리핑 자체의 일부로 들어와 있지 않다.
4. 시그널 카드의 "분석 내용 보기"가 하드코딩된 장식 텍스트로, 클릭에 반응하지 않는다.
5. `watched_addresses` 데이터가 시트에 있는데 유저 홈 큐레이션 섹션에 반영되지 않는다.
6. 고래 스토리가 "방금"으로 끝난다. 사용자는 `YYYY.MM.DD HH:mm:ss`를 원한다.
7. 루트 `.env`에 설정한 텔레그램 변수가 `apps/dashboard`에서 잡히지 않는다.
8. `en` 언어로 전환해도 카드 본문/핼퍼 문구가 한국어로 남는다.

v7의 목적은 위 8개 이슈를 **근본 원인 → 변경 범위 → 구체 코드 변경안 → QA 기준**까지 내려쓰는 것이다. 즉, Claude Code가 세션에 투입되면 그대로 티켓을 열어 구현할 수 있는 수준의 실행 문서로 만든다.

## 1. 일일 브리핑 "무조건 의미있는 데이터 표출"

### 1.1 근본 원인

`src/pipeline/brief.py`의 흐름:

- `_build_brief_request(signals)`가 **시그널만** LLM 입력으로 넘긴다(line 40-61). `top_transactions`, `total_volume_usd`는 시그널이 있어야만 파생된다.
- line 87-96에서 `if not signals: result.update(status="completed_empty"); return result` — **시그널이 0이면 LLM 호출도 트랜잭션 조회도 하지 않고 바로 리턴**한다.
- 그 결과, 시트의 `daily_brief`에 `summary = "주목할 고래 움직임이 없다"` 취지의 기본값이 들어가고, 대시보드는 이를 그대로 렌더링한다.

왜 시그널이 0이 되는가:

- `src/signals/rules.py`의 8개 룰(`cex_inflow_spike`, `cex_outflow_spike`, `cold_to_hot_transfer`, `smart_money_accumulation`, `token_whale_concentration_shift`, `tg_cex_inflow_burst`, `corroborated_move`, `weekly_net_accumulation`)은 **모두 `amount_usd > threshold` 필터를 건다**.
- `amount_usd`는 CoinGecko 가격 enrich에 의존한다. CoinGecko 무료티어 레이트리밋(10–30 req/min)에 걸려 429가 뜨면 enrich 실패 → `amount_usd=0` → 전 룰 탈락 → `signals=0`.
- 따라서 **가격이 한 번 삐끗하는 순간 브리핑이 완전히 무의미해진다**.

### 1.2 개선 원칙

- 브리핑은 **항상** "그 슬롯에서 가장 주목할 만한 사실" 하나를 말해야 한다. 빈 브리핑은 허용하지 않는다.
- 시그널 0건이어도, 원시 트랜잭션과 가격 프록시를 이용해 의미 있는 기본 요약을 만든다.
- 시그널이 있으면 기존 LLM 경로가 우선한다. 없을 때에 한해 fallback이 작동한다.

### 1.3 변경안

`src/pipeline/brief.py`

```python
# before
if not signals:
    result.update(status="completed_empty", message="no signals")
    return result

# after
if not signals:
    fallback = build_transaction_fallback_brief(now)
    if fallback is None:
        # 트랜잭션도 0인 진짜 침묵 슬롯
        save_daily_brief(today, [_empty_but_informative_brief(now)])
        result.update(status="completed_fallback_empty")
        return result
    save_daily_brief(today, [fallback])
    result.update(status="completed_fallback_tx", note="signals=0, tx fallback used")
    return result
```

`build_transaction_fallback_brief(now)` 책임:

1. 최근 60분 트랜잭션을 `load_recent_transactions(window_min=60)`로 읽는다.
2. USD 가격 프록시를 **이중화**한다. `PriceResolver`에 아래 순서로 질의:
   - (a) CoinGecko (기존)
   - (b) Binance spot ticker (공개 엔드포인트, 거의 무제한)
   - (c) CoinMarketCap Free tier (선택)
   - (d) **최근 성공 가격 캐시**(`price_cache` 시트 또는 파일) — 최대 6시간 stale 허용
3. 상위 USD 금액 5건을 `top_transactions`로 선별, 총 `total_volume_usd` 산출.
4. `summary`는 LLM 호출 **없이** 템플릿으로 구성:
   - `"최근 60분 온체인 대형 이동 {n}건 감지. 총 {total}. 최대 이동 {top_asset} {top_amount} ({top_direction})."`
5. `alert_count=0`, `signal_themes=[]`, `highlights=<위 summary의 한 줄 bullet>`, `note="fallback_tx_based"`.

`_empty_but_informative_brief(now)`도 **최소한 시장 맥락**을 담는다:

- 최근 24시간 BTC/ETH 가격 변동률(Binance 공개)
- 최근 24시간 `signals` 테이블 카운트
- 문구: `"최근 1시간 대형 이동은 포착되지 않았습니다. 지난 24시간 기준 BTC {btc_chg}%, ETH {eth_chg}%. 다음 슬롯에서 재갱신됩니다."`

대시보드 측:

- `apps/dashboard/app/page.tsx`의 `buildBriefCopy`는 이제 `brief.note === "fallback_tx_based"`를 구분해 UI에 `"모델 요약 대기 중 · 온체인 기준 요약"` 라벨을 표기한다.
- 브리핑 카드 상단에 `"15분마다 갱신"` 보조 라벨을 항상 표시(§3과 연결).

### 1.4 가격 이중화 모듈 (신규)

`src/enrich/price_resolver.py` (신규)

```python
class PriceResolver:
    def __init__(self, cache: PriceCache):
        self.providers = [
            CoinGeckoProvider(),
            BinanceTickerProvider(),
            CoinMarketCapProvider(optional=True),
        ]
        self.cache = cache

    def resolve(self, symbol: str, at: datetime) -> Optional[float]:
        for p in self.providers:
            try:
                v = p.get(symbol, at)
                if v is not None:
                    self.cache.put(symbol, at, v, source=p.name)
                    return v
            except RateLimited:
                continue
            except Exception:
                log_warn(...)
                continue
        # 마지막 수단: 6시간 이내 캐시
        stale = self.cache.get_recent(symbol, max_age_min=360)
        return stale
```

`PriceCache`는 (a) in-process LRU + (b) `price_cache` 시트 백엔드로 이중 저장. 시트 백엔드는 daily 1회 snapshot으로 충분하다.

### 1.5 QA 기준

- pytest 신규: `tests/test_brief_fallback.py`
  - signals=0, tx=10건 → `fallback_tx_based` 브리핑 생성, `top_transactions`가 비어있지 않음.
  - signals=0, tx=0 → `fallback_empty` 브리핑이지만 summary는 시장 맥락을 포함.
  - signals>0 → 기존 LLM 경로가 우선.

## 2. 뉴스 RSS 스크래핑 재가동

### 2.1 근본 원인

사용자 로그: 슬롯 `15`에서 `['signals', 'curated_balance']`만 실행.

`src/pipeline/run_all.py` (v6 기준):

```python
is_due["news_rss"] = (slot in {0, 30})
```

즉, 스케줄 자체가 30분 간격이라 15·45 슬롯에서는 정상적으로 skip된 것이다. 이것은 버그가 아니라 설계다. 그러나 사용자 인상은 "실행이 안 된다"였으므로, 진짜 문제는 3가지다.

1. **30분 간격이 너무 드물다** — stale 임계(10분)에 금방 걸려 유저 홈에 경고가 뜨니 "안 도는 것처럼" 보인다.
2. **슬롯 0·30 슬롯에서도 실패하는 경우가 있는지** 확인이 안 된다. `system_log`에 `news_rss` status가 누락되면 구분이 안 간다.
3. **CoinGecko 429 로그가 news_rss 옆에 같이 뜨면서 사용자가 news_rss 실패로 오인**했다.

### 2.2 개선안

- **스케줄 조정**: `news_rss`를 매 슬롯 실행(15분)으로 변경. RSS 서버 부하는 크지 않으며, 개별 피드 fetch에 HTTP ETag/Last-Modified 지원을 넣어 trivially idempotent로 만든다.
- **stale 임계 재정의**: 유저 홈에서 뉴스 stale 경고 기준을 `> 25분` → `> 35분`으로 느슨히 한다(슬롯 실행 타이밍 지터 흡수).
- **로깅**: `news_rss` job은 성공 시에도 `items_fetched`, `items_new`, `feeds_ok`, `feeds_failed`를 `system_log.note`에 JSON으로 남긴다. `/admin` Data Source 카드가 이를 파싱해 "12개 피드 중 11개 ok, 1개 실패(reason)"까지 보여주도록 한다.
- **CoinGecko와의 격리**: `news_rss`는 CoinGecko를 호출하지 않는다. 현재 구조에서 엮여 있지 않은지 `src/pipeline/news_rss.py`를 다시 점검(엮여 있지 않으면 코드 주석으로 명시). 429 로그는 signals 경로에서만 나와야 한다.

### 2.3 구체 파일 변경

`src/pipeline/run_all.py`

```python
# before
is_due["news_rss"] = (slot in {0, 30})

# after
is_due["news_rss"] = True  # 매 슬롯
```

`src/pipeline/news_rss.py`

- `fetch_feed(feed)`에 `If-None-Match` / `If-Modified-Since` 지원.
- 결과를 `FeedFetchResult(feed_id, status, items_fetched, items_new, http_status, etag, last_modified)` dataclass로 수집.
- 상위 orchestrator에서 `{"feeds_ok": n1, "feeds_failed": n2, "items_new": sum, "reasons": {...}}`를 `system_log.note`에 기록.

`apps/dashboard/lib/metrics.ts`

- `sourceHealth.news_rss.stalenessMinutes` 임계를 상수화: `NEWS_STALE_WARN_MIN = 35`.

### 2.4 QA 기준

- `pytest tests/test_news_rss.py`(기존) + 신규 케이스: 피드 일부 실패 시 note JSON이 예상대로 직렬화되는지.
- 로컬 dev: 매 슬롯에서 news_rss가 실행되고 30분 이상 `news_feed` 테이블이 비지 않는지.
- `/admin` Data Source 카드에 "feeds_ok/feeds_failed" 뱃지가 뜨는지.

## 3. 시장 분위기 로직을 브리핑에 통합 + "15분 갱신" 표시

### 3.1 근본 원인

`apps/dashboard/app/page.tsx`의 `buildMarketMood`(line 232-276)는 순전히 **시그널 개수만 가지고 한국어 문구를 분기한다**. 즉 실제 시장 관점과 무관한 임시 로직이다. 또 브리핑과 분리되어 있어 일관성이 깨진다.

### 3.2 개선 원칙

- 시장 분위기는 **브리핑 생성 단계**에서 결정되어야 한다. LLM/fallback이 일관된 근거로 값을 생성하고, 대시보드는 그냥 렌더한다.
- 지표는 **객관적**이어야 한다. 단순 시그널 카운트는 신뢰가 낮다.

### 3.3 분위기 산정 로직 (파이프라인 쪽)

`src/pipeline/brief.py`의 출력 스키마에 `market_mood` 추가:

```python
{
  "mood": "risk_on" | "neutral" | "risk_off" | "fear" | "greed",
  "score": -100..100,      # 음수=공포, 양수=탐욕
  "drivers": [              # 대시보드에 표시할 근거 리스트
    {"label": "exchange_net_inflow", "value": "+$312M", "direction": "bearish"},
    {"label": "btc_24h_change", "value": "-1.8%", "direction": "bearish"},
    {"label": "news_sentiment", "value": "mixed", "direction": "neutral"},
  ],
  "as_of": "2026-04-18T14:00:00+09:00",
}
```

score 계산 예(단순 weighted sum, v7 기준):

- exchange net inflow (최근 6h): += 양수이면 risk_off(+=30), 음수이면 risk_on(-=30)
- BTC 24h change: +=10 * clamp(change_pct, -5, +5)
- signal alert count(`alert_level>=warn`): +=5 per, 최대 +30
- news sentiment(RSS에서 LLM 분류 평균; v7에서는 null 허용): +=15 * avg

모두 소스 누락 시 skip해 `drivers`에서 제외, `score`는 기여 항목 가중치로 정규화.

### 3.4 대시보드 렌더

`apps/dashboard/app/page.tsx`에서 `buildMarketMood` 제거, 대신 `brief.market_mood`를 바로 바인딩.

카드 상단에 뱃지 3개를 나열:

- 분위기 라벨(i18n, 컬러 토큰): `RISK_ON=emerald`, `NEUTRAL=slate`, `RISK_OFF=amber`, `FEAR=rose`, `GREED=indigo`.
- `as_of` 상대/절대 표기.
- `"15분마다 갱신"` 설명 뱃지(i18n).

드라이버는 아래에 3개까지 inline chip으로 표시.

### 3.5 "오늘의 고래 브리핑" 헤더 수정

헤더 우측에 **보조 라벨** 추가:

- ko: `"15분마다 갱신 · 마지막 갱신 {HH:mm}"`
- en: `"Refreshes every 15 min · last at {HH:mm}"`

`brief.updated_at`을 신규 필드로 추가(이미 `generated_at`이 있으면 재활용). 사용자는 "오늘"이라는 단어에 걸렸으므로, **"오늘 하루치가 아니라 15분 단위 롤링"**임을 UI로 명시한다.

### 3.6 QA 기준

- pytest: mood score가 drivers 조합에 따라 결정적으로 나오는지.
- 스냅샷: risk_on/off/neutral 각 케이스에서 대시보드가 올바른 컬러·라벨을 출력하는지(Playwright or manual screenshot).

## 4. 감지된 주요 시그널 상세 모달 + "분석 내용 보기" 하드코딩 제거

### 4.1 근본 원인

`apps/dashboard/app/page.tsx:582`:

```tsx
<span className={styles.signalCardLink}>분석 내용 보기 →</span>
```

- `<span>`이며 `onClick`도 `href`도 없다. **장식 텍스트**다.
- 시그널 라벨(`getSignalLabel`), 톤(`getSignalTone`, `getToneLabel`), 내러티브(`buildSignalNarrative`)가 모두 한국어로 하드코딩되어 있다.

### 4.2 개선 원칙

- 시그널 카드의 클릭은 **시그널 상세 모달**을 열어야 한다.
- 모달은 (a) 규칙 해설, (b) 타임윈도우, (c) 증거 트랜잭션 목록, (d) 관련 지갑, (e) 권장 읽을거리(있으면 뉴스·브리핑 링크)를 담는다.
- 내부 "권장 해석"은 AI 요약 한 문단(가능하면 파이프라인에서 사전 생성)과 휴리스틱 해설로 구성한다.

### 4.3 시그널 스키마 확장

시트 `signals`와 API `/api/signals`에 아래 필드를 추가(백엔드 소폭 변경):

- `rule_id`(기존) → `rule_label_key`(i18n key로 매핑, 예: `rules.cex_inflow_spike.label`).
- `window_start`, `window_end` (기존이면 재사용).
- `evidence_tx_hashes: string[]` — 이미 존재.
- `narrative_ai`: string (선택, LLM이 생성). 비어있으면 대시보드에서 규칙 해설 정적 텍스트를 표시.
- `related_wallets: {address, label, chain}[]` — 증거 트랜잭션에서 파생.
- `related_assets: {symbol, direction}[]`.

### 4.4 파이프라인 변경

`src/signals/rules.py` 각 rule이 `narrative_ai=None`인 채로 emit, 후처리 `src/signals/narrate.py`(신규)에서 소수 시그널(`alert_level>=warn`)에만 LLM 요약을 붙인다(비용 통제).

### 4.5 대시보드 변경

- `apps/dashboard/components/signal-detail-modal.tsx` 신규.
- `SignalCard`(또는 `signalCard` 섹션)를 클라이언트 컴포넌트화. `"분석 내용 보기"`를 `<button>`으로 바꾸고 `onClick`으로 모달 오픈.

## 5. `watched_addresses` 기반 큐레이션 감시 지갑 표출

### 5.1 근본 원인 후보

- `apps/dashboard/lib/curated-wallets.ts`의 `loadSheetBackedRegistry()`는 **`curated_wallets`, `wallet_aliases`, `watchlist_overrides`** 탭만 읽는다.
- 사용자가 언급한 **`watched_addresses`** 탭은 현재 로더가 보지 않는다. 탭 이름 불일치가 가장 유력하다.
- 대안: 로더가 예외를 전부 삼키고 seed fallback(바이낸스 14/15/16, 비트피넥스 콜드)으로 가고 있을 가능성.

### 5.2 개선안

1. **탭 이름 통일**: 운영 표준을 결정한다. 권장은 **`curated_wallets`**(현 코드 기준)로 시트 탭명을 맞추고, `watched_addresses`는 legacy로 취급.
   - 선택지 B: 코드가 두 탭명을 모두 지원(union)하도록 확장.
2. **로더 가시성 강화**: `loadSheetBackedRegistry`에서 try/catch 내부에 `log.warn(..., { tab, err })`. `/api/admin/health`의 `operatorChecks`에 "curated_wallets_loaded_ok" 항목을 추가.
3. **UI fallback 표기**: 시트 로드 실패 시 대시보드는 seed 4건을 표시하되, 카드 상단에 `"seed 큐레이션"` 뱃지를 띄워 운영자가 상황을 인지하게 한다.
4. **seed 제거 전 스위치**: `WHALESCOPE_CURATED_DISABLE_SEED=1`이면 seed를 아예 렌더하지 않음. 운영 안정되면 기본값으로 전환.

### 5.3 파일 변경

`apps/dashboard/lib/curated-wallets.ts`

```ts
// before
const CURATED_TAB = "curated_wallets";

// after
const CURATED_TABS = ["curated_wallets", "watched_addresses"] as const;

async function loadSheetBackedRegistry(): Promise<Registry> {
  for (const tab of CURATED_TABS) {
    try {
      const rows = await readSheet(tab);
      if (rows.length > 0) return buildRegistry(rows, /* sourceTab */ tab);
    } catch (err) {
      logger.warn("curated_wallets.load_failed", { tab, err });
    }
  }
  if (process.env.WHALESCOPE_CURATED_DISABLE_SEED === "1") {
    return EMPTY_REGISTRY;
  }
  return SEED_REGISTRY;
}
```

`apps/dashboard/app/page.tsx` 큐레이션 빈 상태:

```tsx
{curated.length === 0 ? (
  <EmptyState text={t.home.watchlistEmpty} />
) : ...}
```

i18n 키 `home.watchlistEmpty` 추가(§8 참조).

### 5.4 QA 기준

- 시트에 `watched_addresses`만 있는 상태에서 카드가 정상 렌더되는지.
- `curated_wallets`만 있을 때도 동일.

### 6.2 개선안

- 모든 고래 스토리 타임스탬프는 **절대시각**으로 통일: `YYYY.MM.DD HH:mm:ss`(KST).
- 상대시각 헬퍼는 제거하거나, 접근성 `aria-label`에만 사용(예: "5분 전"은 aria-only).

### 6.3 파일 변경

`apps/dashboard/lib/whale-stories.ts`

```ts
// after
const DATE_TIME_FORMATTER = new Intl.DateTimeFormat("ko-KR", {
  year: "numeric",
  month: "2-digit",
  day: "2-digit",
  hour: "2-digit",
  minute: "2-digit",
  second: "2-digit",
  hour12: false,
  timeZone: "Asia/Seoul",
});

export function formatStoryTimestamp(iso: string): string {
  const d = new Date(iso);
  const parts = DATE_TIME_FORMATTER.formatToParts(d);
  const get = (t: string) => parts.find(p => p.type === t)?.value ?? "";
  return `${get("year")}.${get("month")}.${get("day")} ${get("hour")}:${get("minute")}:${get("second")}`;
}
```

사용처:

- `apps/dashboard/app/page.tsx`의 고래 스토리 카드 하단 — `formatRelativeGeneratedAt` 호출을 `formatStoryTimestamp`로 교체.
- `en` 모드에서도 동일 포맷(연도 표기는 숫자이므로 언어 독립). prefix만 i18n(`"생성"` → `"Generated"`).

## 7. 루트 `.env` ↔ `apps/dashboard` 통합

### 7.1 근본 원인

Next.js는 빌드/서버 런타임에서 `process.env.NEXT_PUBLIC_*`를 읽되, 그 출처는:

- Vercel 환경변수(배포 환경)
- 빌드 시점의 `apps/dashboard/.env`, `.env.local`, `.env.production`
- `next.config.*`에서 명시적으로 주입한 값

루트 `/02015_reuton_whale/.env`는 **Python 파이프라인용**이며, Next.js는 기본적으로 이를 읽지 않는다. 따라서 `NEXT_PUBLIC_TELEGRAM_*`이 잡히지 않는 건 자연스러운 결과다.

### 7.2 선택지와 트레이드오프

옵션 A — **앱별 `.env` 분리**
- 장점: Next.js 표준, 이해 쉬움, Vercel 배포와 1:1 매칭.
- 단점: 값 중복 관리, 휴먼 에러.

옵션 B — **루트 `.env`를 Next.js 설정에서 명시 로드**
- `apps/dashboard/next.config.ts`에서 `dotenv.config({ path: "../../.env" })` 수동 호출.
- 장점: 값 단일화.
- 단점: Vercel 배포에서는 여전히 Vercel env를 써야 하므로 "로컬 편의"에만 해당. 이중 관리 위험.

옵션 C — **turborepo/pnpm workspace 레벨 `.env` 규약 + symlink**
- `apps/dashboard/.env.local → ../../.env` symlink.
- 장점: 값 단일화, Next.js 원칙도 안 깨짐(로컬만).
- 단점: 윈도우 유저 이슈, git에서 symlink 관리.

옵션 D — **모노레포 공용 env 로더 스크립트**
- `scripts/sync-env.mjs` 실행 시 루트 `.env`를 필터해서 `apps/*/.env.local` 생성.
- `pnpm dev` pre-script로 엮기.
- 장점: 명시적, 배포와 로컬 모두 정합성 확보.
- 단점: 스크립트 유지보수.

**권장: 옵션 D + Vercel 환경변수 동기화**.
- 로컬: 루트 `.env` 단일 소스 → `scripts/sync-env.mjs`가 필요한 키만 `apps/dashboard/.env.local`에 내려씀.
- 배포: Vercel 프로젝트 환경변수에 동일 키 세팅(이 부분은 사람이 한 번만 설정).
- CI: `pnpm env:check`가 필수 키 누락을 검출.

### 7.3 구체 변경

`scripts/sync-env.mjs` (신규)


### 7.4 Vercel 배포 주의

- `NEXT_PUBLIC_*`는 **빌드 시점**에 인라인된다. Vercel UI에서 env를 바꿨다면 **재배포** 필수.
- 배포 환경에서 루트 `.env`는 참고되지 않는다. Vercel 환경변수가 진실이다.

### 7.5 QA 기준

- 배포 환경에서 Vercel env에 동일 키 세팅 확인 체크리스트 추가.

## 8. i18n 전면 확장 — "영어로 바꿨는데 한국어가 남는다" 해결

### 8.1 근본 원인

현재 사전(`ko.ts`, `en.ts`)이 커버하는 범위는 `navbar`, `sidebar`, `home`(핵심 헤딩만), `languageSelector`, `telegram`에 한정된다.

반면 `page.tsx` 내부에는 **본문/카드/헬퍼에서 한국어가 하드코딩**된 지점이 다수다:

- `getSignalLabel(rule_id)` — 룰별 한국어 라벨 매핑.
- `getSignalTone`, `getToneLabel` — 톤 라벨.
- `getCuratedCategoryLabel` — 카테고리 라벨.
- `humanizeChain` — 체인명 정규화.
- `buildBriefCopy` — 브리핑 fallback 한국어.
- `buildMarketMood` — 분위기 문구(§3에서 파이프라인으로 이관 예정).
- `buildSignalNarrative` — 시그널 내러티브.
- 빈 상태 메시지(시그널/감시지갑).

- 카드 본문 텍스트(각종 서브타이틀, 안내 문구).

### 8.2 개선 원칙

- **모든 유저 대면 문자열**은 사전 키로 접근한다. 하드코딩 한국어가 코드에 남아있지 않도록 lint 룰을 건다.

- `en` 디폴트 품질은 한국어와 동등해야 한다. 기계적 재번역이 아니라, 문장 구조/관용표현까지 검수.

- 키보드 내비, 스크린리더 라벨이 해당 언어로 정확히 읽힘.

## 12. 요약

v7은 **"시트에 데이터는 있는데 대시보드가 의미를 못 만든다"**는 문제에 정면으로 대응한다. 핵심은 네 가지다.

첫째, 브리핑이 침묵하지 않는다 — 시그널이 0이어도 트랜잭션·가격 이중화·시장 맥락으로 항상 의미있는 한 문단을 만든다. 둘째, 뉴스·시장 분위기·감시 지갑처럼 **이미 들어와 있는 신호원**을 UI가 제대로 소비하게 한다. 셋째, 시그널 카드의 하드코딩된 "분석 내용 보기"를 실제 모달 흐름으로 올리고, 규칙 해설을 시스템화한다. 넷째, i18n을 chrome 수준에서 본문 수준까지 끌어올리고, 하드코딩 한국어를 lint로 막는다.

부차적으로 루트 env 통합(§7)과 고래 스토리 타임스탬프 포맷(§6)은 작업량 대비 UX 체감이 크므로 우선 처리한다. v7가 끝나면 WhaleScope는 "운영 안정화 + 상태 가시화"에서 "데이터 해석력과 사용성이 제품으로 선 단계"로 올라선다.
