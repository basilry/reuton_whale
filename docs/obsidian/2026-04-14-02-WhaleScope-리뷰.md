## 1. 다관점 리뷰

### 1-1. PM / 투자자 관점

**강점**
- 타겟 정의가 구체적("온체인 분석 못 하는 2030 한국 직장인 투자자")이라 제품 의사결정 기준이 명확함.
- 기존 서비스 비교표(Whale Alert / Arkham / Nansen / 코인업)의 갭 분석이 설득력 있음. "해석의 부재"라는 단일 wedge가 일관되게 제품 결정에 반영됨.
- 북극성 지표(일일 브리핑 열람률 60%)가 측정 가능하고 도전적. D7 Retention, 공유율 등 보조 지표 구성도 합리적.
- 뤼튼 미션("Bring AGI Close to People") 정합성 내러티브가 과제 심사에 유리.

**약점 / 리스크**
- **legal risk 미언급**: "투자 조언 아님" 면책은 OnePager 말미 한 줄. 실제 배포 시 금융당국 해석에 따라 자문업 이슈 소지. 앱/봇 UI에도 고정 표시 필요.
- **Cold start 문제 방치**: 구독자 0명 상태에서 D7 Retention을 어떻게 측정? 런칭 시 초기 시드 구독자 확보 전략이 기획서에 없음.
- **실패 기준 시그널이 늦다**: "2주 후 열람률 30% 미만"은 이미 자원을 다 쓴 뒤. Day 3 정성 인터뷰 같은 선행 지표를 병행해야 함.
- **경쟁 해자 부재**: Claude API + Whale Alert API + Telegram은 누구나 1주일 내 복제 가능. 해자는 "한국어 큐레이션 품질" 하나인데 이는 프롬프트 튜닝에 가깝고 방어가 어렵다.
- **수익 모델 공란**: 과제 범위 밖일 수 있으나 발표에서 질문 나오면 리스크. 뤼튼 플랫폼 연동이라는 단일 카드만으로는 약함.

**투자자 질문 예상**
1. Nansen이 내일 한국어 지원하면? → 답변 준비 필요
2. 브리핑 열람률 60% 근거는? → 실측 데이터 없음, 벤치마크 추정치임을 명시
3. Whale Alert API가 가격을 대폭 인상하거나 끊기면 파이프라인 전체가 멈추는데 대체 소스는? → 문서에 언급됐지만 구현체 없음

### 1-2. 엔지니어링 관점

**강점**
- 계층 분리(collectors / analyzer / storage / distributor / utils) 깔끔함. 새 데이터 소스 추가가 collector만 확장하면 되는 구조.
- 10단계 파이프라인 각 스테이지가 독립 try/except로 감싸져 있어 한 단계 실패가 전체를 중단시키지 않음. `status=completed_with_errors`로 부분 성공을 기록.
- Retry 데코레이터가 지수 백오프 + jitter까지 포함. 외부 API 불안정성 대응 적절.
- `raw_response_hash` 기반 중복 제거로 재실행 안전성 확보.
- 테스트 78개 전부 외부 의존 mock 처리 → CI에서 안정적으로 돌릴 수 있음.

**약점 / 리스크**
- **스키마 불일치 (블로커)**: `WhaleScopeBot`이 `SheetsClient`의 6개 메서드(`get_active_subscribers`, `add_subscriber`, `get_watchlist`, `set_watchlist`, `set_status`, `get_subscriber_info`)를 호출하지만 `SheetsClient`에 존재하지 않음. 테스트는 `MagicMock`으로 Sheets 전체를 대체해서 통과. **프로덕션에서는 즉시 AttributeError**.
- **이중 스코어링 혼란**: `calculate_base_score`(0~10 사이 float, 규칙 기반)와 `importance_score`(Claude가 주는 1~10 int)가 공존. `rank_by_importance`는 후자만 사용 → 전자를 계산하는 코스트가 이후 파이프라인에서 소비되지 않음.
- **분석 캐시 미연결**: `ClaudeAnalyzer.analysis_log`는 프로세스 메모리 dict. 매 실행마다 리셋. `SheetsClient.get_cached_analysis` / `save_analysis`가 있지만 analyzer에서 호출 안 함. 중복 API 비용 발생.
- **Telegram Bot 이벤트 루프 미기동**: `bot.build()`만 호출하고 `run_polling()`은 어디에도 없음. `/start` `/watchlist` 등 유저 명령은 절대 응답 안 됨. 구독자 등록 자체가 불가능.
- **`_filter_brief` 더미**: watchlist 필터링 로직이 `return brief_text` 한 줄로 구현됨 (no-op). 기능 1 "내 관심 코인 맞춤 알림"이 동작 안 함.
- **Streamlit `top_transactions` 파싱 불일치**: `main.py`는 `[hash1, hash2, ...]` 문자열 리스트를 저장하는데, `streamlit_app.py`는 `list[dict]`(symbol/amount_usd/score 포함)로 읽으려 시도. 실측 시 카드 본문이 비어 나옴.
- **`format_daily_brief`의 importance 렌더링**: `f"{importance:.0%}"`는 0~1 범위를 %로 표기. 하지만 값은 1~10 스케일 → 화면에 "700%" 같은 출력.
- **race condition**: `append_transactions`는 read-then-write로 dedup. 동일 크론이 두 번 병렬 실행되면 중복 삽입 가능. GitHub Actions `concurrency` 그룹 지정 필요.
- **Streamlit 캐시 TTL 300초 고정**: 브리핑 직후 접속한 구독자는 이전 데이터를 봄.

**성능 / 비용**
- Claude 분석 30건/일 × batch = 30 API calls. 적절한 규모이나 캐시 미동작으로 같은 거래를 재분석할 가능성 있음.
- CoinGecko freetier는 30 calls/min. 현재 batch 50 단위 안전. 단 재시도 5회 백오프가 쌓이면 장시간 lock.
- Sheets API quota: 쓰기 100/100초/user. `log_run` + `append_transactions` + `save_daily_brief` = 3회 미만으로 여유.

### 1-3. 마케팅 / 사용자 관점

**강점**
- 봇 온보딩 메시지(`format_welcome_message`)가 명확. 4개 명령어 즉시 노출.
- 카드 포맷이 시각적(별표, 중요도 바, USD 포맷 함수).
- 한국어 브리핑이 "2-3문장 요약 → TOP 3 → 시사점" 구조라 모바일 읽기 적합.

**약점 / 리스크**
- **봇 웰컴 메시지가 영어**: 타겟이 한국인이라면 welcome, watchlist 확인, pause 메시지 전부 한국어여야 함. 현재 카드 내부(AI 해설)만 한국어, 포맷은 영어 혼재.
- **CTA 부재**: "3일간 무료로 써보세요" 같은 습관 형성 장치 없음. `/start` 후 "내일 아침 첫 브리핑이 도착합니다" 같은 기대 setting 필요.
- **알림 피로도 보호장치 약함**: `/pause` 명령은 있지만 스누즈(3일 후 재개 등) 없음. 피로도 누적 시 블록 → 영구 이탈.
- **공유 기능 없음**: 보조 지표 "공유율 5%"에 반해 브리핑 메시지에 "이 브리핑이 도움 됐다면 친구에게 공유하세요" 링크/버튼 없음. Telegram 자체 forward에만 의존.
- **브랜드 일관성**: `🐋` 이모지가 Streamlit 제목에는 있지만 Telegram 카드에는 없음. 봇 프로필 사진도 미지정 가능성.
- **가치 증명이 뒤늦게 옴**: `/start` → 다음날 08:00 브리핑까지 최대 23시간 공백. 즉시 최근 1건 샘플 브리핑을 보내주는 "onboarding brief" 필요.

## 2. 코드 리뷰 (세부 이슈)

### 🔴 Critical — 발표/데모 전 반드시 수정

#### C1. `WhaleScopeBot` ↔ `SheetsClient` 인터페이스 불일치

**파일**: `src/distributor/telegram_bot.py` + `src/storage/sheets_client.py`

```python
# telegram_bot.py line 44, 89, 105, 114, 123, 130 — 존재하지 않는 메서드 호출
subscribers = self._sheets.get_active_subscribers()      # ❌
self._sheets.add_subscriber(chat_id=..., username=...)   # ❌
self._sheets.get_watchlist(chat_id=...)                  # ❌
self._sheets.set_watchlist(chat_id=..., coins=...)       # ❌
self._sheets.set_status(chat_id=..., status="paused")    # ❌
self._sheets.get_subscriber_info(chat_id=...)            # ❌
```

`SheetsClient`에는 `get_active_watchlists`, `upsert_watchlist`만 있음. `watchlist` 탭에 `status`, `chat_id` 컬럼도 없음(현재 스키마: user_id, username, coins, active, updated_at).

**영향**: Telegram 발송 시점(파이프라인 10단계)에서 AttributeError → 브리핑 미전송. 발표 데모 실패 확정.

**수정**:
- `subscribers` 개념 도입: `watchlist` 테이블 단일화 or 별도 `subscribers` 탭 추가. 스키마: `chat_id, username, status, watchlist_coins, created_at, updated_at, last_brief_at`.
- `SheetsClient`에 필요한 6개 메서드 구현.
- 테스트를 `MagicMock()`이 아닌 인터페이스 검증 가능한 spec mock으로 변경(`MagicMock(spec=SheetsClient)`).

#### C2. Telegram polling 미기동 → 구독/명령어 영원히 응답 안 함

**파일**: `src/main.py` line 47, `src/distributor/telegram_bot.py`

```python
bot = WhaleScopeBot(config.telegram_token, sheets)
bot.build()   # Application만 생성하고 polling 시작 안 함
```

`run_polling()` 혹은 `run_webhook()` 없이는 handler가 작동하지 않음. 크론 방식에서는 별도 프로세스가 폴링을 유지해야 함.

**수정**:
- **단기**: `scripts/run_bot.py` 별도 실행 스크립트 추가, 백그라운드 상시 폴링.
- **권장**: Telegram Webhook + Cloudflare Worker/Vercel Edge Function 같은 서버리스로 전환. 구독자 온보딩은 웹훅으로, 일일 브리핑 발송은 크론으로 분리.

#### C3. Streamlit 대시보드의 `top_transactions` 파싱 불일치

**파일**: `src/main.py` line 137, `streamlit_app.py` line 134~146

```python
# main.py는 hash만 저장
"top_transactions": json.dumps([tx.get("hash", "") for tx in top5], ...)

# streamlit_app.py는 dict 리스트 기대
for tx in top_txs:
    if isinstance(tx, dict):
        symbol = tx.get("symbol", "?")  # 절대 안 타는 분기
```

**영향**: "오늘의 브리핑" 탭에서 TOP 5 카드가 전부 비어 나옴. 데모 화면 설득력 상실.

**수정**: `main.py`에서 `top5` 전체 객체(symbol, amount_usd, importance_score, interpretation)를 json dump로 저장. Storage 사이즈 증가 무시할 수준.

#### C4. `format_daily_brief`의 importance 출력 버그

**파일**: `src/distributor/formatters.py` line 46

```python
f"Importance: [{_importance_bar(importance)}] {importance:.0%}\n"
# importance는 1~10 int인데 .0% 포맷 → "700%"
```

**수정**: `_importance_bar`에 `score/10.0` 정규화 반영 + `f"{importance}/10"` 표기. 또는 컨벤션을 0~1로 통일.

#### C5. `_filter_brief` 더미 구현 (기능 1 누락)

**파일**: `src/distributor/telegram_bot.py` line 79

```python
def _filter_brief(self, brief_text: str, watchlist: list[str]) -> str:
    return brief_text  # 관심 코인 필터링 미구현
```

**수정**: 브리핑 원본이 모든 코인 대상이라면 필터링은 "관심 코인 거래 강조(별표)"에 가까움. `top5` 객체 단계에서 is_watched 주입 → 포맷터가 사용하도록 파이프라인 변경.

#### C6. 분석 캐시 미연결로 중복 Claude 호출

**파일**: `src/analyzer/claude_analyzer.py`

`self.analysis_log`는 인스턴스 메모리. 매 파이프라인 실행마다 초기화. `SheetsClient.get_cached_analysis` / `save_analysis`는 구현만 있고 호출자가 없음.

**수정**: `ClaudeAnalyzer.__init__`에 `sheets: SheetsClient | None = None` 주입. `analyze_transaction` 초기에 `sheets.get_cached_analysis(prompt_hash)` → 있으면 반환, 없으면 API 호출 후 `save_analysis` 저장.

#### C7. `analyze_batch` 실패 시 `importance_score` 유실

**파일**: `src/main.py` line 100-103

```python
except Exception as e:
    errors.append(f"analyze_batch: {e}")
    analyzed = filtered  # ← importance_score 없는 원본 리스트
```

이후 `rank_by_importance`가 전부 `.get("importance_score", 0)` → 모두 0 → 무의미한 TOP 5.

**수정**: 폴백 시 `base_score`를 `importance_score`로 alias 하거나, 규칙 기반 TOP5 선별 함수 별도 구현.

### 🟡 High — 발표 후 1주일 내 수정

#### H1. API 키 URL query string 전송 (Whale Alert)

`whale_alert.py` line 36-44. `api_key`가 GET 파라미터. 프록시/로그에 기록될 위험.

**수정**: Whale Alert가 `X-WA-API-KEY` 헤더를 지원하면 이전. 미지원 시 최소 HTTPS(이미 됨) + 로그에 URL 마스킹.

#### H2. Streamlit 대시보드 무인증 공개

`streamlit_app.py`. 누구나 URL 알면 구독자 및 거래 데이터 열람. `streamlit-authenticator` 또는 Cloudflare Access 전치 필요.

#### H3. GitHub Actions concurrency 미지정

동시 실행 시 중복 처리 가능. `.github/workflows/daily_brief.yml`에 `concurrency: group: daily-brief, cancel-in-progress: true` 추가.

#### H4. CoinGecko 무료 플랜의 symbol 매핑 하드코딩

`coingecko.py` `SYMBOL_TO_ID`가 15개 고정. 16번째 고래 코인 등장 시 시장 데이터 비보강. 동적 조회 또는 CoinGecko `/coins/list` 캐시 필요.

#### H5. `claude_analyzer._parse_json_response` 파싱 fragile

`split("\n", 1)[1].rsplit("```", 1)[0]` — fence가 없으면 IndexError. fence가 `\`\`\`json{...}\`\`\``처럼 줄바꿈 없이 올 때 실패.

**수정**: 정규식 또는 `json.loads` 선시도 → 실패 시 fence 제거 → 재시도의 2단 구조. 또는 Claude API의 response JSON mode 활용.

#### H6. Welcome / 확인 메시지 언어 혼재

타겟이 한국 투자자인데 `format_welcome_message`, pause 응답이 영어. 번역 필요.

### 🟢 Medium — 후순위

#### M1. `_cell_range = f"A{row}:{chr(64 + len(WATCHLIST_HEADERS))}{row}"` — `chr(64+26+)` 시 AA~ 컬럼 계산 깨짐. 현재 5 컬럼이라 문제 없지만 범용 함수로 분리 권장.

#### M2. 프롬프트가 `src/analyzer/prompts.py`에 있지만 `generate_daily_brief`가 시스템 프롬프트 한 줄을 코드에 하드코딩("You are a professional crypto market analyst..."). prompts.py로 이전해 일원화.

#### M3. 두 스코어링 체계(base_score rules vs Claude importance_score) 중 하나는 제거하거나, "base는 pre-filter, importance는 rank" 역할을 주석으로 명시.

#### M4. 비용 추적 미구현. `ClaudeAnalyzer._call_api`가 token 로그만 남김. 누적 집계 + 예산 알림 (예: Sheets `system_log`의 새 컬럼 `tokens_used`, `cost_usd`) 필요.

#### M5. `CoinGeckoEnricher._fetch_prices`가 `raise Exception(...)` → 커스텀 예외로 교체(`CoinGeckoError`).

#### M6. 로그 포맷이 `%d` / `%s` 스타일로 일관되지만 `structured log`(JSON) 고려. Cloud 환경에서 파싱 편의.

#### M7. `src/distributor/formatters.py`의 `_filter_brief` 미구현과 별개로, `format_daily_brief`는 `briefs: list[dict]`를 받는데 `send_daily_brief`는 `brief_text: str`을 받아 그대로 전송 → 포맷터 사용 경로가 사실상 죽어 있음.
