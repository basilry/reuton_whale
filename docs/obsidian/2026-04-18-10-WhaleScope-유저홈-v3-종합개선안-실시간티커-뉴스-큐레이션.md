## 0. Executive Summary

**핵심 결론**: v2 보고서가 "반영"으로 분류한 §2 체크리스트는 코드와 정합하지만, **Finding A/B/C(Major/Minor)와 §7 P2~P5는 그대로 잔존**한다. 배포 전 반드시 처리해야 할 Major 블로커는 두 건(인증 fail-open · 큐레이션 in-memory 영속화)이며, 이 두 건은 Vercel 프로덕션 배포 시 각각 "운영 API 전면 개방"과 "관리자 토글 유실"이라는 실제 사고 모드를 갖는다.

**본 문서가 추가로 설계하는 것**은 총 9개 섹션이다.

1. 잔존 Findings의 구체 patch 설계 (섹션 1)
2. 현재 크론/스케줄러 상태 브리핑 + 개선 (섹션 2)

## 1. 잔존 Findings 종합 + 구체 patch 설계

v2 보고서가 "잔존"으로 분류한 것을 실제 patch 수준까지 설계한다. 각 항목은 Before(현재) · After(기대) · Why(근거) · Patch Sketch(코드 스케치) 4단으로 기술한다.

### 1.1 Finding A — 대시보드 인증 fail-open (Major)

| 축 | 내용 |
|---|---|
| 파일 | `apps/dashboard/lib/auth.ts:47-73` |
| 현재 | `DASHBOARD_PASSWORD` 미설정이면 `authorized: true` 반환. 모든 운영 API 개방. |
| 기대 | 프로덕션(`NODE_ENV=production`)에서 password 미설정이면 **명시적 401 + 로그**. 타이밍-safe 비교로 시간차 공격 차단. |
| Why | Vercel에 env를 빠뜨리고 배포하면 `/api/dashboard`, `/api/transactions`, `/api/signals`, `/api/system-log`가 완전 개방. Google Sheets에 쌓인 거래/시그널/운영 로그가 공개 노출되는 사고 모드. |

**Before / After / Why**

| Before | After | Why |
|---|---|---|
| `if (!expectedPassword) return { authorized: true, passwordConfigured: false }` | `if (!expectedPassword) { if (process.env.NODE_ENV === "production") return { authorized: false, passwordConfigured: false, reason: "missing-production-password" }; return { authorized: true, passwordConfigured: false }; }` | 프로덕션에서만 fail-closed. 개발 편의는 유지. |
| `expectedPassword === provided` (느슨 비교 없음, 그러나 타이밍 불안전) | `crypto.timingSafeEqual(Buffer.from(expected), Buffer.from(provided))` + 길이 사전 검사 | 타이밍 사이드채널 차단. |
| `requireDashboardAuth` 실패 시 내용 미로깅 | `system_log`에 `run_type=auth`, `level=warn`, `payload={reason, ip, ua, route}` | 공격/오사용 포렌식 근거. |

**Patch Sketch**

```ts
// apps/dashboard/lib/auth.ts (개정)
import { timingSafeEqual } from "node:crypto";

export function getDashboardAuthResult(request: Request): AuthResult {
  const expected = getDashboardPassword();
  const provided = getProvidedPassword(request);

  if (!expected) {
    if (process.env.NODE_ENV === "production") {
      return { authorized: false, passwordConfigured: false, reason: "missing-production-password" };
    }
    return { authorized: true, passwordConfigured: false };
  }
  if (!provided) {
    return { authorized: false, passwordConfigured: true, reason: "missing-credential" };
  }
  const a = Buffer.from(expected, "utf8");
  const b = Buffer.from(provided, "utf8");
  if (a.length !== b.length) {
    return { authorized: false, passwordConfigured: true, reason: "length-mismatch" };
  }
  return {
    authorized: timingSafeEqual(a, b),
    passwordConfigured: true,
    reason: "comparison",
  };
}
```

**수용 기준(AC)**
- `NODE_ENV=production`이면서 `DASHBOARD_PASSWORD` 미설정 시 `GET /api/dashboard` → `401 {"error":"missing-production-password"}`.
- 로컬 개발(미설정)에서는 기존 동작 유지.
- 실패 이벤트가 `system_log`에 `run_type=auth` 카테고리로 기록.

### 1.2 Finding B — 큐레이션 지갑 in-memory 영속화 (Major)

| 축 | 내용 |
|---|---|
| 파일 | `apps/dashboard/lib/curated-wallets.ts:47-227` |
| 현재 | 시드 4개 하드코딩 + `Map<string, boolean>`로 admin 토글 메모리 보관. |
| 기대 | Google Sheets `curated_wallets`, `wallet_aliases`, `watchlist_overrides` 탭으로 영속화. Vercel 함수 인스턴스 간 상태 일관. |

**스키마 (신규 Sheets 탭 3종)**

`curated_wallets` — 정적 시드 + Top 10 Liquid Coins 수입
```
id | chain | address | owner_label | category | source | ref_url | approx_balance | tier | created_at | updated_at
```

`wallet_aliases` — 같은 실체가 여러 주소일 때의 alias 매핑 (예: Binance 14/15/16)
```
alias_id | canonical_id | chain | address | label | note | created_at
```

`watchlist_overrides` — admin UI 토글 이벤트 영속화
```
wallet_id | enabled | actor | reason | updated_at
```

**Before / After / Why**

| Before | After | Why |
|---|---|---|
| `curatedWalletSeed: CuratedWalletEntry[] = [...]` 4개 하드코딩 | `sheets.list_curated_wallets()` + fallback으로 seed | 신규 지갑 추가 시 코드 배포 불필요. Top 10 Liquid Coins 파일 기준으로 seed 일괄 수입 가능. |
| `watchlistOverrides = new Map()` | `sheets.upsert_watchlist_override(wallet_id, enabled, actor)` | Vercel 함수 인스턴스 분리/재시작에도 토글 유실 없음. |
| admin UI가 즉시 반영되어 관리자 사이에서 일관 안 됨 | Sheets write → 다음 ISR/fetch 주기에 반영. UI에는 optimistic update. | 강한 일관성이 아닌 "eventual + optimistic" 모델로 단순화. |

**수용 기준**
- admin에서 토글 후 Vercel 콜드 부트 재기동 시에도 토글 유지.
- 동일 실체의 여러 주소(예: Binance 14 + 15 + 16)가 한 엔티티로 표시되면서 주소 단위 on/off는 개별 설정 가능.

```ts
// apps/dashboard/lib/public-app-config.ts
import QRCode from "qrcode";

export async function getTelegramQrSvg(botUrl: string | null): Promise<string | null> {
  if (!botUrl) return null;
  return QRCode.toString(botUrl, { type: "svg", margin: 0, width: 240 });
```

  - 2차: `https://open.er-api.com/v6/latest/USD` (무료)
  - 3차 폴백: Upbit USDT-KRW 페어를 USD/KRW의 근사값으로 사용 (USDT≈USD 가정)
- 갱신 주기: 5분마다 REST. WS 불필요.
- 캐시: Next.js `unstable_cache` + `revalidate: 300`.

### 3.4 김프 계산식

```ts
// apps/dashboard/lib/market-premium.ts (신규)
export function calcKimchiPremium(params: {
  usdPrice: number;       // Binance BTCUSDT last
  krwPrice: number;       // Upbit KRW-BTC trade_price
  usdKrwFx: number;       // exchangerate.host USD→KRW
}): { premiumPct: number; impliedFairKrw: number } {
  const { usdPrice, krwPrice, usdKrwFx } = params;
  const impliedFairKrw = usdPrice * usdKrwFx;
  const premiumPct = ((krwPrice / impliedFairKrw) - 1) * 100;
  return { premiumPct, impliedFairKrw };
}
```

`JPY premium`, `EUR premium`도 동일 공식. 표기: `+3.21%` (양수 초록/빨강은 컬러블라인드 고려해 숫자 대비 + 아이콘 병기).

### 3.5 차트 라이브러리 선택

| 옵션 | 장점 | 단점 | 결정 |
|---|---|---|---|
| `tradingview/lightweight-charts` (Apache 2.0, 45KB) | 캔버스, 60 FPS, WS 스트리밍 적합, 커스터마이징 자유 | TradingView 로고 attribution 의무 | ✅ **채택** |
| TradingView `Charting Library` (무료 배포판) | 풀 기능 기술분석 | 번들 큼(2MB+), 라이선스 복잡, 데이터피드 구현 필요 | ❌ 과잉 |

**결정**: `lightweight-charts` 단일 채택. 카드 내 미니차트와 펼침 상세 차트 모두 동일 라이브러리로 통일. 런타임 1개 = 번들 크기 최적화.

**구현 스케치**

```ts
// apps/dashboard/components/market-mini-chart.tsx
"use client";
import { createChart, ColorType, type LineData } from "lightweight-charts";
import { useEffect, useRef } from "react";

export function MarketMiniChart({ data }: { data: LineData[] }) {
  const ref = useRef<HTMLDivElement>(null);
  useEffect(() => {
    if (!ref.current) return;
    const chart = createChart(ref.current, {
      width: ref.current.clientWidth,
      height: 40,
      layout: { background: { type: ColorType.Solid, color: "transparent" }, textColor: "transparent" },
      grid: { vertLines: { visible: false }, horzLines: { visible: false } },
      rightPriceScale: { visible: false },
      timeScale: { visible: false },
      crosshair: { vertLine: { visible: false }, horzLine: { visible: false } },
      handleScroll: false,
      handleScale: false,
    });
    const series = chart.addLineSeries({ lineWidth: 2 });
    series.setData(data);
    return () => chart.remove();
  }, [data]);
  return <div ref={ref} aria-hidden className="size-full" />;
}
```

**접근성**: 미니차트는 `aria-hidden`. 실제 수치는 옆 텍스트에 표시. 확장 차트는 `role="img" aria-label="BTC 1분봉 최근 60분 차트, 현재 ..."`와 키보드 포커스링 제공.

### 4.4 집계 전략

| 전략 | 장점 | 단점 | 결정 |
|---|---|---|---|
| 클라이언트에서 4개 거래소 WS 병렬 연결 | 지연 최저 | 브라우저당 4 WS = 모바일 배터리 | ❌ |

### 5.1 investing.com 직접 스크레이핑 판정

- 대체: investing.com RSS(`https://www.investing.com/rss/news_301.rss` 암호화폐 섹션)는 공개 접근 가능. 다만 기사 본문은 링크만 제공.

### 5.5 필터링/큐레이션 로직

1. **소스 신뢰도 가중치** — 1차 편집 매체(CoinDesk, Cointelegraph, The Block) × 1.0, 집계기(CryptoPanic) × 0.7.
2. **키워드 매칭** — 현재 감시 지갑 소유주(Binance, Coinbase, Tether) 또는 티커(BTC/ETH/SOL...) 포함 시 × 1.3.
3. **시간 감쇠** — `weight × exp(-age_hours / 12)`.
4. **중복 제거** — 제목 유사도(`Levenshtein`)가 0.85 이상이면 하나로 병합하고 소스 리스트 병기.
5. **금지어 필터** — "Presale", "Pump", "100x Gem" 등 광고성 키워드 기사 제외.
6. **언어 우선** — 사용자 언어 쿠키(`ko`/`en`/`ja`)에 따라 가중 재정렬.

### 5.6 반응형

- 데스크탑: 사이드바 하단 고정 위젯.
- 태블릿(< 1024px): 사이드바가 햄버거로 접힘 → 위젯도 같이 접힘. `/news` 전용 페이지로 대체.
- 모바일(< 768px): 홈 하단에 별도 섹션으로 이동. 썸존 고려.

### 6.4 큐레이션 점수 (wallet_score)

홈의 `큐레이션 감시 지갑` 목록에 어떤 것을 위로 올릴지 정한다.

```
wallet_score =
    0.35 × normalized_balance_rank         # 큰 지갑이 위로
  + 0.25 × activity_intensity_24h          # inflow+outflow 큰 지갑이 위로
  + 0.15 × owner_visibility                # exchange=1.0, institution=0.9, founder=0.8, ...
  + 0.15 × recency_boost                   # 최근 24h 활동이 있으면 +
  + 0.10 × alignment_with_user_interest    # user_interests 탭의 태그와 교집합
```

`normalized_balance_rank = 1 - (balance_rank - 1) / total_wallets`

## 10. Telegram Broadcast 채널 (`@whalescope_alertz`) 설계

### 10.1 채널 정체성 및 역할

`@whalescope_alertz`는 **WhaleScope가 운영하는 공개 Broadcast 채널**이다. 기존 Telegram 자원들과 역할이 다음과 같이 구분된다.

| 대상 | 종류 | 방향 | 목적 |
|---|---|---|---|
| `@whalescope_demo_bot` | Bot (기존) | 1:1 양방향 | `/start` · `/watchlist` · `/pause` · `/status` 개인 명령 처리, 사용자별 DM 알림 |
| `@whale_alert_io` | 외부 채널 (기존) | 수신 전용 (listener) | Telethon MTProto로 대형 이체 메시지 **ingest**. WhaleScope 입력 소스 |
| **`@whalescope_alertz`** | Broadcast 채널 (신규) | **1:다 송신 전용** | WhaleScope 운영자가 `daily_brief`, `weekly_trend`, `urgent signal`을 구독자 전체에 공개 배포 |

요점: `@whalescope_alertz`는 **입력 소스가 아니라 출력(배포) 대상**이다. 구독자는 읽기 권한만 가지며, WhaleScope 봇이 admin으로 포스트를 자동 게시한다. 1:1 봇이 "개인 맞춤 DM"이라면, 이 채널은 "공개 브리핑 신문"에 해당한다.

### 10.2 3자 역할 분리 아키텍처

```
외부 소스                       WhaleScope Core                        사용자 접점
─────────────                  ───────────────────────                ─────────────
@whale_alert_io    ───Telethon listener──▶   main.py pipeline   ───Bot API────▶   @whalescope_demo_bot (1:1 DM)
(ingest 채널)                  │                                   │
                               ├─ SignalEngine                      └──Bot API─▶   @whalescope_alertz  (공개 1:다)
                               ├─ LLMRouter
                               └─ Sheets 적재
```

- **ingest**(`@whale_alert_io`)는 계속 입력 소스로만 유지. 발행 채널과 결합되지 않는다.
- **1:1 DM**(`@whalescope_demo_bot`)은 사용자별 `watchlist` 기반 맞춤 알림. 개인화·rate limit 보호가 중요.
- **1:다 Broadcast**(`@whalescope_alertz`)는 **동일 메시지를 전 구독자에게 한 번만 게시**. Bot API의 `sendMessage(chat_id=@whalescope_alertz, text=...)`로 단일 호출.

### 10.3 발행 컨텐츠 타입

| 타입 | 트리거 | 주기 | 채널 발행 포맷 |
|---|---|---|---|
| `daily_brief` | `daily_brief.yml` cron (UTC 23:00 = KST 08:00) | 일 1회 | TOP 3 고래 움직임 + LLM 요약 + 대시보드 링크 |
| `weekly_trend` | `weekly_trend.yml` cron (월요일 UTC 23:00) | 주 1회 | 주간 흐름 + 김프 주간 고점/저점 + 주요 주소별 누적 delta |
| `urgent_signal` | `SignalEngine`에서 `severity=major` 시그널 발생 시 | 이벤트 기반 (throttle 필요) | 단일 시그널 요약 + 지갑 라벨 + 근거 트랜잭션 링크 |
| `news_curated_daily` | (옵션, W4 이후) RSS 큐레이션 상위 3건 | 일 1회 | 제목·소스·링크 3개 bullet |

`urgent_signal`은 **중복 발행 방지**가 필수(§10.8). Dry-run 단계에서는 채널 대신 운영자 DM으로 먼저 가도록 분기.

### 10.5 발행 파이프라인

```python
# src/notify/telegram_broadcast.py  (신규)
from typing import Literal
import httpx, os
from .rate_limit import broadcast_throttle  # 토큰버킷

BOT_TOKEN = os.environ["TELEGRAM_BOT_TOKEN"]
BROADCAST_CHAT = os.environ["TELEGRAM_BROADCAST_CHANNEL"]  # @whalescope_alertz

async def post_to_channel(
    text: str,
    kind: Literal["daily_brief","weekly_trend","urgent_signal","news"],
    *,
    dry_run: bool = False,
) -> dict:
    if dry_run:
        return {"ok": True, "dry_run": True, "preview": text[:200]}
    await broadcast_throttle.acquire(kind)           # 채널 타입별 throttle
    async with httpx.AsyncClient(timeout=10) as cli:
        r = await cli.post(
            f"https://api.telegram.org/bot{BOT_TOKEN}/sendMessage",
            json={
                "chat_id": BROADCAST_CHAT,
                "text": text,
                "parse_mode": "HTML",
                "disable_web_page_preview": False,
                "link_preview_options": {"prefer_large_media": True},
            },
        )
    data = r.json()
    if not data.get("ok"):
        # system_log 적재 + 관리자 DM 폴백
        raise RuntimeError(f"broadcast_failed: {data}")
    return data
```

`src/main.py`의 10단계 파이프라인 중 "9. Telegram 발송" 단계를 두 갈래로 확장한다:

1. 기존: `telegram_bot.notify_watchlists(...)` — 1:1 DM (유저별 watchlist 필터)
2. 신규: `telegram_broadcast.post_to_channel(daily_brief_markdown, kind="daily_brief")` — 채널 1회 게시

둘은 서로 독립적으로 성공/실패. 한쪽 실패가 다른 쪽을 막지 않는다.

### 10.8 중복 발행 방지 · Rate Limit

**dedup 키**: `broadcast_dedup_key = sha256(kind + ":" + canonical_content_hash + ":" + yyyymmdd_or_signal_id)`

Sheets `broadcast_log` 탭(신규):

```
broadcast_log
ts | kind | dedup_key | chat_id | message_id | status | error
```

발행 전 동일 `dedup_key`가 있으면 skip + `status=duplicate` 기록. `daily_brief`는 날짜 키 1회/일, `urgent_signal`은 signal id 1회/시그널, `weekly_trend`는 ISO week 키 1회/주.

**rate limit 가드**
- Telegram Bot API 규약: 채널 게시는 `30 messages/sec`까지이지만, 구독자 보호 관점에서 내부 상한 **2 messages/min/kind** 적용.
- `urgent_signal`은 연속 5분 내 동일 엔티티 시그널은 묶어서 1건으로 집계해 발행(burst coalescing).

### 10.9 실패 모드 · 재시도 · 폴백

| 실패 | 조치 |
|---|---|
| `403 Forbidden: bot is not a member of the channel chat` | 봇 admin 권한 제거 감지. system_log `telegram_broadcast_permission_lost` + 관리자 DM 경보. 자동 발행 중단. |
| `400 Bad Request: message is too long` | 메시지 4096자 초과. 자동으로 본문 압축 후 1회 재시도; 실패 시 "… [전체 보기]" 링크로 폴백 |
| `429 Too Many Requests` | `Retry-After` 헤더 준수, 최대 3회 지수 백오프 |
| `502/503/504` | 3회 지수 백오프 (5s → 15s → 45s) |
| 네트워크 단절 | 5분 뒤 재시도 큐에 적재; 만료 30분 |
| LLM 요약 생성 실패 | LLM 미포함 "raw top 3 moves만" 템플릿으로 degrade 후 발행 (운영자에 `llm_degraded=true` 메타 기록) |

모든 실패는 `system_log`에 `category=telegram_broadcast` 로 기록. 일 3회 이상 실패 시 관리자 DM 1회.
