# OpenRouter 채택 다관점 리뷰

WhaleScope의 LLM 호출을 OpenRouter 단일 게이트웨이로 통합하는 결정에 대해 6개 관점에서 평가하고 최종 권고를 도출.

## 0. 검토 대상 결정

- **현 상태**: `ClaudeAnalyzer`가 Anthropic SDK 직접 호출. v2 계획에서는 Groq/Gemini를 추가하는 멀티 공급자 라우터를 자체 구현하기로 했음.
- **제안**: 위 라우터를 자체 구현하지 않고 **OpenRouter**(`https://openrouter.ai/api/v1`)를 단일 엔드포인트로 사용. OpenAI SDK 호환, 마크업 0%, BYOK 50K/월 무료, `:floor`/`:nitro`/`:exacto` 라우팅 모드 제공.
- **최소 진입선**: $5 크레딧 1회 충전(무료 모델 일 200콜 상향 + 유료 모델 pay-as-you-go 활성화).

## 1. Engineering / Architecture 관점

### 긍정
- **추상화 지점이 선명해진다**: 모델 교체가 환경변수 문자열 한 줄(`MODEL=anthropic/claude-3.5-sonnet`). 인터페이스 의존성이 `LLMRouter` 단일 시그니처로 수렴.
- **OpenAI SDK 호환** = 기존 Python 생태계에 학습 곡선 없음. mocking·테스트 도구도 기존 OpenAI 패턴 그대로 사용.
- **공급자 폴백이 1급 시민**: `models=["primary", "fallback"]` 옵션으로 장애 시 자동 전환. 자체 구현 시 retry/circuit-breaker 로직 일체를 직접 짜야 함.
- **프롬프트 실험 비용↓**: `:floor` 모드로 같은 프롬프트를 여러 모델에 던져 품질·비용 트레이드오프를 관측한 뒤 고정.

### 부정
- **외부 의존성 추가**. OpenRouter 자체가 단일 장애점이 됨. 직접 공급자 호출 대비 가용성 곱셈(99.9% × 99.9% ≈ 99.8%).
- **레이어 한 단계 추가** → 응답 latency 평균 +50~150ms. 일간 배치엔 무의미하지만 자연어 개인화 즉답 UX엔 체감.
- **공급자별 고유 기능**(Anthropic prompt caching, OpenAI structured output 등) 일부가 OpenRouter 추상화 뒤에서 제한되거나 지연 도입.

### 점수: **8/10**
구조적 단순화 효과가 단일 장애점·latency 증가를 압도. 단 prompt caching 같은 비용 최적화 기능이 필요해지면 BYOK + 직접 호출 경로를 병행할 여지를 남겨둘 것.

## 2. Cost / FinOps 관점

### 긍정
- **마크업 0%**: 공급자 원가와 동일. "OpenRouter 통과" 자체가 비용 페널티 없음.
- **BYOK 50K/월 무료**: Anthropic 키를 이미 가지고 있다면 OpenRouter 라우팅 비용 사실상 0. 50K 초과 시 1% 수수료(저렴).
- **`:floor` 자동 최저가** = 같은 결과를 더 싼 공급자로. 운영자의 가격 추적 부담이 줄어든다.
- **WhaleScope 예상 비용**: $5 1회 + 운영 $1~3/월 = **월 $2~5**. 직접 호출과 동일 수준.

### 부정
- **무료 모델 20/일 → $5 충전 시 200/일**: 무료만으로는 운영 불가. "완전 무료"라는 슬로건과는 충돌(단, 기존에도 Anthropic API 비용은 있었으므로 새 비용은 아님).
- **크레딧 1년 만료**. 대량 선결제는 비효율.
- **공급자 가격 변동에 노출**: 무료 모델이 갑자기 유료화되거나 라인업에서 사라지는 사례 잦음.

### 점수: **9/10**
비용 페널티 없이 가격 최적화 도구를 얻는 구조. 단점은 "완전 무료" 마케팅 문구 정도이며 실비용 자체는 합리적.

## 3. Product / UX 관점

### 긍정
- **모델 다양화로 톤·길이 실험 용이**: 일간 브리프는 Sonnet, 즉답은 Haiku, 대량 요약은 Flash 등 워크로드별 최적화. 사용자 입장의 "응답 품질"이 올라간다.
- **자연어 개인화 해석**(v2 §3) 같은 분류 task에 저가 모델을 쓸 수 있어 N명 확장 시 비용이 선형이 아니라 거의 평탄.
- **장애 시 무응답 대신 폴백** = 사용자가 "오늘은 알림이 안 왔다"라는 경험을 덜 한다.

### 부정
- 모델 간 출력 일관성 차이. 같은 프롬프트라도 Llama vs Claude vs Gemini가 어조/길이/한국어 자연도가 다름. **브리프 톤이 매번 흔들리는 인상**을 줄 위험. → 동일 작업은 단일 모델 고정 + 프롬프트 버전 로깅으로 완화 필요.
- 사용자에게는 보이지 않는 변화. 마케팅적 "OpenRouter 채택" 자체는 가치가 없음. 가치는 "장애 0회·응답 일관성" 같은 결과로만 증명.

### 점수: **7/10**
장점은 운영자에게, 단점은 사용자에게 노출될 위험. 톤 일관성 가드레일이 필수.

## 4. Security / Privacy / Compliance 관점

### 긍정
- 중앙화된 API 키 관리 = secrets 노출면 축소(키 하나만 회전).
- 사용량 대시보드 단일 창구 → 이상 트래픽 탐지 용이.

### 부정
- **모든 prompt가 OpenRouter 서버를 경유**. 거래 데이터는 공개 정보지만, **사용자 자연어 메시지(개인 성향·관심사)도 흐름**. README/약관에 명시 필요.
- OpenRouter의 데이터 보존·재학습 정책을 공급자별로 다시 확인해야 함(공급자가 Anthropic이면 Anthropic 정책 + OpenRouter 정책 양쪽 적용).
- 서비스 운영자 입장: 한국 PIPA 관점에서 "외부 처리위탁자" 1개 더 추가. 처리위탁 공시 의무 검토.
- **단일 장애점**은 보안 측면에서도 영향: OpenRouter 자체 사고/유출 시 모든 LLM 트래픽 영향.

### 점수: **6/10**
공개 거래 데이터만 보면 무난하지만, 사용자 발화가 흐른다는 점에서 **명시적 정책 문구·약관 보강**이 전제 조건. PIPA 처리위탁 표기는 출시 전 필수.

## 5. Operations / Reliability 관점

### 긍정
- **공급자 장애 자동 폴백**: Anthropic API가 죽어도 OpenRouter가 다른 공급자로 라우팅. 기존엔 cron 한 번 실패로 그날 브리프 누락이 흔했음.
- 사용량·에러율 통합 대시보드.
- Rate limit 분산: 단일 공급자 키의 분당/일일 한도에 묶이지 않음(`:nitro`로 처리량 우선 라우팅).

### 부정
- **OpenRouter SLA가 곧 우리 SLA의 상한**. 그들이 다운되면 우리도 다운. 직접 호출 대비 가용성 합산 하락.
- **결제 끊김 = 전체 LLM 끊김**: 크레딧 0 도달 시 모든 모델 호출 불가. 자동 충전 + 잔액 알림 필수.
- 무료 모델은 SLA가 거의 없음(promo 종료 시 사전 통지 부족 사례 多). 프로덕션 핵심 경로엔 부적합.

### 점수: **7/10**
"공급자 다중화 이점"과 "OpenRouter 자체 SPOF" 트레이드오프가 정직하게 존재. 핵심 경로는 BYOK + Anthropic 직호출 폴백을 병행하는 이중 구조 권장.

## 6. Career / Demo (포트폴리오 맥락) 관점

### 긍정
- **"멀티 LLM 라우팅을 외부 게이트웨이로 위임, 비용 추적과 폴백을 1급 시민으로"** = 면접관에게 어필되는 의사결정. 직접 구현 대신 적절한 추상화를 선택했다는 시그널.
- 코드 리뷰에서 `LLMRouter` 인터페이스가 깨끗하게 보임. 테스트 가능성 높음.
- 비용 의식·운영 의식 동시 강조 가능: "프로덕션이라면 BYOK·캐싱·대시보드 어떻게 운영할지" 질문에 구체 답변 가능.

### 부정
- "왜 직접 안 짜고 외부 의존?"이라는 반대 질문 가능. 답: "프로덕션 운영 가용성·비용 관측·공급자 폴백을 직접 짜는 비용 vs 외부 위임 비용을 평가했고, 데모/MVP 단계에서 위임이 합리적"이라는 트레이드오프 설명을 준비.
- 면접관이 OpenRouter를 모르면 부연 설명 시간 소요.

### 점수: **8/10**
의사결정 자체가 어필 포인트. 단, 이유를 1분 안에 설명할 수 있도록 정리 필요.

### 권고: **조건부 채택**

OpenRouter를 LLM 게이트웨이로 채택한다. 단 다음 4가지 가드레일이 동시에 충족되는 조건에서.

1. **이중 호출 경로 유지**: 핵심 경로(일간 브리프, 주간 트렌드)는 OpenRouter 우선 + Anthropic 직호출 폴백을 환경변수 토글로 즉시 전환 가능하게. 무료 모델은 보조 경로(요약·실험)에만.
2. **톤 일관성**: 동일 산출물은 동일 모델로 고정. `prompt_version` + `model_id`를 `analysis_log`에 함께 기록해 회귀 추적.
3. **개인정보 정책 명시**: README·서비스 약관·`/privacy` 핸들러에 "사용자 메시지가 OpenRouter를 경유한다"는 문구 추가. PIPA 처리위탁 표기 검토.
4. **잔액·사용량 모니터링**: `system_log`에 호출당 (model, tokens_in, tokens_out, cost_estimate) 기록. 잔액 $1 미만 시 Telegram 알림.

### 비채택 조건 (Reject 트리거)
- BYOK 정책이 사라지거나 마크업이 부과되는 경우 → 직접 호출로 회귀
- OpenRouter SLA 사고가 분기 1회 이상 또는 단일 사고 4시간+ 발생 → 핵심 경로 직접 호출로 환원
- 사용자가 1000명을 넘어 자연어 호출이 일 5K+ 도달 시 → BYOK로 전면 전환 + 캐싱 레이어 신설

## 8. 의사결정 후 작업 순서

1. **v2 계획서 §13 신설**: "LLM 공급자 전략" — OpenRouter 게이트웨이, BYOK, 라우팅 룰, 폴백 정책, 비용 모니터링 4개 항목.
2. **`src/analyzer/llm_router.py` 스펙 문서**: 인터페이스(call, stream, embed), 모델 매핑, 폴백 정책, 사용량 계측, 테스트 시나리오.
3. **`ClaudeAnalyzer` → `LLMAnalyzer` 리네이밍 PR**: 모델 비종속 인터페이스로 변경. 기존 sheets 통합·캐시 유지.
4. **README 보강**: 데이터 흐름 다이어그램에 OpenRouter 박스 추가 + 개인정보 처리 안내.

## 9. 보류 및 추가 검토

- **Anthropic prompt caching**을 BYOK 경로로 활용해 주간 트렌드 코멘터리 비용을 줄일 수 있는지 별도 검증.
- Groq Free tier를 OpenRouter 우회 없이 직접 사용할 때의 한도와 비교 검토(이중 통합 가능성).
- OpenRouter `:exacto` 모드의 한국어 출력 안정성 — 샘플 100건 평가 후 결정.
