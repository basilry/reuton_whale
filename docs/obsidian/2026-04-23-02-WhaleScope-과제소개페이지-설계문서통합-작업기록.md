# WhaleScope `/about` 과제소개페이지 — 구현·설계문서 통합 · 작업 기록

> **세션 목표**: `/about` 과제소개페이지의 잔여 이슈를 해결하고, `apps/dashboard/` 루트에 있는 설계 문서(`DESIGN.md`, `.design-context.md`)를 파일 복사 없이 동일 페이지의 **작업 로그 탭**에 노출한다. 심사위원이 코드·기획·설계 세 축을 한 페이지에서 읽을 수 있도록 하는 것이 이번 단계의 목적.

## 1. 로그 API 400 버그 수정

### 증상
사용자가 `/about`의 작업 로그 탭에서 임의의 항목을 클릭하면 다음 에러가 표시됐다.

```
불러오기 실패 / 문서를 불러오지 못했습니다
오류: HTTP 400 — {"error":"Invalid log filename"}
```

### 근본 원인
`app/api/about/doc/route.ts`의 로그 파일명 검증 정규식이 ASCII 전용이었다.

```ts
// Before
const LOG_FILENAME_RE = /^[\w\-. ()]+\.md$/;
```

`\w`는 `[A-Za-z0-9_]`로만 매칭하므로 한글·일본어 등 유니코드 문자는 거부. 실제 manifest의 76개 파일명 대부분이 한글을 포함 (`WhaleScope-진행내역`, `체인-커버리지-적용완료-QA-종합보고서`) → **전수 실패**.

### 패치
유니코드 프로퍼티 이스케이프로 전환.

```ts
// After — path-traversal 방어는 path.resolve + path.relative로 별도 유지
const LOG_FILENAME_RE = /^[\p{L}\p{N}_\-. ()]+\.md$/u;
```

- `\p{L}` — 모든 유니코드 letter (한글·한자·키릴·라틴)
- `\p{N}` — 모든 유니코드 number
- `/u` 플래그 — 유니코드 속성 활성화
- 허용 특수문자 `_ - . ( ) [공백]` 유지 → `/`, `..`, 제어문자는 여전히 거부

### 교훈
파일 시스템 정규식을 작성할 때 ASCII `\w`·`\d`는 다국어 서비스에서 **암묵적 차별**. 처음부터 유니코드 프로퍼티를 쓰는 습관이 필요.

## 3. 로그 본문 폭 확장

### 문제
One Pager/README 탭은 뉴스레터 독서감을 위해 `prose-editorial` 스코프가 본문 너비를 `65ch` ≈ 619 px로 클램프한다. 이 규칙이 작업 로그 탭에도 상속되어 **1078 px 패널에서 로그 본문이 620 px**만 채우고 좌우 약 150 px 씩이 비어 있었다. 작업 로그는 데이터·코드 블록·테이블이 많아 좁은 폭이 오히려 가독성을 해침.

### 해결
`app/about/about.css`에 **스코프 한정 오버라이드** 추가.

```css
/* log 탭에서만 max-width 제약 해제, 다른 탭은 그대로 */
.about-page .ws-log-content .md-content p,
.about-page .ws-log-content .md-content ul,
.about-page .ws-log-content .md-content ol,
.about-page .ws-log-content .md-content blockquote,
.about-page .ws-log-content .md-content table,
.about-page .ws-log-content .md-content hr {
  max-width: none;
}
```

- 로그 탭(`.ws-log-content` 컨테이너)만 영향
- One Pager/README 탭의 `.md-content`는 여전히 에디토리얼 클램프 유지 → **회귀 없음** (DOM 측정으로 확인: 로그 `maxW: none`, 타 탭 `619.531px` 보존)

### 디자인 원칙과의 정합
`.design-context.md`의 "Differentiator: 매일 아침 읽는 한 장의 레터 — Daily Brief" 원칙은 **대시보드의 히어로 카드**에 대한 것이지, 작업 로그 서브 페이지의 기술 문서 렌더링에는 적용되지 않는다. 스코프 분리로 두 맥락이 공존.

## 4. 설계 문서 통합

### 설계 원칙

**"`apps/dashboard/`가 설계 문서의 단일 출처(single source of truth)"** 를 유지. 파일 복사·심볼릭 링크 대신 API 라우트가 원본을 직접 읽도록 구성. 이유:

1. **동기화 비용 제거** — `DESIGN.md`를 수정할 때 Obsidian·repo 두 곳에 복사본을 유지하면 반드시 어느 한쪽이 뒤쳐짐.
2. **git 단순성** — 중복 파일이 없으므로 `git blame`이 한 계통으로 깔끔.
3. **배포 결정의 명시성** — API 화이트리스트에 파일명을 등록하는 **의식적 행위**가 "이 파일은 공개용"이라는 선언이 된다.

### 구현 — 5-파일 최소 변경

| # | 파일 | 변경 요지 |
|---|---|---|
| 1 | `apps/dashboard/lib/about/manifest.ts` | `LogEntry`에 `source?: "obsidian" \| "dashboard"` 추가. undefined 기본값은 기존 obsidian 경로로 폴백 — 76개 기존 엔트리의 manifest를 건드리지 않음. |
| 2 | `apps/dashboard/lib/about/markdown.ts` | `DASHBOARD_DIR = path.join(REPO_ROOT, "apps", "dashboard")` export. `OBSIDIAN_DIR`과 동일한 패턴. |
| 3 | `apps/dashboard/app/api/about/doc/route.ts` | `dashboard/<file>` prefix 분기 추가. `DASHBOARD_ALLOWLIST = {"DESIGN.md", ".design-context.md"}` 명시 Set. path-traversal 가드 이중화. |
| 4 | `apps/dashboard/app/about/AboutClient.tsx` | `fetch()` URL 빌더를 `entry.source === "dashboard"`에 따라 `dashboard/` vs `log/`로 분기. 기존 동작은 source 미지정 엔트리에 대해 완전 보존. |
| 5 | `docs/obsidian/_manifest.json` | 2 엔트리 append — seq 10·11, date 2026-04-17(DESIGN 문서 mtime 기준), category `UX · 디자인`, source `dashboard`. count 76 → 78. |

### 핵심 보안 설계: 디렉토리 노출 대신 파일 화이트리스트

```ts
// apps/dashboard/app/api/about/doc/route.ts
const DASHBOARD_ALLOWLIST = new Set<string>([
  "DESIGN.md",
  ".design-context.md",
]);

if (src.startsWith("dashboard/")) {
  const filename = src.slice("dashboard/".length);
  if (!DASHBOARD_ALLOWLIST.has(filename)) {
    return { error: "Unknown dashboard document", status: 400 };
  }
  const abs = path.resolve(DASHBOARD_DIR, filename);
  const rel = path.relative(DASHBOARD_DIR, abs);
  if (rel.startsWith("..") || path.isAbsolute(rel)) {
    return { error: "Path traversal rejected", status: 400 };
  }
  // ... fs.readFile
}
```

- **1차 방어**: `Set.has(filename)`은 **문자열 동치 검사**. `../`, 절대경로, 인코딩 변형, 다른 확장자 모두 즉시 차단.
- **2차 방어**: 화이트리스트가 미래에 완화되더라도 `path.resolve` + `path.relative`가 `DASHBOARD_DIR` 밖의 어떤 경로도 거부.

디렉토리 공개(`apps/dashboard/**/*.md` 와일드카드) 대신 **개별 파일을 수동 등록하는 구조**가 이 설계의 핵심 안전장치다. 향후 문서를 추가하려면 반드시 화이트리스트에 명시적으로 한 줄을 추가해야 함 → 실수로 `.env` 등 민감 파일을 노출하는 경로가 구조적으로 차단된다.

### 라우트 계약 요약

```
GET /api/about/doc?src=<key>
  • onepager            → <repo>/ONE_PAGER.md
  • readme              → <repo>/README.md
  • log/<filename.md>   → <repo>/docs/obsidian/<filename.md>   [\p{L}\p{N}_\-. ()]+\.md
  • dashboard/<file>    → <repo>/apps/dashboard/<file>         화이트리스트 일치 시에만
```

## 5. 보안성 검토 결과

### 위협 모델
인증 없는 외부 공격자가 임의 `?src=` 페이로드로 호스트 파일·repo 민감 파일을 읽어내려고 시도하는 상황.

### 실제 페이로드 공격 시뮬레이션 (전부 400으로 차단)

| Payload | 브라우저 fetch 응답 |
|---|---|
| `dashboard/DESIGN.md` | ✅ 200 · 34 KB HTML |
| `dashboard/.design-context.md` | ✅ 200 · 5 KB HTML |
| `dashboard/../../README.md` | ❌ 400 `Unknown dashboard document` |
| `dashboard/%2E%2E%2F%2E%2E%2FREADME.md` | ❌ 400 (URLSearchParams가 디코딩 후 화이트리스트 비교) |
| `dashboard/.env` | ❌ 400 (화이트리스트 미등록) |
| `dashboard/.env.local` | ❌ 400 |
| `dashboard/package.json` | ❌ 400 |
| `dashboard/next.config.ts` | ❌ 400 |
| `dashboard/foo.md` | ❌ 400 |

### 위험 등급

| 등급 | 해당 항목 |
|---|---|
| 🟢 안전 | Path traversal (이중 방어), URL 인코딩 우회, 클라이언트 신뢰 우회, XSS(`isomorphic-dompurify` 파이프라인 공유), 정보 노출(문서 내용 비밀 없음), slug 충돌 |
| 🟡 감수 가능(기존 대비 악화 없음) | DoS rate limit 부재 — 기존 `log/` 엔드포인트와 동일 수준. 에러 detail 메시지에 내부 경로가 포함될 수 있음(개선 여지, 범위 외). |
| 🔴 중대 이슈 | 없음 |

### 결론
**배포 가능**. `apps/dashboard/` 디렉토리 전체를 노출하지 않고 개별 파일 화이트리스트로 제한한 설계가 향후 유지보수에서도 이 패턴을 깨지 않는 한 추가 공격 표면을 만들지 않는다.
