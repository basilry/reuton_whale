# WhaleScope 종합보고서

Wrtn Technologies Product Engineer 과제 전형 — 도메인 `C. AI 요약/큐레이션 서비스`.
온체인 고래 지갑 모니터링, 규칙 기반 시그널 탐지, LLM 한국어 해설, Telegram 브리핑을 하나로 묶은 크립토 고래 큐레이션 서비스.

---

## 1. 프로젝트 목적

블록체인 고래 지갑의 대규모 자금 이동을 실시간에 가까운 주기로 수집하고, 규칙 기반 시그널 엔진으로 의미 있는 패턴을 탐지한 뒤, LLM이 한국어 해설을 생성해 Telegram으로 구독자에게 브리핑하는 서비스. 핵심 가치는 "고래 움직임 → 시그널 해석 → 사용자 전달"을 하나의 파이프라인으로 자동화하는 것이다.

과제 전형의 평가 초점(문제 정의, AI 활용, 실행 가능성, 문서화, 코드 품질)에 맞춰, 실제 배포 가능한 수준의 구현과 체계적인 의사결정 기록을 병행했다.

## 3. 아키텍처

### 3.1 시스템 구성도

```
[소스 수집]                         [정규화]             [시그널]                 [해설/배포]

Etherscan API v2  ----+
  ETH/ARB/BASE/BSC/  |
  POLYGON             |
                       +--> Event(dataclass) --> SignalEngine(8 rules) --> LLMAnalyzer --> Telegram Bot
Solscan API v2   -----+                         |                        via LLMRouter    Google Sheets
  SOL                                           |                        Anthropic/
                                                |                        Gemini/Groq
Telethon listener ----+                         |
  public TG channel   +--> tg_whale_events -----+

watched_addresses.csv --> watched_addresses sheet --> collector address scope
address_activity sheet --> baseline builder --> 3-sigma anomaly rules
user_interests sheet --> per-subscriber personalize --> Telegram message variant
```

### 3.2 핵심 설계 원칙

**규칙 기반 탐지 + LLM 해설 분리**: 시그널 판단은 결정론적 규칙 엔진이 담당하고, LLM은 해설/브리핑 생성에만 사용. AI 출력이 변해도 탐지 로직은 불변.

**Storage Protocol 패턴**: 16개 메서드의 Protocol 인터페이스로 저장소를 추상화. 현재 Google Sheets 구현체만 존재하나, SQLite/Postgres 전환 시 구현체만 교체하면 됨. 파이프라인 코드는 프로토콜에만 의존.

**LLM 멀티 프로바이더 라우팅**: task별 preferred/fallback 체인. 하나의 프로바이더 장애가 전체 브리핑 실패로 이어지지 않음. `config/llm_routing.yaml`로 선언적 관리.

**점진적 활성화**: baseline 빌더가 7일 미만 데이터 시 빈 dict 반환 → spike 규칙이 자연스럽게 비활성. 데이터 축적에 따라 점진적으로 규칙이 활성화되는 설계.

**개인화 DI**: `WhaleScopeBot`에 `personalize_fn`을 의존성 주입. 봇 코드가 개인화 로직을 직접 알지 않고, 교체 가능한 함수를 통해 per-subscriber 시그널 필터링.

### 3.3 실행 흐름 (10단계 파이프라인)

1. 감시 주소 목록을 Google Sheets에서 로드
2. Etherscan/Solscan이 감시 주소의 최근 활동 수집
3. Telethon listener가 공개 고래 알림 채널 메시지를 별도 수집
4. 수집 이벤트를 `Event` dataclass로 정규화
5. `address_activity` 기반으로 chain baselines 계산
6. `SignalEngine`이 규칙 기반 시그널 + 교차검증 시그널 생성
7. 시그널 있으면 signal 경로, 없으면 legacy scorer fallback
8. `LLMAnalyzer`가 시그널 목록을 한국어 브리핑으로 변환
9. Telegram Bot이 구독자별 개인화 메시지 발송
10. 거래/시그널/브리핑/시스템 로그를 Google Sheets에 저장

### 3.4 배포 토폴로지

| 구성요소 | Runtime | 프로세스 | 역할 |
|---|---|---|---|
| 정보수집 파이프라인 | Render Cron/Worker | `python -m src.main` | 수집→시그널→브리핑→저장→발송 |
| Telegram bot | Render Worker | `python scripts/run_bot.py` | 사용자 명령 처리 (long-polling) |
| Telegram listener | Render Worker | `python scripts/run_listener.py` | TG 채널 실시간 수신 |
| Dashboard | Vercel | Next.js App Router | 운영/인사이트 대시보드 |
| Legacy dashboard | Local | Streamlit | 로컬 진단용 |

## 8. 기술적 강점 요약

1. **규칙/LLM 분리 아키텍처** — 탐지와 해설을 분리해 결과 일관성과 디버깅 용이성 확보
2. **Storage Protocol 추상화** — 구현체 교체만으로 저장소 전환 가능한 설계
3. **LLMRouter 자체 구현** — 3 provider 라우팅 + task별 preferred/fallback + 비용 추적
4. **통계 기반 이상탐지** — baseline 빌더 + 3σ 규칙 + 점진적 활성화
5. **교차검증 (Corroboration)** — 온체인 + TG 소스 교차 확인으로 시그널 신뢰도 강화
6. **개인화 DI 패턴** — personalize_fn 주입으로 봇/엔진 결합도 최소화
7. **298건 테스트** — 비동기 포함 전수 그린. 외부 API 완전 mock 처리
8. **한국어 네이티브 대시보드** — 15개 humanize 함수로 자연스러운 UX
9. **4차에 걸친 체계적 리뷰 사이클** — 이슈 추적 → 병렬 프롬프트 → 충돌 분석 → 재프롬프트

## 9. 결론

WhaleScope는 과제 전형의 요구사항(문제 정의, AI 활용, 실행 가능성, 문서화, 코드 품질)을 모두 커버하며, 종합 점수 8.6/10을 달성했다. 특히 Operations 영역이 3차 6.0에서 4차 8.0으로 가장 큰 개선을 보였고, 298건 테스트 전체 그린으로 안정성을 확보했다.

즉시 수정 가능한 P0 이슈 2건과 P1 구조 개선 3건을 처리하면 9.0+ 도달이 가능하다. 장기적으로는 Storage 전환, 대시보드 인터랙션, ML 기반 시그널 고도화가 서비스 확장의 핵심 축이 된다.
