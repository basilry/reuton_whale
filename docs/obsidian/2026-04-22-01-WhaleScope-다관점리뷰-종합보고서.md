## 2. 5개 관점 공통 강점 (모두가 인정한 것)

1. **탐지(규칙) ↔ 해설(LLM) 레이어 분리가 실제 코드에 구현됨**
   - `src/signals/rules.py` 8개 규칙이 `src/llm/router.py`의 LLM 해설과 명시적으로 분리.
   - PM 관점: "이 원칙 덕분에 평가 환경에서 키 없이도 `smoke_pipeline.py` SMOKE OK".
   - Engineering 관점: "파이프라인 내부에 `input_fingerprint` 캐시(brief.py L973-1029)까지 내재".

2. **다중 fallback 체인의 완결성**
   - LLM: Anthropic → Gemini → Groq (`src/llm/router.py:22-43`).
   - Price: CoinGecko → Binance(4 quote 순회) → stale cache 6h (`src/enrich/price_resolver.py:52-104`, 5s timeout).
   - Brief: signals→tx fallback→empty fallback, 각 분기가 `brief_cost_ledger`에 다른 `decision` 값으로 기록 (`src/pipeline/brief.py:803-878`).
   - Sheets: batchGet 실패 시 per-tab fallback (`apps/dashboard/lib/sheets.ts:583-642`).

3. **관측성(observability)이 파이프라인과 동등한 시민(first-class)**
   - 모든 job에 `append_service_heartbeat` (`run_all.py:195-236`) — status·run_status·duration_ms·lag_seconds·processed_count 기록.
   - `brief_cost_ledger`에 decision·llm_called·tokens·cost·cumulative_cost.
   - 운영 관점에서 Sheets 한 탭만 열어도 어떤 경로로 갔는지 복원 가능.

4. **L1(Map 45s) + L2(Upstash Redis) + stale 3단 캐시**
   - `lib/sheets.ts:215-256` — L1 hit → L2 hit → Sheets REST → stale(15min).
   - `tabInflight` single-flight로 thundering herd 차단.
   - `redis-cache.ts` `CACHE_REQUEST_TIMEOUT_MS=2000` + AbortController로 Upstash 장애 시 페이지 블로킹 방지.

5. **Next.js 15 App Router 경계와 runtime 힌트 명확성**
   - `api/dashboard/route.ts`: `runtime="nodejs"`, `preferredRegion="icn1"`, `revalidate=60`.
   - `api/stream/route.ts`: `runtime="nodejs"`, `dynamic="force-dynamic"`, `maxDuration=60`, `X-Accel-Buffering: no`.
   - `page.tsx:480` Server Component에서 `Promise.all`로 dictionary/language/state/fearGreed 4-way 병렬 로드.

6. **문서화 문화의 성숙도 (PE 채용 시그널)**
   - ONE_PAGER / README / docs/ / Obsidian 4-layer 문서 체계.
   - `docs/operational-run-verification.md` 653줄 runbook — 1인 프로젝트치고 이례적.
   - `docs/changelog.md` Why/What/How validated/Risks 고정 포맷 (단 v6 이후 drift 발생).

### 3.2 Hardcoded 정책·매직 넘버가 코드 여러 곳에 분산 — Engineering + Maintainability + Performance

- `brief.py:39` `_FULL_BRIEF_HOURS_KST = frozenset({9, 15, 21})` (Python)
- `page.tsx:47` `BRIEF_SCHEDULE_HOURS_KST = [0, 8, 16]` (TypeScript) — **Python과 UI가 서로 다른 시간표를 사용** ⚠️

**영향**: UI가 안내하는 "다음 브리핑 시각"이 실제 full-slot과 **어긋남**. 프로덕션 튜닝마다 3곳 수정 필요 → 누락 시 silent bug.

**공통 처방**: `config/brief_schedule.yaml` + `config/brief_policy.yaml`로 단일 소스화. Python + Next.js 모두 런타임/빌드타임에 참조.

### 3.3 PR-gated CI 부재 — Maintainability + Security

- `.github/workflows/` 9개가 모두 `workflow_dispatch` 수동 트리거.
- "pytest 402 passed"가 CI badge로 증명되지 않음 — 주장일 뿐.
- Security 관점: lint/typecheck가 자동화되지 않으면 `isPublicAdminPreviewEnabled() { return true }` 같은 regression도 merge 전에 잡히지 않음.

**공통 처방**: `.github/workflows/ci.yml`에 `on: [pull_request, push]` 추가 — `pytest -q` + `npm run lint/typecheck/test` + Playwright CT. 작업량 4시간, ROI 최대.

### 3.4 프론트/백엔드 로직 중복 — Engineering + Performance + PM

- `_build_signal_market_mood` (backend, `brief.py:411`) ↔ `buildMarketMood` (frontend, `page.tsx:227-338`).
- 이미 백엔드가 `marketMood` 구조화 필드를 주입하는데, 프론트에는 signal-count 기반 legacy fallback이 500줄 상당 잔존.
- PM 관점: 유지 부담 + DESIGN.md의 "한 장의 레터" 선언과 실제 UI 과밀의 간극.

**공통 처방**: 프론트 `buildMarketMood`를 "structured mood 없으면 대기 중 메시지"로 축약 (≈200줄 제거). `buildBriefCopy/buildMarketMood/buildBriefAnalysis`를 `lib/brief-view-model.ts`로 추출.

### 상충 지점과 권장 균형점

1. **Security "debug endpoint 닫기" ↔ Performance "관측성 확장"**
   - 권장: `/api/debug/*`는 `NODE_ENV !== "production"` + admin auth 둘 다 강제. 프로덕션 관측은 `/api/admin/health`의 admin-auth 영역으로 일원화.
2. **PM "한 장의 레터" ↔ Engineering/Maintainability "page.tsx 분해 비용"**
   - 권장: page.tsx를 지금 분해하지 말고, **CSS 레이어에서** above-the-fold에 hero brief만 노출하고 나머지는 `<details>` 또는 accordion으로 접기. 분해 리팩터링은 제출 후로 미룸.
3. **Engineering "Zod 전면 도입" ↔ Maintainability "리팩터링 비용"**
   - 권장: 즉시 도입은 대시보드 `rowToObject` 경계 1군데로 한정. Python 쪽 Pydantic 이관은 Stage 2로.
