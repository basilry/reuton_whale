# WhaleScope 2차 리뷰 — 개선 검증 + 신규 이슈

1차 리뷰(C1–C7, H1–H6)의 개선 반영 여부를 소스 기준으로 검증하고, 개선 과정에서 새로 드러난 이슈를 정리한다.

## TL;DR

- **1차 Critical 7건 전부 해소.** 봇 폴링 엔트리포인트·subscribers 테이블·Sheets 캐시·중요도 표기·워치리스트 필터·AI 실패 폴백 모두 커밋 `801b687`에서 실제 코드로 반영됨.
- **1차 High 6건 전부 해소.** 중복 방지, 구조화 로그, API 키 마스킹, Streamlit 비번, 동시성 제어 yml, 심볼 매핑 확장까지 적용.
- **1차 구현도 점수 5/10 → 8/10**. 실제 구동 가능한 수준에 도달. 단, 운영 신뢰성(N+1, race, graceful shutdown)과 설계 부채(이중 스코어링, 하드코딩 프롬프트)는 그대로.
- **샌드박스 테스트 106/121 pass**, 실패 15건은 전부 `pytest-asyncio` 플러그인 부재로 인한 마커 미인식(코드 회귀 아님).

## 1. Critical 개선 검증 (C1–C7)

### C1. subscribers 테이블 부재 → 해소
`src/storage/schema.py`에 `TAB_SUBSCRIBERS = "subscribers"`, `SUBSCRIBERS_HEADERS` (chat_id/username/status/watchlist_coins/created_at/updated_at/last_brief_at) 추가. `src/storage/sheets_client.py`에 6개 퍼블릭 메서드 신설: `get_active_subscribers`, `get_subscriber_info`, `add_subscriber`, `get_watchlist`, `set_watchlist`, `set_status`. `_normalize_subscriber`가 chat_id int 캐스팅 + watchlist 파싱을 캡슐화.

### C2. 봇 폴링 진입점 부재 → 해소
`scripts/run_bot.py` 신설 (`app.run_polling(stop_signals=None)`). `/start`·`/watchlist`·`/pause`·`/status` 4개 핸들러가 `WhaleScopeBot.build()`에 등록되어 즉시 동작.

### C3. top_transactions 요약 손실 → 해소
`src/main.py` Step 9에서 `top_transactions`를 `hash/symbol/amount_usd/importance_score/interpretation/type` 전체 dict 리스트로 `json.dumps`. 재현성·디버깅·대시보드 조회 복원.

### C4. 중요도 표시 오류 → 해소
`formatters._importance_bar`가 `score / 10.0`으로 정규화, 라벨은 `"N/10"`로 출력. 이전의 "점수×10%" 버그 사라짐.

### C5. 워치리스트 필터 미구현 → 해소
`telegram_bot._filter_brief`가 re.escape된 심볼을 `\b(...)\b` 패턴으로 컴파일해 줄 단위 매칭, 매치된 라인 앞에 `⭐` prefix. 들여쓰기 보존.

### C6. Claude 캐시 미사용 → 해소
`ClaudeAnalyzer.__init__(..., sheets=None)`, `_load_from_sheets_cache(prompt_hash)` / `_persist_to_sheets(...)` 추가. `main.py`에서 `ClaudeAnalyzer(config.anthropic_api_key, sheets=sheets)`로 주입. `prompt_hash` 기반 analysis_log 재사용 실제 동작.

### C7. AI 실패 시 스코어 0 폴백 → 해소
`main.py` Step 6 except 블록:
```python
tx_copy["importance_score"] = int(tx.get("base_score", 0))
tx_copy["interpretation"] = "(AI 분석 실패, 규칙 기반 점수로 대체)"
tx_copy["confidence"] = "low"
```
완전 침묵 대신 규칙 기반 스코어로 최소 품질 보장.

## 3. 다관점 리뷰 (업데이트)

### Marketing / Product
- 실사용 가능 상태. `/start`→구독 자동 등록→08:00 KST 푸시 플로우가 이제 코드로 완결.
- `⭐` 필터는 기능으로 홍보 가능한 수준으로 구현됨. 단, UX 측면에서 "관심 코인 없는 줄은 숨김" 옵션이 있으면 짧은 브리프에서 더 즉각적.
- 여전히 부재: 온보딩 카피 길이가 길어 모바일 첫 화면에서 짤릴 위험. `/help` 핸들러 없음.

### Engineering
- 계층 경계 일관. `sheets`를 analyzer에 주입하는 방식은 테스트 주입성 확보 면에서 OK.
- 비동기/동기 혼용은 유지됨: analyzer는 sync, distributor는 async. `main.py`에서 `asyncio.run` 한 번으로 처리 — 현 규모에서는 문제 없음.
- `run_polling(stop_signals=None)`은 Ctrl+C에서도 정상 종료를 막는다. 운영 셸/PM2/systemd에서 문제.

### PM / Investor
- MVP 데모 가능. Daily Brief GitHub Actions로 1회 실행 → 스크린샷 캡처 가능.
- 운영 비용: 구독자 N명당 하루 N회 `get_all_values()` 호출하는 경로(§4 N+1) 있으므로 구독자 증가 시 gspread rate limit 주의.
- 확장 리스크: subscribers가 시트 단일 탭 단일 row = 한 유저. 수천명 넘어가면 시트 API QPS에 부딪힘. Postgres 마이그레이션 시점 논의 필요.

## 4. 신규 및 잔존 이슈

### N1. Sheets 기반 캐시의 N+1 성능 리스크 (Medium)
`ClaudeAnalyzer._load_from_sheets_cache`가 분석 요청마다 `analysis_log` 전체를 훑는다(Top5 기준 5회). 현재 규모는 OK지만 로그가 누적될수록 latency와 gspread 쿼터가 동시에 커진다. 최소한 session 단위 in-memory LRU(예: `functools.lru_cache` 또는 dict)를 1번의 시트 조회 결과 위에 얹어야 한다.

### N2. append_transactions race condition 미해소 (High 남음)
read existing_hashes → append_rows 2단계는 atomic하지 않음. `concurrency` yml로 GitHub Actions 동시 실행은 막았지만 수동 `workflow_dispatch`와 schedule이 겹치거나 로컬 실행이 섞이면 중복 가능. 실서비스 DB 이행 전까진 "append 직전 한 번 더 hash 재조회"로 완화.

### N3. 이중 스코어링 시스템 공존 (M3 잔존)
`base_score`(scorer.pre_filter 산정)와 `importance_score`(Claude 산정)가 같은 tx dict에 공존. 폴백 때만 합쳐지고 평시엔 둘 다 유지. 리포트/저장 포맷은 `importance_score`만 쓰는데 `base_score`는 계산 비용만 있고 산출물에 반영되지 않음. scorer 출력 키 통일 필요.

### N4. `generate_daily_brief` 시스템 프롬프트 하드코딩 (M2 잔존)
`ClaudeAnalyzer.generate_daily_brief`가 프롬프트 문자열을 직접 포함. 톤/포맷 실험이 코드 수정을 강요한다. `prompts/daily_brief.txt` 분리 + 버전 스트링 로깅 권장.

### N5. `coingecko._fetch_coins_list` bare Exception (M5 잔존)
`except Exception`만 걸려 네트워크 예외와 파싱 예외가 구분되지 않는다. `requests.exceptions.RequestException` / `ValueError` 분리 catch, 또는 `CollectorError`로 래핑.

### N6. `run_bot.py` graceful shutdown 불가
`app.run_polling(stop_signals=None)` 때문에 SIGTERM/SIGINT가 무시된다. systemd/PM2/도커 `stop` 시 강제종료로 이어져 대기 중 메시지 유실 가능. 기본값(=None 넘기지 않음) 또는 `stop_signals=(signal.SIGINT, signal.SIGTERM)` 권장. `sys.path.insert(0, ".")`도 제거 가능(루트에서 `python -m scripts.run_bot` 사용).

### N7. Streamlit 비번 인증 rate limit 없음
`_check_password()`에 시도 횟수 제한 없음. 비밀번호 단일 문자열이면 brute-force가 용이. 최소 `time.sleep(2)`+실패 카운터, 궁극은 OAuth/ Cloudflare Access.

### N8. `_find_subscriber_row` 중복 가드
```python
if i - 2 < len(all_values) - 1 and col < len(row) and row[col] == target:
```
`enumerate(all_values[1:], start=2)` 하에서 `i - 2 < len(all_values) - 1`은 항상 참. 제거 가능, 가독성만 떨어뜨림.

### N9. `handle_start` 실패 시 사용자 UX
`add_subscriber` 실패하면 `DistributorError`로 예외가 튀어 사용자 메시지 없이 종료된다. PTB가 로그만 남기고 `/start`가 무반응처럼 보임. try/except 안에서 사용자에게 "등록 실패, 잠시 후 다시 시도" 안내 메시지 선전달 후 재던지는 패턴이 안전.

### N10. 테스트 환경 의존성
`requirements.txt`/`requirements-dev.txt`에 `pytest-asyncio` 명시 상태 재확인 필요. 샌드박스에서 15건 async 테스트가 `PytestUnknownMarkWarning`로 스킵되는 원인. dev 환경에선 통과한다고 했으나 CI 스펙에도 명시되어야 한다.
