# WhaleScope 리뷰 기반 개선안

## 1. 문서 목적

이 문서는 2026-04-16 코드리뷰에서 확인된 5개 finding을 기준으로 WhaleScope의 다음 개선 순서와 구현 기준을 정리한다.

이번 리뷰의 핵심 결론은 다음과 같다.

- 현재 구조는 Render worker, Google Sheets, Vercel dashboard 분리가 타당하다.
- 하지만 Telegram listener가 수집한 데이터가 daily pipeline으로 되돌아오지 않아 문서화된 "온체인 + Telegram 교차검증" 가치가 아직 실제 서비스 경로에서 완성되지 않았다.
- Vercel dashboard는 운영자용 데이터와 사용자용 화면이 동시에 존재하므로, 배포 전 API 인증과 에러 메시지 sanitization이 필요하다.
- Listener와 dashboard health는 실제 상태를 나타내도록 heartbeat 기반으로 보강해야 한다.

검증 상태:

- `python -m pytest -q`: 265 passed, 6 warnings
- `npm run dashboard:typecheck`: passed
- `npm run dashboard:lint`: passed
- `npm run dashboard:build`: passed
- `git diff --check`: passed

## 2. 리뷰 Findings 요약

| 우선순위 | Finding | 영향 | 핵심 조치 |
|---|---|---|---|
| P1 | Listener data is never fed back into the signal pipeline | Telegram 채널 수집 데이터가 시그널, 브리핑, 교차검증에 반영되지 않음 | `tg_whale_events` read path 추가 후 `Event(source="tg")`로 병합 |
| P1 | Public API exposes operational data and raw error messages | Vercel 공개 URL에서 운영 데이터와 내부 오류가 노출될 수 있음 | 운영 API 인증, 사용자 API 분리, generic error response 적용 |
| P1 | User page leaks internal load errors | `/insights`에서 Sheets/Auth/env 오류가 사용자에게 노출될 수 있음 | 서버 로그만 상세 유지, UI는 일반 안내문 표시 |
| P2 | Telethon handler blocks the event loop on LLM and Sheets I/O | burst 상황에서 메시지 처리 지연 또는 누락 가능성 | blocking I/O를 `asyncio.to_thread` 또는 queue worker로 분리 |
| P2 | Listener health card is inferred from Sheets connectivity | listener 미실행/미인증 상태도 정상처럼 보일 수 있음 | heartbeat 또는 최신 `tg_whale_events` 기준 상태 산출 |

## 3. 개선 목표

### 3-1. 과제 제출 관점

뤼튼 Product Engineer 과제에서는 "실제로 동작하는 AI 요약/큐레이션 서비스"로 보이는 것이 중요하다. 따라서 단순히 화면을 예쁘게 만드는 것보다, 아래 흐름이 실제 데이터로 연결되어야 한다.

```text
온체인 수집
→ Telegram 공개 채널 수집
→ Event 정규화
→ SignalEngine 교차검증
→ LLM 한국어 브리핑
→ Telegram 발송
→ Vercel dashboard 표시
```

현재 가장 큰 gap은 `Telegram 공개 채널 수집 → SignalEngine 교차검증` 구간이다.

### 3-2. 운영 안정성 관점

Render와 Vercel로 배포하려면 다음이 필요하다.

- long-running listener가 block되지 않아야 한다.
- worker별 마지막 성공/실패 상태가 기록되어야 한다.
- dashboard는 실제 worker 상태를 표시해야 한다.
- 사용자용 route는 내부 장애 정보를 노출하지 않아야 한다.
- 운영자용 API는 인증 없이 공개되면 안 된다.

### 3-3. 사용자 UX 관점

사용자용 `/insights`는 raw JSON, traceback, provider error가 아니라 사람이 이해할 수 있는 상태를 보여줘야 한다.

예:

- 나쁜 예: `Missing required environment variable: GOOGLE_CREDENTIALS_JSON`
- 좋은 예: `데이터 연결을 확인 중입니다. 잠시 후 다시 확인해 주세요.`

## 4. 개선 순서

## Phase 1. 보안/노출 차단

목표:

- 배포 전 사용자에게 내부 오류와 운영 API가 노출되는 문제를 먼저 막는다.

작업:

1. `/insights`에서 `state.errorMessage` 렌더링 제거
2. `loadInsightState()`의 catch에서는 서버 로그만 남기고 UI에는 generic 상태만 전달
3. dashboard API route의 `errorResponse()`를 generic message로 변경
4. 운영자 API와 사용자 API를 분리
5. 운영자 API에 `DASHBOARD_PASSWORD` 또는 Bearer token 기반 server-side guard 추가

권장 구현:

- `apps/dashboard/lib/auth.ts` 추가
- `requireDashboardAuth(request)` 또는 `isDashboardAuthorized(request)` 구현
- `/api/dashboard`, `/api/transactions`, `/api/signals`, `/api/system-log`는 운영자 API로 두고 인증 적용
- 사용자용 API가 필요하면 `/api/public/insights`처럼 별도 route를 만들고, 주소/hash/system log를 제거한 sanitized payload만 반환

검증 기준:

- 인증 없이 운영 API 요청 시 401
- 인증 실패 응답에 env key, Google API body, stack trace가 없음
- `/insights` 데이터 로드 실패 시 내부 에러 문자열이 화면에 표시되지 않음
- `npm run dashboard:typecheck`, `npm run dashboard:lint`, `npm run dashboard:build` 통과

## Phase 2. Telegram listener 데이터를 pipeline에 병합

목표:

- listener가 저장한 `tg_whale_events`를 daily pipeline의 `raw_events`에 포함해 실제 교차검증과 daily brief에 반영한다.

작업:

1. `SheetsClient.list_tg_whale_events(since: datetime | None = None)` 추가
2. `tg_whale_events` row를 `Event(source="tg")`로 변환하는 adapter 추가
3. `src/main.py` Stage 3에서 Etherscan/Solscan 이벤트 수집 후 TG 이벤트를 함께 읽어 `raw_events`에 merge
4. duplicate 방지 기준 정의
5. `SignalEngine` corroboration 테스트에 실제 Sheets row 변환 케이스 추가

권장 adapter 필드 매핑:

| tg_whale_events | Event |
|---|---|
| `tg_msg_id` | `tx_hash`는 없으면 `None`, 별도 `raw_response_hash`는 사용하지 않음 |
| `tg_date` | `block_time` |
| `blockchain` | `chain` |
| `symbol` | `token` |
| `amount` | `amount_token` |
| `amount_usd` | `amount_usd` |
| `from_owner` | `from_addr` 또는 owner label |
| `to_owner` | `to_addr` 또는 owner label |
| `from_owner_type`, `to_owner_type` | `counterparty_category`, `direction` 추론에 사용 |
| `collected_at` | `collected_at` |

주의:

- TG 이벤트는 실제 tx hash가 없는 경우가 많으므로 `tx_hash` match보다 시간/금액 tolerance 기반 corroboration이 중요하다.
- owner label은 address가 아닐 수 있으므로 `from_addr`, `to_addr`에 그대로 넣을 경우 dashboard에서 주소처럼 보이지 않게 해야 한다.
- `tg_whale_events`가 많아지면 24시간 window 또는 configurable lookback으로 제한해야 한다.

검증 기준:

- `tg_whale_events` fixture 1건 + chain event 1건이 같은 window/금액 tolerance에 있으면 `source="both"` signal 생성
- pipeline 결과의 `signal_count`가 TG 이벤트 병합 전후로 달라지는 테스트 추가
- `daily_brief.top_transactions` 또는 signal cards에 TG corroboration이 표시됨

## Phase 3. Listener 비동기 처리 안정화

목표:

- Telethon event loop가 LLM fallback 또는 Google Sheets I/O로 block되지 않게 한다.

단기 구현:

- `_handle_message()` 내부의 blocking 작업을 `asyncio.to_thread()`로 감싼다.
- storage append 실패는 warning/error로 기록하되 listener loop는 계속 살아있게 유지한다.

중기 구현:

- `asyncio.Queue` 도입
- handler는 message를 queue에 넣고 즉시 반환
- worker task가 queue에서 꺼내 parse/storage 처리
- queue size 제한과 drop/backpressure 정책 설정

권장 구조:

```text
Telethon NewMessage handler
→ queue.put_nowait(raw_message)
→ background worker
→ regex parse
→ optional LLM fallback
→ Sheets append
→ heartbeat/system_log update
```

검증 기준:

- `_handle_message`가 storage 지연 상황에서도 event loop를 장시간 점유하지 않음
- storage exception 발생 후에도 listener process가 종료되지 않음
- burst 메시지 테스트에서 append 호출이 순차 처리됨

## Phase 4. Listener heartbeat와 dashboard health 보정

목표:

- 운영자 대시보드가 "Sheets 연결됨"이 아니라 실제 worker 상태를 보여주게 한다.

작업:

1. listener start 시 `system_log`에 `run_type=telegram_listener`, `status=running` 또는 heartbeat 기록
2. 메시지 수신 시 `tg_whale_events.collected_at` 업데이트
3. dashboard metric 계산 시 최신 listener heartbeat 또는 최신 `tg_whale_events` timestamp 읽기
4. `buildServiceCards()`에서 listener card를 `sourceState`가 아니라 listener status로 판단

상태 산출 예:

| 조건 | 표시 |
|---|---|
| 최근 heartbeat 5분 이내 | 정상 |
| heartbeat는 있으나 최근 수신 없음 | 대기 중 |
| heartbeat 15분 이상 없음 | 확인 필요 |
| 인증 오류 기록 | 인증 필요 |
| env 누락 | 설정 필요 |

검증 기준:

- listener가 꺼져 있으면 dashboard에 `확인 필요` 표시
- 인증 실패 시 `인증 필요` 표시
- Sheets 연결만 되어 있어도 listener가 자동으로 정상 처리되지 않음

## 7. Definition of Done
