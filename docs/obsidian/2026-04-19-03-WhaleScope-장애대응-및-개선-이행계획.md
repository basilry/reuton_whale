## 1. [P0-A] `news_rss` 파이프라인 완전 실패

### 증상

```
src.utils.errors.StorageError: Failed to append news_feed:
APIError: [400]: Range (news_feed!K1) exceeds grid limits.
Max rows: 1000, max columns: 10
```
- `2026-04-18 16:32:45` 이후 매 15분 run_all 실행 시 `news_rss` 단일 잡이 실패.
- `signals`, `curated_balance`는 정상 실행. run_all 전체 상태는 `completed_with_errors`.
- 결과: `news_feed` 시트에 신규 행이 들어가지 않고, 기존 행의 `last_seen_at`도 갱신 불가 → 대시보드 뉴스 위젯의 `pipeline_stale` 경고가 정상적으로 뜨고 있다(이건 구현이 맞게 동작한 것).

### 근본 원인

`src/storage/sheets_client.py:1138`의 `_ensure_news_feed_schema`:

```python
cell_range = f"{start_col}1:{end_col}1"   # "K1:K1"
ws.update(cell_range, [missing], value_input_option="RAW")
```

- `NEWS_FEED_HEADERS`는 11개(`last_seen_at` 포함, 인덱스 0-10 → 열 A-K).
- 기존 `news_feed` 워크시트의 **그리드 물리 크기가 10열**(A-J)로 생성되어 있어 `K1`은 존재하지 않는 셀.
- gspread는 `values.update` 호출 전 그리드 자동 확장을 하지 않음 → 400 에러.
- 어제 리뷰의 B3 지적("헤더가 길 경우 덮어쓴다")과는 다른 경로이지만, **같은 근본 원인**(마이그 함수가 그리드 상태를 확인/보정하지 않음)의 자매 버그.

### 수정안

**패치 1 — 그리드 사이즈 확장**

```python
# src/storage/sheets_client.py, _ensure_news_feed_schema 내부

def _ensure_news_feed_schema(
    self, ws: "gspread.Worksheet", all_values: list[list[str]]
) -> None:
    if not all_values:
        # 빈 시트: 헤더만 써도 gspread가 그리드를 알아서 늘리지 않으므로 명시 확장.
        if ws.col_count < len(NEWS_FEED_HEADERS):
            ws.resize(cols=len(NEWS_FEED_HEADERS))
        ws.append_row(list(NEWS_FEED_HEADERS))
        return

    header = all_values[0]
    expected = list(NEWS_FEED_HEADERS)
    if header == expected:
        return

    # B3 방어: 헤더가 expected보다 길면 운영자 커스텀 컬럼 존재 → abort.
    if len(header) > len(expected):
        logger.warning(
            "news_feed header has unexpected extra columns (%d vs %d), "
            "skipping auto-extension. Manual review required: %s",
            len(header), len(expected), header,
        )
        return

    # 접두 일치 검사
    if expected[: len(header)] != header:
        logger.warning(
            "news_feed header layout unexpected, skipping auto-extension: %s",
            header,
        )
        return

    missing = expected[len(header):]
    if not missing:
        return

    # ★ 핵심 수정: 그리드 물리 크기 확장 (K1이 존재하도록)
    if ws.col_count < len(expected):
        ws.resize(cols=len(expected))
        logger.info(
            "Resized news_feed worksheet grid to %d cols",
            len(expected),
        )

    start_col = _column_letter(len(header) + 1)
    end_col = _column_letter(len(expected))
    cell_range = f"{start_col}1:{end_col}1"
    ws.update(cell_range, [missing], value_input_option="RAW")
    logger.info(
        "Extended news_feed header with columns: %s",
        ",".join(missing),
    )
```

**패치 2 — 원자성 강화(선택)**

현재 구조는 "read → compare → resize → update" 비원자적. 동시에 두 워커가 붙으면 레이스. 단기적으로는 `self._write_lock` 안에서 돌고 있으니 프로세스 내 동기화는 OK, 하지만 두 Render cron 인스턴스가 동시에 실행될 여지가 있으면 Sheets 수준에서는 보호 없음. 장기적으로 `service_health` 시트에 `migration_lock` row를 두는 방식 권장 (별도 PR).

## 2. [P0-B] 리뷰 블로커 B4·S3 병행 반영

### B4 — `getNewsFeedLastPollAt` 폴백 레이어링

**현재 (flat max)**: `apps/dashboard/lib/news.ts`
```ts
return Math.max(
  ...lastSeenAts.filter(Boolean).map(toMs),
  ...fetchedAts.filter(Boolean).map(toMs),
);
```

**수정 (staged fallback)**:
```ts
export function getNewsFeedLastPollAt(
  rows: NewsFeedRow[],
  systemLogRows: SystemLogRow[],
): string | null {
  // Layer 1: last_seen_at (신뢰도 1위, 파이프라인이 15분마다 갱신)
  const lastSeens = rows.map((r) => r.last_seen_at).filter(Boolean);
  if (lastSeens.length > 0) {
    return lastSeens.sort().reverse()[0];
  }
  // Layer 2: fetched_at (마이그 이전 데이터용)
  const fetcheds = rows.map((r) => r.fetched_at).filter(Boolean);
  if (fetcheds.length > 0) {
    return fetcheds.sort().reverse()[0];
  }
  // Layer 3: system_log 의 news_rss 엔트리
  const logs = systemLogRows
    .filter((r) => r.run_type === "news_rss")
    .map((r) => r.finished_at || r.started_at)
    .filter(Boolean);
  if (logs.length > 0) {
    return logs.sort().reverse()[0];
  }
  return null;
}
```

### 근거 (`src/pipeline/run_all.py:95-111`)

### 결정 (확정)

**→ 옵션 A 채택: 크론 주기 1시간으로 단축.** (기존 옵션 B "UI 배지"는 단기 완화책일 뿐 근본 해결이 아니라고 판단.)

### 수정안

**패치 1 — `due_job_names` 매시간 브리핑**

```python
def due_job_names(now: datetime | None = None) -> list[str]:
```

```python
def _should_skip_brief_due_to_inactivity(sheets, now: datetime) -> bool:
```

## 4. [P1-A] 실시간 시장 티커 — Upbit/fx/snapshot 이상

### 4-1. Upbit 칩 "중단" (배포 환경 한정)

**코드 위치**: `apps/dashboard/components/market-ticker-strip.tsx:579-660`

**연결 흐름**:
1. 클라이언트에서 `new WebSocket("wss://api.upbit.com/websocket/v1")` 생성
2. `onopen`에서 subscription 페이로드 전송
3. `onmessage`에서 ArrayBuffer 디코딩 → ticker 갱신
4. 2분 이상 메시지 미수신 시 `updateSource("upbit", {errorAt})` → 칩 상태 `down`

**추정 원인 (3가지)**

| 가설 | 가능성 | 검증 방법 |
|------|--------|----------|
| Upbit가 `Origin` 화이트리스트 정책으로 `reuton-whale-dashboard.vercel.app` 거부 | 🟠 중 | 브라우저 Network 탭에서 `101 Switching Protocols` 성공 여부 확인 |
| CSP(Content-Security-Policy) `connect-src`에 `wss://api.upbit.com` 누락 | 🔴 높음 | `next.config.ts`의 CSP 설정 검토 |
| Vercel Edge의 프록시 레이어가 WebSocket 업그레이드 방해 | 🟢 낮음 | 로컬에서 Upbit 직접 연결은 정상인지 재확인 |

**가장 유력**: **CSP `connect-src` 누락**. Next.js 프로덕션 빌드는 흔히 `connect-src 'self'`를 강제해서 외부 wss를 막는다.

### 4-2. fx/snapshot — 배포: 지연(stale) / 로컬: 중단(down)

**비대칭 증상의 의미**:
- **배포**: 초기 snapshot은 받았으나 후속 갱신이 느림 → 로더는 돌지만 업데이트 간격이 임계치 초과 → `stale`
- **로컬**: snapshot 페치 자체 실패 → `down`

**추정 원인**:
- **로컬**: `.env.local` 또는 `.env`에 fx API 키(`EXCHANGE_RATE_API_KEY`, `FRANKFURTER_API_URL` 등) 누락 → `fetch` 실패.
- **배포**: Vercel 환경변수는 설정되어 있어서 초기 1회 성공, 하지만 **Next.js ISR 캐시**가 TTL 길게 잡혀 있어서 갱신이 안 됨 → `stale`.

### 4-3. 수정안

**패치 1 — CSP `connect-src` 확장** (`next.config.ts` 또는 `middleware.ts`)

```ts
  "https://api.coingecko.com",     // fallback

const headers = [
  {
    key: "Content-Security-Policy",
    value: `default-src 'self'; connect-src ${CSP_CONNECT_SRC.join(" ")}; ...`,
  },
```

**패치 2 — 신규 fetch 함수 (snapshot 방식, REST polling)**

WebSocket 지원 여부와 인증 요구가 복잡하므로 1차는 **REST 스냅샷 폴링**(120s 주기).

```ts
// lib/market-ticker.ts
type BitflyerTickerResponse = {
  product_code: string;
  ltp: number;              // last traded price
  volume_by_product: number;
  timestamp: string;
  best_bid: number;
  best_ask: number;
};

export async function fetchBitflyerTickerSnapshot(
  definitions: MarketTickerDefinition[] = DEFAULT_MARKET_TICKER_SYMBOLS,
  options?: { signal?: AbortSignal },
): Promise<Map<string, BitflyerTickerQuote>> {
  const result = new Map<string, BitflyerTickerQuote>();
  await Promise.all(
    definitions
      .filter((d) => d.bitflyerProductCode)
      .map(async (d) => {
        const res = await fetch(
          `https://api.bitflyer.com/v1/ticker?product_code=${d.bitflyerProductCode}`,
          { cache: "no-store", signal: options?.signal },
        );
        if (!res.ok) return;
        const payload = (await res.json()) as BitflyerTickerResponse;
        result.set(d.id, {
          priceJpy: payload.ltp,
          updatedAt: new Date(payload.timestamp + "Z").toISOString(),
          // change24hPct는 별도 엔드포인트(/v1/getchats 또는 /v1/board) 필요 → 1차는 생략
        });
      }),
  );
  return result;
}

// Kraken은 multi-pair 한 번에 조회 가능
export async function fetchKrakenTickerSnapshot(
  definitions: MarketTickerDefinition[] = DEFAULT_MARKET_TICKER_SYMBOLS,
  options?: { signal?: AbortSignal },
): Promise<Map<string, KrakenTickerQuote>> {
  const pairs = definitions.map((d) => d.krakenPair).filter(Boolean).join(",");
  if (!pairs) return new Map();
  const res = await fetch(
    `https://api.kraken.com/0/public/Ticker?pair=${pairs}`,
    { cache: "no-store", signal: options?.signal },
  );
  if (!res.ok) return new Map();
  const payload = (await res.json()) as {
    error: string[];
    result: Record<string, {
      c: [string, string];   // last trade closed price
      v: [string, string];   // volume
      p: [string, string];   // vwap
      o: string;             // today's opening price
    }>;
  };
  const result = new Map<string, KrakenTickerQuote>();
  for (const d of definitions) {
    if (!d.krakenPair) continue;
    const entry = payload.result[d.krakenPair];
    if (!entry) continue;
    const last = Number(entry.c[0]);
    const open = Number(entry.o);
    const change24hPct = open > 0 ? ((last - open) / open) * 100 : null;
    result.set(d.id, {
      priceUsd: last,   // XBTUSD는 USD 페어
      change24hPct,
      updatedAt: new Date().toISOString(),
    });
  }
  return result;
}
```

**패치 5 — 김치 프리미엄 외 "와규 프리미엄" 지표 (선택)**

Binance BTC/USDT와 Bitflyer BTC/JPY × USDJPY(Frankfurter)를 비교해 JPY 프리미엄 계산. 대시보드 미니 위젯으로 노출하면 일본 투자자 관심을 끌 수 있는 시그너처 지표.

```ts
export function formatJapanPremium(
  binanceUsd: number | null | undefined,
  bitflyerJpy: number | null | undefined,
  usdJpy: number | null | undefined,
): string {
  if (!binanceUsd || !bitflyerJpy || !usdJpy) return "—";
  const bitflyerInUsd = bitflyerJpy / usdJpy;
  const pct = ((bitflyerInUsd - binanceUsd) / binanceUsd) * 100;
  return `${pct >= 0 ? "+" : ""}${pct.toFixed(2)}%`;
}
```

```ts
function sortWallets(entries: CuratedWalletEntry[]): CuratedWalletEntry[] {
  return [...entries].sort((a, b) => {
    const aPriority = a.displayPriority ?? a.priority;
    const bPriority = b.displayPriority ?? b.priority;
    if (aPriority !== bPriority) return aPriority - bPriority;
    // 개인 고래를 거래소보다 앞으로
    const categoryWeight = (c: CuratedWalletCategory): number =>
      c === "celebrity" ? 0
      : c === "founder" ? 1
      : c === "protocol_treasury" ? 2
      : c === "fund" ? 3
      : 5;
    return categoryWeight(a.category) - categoryWeight(b.category);
  });
}
```

```ts
// lib/whale-stories.ts — 카드 body 생성 helper
function buildStoryBody(
  tx: TransactionLike,
  fromLabel: string,
  toLabel: string,
): string {
  const amount = parseFloatSafe(tx.amount) ?? 0;
  const amountUsd = parseFloatSafe(tx.amount_usd) ?? 0;
  const symbol = compactString(tx.symbol).toUpperCase() || "TOKEN";
  const amountStr = formatTokenAmount(amount, symbol);  // 예: "12.5K ETH"
  const usdStr = formatCompactUsd(amountUsd);           // 예: "$38.4M"
  return `${fromLabel} → ${toLabel} • ${amountStr} (${usdStr})`;
}

// formatTokenAmount는 lib/format.ts에 신규 추가
export function formatTokenAmount(amount: number, symbol: string): string {
  if (!Number.isFinite(amount) || amount <= 0) return `- ${symbol}`;
  const notation = amount >= 1000 ? "compact" : "standard";
  const fractionDigits =
    amount >= 1000 ? 1 :
    amount >= 1 ? 2 :
    amount >= 0.01 ? 4 : 8;
  return `${new Intl.NumberFormat("en-US", {
    notation,
    maximumFractionDigits: fractionDigits,
  }).format(amount)} ${symbol}`;
}
```

   - curated match면 해당 지갑 클릭 시 §6 모달로 전환
4. **Amount 섹션**:
   - `amountToken` 원문 + `symbol`
   - `amountUsd` 환산 + 당시 환율(있으면)
   - 동일 심볼의 24h 평균 tx size 대비 비율 바 차트 (예: "이 이동은 평균의 8.2× 규모")
5. **Related Signals**: `supportingSignalIds`에 매칭된 시그널 카드 (최대 5개)
6. **Narrative 맥락 (선택)**: LLM이 생성한 한 문장 ("이 이동은 Vitalik의 최근 7일 평균보다 큰 분산 기부 패턴과 일치합니다.") — brief pipeline과 공유

**데이터 소스**:
- 기본 데이터는 `whale_stories` 생성 시 이미 알고 있는 값 → extra fetch 불필요.
- "평균 대비 배율"은 `/api/market/avg-tx-size?symbol=ETH&chain=ethereum` 신규 엔드포인트. `transactions` 시트에서 7d 평균 계산, 60s cache.

**상호작용**:
- 카드 전체에 `role="button"` + `tabindex="0"`.
- Enter/Space 키 또는 클릭으로 모달 오픈.
- 모달 내 "related signal" 클릭 시 해당 시그널 상세로 링크 이동 (동일 페이지 탭 또는 새 탭).

**디자인 상세**:
- From → To 사이에 애니메이션 화살표(길이 300ms, stagger 50ms) — 단, `prefers-reduced-motion`에서는 정적.
- Amount 섹션의 배율 바는 tone 색상으로 채움 (critical = red, watch = amber, positive = green, neutral = slate).
- Explorer 링크 아이콘은 Lucide `ExternalLink` 사용 (이모지 금지).

### 검증
- `whale-stories.ts` 단위 테스트: body 문자열에 symbol·amount·usd 포함 확인.
- 모달 렌더링: Storybook or Playwright test로 critical/watch/positive/neutral 4개 tone 스냅샷.

## 10. [P1-F] 실시간 자동 갱신 (뉴스·지갑·고래스토리 동기화)

### 증상
구글시트에 새 row가 append되어도 대시보드는 새로고침 전까지 stale. IR 심사 및 실사용 데모에서 "실시간 서비스"라는 서사와 모순.

### 아키텍처 제약

| 제약 | 영향 |
|------|------|
| 데이터 원천이 Google Sheets | **Sheets는 push notification 없음**. webhook/Apps Script trigger는 rate limit + 지연 커서 부적합. |
| Next.js on Vercel (serverless) | 장시간 연결(WebSocket)은 별도 서버 필요. Vercel Edge에서 SSE는 지원(max 300s per connection, 반복 재연결 가능). |
| Pipeline은 Render cron (15분) | 푸시 주체가 "서버 사이드 이벤트 생성자" 역할 필요. |

### 결정: **SSE(Server-Sent Events) + Pipeline → Upstash Redis pub/sub → Next.js 스트리밍 엔드포인트**

WebSocket은 양방향이 불필요한 구독형 데이터에 과한 복잡도. 폴링은 Google Sheets read quota 압박 + N×client 증가. SSE는 단방향, HTTP 스택 위에서 작동, Vercel Edge Runtime 지원.

### 데이터 흐름

```
[Render cron] run_all.py
   │ run 완료 시
   ▼
[Upstash Redis] PUBLISH whalescope:updates {"kind":"news","run_id":"...", "ts":"..."}
   │
   ▼
[Vercel Edge] /api/stream  (ReadableStream, Edge Runtime)
   │ SUBSCRIBE whalescope:updates
   │ (서버 이벤트를 EventSource 포맷으로 flush)
   ▼
[Browser] new EventSource("/api/stream")
   │ onmessage → kind 분기
   ▼
[React]
  kind === "news"         → router.refresh() 전체 또는 NewsWidget만 re-fetch
  kind === "whale_story"  → router.refresh()
  kind === "curated"      → CuratedWatchlist 재조회
  kind === "signal"       → Signal 카드 갱신
  kind === "brief"        → BriefPanel 재조회
```

### 이유: WebSocket만 가능한가?

**아니다.** 실시간 "푸시" 요구는 다음 세 가지 중 하나로 해결 가능:

| 방식 | 장점 | 단점 | 적합도 |
|------|------|------|-------|
| **WebSocket** | 양방향, 낮은 latency | Vercel Edge 제한적, Durable Object급 인프라 필요, 코드 복잡도 높음 | ❌ 과한 선택 |
| **SSE** | 단방향 구독에 최적, HTTP 기반, Vercel Edge 지원, 자동 재연결 | 일부 모바일 브라우저에서 BFCache 간섭 | ✅ **권장** |
| **Short polling** | 구현 단순 | Google Sheets quota, 클라이언트 N× 부하, latency 가변 | 🟠 Fallback |
| **`router.refresh()` + heartbeat API** | Next.js 13+ 에서 간단 | 여전히 polling, 세밀 제어 부족 | 🟠 2차 fallback |

**결론**: SSE가 구독형·단방향·Vercel 친화의 모든 조건을 만족. WebSocket은 오버엔지니어링이라 배제.

#### 패치 1 — Pipeline에서 이벤트 발행

```python
# src/pipeline/common.py 또는 신규 src/notify/event_bus.py
import os
import json
import time
from redis import Redis

_redis: Redis | None = None

def _redis_client() -> Redis | None:
    global _redis
    if _redis is not None:
        return _redis
    url = os.getenv("UPSTASH_REDIS_URL", "").strip()
    token = os.getenv("UPSTASH_REDIS_TOKEN", "").strip()
    if not url:
        return None
    _redis = Redis.from_url(url, password=token, decode_responses=True)
    return _redis

def publish_pipeline_event(kind: str, payload: dict) -> None:
    """kind: news | whale_story | curated | signal | brief"""
    client = _redis_client()
    if client is None:
        return
    try:
        message = json.dumps({"kind": kind, "ts": int(time.time()), **payload})
        client.publish("whalescope:updates", message)
    except Exception as exc:
        # 이벤트 발행 실패는 파이프라인을 막지 않는다 (best-effort)
        from src.utils.logger import get_logger
        get_logger("event_bus").warning("publish failed: %s", exc)

# run_all.py 각 잡 완료 직후 호출
if job_name == "news_rss" and outcome.get("status") == "completed":
    publish_pipeline_event("news", {"run_id": outcome.get("run_id")})
```

#### 패치 2 — Next.js Edge SSE 엔드포인트

```ts
// apps/dashboard/app/api/stream/route.ts
export const runtime = "edge";

import { Redis } from "@upstash/redis";

export async function GET(request: Request): Promise<Response> {
  const redis = new Redis({
    url: process.env.UPSTASH_REDIS_REST_URL!,
    token: process.env.UPSTASH_REDIS_REST_TOKEN!,
  });

  const encoder = new TextEncoder();
  const stream = new ReadableStream({
    async start(controller) {
      // 접속 확인 주석
      controller.enqueue(encoder.encode(": connected\n\n"));

      // Upstash는 Redis pub/sub 직접 지원 안 함 → SUBSCRIBE 대신 stream 기반
      // 실무: @upstash/redis의 `xread` 또는 별도 Upstash QStash 활용
      // 여기서는 "push key" 방식: run_all이 KV에 마지막 이벤트를 set하고
      // 클라이언트는 SSE로 주기(예: 5초) 체크 후 delta만 push
      let lastSeenTs = 0;
      const interval = setInterval(async () => {
        try {
          const latest = await redis.get<{ kind: string; ts: number }>("whalescope:last_update");
          if (latest && latest.ts > lastSeenTs) {
            lastSeenTs = latest.ts;
            controller.enqueue(encoder.encode(`data: ${JSON.stringify(latest)}\n\n`));
          }
        } catch {
          // silent
        }
      }, 5000);

      request.signal.addEventListener("abort", () => {
        clearInterval(interval);
        controller.close();
      });
    },
  });

  return new Response(stream, {
    headers: {
      "Content-Type": "text/event-stream",
      "Cache-Control": "no-cache, no-transform",
      Connection: "keep-alive",
      "X-Accel-Buffering": "no",
    },
  });
}
```

> Upstash Redis REST API는 pub/sub를 지원하지 않으므로 **"마지막 이벤트 key" 풀링** 방식으로 단순화. 클라이언트 한 명당 5초 interval이라 quota 부담 낮음. 만약 pub/sub가 꼭 필요하면 Upstash QStash 또는 Redis Cloud로 전환.

#### 패치 3 — 클라이언트 구독 훅

```tsx
// apps/dashboard/hooks/use-live-updates.ts
"use client";

import { useEffect } from "react";
import { useRouter } from "next/navigation";

type UpdateKind = "news" | "whale_story" | "curated" | "signal" | "brief";

export function useLiveUpdates(handlers?: Partial<Record<UpdateKind, () => void>>) {
  const router = useRouter();
  useEffect(() => {
    if (typeof window === "undefined") return;
    if (typeof EventSource === "undefined") return;

    const es = new EventSource("/api/stream");
    es.onmessage = (ev) => {
      try {
        const data = JSON.parse(ev.data) as { kind: UpdateKind; ts: number };
        const handler = handlers?.[data.kind];
        if (handler) handler();
        else router.refresh();  // 디폴트: 전체 서버 컴포넌트 재검증
      } catch {}
    };
    es.onerror = () => {
      // 브라우저가 자동 재연결 시도 (EventSource 기본 동작)
    };
    return () => es.close();
  }, [router, handlers]);
}
```

**장착 지점**: `app/providers.tsx` 또는 `app/layout.tsx`의 최상위 client component에서 훅 호출. 페이지 모든 서버 컴포넌트가 `router.refresh()`로 재검증됨.

#### 패치 4 — BFCache·멀티탭 대응

- 탭 비활성 시 `document.hidden` 감지해 EventSource close → 활성 전환 시 재오픈.
- 여러 탭 열린 경우 모든 탭이 각자 연결 — Upstash 5초 polling에서 per-connection 부담 ≈ 1 req/5s/tab. Dev 환경에서 수용 가능, Production에서 10k concurrent 이상이면 QStash 전환.

### 검증
- pipeline 완료 후 `redis.get("whalescope:last_update")` 값 변경 확인.
- SSE 수신 테스트: `curl -N https://.../api/stream` → 5초 간격 데이터 수신 로그.
- Chrome DevTools Network 탭에서 `/api/stream`이 "eventsource" 타입으로 연결 지속 확인.
- 테스트 시나리오: 브라우저 열어둔 상태에서 pipeline을 강제 실행 → 수초 내 카드 자동 갱신.
- `prefers-reduced-motion` 사용자도 동일하게 동작(모션은 갱신과 무관).

### 리스크
- **Upstash Redis 무료 tier 한계**: 10,000 commands/day. 5s interval × N users × 주요 페이지 = 충분히 쓸 수 있음 (N<50명 내 데모 환경 기준).
- **브라우저 백그라운드 탭**: Safari는 숨겨진 탭의 JS 타이머를 throttling. visibility API로 명시적 관리 필요.
- **SSE vs BFCache 간섭**: Next.js router.refresh 호출 시 cache invalidation 포인트 주의. `revalidatePath` 서버 액션과 궁합 확인.

### 수정안

#### 패치 1 — 신규 job `broadcast_periodic`

```python
# src/pipeline/broadcast_periodic.py (신규 파일)
from __future__ import annotations

import json
from datetime import datetime, timedelta, timezone
from zoneinfo import ZoneInfo

from src.notify.telegram_broadcast import TelegramBroadcastAdapter
from src.observability.service_health import append_service_heartbeat, pipeline_status_to_health
from src.pipeline.common import (
    build_sheets_client,
    coerce_json_list,
    init_run_result,
    load_pipeline_env,
    safe_float,
)
from src.storage.queries import now_iso
from src.utils.logger import get_logger

logger = get_logger("pipeline.broadcast_periodic")
_KST = ZoneInfo("Asia/Seoul")

def _collect_recent_movements(sheets, window_minutes: int = 15) -> list[dict]:
    """직전 N분의 amount_usd >= $100k 이동."""
    since = datetime.now(timezone.utc) - timedelta(minutes=window_minutes)
    rows = sheets.fetch_transactions_since(since)
    filtered = [r for r in rows if safe_float(r.get("amount_usd")) >= 100_000]
    filtered.sort(key=lambda r: safe_float(r.get("amount_usd")), reverse=True)
    return filtered[:3]

def _collect_top_signals(sheets, window_minutes: int = 15) -> list[dict]:
    since = datetime.now(timezone.utc) - timedelta(minutes=window_minutes)
    rows = sheets.fetch_signals_since(since)
    # severity가 높은 것부터, 동률이면 score 높은 것부터
    rows.sort(key=lambda r: (r.get("severity_weight", 0), safe_float(r.get("score"))), reverse=True)
    return rows[:3]

def _format_movement_line(row: dict) -> str:
    symbol = str(row.get("symbol", "TOKEN"))
    amount = safe_float(row.get("amount"))
    amount_usd = safe_float(row.get("amount_usd"))
    from_owner = str(row.get("from_owner", ""))[:20] or "unknown"
    to_owner = str(row.get("to_owner", ""))[:20] or "unknown"
    return f"• {amount:,.2f} {symbol} (${amount_usd:,.0f}) · {from_owner} → {to_owner}"

def _next_hourly_brief_time_kst(now: datetime) -> str:
    kst_now = now.astimezone(_KST)
    next_brief = (kst_now + timedelta(hours=1)).replace(minute=0, second=0, microsecond=0)
    return next_brief.strftime("%H:%M KST")

def _build_message(movements: list[dict], signals: list[dict], now: datetime) -> str:
    kst = now.astimezone(_KST).strftime("%Y-%m-%d %H:%M KST")
    lines = [f"<b>🐋 WhaleScope 15분 업데이트</b>  {kst}", ""]

    if movements:
        lines.append("<b>최근 15분 특이 이동</b>")
        for row in movements:
            lines.append(_format_movement_line(row))
    else:
        lines.append("<i>최근 15분간 $100k 이상의 특이 이동은 관측되지 않았습니다.</i>")

    lines.append("")
    if signals:
        lines.append("<b>Top 3 시그널</b>")
        for s in signals:
            summary = str(s.get("summary", ""))[:120]
            rule = str(s.get("rule", ""))
            lines.append(f"• <b>[{rule}]</b> {summary}")

    lines.append("")
    lines.append(f"<i>다음 정시 브리핑: {_next_hourly_brief_time_kst(now)}</i>")
    lines.append("")
    lines.append('<a href="https://reuton-whale-dashboard.vercel.app">대시보드 열기 →</a>')
    return "\n".join(lines)

def run_broadcast_periodic(*, force: bool = False) -> dict:
    result = init_run_result("broadcast_periodic")
    env = load_pipeline_env(require_telegram=True)
    sheets = build_sheets_client(env)
    now = datetime.now(timezone.utc)

    # dedup_key: 15분 슬롯 단위
    slot_key = now.strftime("%Y%m%dT%H") + f"{(now.minute // 15) * 15:02d}"
    movements = _collect_recent_movements(sheets)
    signals = _collect_top_signals(sheets)

    # 움직임도 시그널도 없으면 skip (채널 피로도 방지)
    if not movements and not signals and not force:
        result.update(status="skipped_empty", finished_at=now_iso(), details="no movements or signals")
        sheets.log_run(result)
        return result

    message = _build_message(movements, signals, now)
    broadcaster = TelegramBroadcastAdapter.from_env(env)
    attempt = broadcaster.broadcast_text(
        text=message,
        kind="broadcast_periodic",
        dedup_key=f"periodic:{slot_key}",
    )

    result.update(
        status="completed" if attempt.status in {"ok", "dry_run"} else "completed_with_errors",
        finished_at=now_iso(),
        details=f"telegram_status={attempt.status}",
    )
    sheets.log_run(result)
    append_service_heartbeat(
        sheets,
        service="pipeline.broadcast_periodic",
        component="pipeline",
        status=pipeline_status_to_health(result["status"]),
        heartbeat_key=str(result.get("run_id", "")),
        details={"status": result["status"], "message_size": len(message)},
        error="",
    )
    return result

if __name__ == "__main__":
    run_broadcast_periodic(force=True)
```

배포 환경(Render + Vercel)에서 값이 맞는지 대시보드 ops 섹션에서 확인 (`service_health.telegram.broadcast`).

#### 패치 5 — 디자인 가드

- 15분마다 포스팅하면 채널이 "스팸 봇"처럼 보일 위험 → 메시지 양식을 **상수 브랜딩**(헤더 `🐋 WhaleScope`) + **정량 수치 위주**로 유지.
- 메시지 길이 cap: 1,500자. 초과 시 Top 3 시그널만, movements는 1개로 축약.
- hour 24 기준 1일 최대 96건. 채널 구독자가 notification off 할 수 있도록 고지 (채널 pinned message에 안내).

### 검증
- `python -m src.pipeline.broadcast_periodic` (force=True)로 로컬 발송 테스트.
- `@whalescope_alertz`에 포맷 정상 표시 확인 (HTML parse_mode).
- Render cron 15분 슬롯에서 `run_all` 실행 로그에 `broadcast_periodic` 잡 출현.
- `service_health` 시트에 `pipeline.broadcast_periodic` 하트비트 기록.
- movements/signals 둘 다 없는 슬롯에서 skip 확인.
- 동일 hash 중복 방지 확인.

### 리스크
- **채널 구독자 이탈**: 15분 주기가 과할 수 있음. 2주 관측 후 30분 또는 동적 주기(시그널 있을 때만) 전환 옵션 열어둠.
- **Telegram API rate limit** 초과 시 429 응답 → 백오프 로직 필요. `TelegramBroadcastAdapter._clip_message` 옆에 `_retry_with_backoff` 추가.
- **브리핑 vs 주기 브로드캐스트 충돌**: 정시에 `brief` 잡과 `broadcast_periodic`이 동시 실행. 기존 `run_all`은 순차 실행이므로 문제 없음, 다만 전체 슬롯 처리 시간이 단일 슬롯 window(15분)를 넘지 않도록 측정 필요.

## 12. [P1-H] 크립토 공포탐욕지수 (Fear & Greed Index) 게이지

### 배경 — 기존 "안정/주의/관찰" 배지의 문제

현재 `app/page.tsx:256-370` `buildMarketMood()` 함수는 고래브리핑의 `marketMood` 구조화 데이터(`data.latestBrief.marketMood.mood = "risk_on" | "risk_off" | "neutral"`) 또는 signals/transactions 기반 fallback으로 "안정(Stable) / 주의(Caution) / 관찰(Watch)" 배지를 렌더한다 (`BriefAnalysisItem[0]`의 description/tone). 위치는 고래브리핑 카드 오른쪽.

**문제**:
- 데이터 출처가 고래브리핑 본문과 동일 → 같은 내용이 두 번 노출되어 정보 밀도가 낮다.
- 지표가 **내부 파이프라인 산출값**이라 "외부 시장 센티먼트"를 보여주지 못한다.
- "안정"이라는 단어가 시장이 실제로 어떠한지와 무관하게 signals 숫자만으로 결정되어 오해를 유발할 수 있다.

### 해결 — Alternative.me Fear & Greed Index 게이지로 교체

**선정 근거** (사용자 제공 레퍼런스 + 본 문서 판단):

| 후보 | 장점 | 단점 | 채택 |
|------|------|------|------|
| **Alternative.me** | 원조·무료·API key 불필요·BTC 중심 · 알고리즘 공개(Volatility 25 / Volume 25 / Social 15 / Dominance 10 / Trends 10 / Survey 15) | BTC 위주 | **주(primary)** |
| CoinMarketCap | Top 10 종합 · Bloomberg/CNBC 인용 · 실시간 갱신 | API key 필요 · quota 관리 | M2 이후 선택적 병용 |
| CoinGlass | 파생상품 데이터 반영 | 공식 API 문서 불명확 | M3 이후 검토 |

### API 계약 (Alternative.me)

실측 엔드포인트 (`curl https://api.alternative.me/fng/?limit=30&format=json`):

```json
{
  "name": "Fear and Greed Index",
  "data": [
    {
      "value": "27",
      "value_classification": "Fear",
      "timestamp": "1776556800",
      "time_until_update": "71456"
    },
    { "value": "26", "value_classification": "Fear", "timestamp": "1776470400" },
    { "value": "21", "value_classification": "Extreme Fear", "timestamp": "1776384000" }

  "metadata": { "error": null }

```

규격:
- `value`: `"0"~"100"` 문자열 (정수 권고, parseInt 필수).
- `value_classification`: `"Extreme Fear" | "Fear" | "Neutral" | "Greed" | "Extreme Greed"`.
- `timestamp`: Unix epoch(초), UTC 자정 기준.
- `time_until_update`: 다음 업데이트까지 남은 초. **최신 row에만 포함**.

권장 파라미터: `?limit=30&format=json` — 30일치 히스토리를 한 번의 호출로 받아 "어제 / 7일 전 / 30일 전" 비교 그래프를 즉시 그릴 수 있다.

#### 12-1. 데이터 페처 (`lib/fear-greed.ts`)

```typescript
// apps/dashboard/lib/fear-greed.ts
export type FearGreedClassification =
  | "extreme_fear" | "fear" | "neutral" | "greed" | "extreme_greed";

export type FearGreedSnapshot = {
  value: number;                 // 0-100
  classification: FearGreedClassification;
  takenAt: string;               // ISO
};

export type FearGreedData = {
  current: FearGreedSnapshot;
  yesterday?: FearGreedSnapshot;
  weekAgo?: FearGreedSnapshot;
  monthAgo?: FearGreedSnapshot;
  nextUpdateAt?: string;         // ISO — 현재값 기준
  source: "alternative_me" | "stale_cache";
  fetchedAt: string;
};

const ENDPOINT = "https://api.alternative.me/fng/?limit=30&format=json";
const REVALIDATE_SECONDS = 600;  // 10분

function toClassification(s: string): FearGreedClassification {
  switch (s) {
    case "Extreme Fear": return "extreme_fear";
    case "Fear":         return "fear";
    case "Neutral":      return "neutral";
    case "Greed":        return "greed";
    case "Extreme Greed":return "extreme_greed";
    default:             return "neutral";
  }
}

function toSnapshot(row: { value: string; value_classification: string; timestamp: string }): FearGreedSnapshot {
  return {
    value: Math.max(0, Math.min(100, parseInt(row.value, 10) || 0)),
    classification: toClassification(row.value_classification),
    takenAt: new Date(parseInt(row.timestamp, 10) * 1000).toISOString(),
  };
}

export async function getFearGreedData(): Promise<FearGreedData | null> {
  try {
    const res = await fetch(ENDPOINT, {
      next: { revalidate: REVALIDATE_SECONDS, tags: ["fear-greed"] },
      // Edge 런타임에서도 동작하는 기본 fetch
    });
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const json = await res.json() as {
      data: Array<{ value: string; value_classification: string; timestamp: string; time_until_update?: string }>;
      metadata?: { error: string | null };
    };
    if (json.metadata?.error) throw new Error(json.metadata.error);
    const rows = json.data;
    if (!rows?.length) throw new Error("empty data");

    const current = toSnapshot(rows[0]);
    const yesterday = rows[1] ? toSnapshot(rows[1]) : undefined;
    const weekAgo   = rows[6] ? toSnapshot(rows[6]) : undefined;
    const monthAgo  = rows[29] ? toSnapshot(rows[29]) : undefined;

    const secondsUntil = parseInt(rows[0].time_until_update ?? "0", 10);
    const nextUpdateAt = secondsUntil > 0
      ? new Date(Date.now() + secondsUntil * 1000).toISOString()
      : undefined;

    return {
      current, yesterday, weekAgo, monthAgo, nextUpdateAt,
      source: "alternative_me",
      fetchedAt: new Date().toISOString(),
    };
  } catch (err) {
    console.error("[fear-greed] fetch failed", err);
    return null;  // 상위에서 fallback 처리
  }
}
```

**캐시 정책**: `next.revalidate = 600` (10분). Alternative.me는 12시간 단위 갱신이므로 10분 캐시는 사용자 체감상 "실시간"이면서 서버 호출은 일 144회 이하로 제한 → 외부 API 부하 전혀 없음.

**실패 처리**: `getFearGreedData()`가 `null` 반환 시 `app/page.tsx`에서 기존 `buildMarketMood()` 결과로 graceful fallback. 즉 **본 기능 장애가 페이지 전체를 깨뜨리지 않는다**.


**게이트 조건 (절대적)**:
1. M0 완료 없이 M1 시작 금지. `news_rss` 복구되지 않으면 전체 동결.
2. M1 트랙 A (티커) close code가 `1008 Policy Violation`이면 Upbit REST fallback 영구화 + 심사보고서에 명시.
3. M3 SSE는 반드시 단독 PR — 실패 시 즉시 롤백 가능하도록 다른 기능과 섞지 않는다. Upstash 비용 모니터링 24h 선행.
4. M4 `broadcast_periodic`은 반드시 **dry-run 24h → 실전환**. 첫 48시간 skip rate / 비용 / 구독 해지율 3가지 모두 게이트 통과해야 live.

**롤백 플랜**:
- M1~M2: 개별 feature flag로 OFF 가능 (`NEXT_PUBLIC_ENABLE_BITFLYER`, `NEXT_PUBLIC_ENABLE_FEAR_GREED` 등). P1-H는 `FearGreedGauge` 내부 `data === null` 경로가 곧 fallback이므로 env OFF 시 즉시 기존 mood 배지로 복귀.

- M4: `brief` 크론은 8h로 복원, `broadcast_periodic`은 `due_job_names`에서 제거.
