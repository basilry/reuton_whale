# WhaleScope 다관점 리뷰 기반 개선계획

## 1. 목적

이 문서는 `[[2026-04-16-03-WhaleScope-다관점-코드리뷰-및-개선내역]]`에서 확인된 High, Medium, Low finding을 실제 개발 순서로 재정렬한 실행 계획이다.

목표는 단순한 코드 정리가 아니라 뤼튼 Product Engineer 과제 제출 기준에서 다음 3가지를 안정화하는 것이다.

- 실제 서비스처럼 동작하는 정보수집, 시그널, AI 브리핑, 대시보드 흐름
- 운영자/사용자 화면에서 오해를 줄이는 상태 표시와 에러 처리
- 커밋/배포 전 회귀 위험을 줄이는 테스트와 문서화

## 2. 현재 상태 요약

이미 반영된 주요 개선:

- Telegram listener 수집 데이터가 `tg_whale_events`에서 daily pipeline으로 다시 들어간다.
- 운영자 API에 `DASHBOARD_PASSWORD` 기반 인증과 generic error response가 적용됐다.
- `/insights` 사용자 화면은 내부 에러 메시지를 노출하지 않는다.
- Telethon listener의 blocking LLM/Sheets I/O는 `asyncio.to_thread()`로 분리됐다.
- listener heartbeat가 `system_log`에 기록되고, 운영 대시보드는 이를 기준으로 listener 상태를 표시한다.
- `/insights` story list의 duplicate React key 경고가 수정됐다.

남은 핵심 리스크:

- TG 이벤트 direction 추론이 모호한 edge case가 있다.
- listener start heartbeat가 Telegram 연결 성공 전 기록될 수 있다.
- Signal 기반 top transaction 저장 구조, Telegram 개인화, analysis_log telemetry는 이전 Major finding으로 남아 있다.
- dashboard 파일이 커지고 중복 formatter/type이 생겨 유지보수성이 낮아지고 있다.

## 3. 개선 원칙

### 3-1. 과제 제출 우선순위

과제 제출 관점에서는 "멋진 화면"보다 "실제로 end-to-end로 연결된 서비스"가 우선이다.

우선순위:

1. 잘못된 데이터/상태를 보여주는 문제 제거
2. 사용자에게 AI 요약 가치가 보이는 문제 해결
3. 운영자가 배포 후 장애 원인을 추적할 수 있게 만들기
4. 유지보수성 개선

### 3-2. 구현 원칙

- 현재 MVP 저장소인 Google Sheets는 유지한다.
- Render worker와 Vercel dashboard 경계는 유지한다.
- 대규모 리팩터링보다 high-risk bug를 먼저 제거한다.
- 리뷰 finding 해결마다 테스트 또는 명시적 검증 명령을 붙인다.
- secret, session, raw upstream error는 문서와 UI에 노출하지 않는다.

```typescript
import { timingSafeEqual } from "node:crypto";

function safeEqual(left: string, right: string): boolean {
  const leftBuffer = Buffer.from(left);
  const rightBuffer = Buffer.from(right);
  return leftBuffer.length === rightBuffer.length && timingSafeEqual(leftBuffer, rightBuffer);
}
```

Definition of Done:

### 7-1. `_signals_to_top5()` signal-first 구조

문제:

- aggregate signal은 특정 tx hash나 대표 거래가 없을 수 있다.
- 기존 top transaction 저장 구조는 hash/symbol/amount를 거래 중심으로 기대해 빈 값이 생길 수 있다.

계획:

- top item 모델을 `transaction-like item`이 아니라 `signal evidence item`으로 재정의한다.
- signal 자체 필드(`rule`, `severity`, `score`, `summary`, `source`, `confidence`)를 우선 표시한다.
- evidence event가 있으면 부가 정보로 붙인다.
- evidence가 없어도 dashboard에 의미 있는 signal card가 나오게 한다.

검증:

- signal만 있고 tx hash가 없는 경우에도 `top_transactions` JSON에 summary/rule/source가 채워진다.
- dashboard에서 `UNKNOWN`, `0 USD`, 빈 hash가 주요 card로 보이지 않는다.

### 7-2. Telegram 개인화와 LLM brief 보존

문제:

- 개인화 경로가 LLM brief를 우회하면 사용자에게 "AI 요약/큐레이션" 가치가 약해진다.

계획:

- 기본 daily brief text는 보존한다.
- 개인화 신호는 brief 뒤에 "내 관심 항목" 섹션으로 append한다.
- 사용자별 watchlist/interests 기반 필터링은 보조 레이어로 둔다.

검증:

- `WhaleScopeBot.send_daily_brief()` 또는 formatting test에서 LLM brief 본문이 유지된다.
- 관심 코인 사용자는 개인화 섹션을 추가로 받는다.

### 7-3. analysis_log telemetry schema 갱신

문제:

- LLM 비용, latency, token usage 추적이 제대로 되지 않으면 운영성과 비용 관리가 어렵다.

계획:

- `ANALYSIS_LOG_HEADERS`와 실제 저장 payload를 맞춘다.
- LLM provider별 `model_id`, `tokens_in`, `tokens_out`, `cost_usd`, `latency_ms`를 저장한다.
- Gemini/Groq 등 fallback provider 비용 추정값이 없다면 `cost_usd=""` 또는 `unknown`으로 구분한다. `0.0`은 실제 무료와 구분이 어렵다.

검증:

- `scripts/smoke_pipeline.py` 또는 pipeline dry-run에서 analysis_log row가 schema에 맞게 저장된다.
- cost unknown과 zero cost가 구분된다.

## 14. 최종 판단

현재 WhaleScope는 핵심 아키텍처가 제출 가능한 수준에 가까워졌지만, 리뷰 기준으로는 아직 "운영 정확도"와 "AI 요약 가치 전달"에서 남은 gap이 있다.

가장 먼저 해야 할 것은 Phase 0이다. 이 단계는 수정량이 작고 회귀 위험도 낮지만, 커밋 전 코드 품질과 운영 신뢰도를 바로 올린다.

그 다음 과제 제출 품질에 가장 큰 영향을 주는 것은 Phase 3이다. 특히 `_signals_to_top5()`의 signal-first 구조와 Telegram brief_text 보존은 사용자에게 실제 AI 큐레이션 서비스처럼 보이게 만드는 핵심이다.
