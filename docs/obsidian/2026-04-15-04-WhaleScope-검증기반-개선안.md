## 3. 검증 과정에서 확인한 주요 사실

### 3-1. 대시보드 에러

사용자가 Streamlit 화면에서 아래 오류를 확인했다.

```text
requests.exceptions.ConnectionError:
('Connection aborted.', RemoteDisconnected('Remote end closed connection without response'))
```

발생 위치:

- `streamlit_app.py`
- `load_transactions()`
- `ss.worksheet(TAB_TRANSACTIONS)`
- `gspread.spreadsheet.fetch_sheet_metadata()`

해석:

- 대시보드 자체 렌더링 문제가 아니라 Google Sheets metadata 조회 중 네트워크 연결이 끊긴 것이다.
- 외부 저장소를 직접 읽는 Streamlit app은 이 예외를 사용자 화면에 그대로 노출하면 안 된다.

이미 적용한 개선:

- `get_spreadsheet()` 예외 방어
- `load_transactions()` 예외 방어
- `load_daily_briefs()` 예외 방어
- 실패 시 빈 DataFrame 반환 + Streamlit warning 표시

추가 개선 필요:

- Google Sheets read retry 추가
- 마지막 정상 로드 결과를 session/cache에 보관
- dashboard 상단에 "데이터 소스 상태" 표시

### 3-2. 대시보드 버튼

요구사항:

- 대시보드에 버튼을 만들어 `python -m src.main`을 실행한다.

이미 적용한 개선:

- Streamlit sidebar에 `운영 액션` expander 추가
- `일일 파이프라인 실행` 버튼 추가
- 내부에서 `subprocess.run([sys.executable, "-m", "src.main"])` 실행
- stdout/stderr tail 표시
- timeout 처리
- 성공 시 cache clear

현재 한계:

- Streamlit 프로세스 안에서 long-running batch job을 직접 실행하는 구조라 UI request가 오래 묶인다.
- 300초 timeout으로 보호하지만, 운영 관점에서는 별도 job queue가 더 적합하다.
- Vercel로 옮길 경우 이 방식은 적합하지 않다.

권장 후속:

- 단기: 버튼은 유지하되 timeout과 로그 표시로 운영 확인용으로 사용
- 중기: `pipeline_runs` 탭 또는 파일 lock으로 중복 실행 방지
- 장기: Vercel/Next.js 전환 시 API route에서 job queue 또는 external runner trigger로 변경

### 3-3. 온체인 수집 지연

초기 관찰:

- `python -m src.main` 실제 실행 시 Stage 3 `Collecting whale events`에서 5분 이상 정체
- 로그가 없어 멈춘 것처럼 보임

원인 1: Etherscan 전체 히스토리 조회

기존 collector는 주소마다 다음 방식으로 조회했다.

- `startblock=0`
- `endblock=99999999`
- `sort=desc`
- page/offset 없음
- 응답 결과 최대 10,000행을 받아온 뒤 로컬에서 24시간 필터링

단일 주소 검증 결과:

| action | 기존 시간 | rows | recent24h |
|---|---:|---:|---:|
| `txlist` | 약 11.5초 | 10,000 | 7 |
| `tokentx` | 약 11.3초 | 10,000 | 27 |

즉 주소 1개만 약 22초가 걸릴 수 있고, ETH 주소 78개에 대해 2개 endpoint를 직렬 호출하면 전체 시간이 수십 분까지 늘어난다.

이미 적용한 개선:

- Etherscan `block/getblocknobytime`로 24시간 전 block 조회
- `startblock`에 해당 block을 넣어 최근 범위만 조회
- `page`, `offset`, `max_pages` 도입
- 단일 주소 기준 약 3.5초에 34 events 수집 확인

남은 문제:

- 여전히 78개 ETH 주소를 직렬 처리한다.
- chain별/주소별 진행률 로그가 없다.
- 실패한 주소와 오래 걸리는 주소가 구분되지 않는다.

권장 후속:

- 주소 병렬 처리 도입. 단, Etherscan rate limit을 고려해 concurrency 2-3부터 시작
- chain별 수집 결과 count와 elapsed 로그 추가
- 주소별 timeout과 실패 기록 추가
- demo용 lookback/window 옵션 분리

### 3-4. CoinGecko rate limit

실제 pipeline 300초 제한 실행에서 다음 로그가 확인됐다.

```text
CoinGecko rate limited (429)
Enrichment failed (continuing): CoinGecko rate limited (429)
```

원인:

- 이벤트 정규화 중 `PriceService.get_usd()`가 심볼별 가격을 CoinGecko에 조회한다.
- 실패 결과를 캐시하지 않아 429가 난 심볼도 이벤트마다 반복 요청했다.
- Stage 4 `CoinGeckoEnricher`도 별도 가격 보강을 수행해 추가 429가 발생했다.

이미 적용한 개선:

- `PriceService`에 `_miss_cache` 추가
- CoinGecko 실패도 TTL 동안 negative cache
- 같은 심볼 반복 실패 요청을 줄임

남은 문제:

- Stage 4의 `CoinGeckoEnricher`는 별도 retry/backoff를 수행하며 여전히 429로 시간을 쓴다.
- 정규화 단계와 enrich 단계가 중복으로 가격 API를 호출한다.
- 가격이 없어도 signal/transaction 저장은 먼저 가능해야 하는데 현재 저장은 Stage 9에 몰려 있다.

권장 후속:

- 가격 enrich를 optional best-effort로 격리
- 수집 원천 데이터는 가격 실패와 무관하게 먼저 저장
- `PriceService`와 `CoinGeckoEnricher` 통합 또는 shared cache 적용
- 주요 심볼은 static fallback price 또는 "unknown" 허용

### 3-5. 실제 파이프라인 진행 상황

최적화 후 실제 실행은 다음 단계까지 진행됐다.

```text
Stage 3: Collecting whale events
Stage 4: Enriching with market data
Enrichment failed (continuing): CoinGecko rate limited (429)
Stage 5: Running SignalEngine
SignalEngine produced 0 signals
Stage path: signals=0, using legacy_fallback
Stage 6: Pre-filtering transactions
Pre-filtered 3587 -> 30 transactions
Stage 7: Analyzing 30 candidates
```

이후 300초 timeout으로 종료됐다.

중요한 해석:

- 온체인 이벤트 수집 자체는 이제 된다. 3,587개 transaction 후보까지 왔다.
- 하지만 signal이 0건이라 legacy fallback으로 넘어갔다.
- legacy fallback은 30개 후보를 LLM으로 개별 분석하려 해서 오래 걸린다.
- 저장은 Stage 9라서, Stage 7에서 timeout되면 dashboard에 아무 데이터도 쌓이지 않는다.

즉 현재 가장 큰 구조적 문제는 "수집과 저장이 LLM 분석 뒤에 있다"는 점이다.

## 4. Signal dashboard 데이터가 없는 이유

현재 `signals`가 0건인 이유는 단순히 UI가 없어서만은 아니다. 데이터 생성 조건도 맞지 않고, 저장 경로도 부족하다.

### 4-1. SignalEngine 결과가 0건

실행 로그:

```text
Loaded 0 baseline bucket(s)
SignalEngine produced 0 signals
```

원인:

- `address_activity`가 0건이라 baseline 기반 spike rule이 의미 있게 작동하지 않는다.
- `tg_whale_events`가 0건이라 Telegram 기반 burst/corroboration rule이 작동하지 않는다.
- 현재 seed category는 `cex`, `smart_money`, `token_whale`, `bridge` 중심이고 `hot` category가 없어 `cold_to_hot_transfer` 조건이 잘 맞지 않는다.
- `smart_money_accumulation` rule은 `RuleContext.watched_index`를 참조하지만 현재 engine context에 watched_index가 충분히 주입되지 않는다.

### 4-2. signals 탭 저장이 불충분

`SheetsClient.append_signal()`은 구현되어 있다. 하지만 `src.main`에서 SignalEngine 결과를 `signals` 탭에 저장하는 호출이 없다.

결과:

- signal이 생성되어도 dashboard용 `signals` 탭에는 안 쌓일 수 있다.
- daily brief의 `top_transactions` JSON에는 signal metadata가 들어가지만, 별도 signal dashboard를 만들기에는 부족하다.

### 4-3. Streamlit에 signals UI가 없음

현재 Streamlit tab:

- `오늘의 브리핑`
- `거래 히스토리`
- `통계`

signals 전용 tab은 없다.

따라서 signal dashboard를 만들려면 다음이 모두 필요하다.

1. `signals` 탭에 SignalEngine 결과 저장
2. `streamlit_app.py`에서 `load_signals()` 구현
3. `시그널` 탭 추가
4. rule/severity/source/score/evidence_count 필터 제공

## 5. Telegram 인사 무응답 원인

검증 결과:

```text
webhook_url_set: False
pending_update_count: 2
updates_returned: 2
```

해석:

- Telegram Bot API는 정상이다.
- webhook은 설정되어 있지 않다.
- pending update가 있다는 것은 bot이 메시지를 받아가지 않았다는 뜻이다.
- 즉 `scripts/run_bot.py` long polling 프로세스가 실행 중이지 않았을 가능성이 높다.

추가 원인:

- 기존 bot은 `CommandHandler`만 등록했다.
- `/start`, `/watchlist`, `/pause`, `/status`만 처리한다.
- 일반 텍스트 인사에는 handler가 없어서, bot이 실행 중이어도 답하지 않는다.

이미 적용한 개선:

- 일반 텍스트 fallback handler 추가
- 사용자가 “안녕”처럼 입력하면 `/start`, `/watchlist` 안내를 답하도록 수정

남은 운영 조건:

- 실제 답장을 받으려면 반드시 아래 프로세스가 떠 있어야 한다.

```bash
python scripts/run_bot.py
```

주의:

- `python -m src.main`은 daily brief 발송용이다.
- `scripts/run_bot.py`는 사용자 명령/인사 응답용 long polling이다.
- 둘은 역할이 다르다.

## 6. 우선순위별 개선안

### P0. 수집 원천 데이터 선저장

문제:

- 현재 Stage 9에서 저장한다.
- LLM 분석이 느리거나 timeout되면 수집한 3,587개 이벤트도 dashboard에 남지 않는다.

개선:

- Stage 3 수집 직후 `address_activity`와 `transactions`를 먼저 저장한다.
- LLM 분석/브리핑은 그 다음 단계로 둔다.
- 저장 성공 후 LLM이 실패해도 dashboard에는 거래 히스토리가 표시되어야 한다.

기대 효과:

- dashboard 빈 화면 문제 즉시 완화
- 운영 중 LLM 장애와 데이터 수집 장애를 분리
- 과제 데모에서 “실제 데이터가 들어온다”를 먼저 증명 가능

구현 순서:

1. `src.main` Stage 3 직후 raw event 저장 함수 추가
2. 중복 저장 방지를 기존 `append_transactions`, `append_address_activity`에 맡김
3. Stage 9에서는 daily brief/system log 중심으로 축소
4. 테스트에서 LLM timeout/실패 시에도 transactions 저장 확인

### P0. legacy fallback LLM 개별 분석 제거 또는 제한

문제:

- signal이 0건이면 30개 transaction을 개별 LLM 분석한다.
- 이 경로가 timeout의 직접 원인이다.

개선:

- fallback에서는 LLM 개별 분석을 하지 않고 규칙 기반 score만으로 top5 생성
- 또는 최대 3개만 LLM 분석
- daily brief는 top5 묶음을 한 번에 `generate_daily_brief()`로 처리

기대 효과:

- pipeline 완료 시간 급감
- LLM 비용 감소
- dashboard 데이터 생성 안정화

권장 정책:

- 과제 데모에서는 `signal-first`, `fallback-summary-only`가 적합하다.
- LLM을 30번 호출하는 방식은 MVP에 과하다.

## 9. 결론

현재 프로젝트는 외부 연결과 기본 구조는 살아 있다. 문제는 “아예 안 된다”가 아니라, 운영 파이프라인의 순서와 병목 때문에 dashboard에 표시될 데이터가 저장 단계까지 도달하지 못하는 것이다.

가장 먼저 고쳐야 할 것은 LLM이 아니라 저장 순서다.

핵심 개선 방향:

1. 수집 원천 데이터는 LLM 전에 먼저 저장한다.
2. legacy fallback에서 transaction별 LLM 30회 호출을 제거하거나 제한한다.
3. signal 결과를 `signals` 탭에 저장하고 dashboard tab을 만든다.
4. Telegram bot은 long polling 프로세스를 별도로 켜고 일반 텍스트 handler를 유지한다.
5. Vercel 배포는 Streamlit 유지가 아니라 Next.js 전환으로 해결한다.

이 순서대로 가면 대시보드 빈 화면 문제, signal dashboard 부재, Telegram 무응답 문제를 각각 독립적으로 해소할 수 있다.
