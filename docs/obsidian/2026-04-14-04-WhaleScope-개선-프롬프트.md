# WhaleScope 개선 프롬프트 (Claude Code용)

2차 리뷰의 신규/잔존 이슈 N1–N10을 한 번의 Claude Code 세션에서 처리하도록 설계된 프롬프트. 우선순위별로 3개 트랙으로 분리. 순서대로 실행하거나, 트랙별 독립 PR로 쪼개도 OK.

---

## 사용법

Claude Code 세션 시작 후 트랙 하나를 통째로 복사해 붙여넣는다. 트랙 사이에는 `git status` 클린 상태 + 테스트 통과를 확인하고 다음 트랙으로 이동.

각 트랙은 "원인 → 범위 → 수락 기준 → 금지 사항" 순서로 구성. 파일 경로/라인은 리뷰 기준이며 변경 시 Grep으로 재탐색할 것.

## 트랙 B — 운영 안정성 (반나절)

```
WhaleScope 프로젝트에서 운영 신뢰성에 직접 영향을 주는 3건을 처리한다.
feature/track-b-ops 브랜치에서 작업.

[N1] Claude 분석 캐시 N+1 제거
- 파일: src/analyzer/claude_analyzer.py
- 현재: _load_from_sheets_cache 가 매 호출마다 analysis_log 전체 get_all_values() 수행.
  Top5 루프에서 5회 시트 읽기 → 쿼터/레이턴시 리스크.
- 수정 방향:
  1. ClaudeAnalyzer 인스턴스에 `self._cache: dict[str, dict] | None = None` 필드 추가
  2. 첫 analyze 호출 시 한 번만 시트 전체를 읽어 prompt_hash 기준 dict으로 구성
  3. 이후 lookup은 self._cache.get(prompt_hash) 로 O(1)
  4. _persist_to_sheets 직후 self._cache 에도 갱신 반영
  5. sheets 가 None 일 때는 캐시 비활성(기존 동작 유지)
- 수락 기준:
  - 같은 pipeline 실행 내에서 analysis_log 시트 전체 읽기가 최대 1회
  - 캐시 hit/miss 로깅 추가
  - 기존 unit test 통과 + 캐시 miss→hit 전환 시나리오 테스트 추가

[N2] append_transactions race condition 완화
- 파일: src/storage/sheets_client.py
- 현재: existing_hashes 조회 → new_rows 계산 → append_rows. 2단계가 atomic 하지 않음.
- 수정:
  1. @retry 안쪽에서 append 직전 existing_hashes를 한 번 더 재조회
  2. 재조회 결과로 new_rows를 한 번 더 필터링(있으면 제거)
  3. 로그에 "refiltered=N" 추가
  4. 완전 해소는 아니며 "Postgres 이행 시 트랜잭션으로 대체" TODO 주석 남길 것
- 수락 기준:
  - 동일 tx를 2회 연속 append_transactions 해도 중복 row 생성되지 않는 케이스를 시뮬레이션한 테스트 추가
    (첫 호출 후 두 번째 호출에서 existing_hashes가 업데이트되어 있음을 mock으로 표현)

[N7] Streamlit 비밀번호 brute-force 완화
- 파일: streamlit_app.py
- 현재: _check_password() 단순 문자열 비교
- 수정:
  1. st.session_state 에 실패 카운터와 last_attempt_at 유지
  2. 3회 이상 실패 시 잠금: 5분간 재시도 불가. 남은 시간 카운트다운 표시
  3. 매 실패 후 time.sleep(1)
  4. 비번 로딩 실패(env 미설정) 시 "운영자에게 문의" 메시지 후 st.stop()
- 수락 기준:
  - 수동 시나리오: 잘못된 비번 3회 → 잠금 문구 → 5분 후 다시 시도 가능
  - 잠금 상태에서도 앱 로직이 절대 실행되지 않음

작업 규칙:
- 각 이슈 독립 커밋.
- N1/N2는 unit test 필수. N7은 수동 확인 + docstring.
- 마지막에 `pytest -q` 결과 보고.
```

## 트랙 C — 설계 부채 (1~2일)

```
WhaleScope 프로젝트 설계 부채 3건을 정리한다. 리팩터링 성격이므로 별도 PR 권장.

[N3] 이중 스코어링 시스템 통합
- 배경: scorer.pre_filter 가 base_score를 부여하고, Claude 분석이 importance_score를 부여한다.
  폴백 때만 base_score→importance_score 이관, 평시엔 base_score가 버려진다.
- 목표: 산출물에 쓰이는 키를 importance_score 하나로 통일하되 base_score의 "규칙 기반 1차 점수"라는 의미를 보존한다.
- 구현:
  1. scorer.pre_filter 는 기존대로 base_score 부여(내부 기준으로 사용)
  2. ClaudeAnalyzer.analyze_batch 가 Claude 결과를 적용할 때
     `tx["importance_score"] = claude_score` 로 덮어쓰되, 덮어쓰기 전 값을 `tx["rule_score"]` 로 보존
  3. main.py Step 9에서 top_transactions 에 rule_score도 같이 저장
  4. README "스코어링 방식" 섹션 신설(규칙 점수 → AI 최종 점수 흐름 도식화)
- 수락 기준:
  - 모든 저장 포맷/로그에서 base_score 키 제거 또는 rule_score로 renamed
  - 폴백 경로에서도 importance_score == rule_score 로 일관

[N4] 데일리 브리프 프롬프트 외부화
- 파일: src/analyzer/claude_analyzer.py generate_daily_brief
- 문제: 시스템 프롬프트/유저 프롬프트가 메서드 안에 하드코딩
- 수정:
  1. prompts/daily_brief.system.txt, prompts/daily_brief.user.txt 신설
  2. 프롬프트 로더 유틸 `src/analyzer/prompt_loader.py` 작성:
     - 파일을 읽고 {placeholder} 치환
     - 파일 mtime 또는 sha1 앞 8자리를 "prompt_version" 으로 반환
  3. generate_daily_brief 시그니처는 그대로, 내부에서 loader 사용
  4. analysis_log 에 prompt_version 기록(스키마 이미 허용 범위인지 확인 후 필요시 header에 추가)
- 수락 기준:
  - 프롬프트 수정이 코드 변경 없이 가능
  - prompt_version이 로그에 남음
  - 기존 테스트는 MagicMock 으로 load_prompt를 모킹해 통과

[N5] coingecko bare Exception 축소
- 파일: src/collectors/coingecko.py 약 line 133 부근 _fetch_coins_list

  1. `except requests.exceptions.RequestException as e:` 네트워크 예외 분리
```

## 공통 준수 사항 (모든 트랙)

- 커밋 메시지: `type(scope): subject` (ex. `fix(analyzer): cache analysis_log in-memory for the pipeline run`)
- 시크릿·토큰·키 절대 하드코딩 금지
- 기존 공개 API 시그니처 변경 시 호출부 전부 업데이트
- 작업 종료 전 반드시:
  1. `ruff check .` / `black --check .` (설정 파일 있을 때만)
  2. `pytest -q`
  3. `git diff --stat`으로 변경 범위 요약
- PR 설명 템플릿:
  - ## 무엇을 (What)
  - ## 왜 (Why)
  - ## 검증 (How verified)
  - ## 롤백 방법 (Rollback)
