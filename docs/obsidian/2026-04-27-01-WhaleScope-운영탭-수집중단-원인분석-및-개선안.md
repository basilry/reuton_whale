# WhaleScope 운영탭 수집 중단 원인분석 및 개선안

## 1. 결론

2026-04-27 KST 기준 배포된 운영 탭의 수집 데이터가 멈춘 것처럼 보이는 판단은 맞다. 다만 Render cron 자체가 죽은 것은 아니다. `python -m src.pipeline.run_all`은 15분마다 실행 중이고, `news_rss`, `curated_balance`, `broadcast_periodic`, `brief`, `stories`도 스케줄에 맞게 돌아간다.

직접 장애는 **온체인 수집의 seed인 `watched_addresses`를 Postgres에서 읽지 못하는 것**이다. `signals` job이 매번 `completed_empty`로 끝나며, 그 결과 `transactions`, `signals`, `brief`, `broadcast_periodic`가 연쇄적으로 빈 상태가 된다.

핵심 원인:

```text
PostgresClient._select_rows()가 모든 테이블에 id 컬럼이 있다고 가정
-> watched_addresses / subscribers에는 id 컬럼이 없음
-> list_watched_addresses(), list_subscribers()가 SQL 에러
-> signals 수집 후보 없음
-> transactions/signal 신규 적재 중단
-> brief skipped_inactive
-> broadcast_periodic skipped_empty
```

## 3. 코드 레벨 근본 원인

### 3.1 문제 코드

파일:

- `src/storage/postgres_client.py`

문제 함수:

```python
def _select_rows(...):
    ...
    if limit is not None and limit > 0:
        sql = (
            f"SELECT {columns} FROM {table}{where_clause} "
            f"ORDER BY {time_expr} DESC, id DESC LIMIT %s"
        )
    else:
        sql = f"SELECT {columns} FROM {table}{where_clause} ORDER BY {time_expr} ASC, id ASC"
```

문제:

- 모든 테이블에 `id` 컬럼이 있다고 가정한다.
- 하지만 `watched_addresses`와 `subscribers`는 자연키 PK 구조다.

### 3.2 충돌 테이블

파일:

- `src/storage/postgres_schema.py`
- `src/storage/schema.py`

`watched_addresses`:

```sql
CREATE TABLE IF NOT EXISTS watched_addresses (
  address text PRIMARY KEY,
  chain text,
  category text,
  label text,
  source text,
  confidence numeric,
  enabled text,
  added_at timestamptz,
  notes text
)
```

`subscribers`:

```sql
CREATE TABLE IF NOT EXISTS subscribers (
  chat_id text PRIMARY KEY,
  username text,
  status text,
  ...
)
```

따라서 `ORDER BY ..., id ASC`가 붙으면 운영 DB에서 실패한다.

### 3.3 왜 테스트에서 놓쳤는가

가능한 원인:

1. `PostgresClient._select_rows()` 공통 쿼리의 `id` tie-breaker를 테이블별로 검증하는 테스트가 없었다.
2. `list_watched_addresses()`와 `list_subscribers()`의 SQL 생성 테스트가 없거나, mock cursor가 SQL 실행 실패를 재현하지 않았다.
3. Sheets 기반에서 시작한 테이블들이 Postgres로 이전되면서 자연키 PK 구조와 공통 조회 helper의 가정이 충돌했다.
4. Render production DB migration 이후 실제 테이블 구조와 애플리케이션 조회 helper 간 정합성 검증이 부족했다.

## 5. 개선안

## 5.1 P0 — Postgres 조회 helper 정합성 수정

### 목표

`_select_rows()`가 모든 테이블에 `id`가 있다고 가정하지 않게 만든다.

### 권장 구현

`_select_rows()`에 `tie_breaker` 인자를 추가한다.

```python
def _select_rows(
    self,
    table: str,
    headers: list[str],
    *,
    time_expr: str,
    since: datetime | None = None,
    limit: int | None = None,
    filters: list[str] | None = None,
    params: list[object] | None = None,
    tie_breaker: str | None = "id",
) -> list[dict[str, object]]:
    order_terms = [time_expr]
    if tie_breaker:
        order_terms.append(tie_breaker)
```

호출부:

```python
def list_watched_addresses(self) -> dict[str, dict]:
    rows = self._select_rows(
        "watched_addresses",
        WATCHED_ADDRESSES_HEADERS,
        time_expr="COALESCE(added_at, now())",
        tie_breaker="address",
    )

def list_subscribers(self, statuses=None) -> list[dict]:
    rows = self._select_rows(
        "subscribers",
        SUBSCRIBERS_HEADERS,
        time_expr="COALESCE(updated_at, created_at)",
        tie_breaker="chat_id",
        filters=filters,
        params=params,
    )
```

대안:

- 모든 테이블에 `id bigserial`을 추가하는 migration
- 단, 자연키 PK 테이블에 불필요한 surrogate key를 추가하는 것은 현재 문제보다 변경 범위가 크다.
- 이번 장애 복구에는 helper 수정이 더 안전하다.

### 테스트

- `test_list_watched_addresses_uses_address_tie_breaker`
- `test_list_subscribers_uses_chat_id_tie_breaker`
- `test_select_rows_defaults_to_id_for_id_tables`

## 6. 권장 실행 순서

### 2026-04-27 QA 상태 업데이트

- 로컬 구현 단계: `_select_rows()`의 테이블별 tie-breaker 적용은 코드상 반영된 상태로 확인했다.

## 9. 최종 권장안

이번 장애는 인프라 다운보다 **Postgres 전환 후 공통 조회 helper의 테이블 PK 가정이 깨진 문제**다. 가장 먼저 `_select_rows()`의 `id` 고정 정렬을 제거하고, 자연키 테이블에 맞는 tie-breaker를 넘기도록 고쳐야 한다.

그 다음 운영 DB에 schema init을 재실행하고, `watched_addresses` seed가 실제로 들어 있는지 확인해야 한다. 이 두 가지가 끝나야 `signals -> transactions -> brief -> broadcast` 흐름이 다시 살아난다.

listener와 bot 문제는 중요하지만 2순위다. 현재 수집 중단의 핵심은 `watched_addresses` 조회 실패다.

### 10.1 현재 운영 값

`watched_addresses` seed import와 `PostgresClient.list_address_activity()` 패치 이후 수집 자체는 회복됐다.

```text

latest_transaction_at = 2026-04-27T04:46:52.384Z
latest_transaction_age_hours = 0
transactionCount = 5158
signals_last_run_at = 2026-04-27T04:47:01.067Z
```

따라서 현재 상태는 "수집 파이프라인이 죽음"이 아니라 다음 3가지가 남은 상태다.

1. 일부 chain seed가 있지만 feature flag가 꺼져 있어 `signals` job이 `completed_with_errors`로 기록된다.
2. 신규 transaction은 쌓이고 있으나 signal rule이 새로 발화하지 않아 `signals` 테이블에는 2026-04-27 row가 없다.
3. 09:00 Telegram 발송은 성공했지만, 새 `daily_brief` row 생성이 아니라 기존 brief/news fallback 발송이라 `daily_brief` freshness는 오래됐다.

### 10.2 signals 카드가 정상 칩을 받지 못하는 이유

정확히는 "코인별 기록 flag"라기보다 **체인별 수집 feature flag** 문제다.

현재 seed 상태:

| Chain | Seed | Collector | 현재 상태 |
|---|---:|---|---|
| ETH | 78 | always_on | 수집 가능 |
| SOL | 2 | always_on | 수집 가능, 현재 Solscan 404 일부 발생 |
| XRP | 2 | `ENABLE_CHAIN_XRP` | flag off |
| TRX | 2 | `ENABLE_CHAIN_TRX` | flag off |
| BTC | 5 | `ENABLE_CHAIN_BTC` | flag off, partial view |
| DOGE | 3 | `ENABLE_CHAIN_DOGE` | flag off, partial view |

`signals_last_error`에는 다음 값이 남는다.

```text
unsupported_chains=BTC=5,DOGE=3,TRX=2,XRP=2
```

이 값은 fatal collector error가 아니다. "seed는 있는데 운영자가 해당 체인 수집을 아직 열지 않았다"는 rollout 상태다. 그런데 현재 `/admin`은 이를 `signals_last_error`로 읽어 signal 관련 카드를 노랑/빨강으로 보이게 만든다.

#### 정상 칩이 붙는 방법

선택지는 2개다.

**A안 — 실제 수집 체인으로 승격**

Render pipeline 환경변수에 아래 값을 켠다.

```text
ENABLE_CHAIN_XRP=true
ENABLE_CHAIN_TRX=true
ENABLE_CHAIN_BTC=true
ENABLE_CHAIN_DOGE=true
```

그리고 각 수집기의 외부 API endpoint/key가 운영 품질을 만족해야 한다.

```text
XRPSCAN_API_BASE
TRONGRID_API_BASE
TRONGRID_API_KEY
BTC_INDEXER_BASE
BTC_INDEXER_KEY
DOGE_INDEXER_BASE
DOGE_INDEXER_KEY 또는 BLOCKCHAIR_API_KEY
```

장점:

- Chain rollout 카드에서 `seed_flag_off`가 사라진다.
- `unsupported_chains` 경고가 제거된다.
- BTC/TRX/XRP/DOGE도 실제 수집 대상이 된다.

리스크:

- 무료 API rate limit, partial indexer 품질, 주소별 응답 형식 차이 때문에 새 장애 지점이 늘어난다.
- 과제 제출 직전에는 범위를 넓히는 것보다 안정성을 해칠 수 있다.

**B안 — 현재 MVP 범위로 명시하고 error에서 제외**

BTC/DOGE/TRX/XRP seed는 "향후 rollout 후보"로 유지하되, disabled chain은 `signals_last_error`가 아니라 `chainRollout.seedButFlagDisabled`의 warning으로만 표시한다.

권장 코드 변경:

- `collect_recent_events()` 또는 `signals` 결과 조립부에서 `unsupported_chains`를 fatal/error 배열에 넣지 않는다.
- `service_health.supported_chains`, `unsupported_chain_names`에는 계속 남긴다.
- `/admin`의 `signals` 카드 tone은 `signals_last_status`, transaction freshness, fatal collector errors를 기준으로 판단한다.
- Chain rollout 카드에서만 `seed 있는데 flag 꺼짐 BTC/DOGE/TRX/XRP`를 노랑으로 표시한다.

권장안:

- 제출 안정성 기준으로는 **B안 우선**.
- 이후 운영 확장 phase에서 A안을 chain별 canary로 연다.

### 10.3 daily_brief가 09:00 Telegram 발송 후에도 빨간 이유

사용자가 2026-04-27 09:00 KST에 Telegram에서 본 메시지는 "새 daily brief 생성"과 동일하지 않다.

현재 운영 API 기준 최신 `daily_brief` row는 다음이다.

```text
latestBrief.generatedAt = 2026-04-23T04:07:34.020Z
```

관리자 수집 데이터 카드의 `daily_brief` 판정 로직은 다음이다.

```tsx
status: freshnessStatus(data.latestBrief.generatedAt, 8 * 60)
tone: freshnessTone(data.latestBrief.generatedAt, 8 * 60)
```

즉, `daily_brief.generatedAt`이 8시간 이내면 정상, 16시간 이내면 지연, 그 이상이면 빨강이다. 현재 latest brief는 2026-04-23 생성물이므로 빨강이 맞다.

반면 Telegram 09:00 메시지는 `broadcast_periodic` 또는 channel broadcast가 기존 `daily_brief + news` fallback을 조합해 보낸 것이다.

```text
fallback=daily_brief+news
signals=0
tx=0 또는 candidate tx만 사용
```

따라서 "텔레그램에 올라왔다"는 것은 `broadcast_log` 정상의 근거이지, `daily_brief` 신규 생성 정상의 근거가 아니다.

#### daily_brief 정상화 조건

현재 brief observability:

```text
brief.totalRuns(24h) = 24
generated = 0
cached = 0
skipped_inactive = 24
llmCallCount = 0
```

정상 칩이 붙으려면 둘 중 하나가 필요하다.

1. `brief` pipeline이 최근 transaction/signal을 기반으로 새 `daily_brief` row를 생성한다.
2. UI에서 `daily_brief`와 `broadcast_log`의 의미를 분리해, `daily_brief`는 "브리핑 생성 freshness", `broadcast_log`는 "채널 발송 freshness"로 명확히 표기한다.

권장 개선:

- `daily_brief` 카드 detail에 "최근 채널 발송은 정상이나, 새 daily_brief 생성은 2026-04-23 이후 없음"을 표시한다.
- `broadcast_log` 카드가 정상일 때 `daily_brief` 빨강을 가리지 않도록 한다.
- `brief`가 `skipped_inactive`일 때도 최근 transaction이 존재하면 `completed_empty`가 아니라 `skipped_no_signal` 또는 `fallback_tx_available`처럼 더 정확히 기록한다.
- 09:00 발송 메시지가 기존 brief fallback인지, 새 brief generated인지 `broadcast_log.fallback_source`와 `brief_cost_ledger.slot_key`를 연결해 표시한다.

### 10.4 brief_cost_ledger가 정상이 아닌 이유

`brief_cost_ledger` 카드는 현재 `summary.brief.latestGeneratedAt`을 기준으로 8시간 SLA를 적용한다.

```tsx
status: freshnessStatus(summary?.brief.latestGeneratedAt ?? briefSection?.lastUpdatedAt, 8 * 60)
tone: freshnessTone(summary?.brief.latestGeneratedAt ?? briefSection?.lastUpdatedAt, 8 * 60)
count: `${summary.brief.llmCallCount}회 호출`
detail: `generated ${summary.brief.generated.count} · cached ${summary.brief.cached.count}`
```

현재 `summary.brief.latestGeneratedAt`은 `latestGeneratedBriefAt()` 함수가 계산한다.

```ts
generated decision만 먼저 찾고,
없으면 latestBrief.generatedAt으로 fallback한다.
```

문제는 최근 24시간 brief ledger가 모두 `skipped_inactive`라는 점이다.

```text
brief.totalRuns = 24
generated = 0
cached = 0
skippedInactive = 24
llmCallCount = 0
latestGeneratedAt = 2026-04-22T18:03:34.832Z
```

즉 `brief_cost_ledger` 자체가 비어 있거나 읽히지 않는 문제가 아니라, **최근 24시간 동안 LLM brief 생성 비용이 발생하지 않았고 generated row가 없어서** 정상 칩을 받지 못한다.

#### brief_cost_ledger 개선 방향

현재 카드명과 판정 기준이 섞여 있다.

- `brief_cost_ledger`는 비용/결정 원장이다.
- 그런데 UI는 "최근 generated brief freshness"로 정상 여부를 판단한다.
- 따라서 skipped가 정상 정책일 때도 빨강으로 보인다.

권장 분리:

1. `brief_cost_ledger` 카드
   - 기준: 최근 8시간 내 ledger row 존재 여부.
   - 정상: `generated`, `cached`, `skipped_inactive`, `skipped_budget` 중 하나라도 최근 cadence에 맞게 기록됨.
   - 경고: ledger row 자체가 없음.
   - 빨강: ledger write 실패 또는 budget 계산 실패.

2. `brief_generation` 카드
   - 기준: 최근 generated/cached brief 존재 여부.
   - 정상: generated/cached.
   - 노랑: skipped_inactive가 정책상 발생.
   - 빨강: error, budget cap, LLM provider failure.

3. `telegram_broadcast` 카드
   - 기준: `broadcast_log.lastBroadcastAt`, `delivery_mode`, `status`.
   - 정상: channel sent.
   - daily_brief 생성 여부와 분리.
