# WhaleScope `/about` 과제소개페이지 개발 기획안

> **목적**: 뤼튼 PE 심사위원과 일반 방문자가 **배포 URL 한 곳(`https://whalescope.6esk.com/about`)에서** 과제의 문제 정의·해법·기술·운영 근거를 파악할 수 있도록 README + ONE_PAGER 내용을 구조화해 노출한다.
> **왜 지금인가**: 제출본의 심사 경로는 (1) `README.md`·`ONE_PAGER.md` 읽기, (2) `/` 홈 방문, (3) Telegram 채널 구독 3가지. 현재 2·3은 배포된 URL로 가능하지만 1은 GitHub 진입이 필요하다. 심사위원 상당수가 **배포 URL → 홈 → 복귀**만 보고 GitHub을 열지 않을 가능성이 실재한다. `/about`은 이 단절을 메운다.
> **핵심 원칙**: 원본 문서(`README.md`·`ONE_PAGER.md`)는 단일 소스. 페이지는 **렌더러**이며 내용 복제·파생 편집 금지. 문서 수정 → 재빌드 → 자동 반영.

## 0. 배경과 문제 정의

### 0-1. 현재 상태의 심사 동선

| 단계 | 경로 | 소요 | 이탈 위험 |
|-----|------|----:|----------|
| 1 | Wrtn 제출 링크 수신 → GitHub repo 진입 | 5~10초 | 낮음 |
| 2 | README.md 스캔 (76KB · 긴 기술 문서) | 3~5분 | **높음** — 심사위원이 "기술 README"로 인식하면 ONE_PAGER를 놓침 |
| 3 | ONE_PAGER.md 이동 (43KB · 9개 섹션) | 5~7분 | 중간 — 클릭 1회 추가 |
| 4 | 배포 URL 접속 → `/` 사용자 홈 | 10~15초 | 낮음 |
| 5 | `/admin` 운영 대시보드 (옵션) | 20초 | — |
| 6 | Telegram 채널 `@whalescope_alertz` | 30초 | — |

**문제**: (a) 심사위원이 GitHub 대신 **배포 URL만 방문**하면 과제의 핵심 기획 문서(ONE_PAGER)를 못 본다. `/`는 사용자 홈이지 과제 문서가 아니다. (b) README·ONE_PAGER가 각각 76KB·43KB여서 모바일에서 스크롤 피로가 크다. (c) 둘 중 무엇을 먼저 봐야 하는지 제시되지 않는다 — README §0에 "평가자용 빠른 동선: ONE_PAGER → README" 안내가 있지만 실제로는 README를 먼저 만나게 된다.

### 0-2. 해결하려는 것

1. **단일 진입점**: `whalescope.6esk.com/about` 한 URL로 과제 문서 전체 접근.
2. **우선순위 있는 정보 배치**: 첫 스크롤에 "도메인 C 선택·선택 근거·배포 URL·Telegram 채널" 4개 정보가 보이도록. ONE_PAGER가 README보다 앞선다는 것을 시각적으로 전달.
3. **TOC 기반 스캔 가능성**: 심사위원이 "Day 7 판단 기록" 또는 "실패 판단 기준" 같은 특정 섹션을 찾을 때 3클릭 이내 도달.
4. **원본 존중**: README·ONE_PAGER는 여전히 GitHub의 single source of truth. 페이지는 렌더링 레이어로 유지(수정 UI 없음, edit-on-github 링크만).
5. **성능**: 홈(`/`) 수준의 LCP < 2.0s 유지. 대용량 마크다운을 빌드 타임에 정적 HTML로 변환하여 런타임 파싱 비용 0.

### 0-3. 설계 비-목적 (명시적 Out of Scope)

- **페이지 내부에서 마크다운 편집 금지** — 편집은 GitHub PR으로만. 편집 UI 구현 시 인증·권한·감사 로그 부담이 MVP 범위를 초과.
- **다관점 리뷰 보고서(2026-04-22-01·02)는 비공개** — 내부 개선 계획으로 외부 노출하지 않음. 심사위원이 요구하면 별도 pdf 전달.
- **관리자 runbook (`docs/operational-run-verification.md`) 미노출** — 운영 시크릿 언급 가능성 + 대상이 아님.
- **i18n 전면 번역 금지** — 원본이 한국어이므로 `en` 선택 시 "일부 섹션 영어 미번역" 안내 배너. 자동 번역은 품질 리스크.

## 1. 타겟 페르소나와 시나리오

### 1-1. 페르소나 A — 심사위원 (주 타겟, 무게 70%)

- **도착 경로**: Wrtn 제출 링크 → 배포 URL 리다이렉트 또는 직접 `/about` 링크 클릭.
- **시간 예산**: 5~15분.
- **원하는 답**: (1) 도메인 C 선택 근거와 합리성, (2) 탐지/해설 분리 아키텍처의 설명, (3) AI 활용 방식의 판단력, (4) 운영 실측 증빙, (5) Day-by-day 판단 기록.
- **핵심 행동**: 스캔 → 관심 섹션 클릭 → 3~5 섹션 읽기 → `/`·`/admin` 이동.
- **이탈 트리거**: 첫 화면에 "긴 기술 README"만 보이면 3초 이탈.

### 1-2. 페르소나 B — 일반 방문자·잠재 사용자 (보조, 무게 20%)

- **도착 경로**: Telegram 채널 → "이 서비스 뭐야?" 궁금 → `/` → 상단 nav "과제 소개" 클릭.
- **시간 예산**: 1~3분.
- **원하는 답**: (1) 왜 만들었는가, (2) 믿을 만한가(투자 자문 면책), (3) 어떤 데이터 쓰는가.
- **핵심 행동**: ONE_PAGER §1~2 훑고 Telegram CTA로 복귀.

### 1-3. 페르소나 C — 개발자·오픈소스 기여 관심자 (보조, 무게 10%)

- **도착 경로**: 블로그·X 언급 → 궁금 → `/about` → README 탭.
- **시간 예산**: 10~30분.
- **원하는 답**: 기술 스택, 모듈 구조, 테스트, 환경변수.
- **핵심 행동**: README 탭 → "모듈 구조"·"테스트" 섹션 → GitHub clone.

**의미**: 3개 페르소나 모두 **첫 화면은 One Pager 성격 요약**이 적절. README 기술 상세는 탭으로 분리하여 B·C가 방해받지 않도록.

### 2-2. URL 설계

| URL | 렌더 대상 | 기본 탭 | 딥링크 예시 |
|-----|----------|--------|-----------|
| `/about` | 전체 · ONE_PAGER 기본 | One Pager | — |
| `/about?tab=readme` | README 활성 | README | 공유 가능 |
| `/about?tab=one-pager#3-핵심-기능과-근거` | ONE_PAGER §3로 스크롤 | One Pager | 심사위원 구체 섹션 공유 |
| `/about/one-pager` (대안) | ONE_PAGER 전용 라우트 | — | SEO 독립 페이지 |
| `/about/readme` (대안) | README 전용 라우트 | — | SEO 독립 페이지 |

**선택**: 서브 라우트(`/about/one-pager`) + 쿼리 폴백(`/about?tab=…` redirect) 조합.

**근거**:
- 서브 라우트는 각 탭이 독립 SEO 단위로 인덱싱됨. Google은 `whalescope ONE_PAGER` 검색어로 해당 하위 페이지를 노출 가능.
- 쿼리 폴백은 기존 외부 링크 호환.
- Next.js App Router에서 `app/about/layout.tsx`에 공통 chrome, `app/about/page.tsx`가 기본 리다이렉트(`redirect('/about/one-pager')`), `app/about/[slug]/page.tsx`가 실제 렌더.

### 2-3. 탭 전환 UX

- 탭 클릭 → URL 변경(push) + 스크롤 top.
- 탭 간 유지되는 상태 없음 (단순). TOC는 탭별로 새로 생성.
- 브라우저 뒤로가기 → 이전 탭/섹션 복원.
- 모바일에선 탭이 horizontal scroll 가능한 chip으로 표시.

### 5-2. 마크다운 렌더 파이프라인

**선택 스택**:
- `remark` + `remark-gfm` (GitHub Flavored Markdown: 테이블, task list, strikethrough)
- `remark-frontmatter` (YAML block 처리 — ONE_PAGER에는 없지만 확장 대비)
- `remark-rehype` (mdast → hast 변환)
- `rehype-slug` (heading에 id 자동 부여 — TOC용)
- `rehype-autolink-headings` (heading 우측 `#` 앵커 링크)
- `rehype-highlight` (syntax highlight · `highlight.js` 엔진)
- `rehype-stringify` (최종 HTML)

**대안 검토**:
- `react-markdown` → 런타임 파싱 · JSX 매핑 유연하지만 **번들 사이즈 +80KB**. 빌드 타임 static이면 불필요.
- `@next/mdx` → MDX(JSX 임베드 가능)이지만 원본이 plain .md이고 수정 원치 않음. 오버킬.
- **선택**: unified 파이프라인(remark→rehype) **빌드 타임만** 실행, 결과 HTML을 `dangerouslySetInnerHTML`로 삽입.

**파이프라인 구조**:
```ts
// apps/dashboard/lib/markdown/render.ts
import { unified } from "unified";
import remarkParse from "remark-parse";
import remarkGfm from "remark-gfm";
import remarkRehype from "remark-rehype";
import rehypeSlug from "rehype-slug";
import rehypeAutolinkHeadings from "rehype-autolink-headings";
import rehypeHighlight from "rehype-highlight";
import rehypeStringify from "rehype-stringify";
import { remarkRewriteLinks } from "./remark-rewrite-links";
import { remarkExtractToc } from "./remark-extract-toc";

export type RenderedMarkdown = {
  html: string;
  toc: TocNode[];
  wordCount: number;
  readingMinutes: number;
};

export async function renderMarkdown(source: string): Promise<RenderedMarkdown> {
  const toc: TocNode[] = [];
  const file = await unified()
    .use(remarkParse)
    .use(remarkGfm)
    .use(remarkExtractToc, { collector: toc })
    .use(remarkRewriteLinks, {
      githubBase: process.env.NEXT_PUBLIC_GITHUB_REPO_URL,
    })
    .use(remarkRehype, { allowDangerousHtml: false })
    .use(rehypeSlug)
    .use(rehypeAutolinkHeadings, { behavior: "append" })
    .use(rehypeHighlight, { detect: true, ignoreMissing: true })
    .use(rehypeStringify)
    .process(source);

  const text = String(file);
  const words = countWords(text);
  return {
    html: text,
    toc,
    wordCount: words,
    readingMinutes: Math.max(1, Math.round(words / 300)),
  };
}
```

### 5-3. 빌드 타임 vs 런타임

**선택**: **빌드 타임(static) + ISR(revalidate=3600)**

| 방식 | Pros | Cons | 적합? |
|------|------|------|:----:|
| 완전 static (build only) | 최고 성능, CDN 캐시 | 문서 수정 → 재배포 필요 | △ |
| **ISR (revalidate=1h)** | 1h마다 자동 갱신, CDN 캐시 | 수정 반영 최대 1h 지연 | **✓** |
| 런타임 fetch (no cache) | 최신 즉시 반영 | LCP 저하, Edge 파일 접근 복잡 | ✗ |
| Static Props + webhook 갱신 | 즉시 + 성능 | GitHub webhook 구현 | △ (과투자) |

**설계**:
```ts
// apps/dashboard/app/about/one-pager/page.tsx
import { readFileSync } from "node:fs";
import { resolve } from "node:path";
import { renderMarkdown } from "@/lib/markdown/render";

export const revalidate = 3600; // ISR 1h

export default async function OnePagerTab() {
  const mdPath = resolve(process.cwd(), "..", "..", "ONE_PAGER.md");
  const source = readFileSync(mdPath, "utf8");
  const { html, toc, readingMinutes } = await renderMarkdown(source);

  return (
    <AboutPageLayout toc={toc} readingMinutes={readingMinutes}>
      <MarkdownView html={html} />
    </AboutPageLayout>
  );
}
```

**주의 — Vercel deploy 타이밍**: 빌드 타임 `readFileSync`는 Vercel 빌드 컨테이너에서 실행되므로 저장소 루트 파일을 읽을 수 있다. 빌드 결과가 정적 HTML이면 Edge 런타임에서도 문제 없음. Render 배포 시에도 동일.

### 5-4. 링크 재매핑 (remark plugin 세부)

```ts
// apps/dashboard/lib/markdown/remark-rewrite-links.ts
import type { Plugin } from "unified";
import { visit } from "unist-util-visit";

type Options = {
  githubBase: string; // e.g. https://github.com/{owner}/{repo}/blob/main
};

const INTERNAL_MD_MAP: Record<string, string> = {
  "ONE_PAGER.md": "/about/one-pager",
  "README.md": "/about/readme",
  "apps/dashboard/README.md": "/about/readme#dashboard",
};

export const remarkRewriteLinks: Plugin<[Options]> = (options) => (tree) => {
  visit(tree, "link", (node) => {
    const url = node.url;
    if (!url || url.startsWith("http") || url.startsWith("/") || url.startsWith("#")) return;

    // 1) 내부 주요 MD → /about/* 매핑
    const mapped = INTERNAL_MD_MAP[url.replace(/^\.\//, "")];
    if (mapped) {
      node.url = mapped;
      return;
    }

    // 2) docs/* 등 기타 내부 파일 → GitHub blob 링크
    if (url.endsWith(".md") || url.includes("/")) {
      node.url = `${options.githubBase}/${url}`;
      return;
    }

    // 3) #anchor만 있으면 그대로
  });
};
```

### 5-5. TOC 추출 & Scroll Spy

**추출 (빌드 타임)**:
```ts
// remark-extract-toc.ts
export const remarkExtractToc: Plugin<[{ collector: TocNode[] }]> = (opts) => (tree) => {
  visit(tree, "heading", (node) => {
    if (node.depth < 1 || node.depth > 3) return; // h1~h3만
    const text = toString(node); // unist-util-to-string
    opts.collector.push({ depth: node.depth, text, slug: slugify(text) });
  });
};
```

**Scroll Spy (클라이언트)**:
```tsx
// TableOfContents.tsx
"use client";
import { useEffect, useState } from "react";

export function TableOfContents({ toc }: { toc: TocNode[] }) {
  const [activeSlug, setActiveSlug] = useState<string | null>(null);

  useEffect(() => {
    const headings = toc.map((t) => document.getElementById(t.slug)).filter(Boolean) as HTMLElement[];
    const observer = new IntersectionObserver(
      (entries) => {
        const visible = entries.filter((e) => e.isIntersecting).sort((a, b) => a.boundingClientRect.top - b.boundingClientRect.top);
        if (visible[0]) setActiveSlug(visible[0].target.id);
      },
      { rootMargin: "-80px 0px -60% 0px" },
    );
    headings.forEach((h) => observer.observe(h));
    return () => observer.disconnect();
  }, [toc]);

  return (
    <nav aria-label="문서 목차" className={styles.toc}>
      <ol>
        {toc.map((node) => (
          <li key={node.slug} data-depth={node.depth} data-active={activeSlug === node.slug}>
            <a href={`#${node.slug}`}>{node.text}</a>
          </li>
        ))}
      </ol>
    </nav>
  );
}
```

## 9. 성능 지표

### 9-1. 목표 (Lighthouse mobile)

| 지표 | 목표 | 현재 홈(`/`) | 비고 |
|-----|------|:-----------:|-----|
| LCP | < 2.0s | 1.8s | Hero 텍스트 타이밍 |
| INP | < 200ms | 150ms | 탭 전환 |
| CLS | < 0.05 | 0.02 | TOC sticky로 안정 |
| TBT | < 200ms | 180ms | 클라이언트 JS 경량 |
| First Load JS | < 200KB | 206KB | 본 페이지는 **더 작아야** (마크다운 static) |
| Total page weight | < 500KB | ~600KB | 이미지 없음 유리 |

### 9-2. 최적화 전략

1. **마크다운은 빌드 타임 HTML → 런타임 파싱 0**. react-markdown 대비 번들 −80KB.
2. **TOC Scroll Spy만 클라이언트 JS** (약 2KB gzip).
3. **DocTabs는 Client**이지만 단순 URL push + CSS display toggle → 5KB 이하.
4. **이미지 없음** 원본이 텍스트 중심. 있으면 `next/image` + `sizes` 설정.
5. **코드블록 highlight는 빌드 타임**. 런타임 highlight.js 로드 0.
6. **Revalidate=3600** — 1h CDN 캐시 히트율 높임.

### 9-3. 예산

```
[빌드 산출물]
/about/one-pager.html     target ≤ 120KB (ONE_PAGER 43KB → HTML ~60KB gzip ~20KB)
/about/readme.html        target ≤ 180KB (README 76KB → HTML ~100KB gzip ~35KB)
/about _shared.js         target ≤ 20KB (TOC + DocTabs)
```

### 9-4. 측정

- Vercel Speed Insights (이미 layout.tsx에 통합됨) — `/about` 별도 세그먼트로 필터링.
- Core Web Vitals 대시보드 주간 리뷰.

## 17. 대안 검토 · 기각 사유

### 17-1. 대안 A — 기존 README/ONE_PAGER를 GitHub pages로 노출

- 장점: 영(0) 개발 비용.
- 단점: 도메인 분리(github.io), 브랜드 일관성 저하, nav 통합 불가, 실측 주입 불가.
- **기각**: 배포 URL 한 곳이라는 목표와 상충.

### 17-2. 대안 B — notion public 페이지

- 장점: 편집 편의.
- 단점: 원본 단일 소스 원칙 파괴, 두 곳 동기화 부담.
- **기각**: "원본 = 저장소 내 md" 원칙 위반.

### 17-3. 대안 C — `/` 홈 내부에 과제 소개 섹션 추가

- 장점: 별도 라우트 불필요.
- 단점: 홈이 사용자용으로 설계됨, 과제 문서가 사용자에게 방해.
- **기각**: 페르소나 혼재.

### 17-4. 대안 D — `/about`을 단일 페이지 long-form (탭 없음)

- 장점: 구조 단순.
- 단점: ONE_PAGER+README 둘 다 스크롤해야 함, 모바일 피로.
- **기각**: 스캔 효율 낮음.

### 17-5. 대안 E — `react-markdown` 런타임 렌더

- 장점: 간단한 설정.
- 단점: 번들 +80KB, 런타임 파싱 비용.
- **기각**: 빌드 타임 unified가 우수.

### 17-6. 대안 F — PDF 생성을 사전 빌드 (CI에서 pandoc)

- 장점: 즉시 다운로드.
- 단점: pandoc+latex 의존(CI +5min), 빌드 복잡.
- **기각**: print stylesheet로 충분.
