## 2. 다관점 리뷰

### 2.1 엔지니어링 관점

**강점**

1. **T3 잠금 설계의 이론적 정당성이 구체적이다.** `SheetsClient.__init__`의 `self._write_lock = threading.Lock()` 주석(L101-L102)이 "gspread mutations across threads (async callers via asyncio.to_thread may schedule multiple writes concurrently)"로 동기화 필요성을 명시한다. Lock을 쓰지 않고 RLock을 쓰지 않은 선택도 일관적이다 — 내부에서 재진입이 없고(쓰기 메서드는 서로 호출하지 않음), Lock이 더 엄격한 불변식을 강제한다.

2. **T6 `_staleness_threshold_seconds()` 폴백 로직이 방어적이다.** L17-L32의 세 단계 분기 (unset/blank → `int()` 실패 → `<=0`)는 운영 환경의 공백 문자열·오타·음수 값 모두를 하나의 기본값(900s)으로 수렴시킨다. 예외 대신 폴백을 택한 것은 헬스 체크 경로의 정상성을 깨지 않기 위한 올바른 선택이다.

3. **W1-C rate limit의 윈도우 정리 패턴.** `lib/rate-limit.ts:29`에서 요청마다 `filter((ts) => ts > horizon)`를 수행해 idle 버킷의 타임스탬프가 누적되지 않도록 한다. `retryAfter` 계산이 `(oldest + windowMs - now)`로 정확해 Retry-After 헤더와 대시보드 UX가 자연스럽다.

4. **W2-E `SignalActionCard`의 재진입 가드.** L27-L28 `if (status === "saving") return;`가 버튼 더블클릭으로 인한 중복 PATCH를 차단하고, `encodeURIComponent(props.signalId)`로 경로 주입을 방지한다. `dismissed` 상태일 때 `return null`로 언마운트해 DOM 잔여 처리를 단순화한 것도 명료하다.

5. **W1-A `tg_direction`의 의도 문서화.** `src/ingestion/tg_normalizer.py:52-63`에서 wallet-to-wallet 폴백이 `direction="out"`로 수렴하는 이유(`Literal["in","out"]` 타입 일관성, `counterparty_category=None`으로 시그널 룰이 필터링 가능)를 주석으로 박아두어 다음 개발자의 "이 default가 맞나?" 반복 질문을 차단한다.

6. **`safe_float`의 옵션 설계.** `src/utils/number_utils.py`는 기존 두 구현(`main.py` WARNING + strip_commas, `baseline.py` DEBUG) 각각의 콜 사이트를 해치지 않고도 통합되도록 `log_level` / `strip_commas` / `field_name` / `logger` 인자를 노출했다. `tg_normalizer.py:88-101`에서 `logging.WARNING`, `strip_commas=True`, `field_name="amount_usd"`로 정확히 이식된 것을 확인.

**우려 사항 (중-저 영향)**

7. **[MEDIUM] `SheetsClient` 락 커버리지 부분 적용.** `grep -c "with self._write_lock" == 9`이지만 `@retry` 데코레이터가 붙은 쓰기 메서드는 16개 이상이다. 잠금 없이 남아있는 쓰기 메서드:
   - `save_daily_brief` (L149)
   - `add_subscriber` (L222)
   - `set_watchlist` (L276)
   - `set_status` (L314)
   - `update_subscriber_language` (L336)
   - `save_analysis` (L428)
   - `log_run` (L443)

   대부분 Telegram 봇 스레드에서만 호출되는 경로(구독자 관리 · 커맨드 핸들러)이긴 하나, `log_run`과 `save_analysis`는 파이프라인(`src/main.py`)에서 `asyncio.to_thread`를 경유해 호출될 소지가 있다. 현재 설계는 "hot path write만 보호" 정책으로 보이는데, 이 정책이 *어느 경로를 hot으로 정의*했는지는 코드 주석/docstring 어디에도 쓰여 있지 않다. 락 정책을 명문화하거나 균질하게 적용할 필요가 있다. → **FU-03**.

8. **[MEDIUM] In-memory cache/rate-limit의 멀티 워커 취약성.** `lib/cache.ts`, `lib/rate-limit.ts` 모두 모듈 스코프 `Map`을 쓴다. Render/Vercel의 수평 스케일 환경에서는 워커별로 분리되어 (a) 캐시 히트율 저하, (b) rate limit 회피(`maxRequests * N_workers`)가 가능하다. T5 changelog의 "Risks" 섹션은 `threading.Lock`만 언급했고 이 제약은 기록되지 않았다. 현재 배포(Render Starter, 1 워커)에서는 문제가 현재화되지 않으나, 스케일 업 전에 Redis 전환 혹은 Edge runtime cache로 대체 필요. → **FU-04**.

9. **[LOW] `SystemLogPanel` 데이터 흐름 — SSR 모드.** `app/page.tsx:30` `export const dynamic = "force-dynamic"`이 `/api/dashboard/route.ts:9` `export const revalidate = 60`과 공존한다. 페이지 자체는 매 요청 런타임 렌더이므로 ISR은 API 레이어에만 효과가 있고, 인메모리 `DASHBOARD_CACHE_TTL=60`과 ISR의 60s가 동일값이라 단일 워커에서는 완전 중복된 두 캐시 계층이 된다. 기능 버그는 아니지만 어느 한 쪽을 제거해도 동작은 같다. → **FU-05**.

10. **[LOW] `tg_direction` wallet→wallet 기본값.** 수집된 원본 TG 이벤트 중 `from_owner_type != exchange && to_owner_type != exchange`인 케이스는 모두 `direction="out"`으로 수렴한다. TG 소스에 `watched_address`가 없어 실질적 in/out 의미가 없다는 주석이 있지만, 만약 향후 Telegram 룰이 `direction`을 필터 키로 쓰기 시작하면 "out" 편향이 데이터 왜곡을 일으킨다. 이 경우엔 `direction="unknown"` 허용이 필요. 지금은 타입(`Literal["in","out"]`)이 이를 막고 있음.

11. **[LOW] Language 쿠키 `httpOnly: false`.** `/api/language/route.ts:73`은 SameSite=Lax / Max-Age=1년 쿠키를 `httpOnly: false`로 설정한다. `LanguageSelector`가 `document.cookie`로 직접 읽고 쓰기 위한 의도적 설계이며, 쿠키의 민감도가 낮아(언어 선택) XSS 노출 시의 위험이 작다. 단, 추후 동일 쿠키에 구독자 ID 등이 얹히면 즉시 `httpOnly: true`로 전환해야 한다.

**운영 가치 관점 강점**

"`cex_venue_rebalance` 같은 별도 룰을 만들라"는 탈출구까지 명시한 점이 특히 좋다.

3. **T5 changelog의 "Why → What → How validated → Risks" 4부 구조가 스프린트 복기 템플릿으로 기능한다.** 특히 "Risks" 섹션에 `threading.Lock`의 프로세스 경계 한계, Gemini 2.5 단가 미확정, `cex_to_cex` 소비처 부재까지 명시해 다음 스프린트의 백로그 후보를 자동으로 만든다.

### 2.3 마케팅 관점

**포지셔닝 · 메시징 관점 강점**

1. **"노이즈를 분리한 시그널"은 경쟁 우위 카피가 된다.** Whale Alert 류의 원천은 raw 이벤트를 그대로 뿌리는 반면, WhaleScope는 (a) `cex_to_cex` 분리로 거래소 리밸런싱을 제거하고 (b) `score > 0` 게이트로 저가치 알림을 잘라냈다. 이는 "단순 알림"이 아닌 "Curated Whale Intelligence"로 포지션할 수 있는 차별점이며, changelog의 "CEX 노이즈 분리"는 그대로 마케팅 헤드라인으로 재활용 가능하다.

2. **다국어 브리핑이 리테일 글로벌화 스토리를 연다.** ko/en/ja 동시 지원은 경쟁 서비스 대비 포지션을 "Asian-first crypto whale intelligence"로 압축할 수 있게 한다. 랜딩 페이지에서 "한국·일본 시장 전용 온체인 고래 시그널" 메시지와 매칭된다.

3. **대시보드 한국어 번역(`humanizeLogMessage`)이 현지화 완성도를 보여준다.** raw `system_log` JSON이 아니라 "[telethon_listener] 정상 완료 (3건 처리)" 같은 자연어 출력은 "엔지니어가 아닌 트레이더도 읽을 수 있는 대시보드"라는 데모 영상 소구점이 된다.

**메시징 간극**

4. **[MEDIUM-MKT] 데모 한계를 숨기지 않는 것이 중요하다.** 인메모리 스토어 한계(FU-06)를 마케팅 카피로 "Production-ready"로 오버셀하면 영업 파이프라인에서 POC 단계에서 불일치가 드러난다. "1차 출시는 데모 스코프, 2차에 영속 스토어" 로드맵을 PM과 마케팅이 동기화해야 한다.

5. **[LOW-MKT] 경쟁 키워드 선점 기회.** `cex_venue_rebalance`(FU-02) 룰이 추후 출시되면 "Whale vs Venue Rebalance Differentiator"가 프레스 릴리즈 훅이 될 수 있다. 현재 Whale Alert, Arkham 등 경쟁 서비스는 이 구분을 명시적으로 내세우지 않는다.

## 6. 전체 판단

**스프린트는 계획의 초과 달성이다.** T1-T6의 DoD를 모두 충족한 상태에서 W1-W3 웨이브까지 같은 세션에 적용되어, 리팩토링(W1)·기능 확장(W2)·패키징(W3) 세 측면의 성과가 병렬로 누적됐다. 설계 의도 고정(T4)과 변경 기록(T5)이 짝지어진 덕분에 회귀 방지와 지식 공유 모두 유지된다.

주요 리스크는 세 가지:
1. **락 커버리지 비대칭**(FU-03) — 정책을 명문화하거나 전 메서드로 확장해야 운영 확신이 커진다.
2. **인메모리 상태의 데모 스코프**(FU-06) — 특히 W2-E 기능이 "프로덕션으로 오해될 여지"가 있다.
3. **멀티 워커 스케일 가정 부재**(FU-04) — 현재는 단일 워커 Render 환경이라 드러나지 않는다.

WhaleScope 2026-04-17 개선 스프린트의 모든 원안(T1-T6)이 병합됐으며 동일 브랜치에서 후속 W1-W3 웨이브(대시보드 리팩토링·다국어·인터랙티브·패키징)까지 누적 적용됐다. 현재 `main`은 origin/main 대비 26커밋 앞섬. 코드 품질은 전반적으로 양호하나, `SheetsClient._write_lock`이 9/16 쓰기 메서드에만 적용된 비대칭(FU-03), `lib/metrics.ts`의 인메모리 상태(FU-06), 단일 워커 가정(FU-04) 세 가지 후속 리스크가 있다. 다음 작업 시에는 FU-06을 Sheets 영속화로 해소하거나, FU-03을 락 정책 명문화로 마무리하는 것이 자연스럽다. 테스트 로컬 실행은 샌드박스 파이썬 3.10 제약으로 불가했고, CI에서 통과 수(277 or 298) 교차 확인이 필요하다. 관련 파일: `src/storage/sheets_client.py`, `apps/dashboard/lib/{cache,rate-limit,metrics}.ts`, `docs/changelog.md`, `config/signals.yaml`, `src/signals/rules.py`, `src/ingestion/telethon_listener.py`.
