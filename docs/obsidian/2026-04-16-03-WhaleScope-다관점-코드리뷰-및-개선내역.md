### 4-1. High Priority (버그 또는 유의미한 gap)

#### H1. `_tg_direction` dead code와 exchange-to-exchange 미정의

위치: `src/main.py` (`_tg_direction` 함수)

내용:

```python
def _tg_direction(from_owner_type: str, to_owner_type: str) -> tuple[str, str | None]:
    from_type = from_owner_type.lower()
    to_type = to_owner_type.lower()
    if to_type == "exchange" and from_type != "exchange":
        return "in", "cex"
    if from_type == "exchange":
        return "out", None
    if to_type == "exchange":      # dead code: 첫 번째 branch에서 이미 처리
        return "in", "cex"
    return "out", None
```

- 세 번째 `if to_type == "exchange":` branch는 도달 불가능한 dead code다.
- `from_type == "exchange"` and `to_type == "exchange"` (거래소 간 이체)인 경우 첫 번째 branch를 건너뛰고 두 번째 branch에서 `("out", None)`을 반환한다. 의도인지 불명확하다.
- wallet-to-wallet 경우도 `("out", None)`이 기본값인데, TG 이벤트에는 `watched_address`가 없으므로 "outflow" 의미가 모호하다.

영향:

- corroboration rule에서 direction 기반 매칭이 부정확할 수 있다.

권장:

- dead code 제거
- exchange-to-exchange 케이스를 `("transfer", "cex")` 또는 별도 값으로 정의
- 최소한 주석으로 의도 명시

#### H4. `auth.ts` timing-safe comparison 미사용

위치: `apps/dashboard/lib/auth.ts` `getDashboardAuthResult()`

내용:

- `suppliedPassword === expectedPassword`는 JavaScript 문자열 동등 비교로, 이론적 timing oracle이 존재한다.
- 서버사이드 Node.js에서 네트워크 latency가 지배적이므로 실제 exploit 가능성은 낮다.

권장:

```typescript
import { timingSafeEqual } from "crypto";
const a = Buffer.from(suppliedPassword);
const b = Buffer.from(expectedPassword);
const authorized = a.length === b.length && timingSafeEqual(a, b);
```

## 5. 다관점 리뷰

### 5-1. 제품/사용자 가치

개선된 점:

- TG 채널 데이터가 signal pipeline에 합류하면서 "온체인 + Telegram 교차검증"이라는 제품 가치가 실제 코드에서 작동하기 시작했다.
- `/insights` 사용자 화면이 raw JSON이 아니라 사람이 읽는 한국어 요약으로 전환됐다.
- 운영자/사용자 화면이 분리되어 각 대상에 맞는 정보 밀도를 제공한다.

남은 리스크:

- `_signals_to_top5()`의 빈값 저장 문제(이전 Major 2)가 미해소. signal dashboard에서 의미 있는 시그널이 빈 거래처럼 보일 수 있다.
- Telegram 개인화가 LLM brief를 우회하는 문제(이전 Major 3)가 미해소. 사용자에게 전달되는 브리핑 품질이 의도보다 낮을 수 있다.
- `buildWatchlist` 하드코딩으로 인해 실제 signal 상태와 watchlist tone이 괴리된다.

### 5-2. 아키텍처/엔지니어링

개선된 점:

- `chain_events` vs `raw_events` 분리로 TG 이벤트가 chain 저장 경로를 오염시키지 않는 구조가 됐다.
- `asyncio.to_thread`로 blocking I/O를 event loop에서 분리해 listener 안정성이 높아졌다.
- heartbeat 기반 health model로 운영 대시보드 정확도가 개선됐다.

남은 리스크:

- `_parse_dt` 3곳 중복은 diverge 위험이 높다.
- `analysis_log` schema mismatch(이전 Major 4)가 미해소. LLM 비용/latency 추적이 불가능하다.
- `page.tsx` 1200줄 이상 단일 파일. 향후 auth, heartbeat, public API 추가 시 복잡도 급증 예상.

### 5-3. 보안

개선된 점:

- 운영 API에 `DASHBOARD_PASSWORD` 기반 인증 적용.
- raw error message를 client에 반환하지 않는 `createGenericErrorResponse` 패턴 적용.
- `.gitignore`에 `*.session` 추가로 Telethon 세션 파일 커밋 방지.

남은 리스크:

- timing-safe comparison 미사용 (H4).
- `x-dashboard-password` 평문 헤더가 로깅될 가능성 (L6).
- rate limiting 없음. 약한 비밀번호에 대한 brute force 가능.
- `/insights`가 인증 없이 전체 Sheets 데이터에 접근 가능. 의도적 설계이나 README에 명시 필요.

### 5-4. 운영/SRE

개선된 점:

- listener heartbeat가 `system_log`에 구조화되어 기록된다.
- dashboard가 Sheets 연결이 아니라 실제 listener 활동 기준으로 상태를 판단한다.
- storage/LLM 실패가 listener 프로세스를 죽이지 않는다.

남은 리스크:

- heartbeat 기록 자체가 Sheets에 의존하므로 Sheets 장애 시 heartbeat도 중단된다.
- `_record_system_log`가 `connect()` 전에 호출되어 허위 시작 로그 가능성 (H6).
- 15분 staleness 임계값이 하드코딩 (M9).
- `run()` 메서드 테스트 부재로 인증 실패/세션 만료 시 동작 불확실 (H3).

### 5-5. 데이터/시그널 품질

개선된 점:

- chain-only rule에 `e.source == "chain"` 가드 추가로 TG 데이터가 chain rule을 오염시키지 않는다.
- corroboration은 `corroborated_move` rule과 SignalEngine의 `source="both"` 승격에서만 처리된다.

남은 리스크:

- `_tg_direction`의 exchange-to-exchange 방향이 모호 (H1).
- `_safe_float`가 파싱 실패 시 0.0을 반환하여 실제 0과 구분 불가 (M8).
- `_signals_to_top5()`의 aggregate signal에서 빈 hash/symbol/0 USD 저장 (이전 Major 2 미해소).

### 5-6. 비용/FinOps

개선된 점:

- TG 이벤트가 signal pipeline에 들어가면서 chain-only 수집에 의존하지 않아도 시그널 생성 가능성이 높아졌다.
- `asyncio.to_thread`로 LLM fallback이 listener를 block하지 않아 불필요한 retry/timeout 비용이 줄었다.

남은 리스크:

- `analysis_log` telemetry 미저장으로 실제 LLM 비용 추적이 불가 (이전 Major 4).
- Gemini 2.5 모델 비용 테이블 미등록으로 cost가 0.0으로 기록되는 기존 이슈 유지.

## 9. 결론

이번 적용내역은 이전 리뷰의 P1 Finding 5개 중 5개를 모두 코드에 반영했다. 특히 TG 데이터 pipeline 병합, 운영 API 인증, listener 비동기 안정화는 WhaleScope의 "온체인 + Telegram 교차검증 AI 큐레이션 서비스"라는 핵심 가치를 코드 수준에서 완성하는 의미 있는 진전이다.

3. Telegram brief_text 보존 (이전 Major 3) - 과제 핵심인 "AI 요약"이 실제 사용자에게 전달되어야 함
