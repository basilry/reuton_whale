## 1. Code Review — Blockers & High Issues

### B1. 테스트 스위트 적색 (Blocker)

`pytest` 결과 **190 passed / 20 failed / 4 errors**. 실패 분포:

- `test_main.py` 5건 — 파이프라인 통합 경로. 리팩터 이후 시그니처/목 불일치.
- `test_telethon_parser.py` 5건 — Telethon 리스너 메시지 핸들러 테스트 전반 실패.
- `test_distributor.py::test_handle_status_not_subscribed_korean` 1건.
- `scripts/test_connection.py` 4건 ERROR — 실 API 호출(기대된 수집 단계 에러; 분리 가능).

**위험**: 메인 파이프라인 통합 테스트가 빨간 상태로 main 머지되면 회귀 탐지 불가. 특히 `test_main.py` 실패는 `run_daily_pipeline`의 실제 호출 경로에 기인 — **프로덕션 경로가 테스트와 어긋남**.

**조치**: 머지 전 `test_main.py`·`test_telethon_parser.py` 그린 복구. `scripts/test_connection.py`는 `@pytest.mark.skip` 또는 `pytest.ini` 기본 제외로 CI 분리.

### B2. Storage Protocol 우회 (High)

`main.py:170`에서 `load_watched(sheets)` 직접 호출. `src/ingestion/registry.py`는 `sheets` 인스턴스의 내부 메서드를 직접 읽는다.

```python
watched_index = load_watched(sheets)  # Protocol이 아닌 직접 접근
```

ADR-003(Storage ladder Sheets→SQLite→Postgres)의 전제인 **Protocol 인터페이스**가 깨진다. 미래에 SQLite로 교체할 때 이 지점이 회귀 원인.

**조치**: `Storage` Protocol에 `list_watched_addresses() -> dict[str, dict]` 추가 후 `load_watched`를 그 호출로 변경.

### B3. 이중 스코어링 경로 (High)

main.py에 Stage 5(`SignalEngine.run`)와 Stage 6-7(`TransactionScorer.pre_filter` + `analyzer.analyze_batch`)가 동시에 존재.

- Stage 8의 `brief_text = analyzer.generate_daily_brief(signals)`는 **SignalEngine 결과만** 사용.
- 하지만 Stage 9의 `save_daily_brief`에 저장되는 `top5`는 **레거시 scorer 결과**.
- Telegram 분배는 Stage 8 brief_text 기반.

**영향**: 사용자가 받는 브리프(Signal 기반)와 Sheets `daily_brief` 탭에 저장되는 top5(레거시 기반)가 **다른 근거**를 참조. 데이터 추적·감사 시 불일치.

**조치**:
- 단기: Stage 6-7을 SignalEngine fallback으로만 명시적 표기(주석 + 조건). `engine.run`이 시그널을 냈으면 레거시 경로 스킵.
- 장기: `TransactionScorer`와 `prompts.py`(하드코딩 상수) 제거.

### B4. 엔진 corroboration Path 3 로직 혼탁 (Medium)

`signals/engine.py:65-73`:

```python
if sig.source == "chain":
    for ce in chain_ev:
        if ce.tx_hash not in sig.evidence_tx_hashes and ce.tx_hash not in (None,):
            continue
        for te in tg_ev:
            ...
```

`not in (None,)`은 `is not None`과 동치이나 가독성이 낮고, **연속 AND의 결과가 "evidence에 포함되거나 tx_hash가 None인 chain event"를 처리** — 즉 해시가 없는 체인 이벤트가 휴리스틱 매칭 루프를 타는 것이 의도인지 불명.

**조치**: 리팩터.
```python
candidates = [ce for ce in chain_ev if ce.tx_hash in sig.evidence_tx_hashes]
for ce in candidates:
    ...
```

### B5. `chain_baselines` 미주입 (Medium)

`RuleContext.chain_baselines`가 `field(default_factory=dict)`로 비어있음. main.py:229에서 `engine.run(raw_events, now)` 호출 시 ctx를 기본값으로 생성. 결과:
- `cex_outflow_spike`/`cex_inflow_spike`의 3σ 임계가 작동하지 않음 (기준 분포 부재).
- `weekly_net_accumulation`도 동일.

**현재 완화**: TODO 주석으로 기록 (main.py:231).

**조치**: 단기 대안 — Sheets `address_activity` 탭에서 과거 N일 집계를 계산해 baseline 주입하는 `_build_context()` 헬퍼 추가. 최소 7일 데이터 축적 후 활성화.

### B6. `engine.personalize(signals, [])` 빈 호출 (Medium)

Stage 5 마지막에 `signals = engine.personalize(signals, [])` 하드코딩 빈 리스트. `user_interests` 탭에서 실제 데이터를 읽지 않음.

**영향**: 개인화가 noop. 사용자 차별화된 경험 제공 불가.

**조치**: `sheets.list_user_interests(user_id)` API + Telegram 명령(`/interests add rule=cex_outflow_spike weight=1.5`) 기반 주입. 단, 멀티유저 시 시그널 복제가 발생하므로 `bot.send_daily_brief`에서 per-user 개인화를 돌리도록 책임 이동 필요.

## 2. Medium / Low Issues

### M1. Dead code — `src/analyzer/prompts.py`
- `SYSTEM_PROMPT` 상수 54 LOC. `prompt_loader`로 대체되었으나 파일 제거 안 됨. 
- **조치**: 파일 삭제 + import 흔적 제거.

### M2. `_event_to_dict` 필드 이름 불일치
main.py:77-92에서 `Event` → dict 변환 시 `tx_hash`를 `hash`, `from_addr`를 `from_address`로 리매핑. 레거시 스코어러 호환용이지만 **두 이름 시스템 공존**은 신규 기여자 혼란 유발.

**조치**: 레거시 제거 시 이 변환기도 함께 제거.

### M3. Etherscan rate limit 신호 미활용
`src/ingestion/etherscan.py`는 단순 throttle(`rate_limit_per_sec=3.0`)만 적용. HTTP 429·응답의 `Max rate limit reached` 메시지를 파싱해 backoff 하지 않음.

**조치**: `requests` 응답 본문에서 `"status": "0"` + 메시지 패턴 시 지수 백오프 추가.

### M4. `price_service` 미지원 심볼 silent fallback
`_COINGECKO_IDS`에 없는 심볼은 None 반환. 신규 토큰 추가 시 조용히 `amount_usd=0`으로 처리되어 시그널 누락.

**조치**: 미지원 심볼 카운터 + 주간 상위 N개 심볼 리포트를 `system_log`에 적재해 화이트리스트 업데이트 근거 생성.

### M5. `prompt_loader` 해시 캐시 중복 로드
동일 프롬프트를 같은 run 내 여러 번 호출 시 파일 I/O 반복. mtime 캐시가 있으나 sha1 재계산이 호출마다 발생.

**조치**: `(mtime, sha1)` 튜플 캐시. 성능 영향은 작으나 청결도 이슈.

### M6. `telegram_bot.send_daily_brief` 백오프 없음
`python-telegram-bot` 라이브러리 자체 재시도 없음. 블록/비활성 계정만 휴리스틱 감지(`"blocked"`, `"deactivated"` 문자열). 429·5xx에 대한 재전송 큐 부재.

**조치**: `tenacity` 또는 기존 `@retry` 재사용.

### L1. Import 순서 PEP 8 위반
main.py:28 `from src.analyzer.price_service import PriceService`가 다른 src 임포트 블록과 공백 분리. `isort` 도입 시 자동 해소.

### L2. `TransactionScorer` 경로 테스트 커버리지
레거시지만 아직 호출 경로에 존재. 제거 전까지 최소 회귀 테스트 유지 필요.

## 3. Positive Findings

- **LLMRouter 품질**: Provider Protocol + fallback + usage 추적까지 깔끔. 60 LOC 미만으로 설계 목표 달성.
- **8개 룰 외부화**: `config/signals.yaml`에서 전량 YAML 구동. 룰 추가 시 코드 변경 최소.
- **프롬프트 버저닝**: sha1 해시로 implicit 버전 관리. `analysis_log`에 `prompt_version` 기록되어 회귀 추적 가능.
- **190개 테스트 통과**: 신규 signals/llm/storage 단위 테스트 밀도 높음(신규 테스트 약 80+건 추가).
- **에러 격리**: 각 Stage try/except로 부분 실패 허용 — 프로덕션 안정성에 유리.
- **Protocol 기반 Storage**: (일부 우회 있지만) 기반은 마련. 마이그레이션 비용 감소.

## 4. 다관점 리뷰

### 4.1 Engineering

**긍정**
- 레이어 경계 명확(ingestion → signals → analyzer → distributor).
- LLMRouter는 3 Provider 지원하면서 총 205 LOC. 의도한 스코프 봉쇄 성공.
- 테스트 수 214건으로 2차 리뷰 시점 대비 약 2배.

**부정**
- **test_main 적색** — 리팩터 테스트 동기화 누락.
- 이중 스코어링 경로 → 유지보수 부채.
- Storage Protocol 부분 우회.

**점수: 7/10** (2차 7.0 유지. 구조 개선은 있으나 통합 테스트 적색이 상쇄.)

### 4.2 Cost / FinOps

**긍정**
- 자체 LLM 라우터 + Gemini Flash/Groq 프리퍼드 경로 → 일간 파이프라인 무료 티어 충분.
- `config/llm_routing.yaml` 작업별 모델 매핑으로 비용/품질 분리 최적화 가능.
- `usage.py` 호출당 비용 추정 기록 → 월말 실측 근거.

**부정**
- Anthropic prompt caching 미활용(구조는 준비됨).
- CoinGecko 호출이 토큰당 1회 — 티커 많을 시 레이트 리밋 접근.

**점수: 9/10** (2차 9 유지. 라우터 구현으로 이론이 실체화.)

### 4.3 Product / UX

**긍정**
- 시그널 기반 브리프로 "고래 거래 단순 나열"에서 "패턴 해석"으로 진화.
- 프롬프트 외부화로 톤 튜닝 민첩성 확보.

**부정**
- `personalize(signals, [])` — 사용자 맞춤 미작동.
- Stage 9 레거시 top5와 Stage 8 시그널 브리프의 근거 불일치 → 감사 요청 시 설명 곤란.
- Telegram 분배 async 경로의 통합 테스트 부재.

**점수: 7/10** (2차 7 유지. 개인화 scaffolding은 있으나 작동 안 함.)

### 4.4 Security / Privacy

**긍정**
- 시크릿 env-var + 로깅에서 키 노출 없음.
- Telethon 세션 파일 격리.
- `tx_hash`·주소 외 PII 로깅 없음.

**부정**
- `GOOGLE_CREDENTIALS_JSON` 단일 라인 JSON — env dump 사고 시 서비스 계정 전체 노출 위험.
- `price_service` 미지원 심볼에 대한 silent 0 — 금액 왜곡을 통한 시그널 조작 가능성(저위험).
- Telethon 로그인 채널 메시지에 사용자 ID 포함 시 개인정보 처리위탁 재검토 필요.

**점수: 8/10** (2차 동일.)

### 4.5 Operations / Reliability

**긍정**
- `@retry(max_retries=3, base_delay=2.0)` Sheets 전역 적용.
- 파이프라인 Stage별 에러 격리.
- weekly_trend GitHub Actions 워크플로우 추가.

**부정**
- **통합 테스트 적색 상태로 CI 머지 위험**.
- Etherscan 429 핸들링 부재.
- Telegram 분배 재시도 큐 없음.
- `chain_baselines` 빈 상태 — 임계 기반 룰 사실상 대기 상태.

**점수: 6/10** (2차 7 → 6, 통합 테스트 적색 감점. 수정 시 즉시 복구.)

### 4.6 Career / Demo

**긍정**
- ADR 5건 문서화 + 리뷰 문서 3건 + 구현 프롬프트로 "의사결정 자취" 완비.
- 자체 LLM 라우터 구현 실체 → 면접 어필 재료.
- 마이그레이션을 14커밋으로 단계적·원자적으로 분리 — PR 리뷰 품질.

**부정**
- 통합 테스트 적색 상태 PR은 시니어 리뷰어에게 감점.
- 레거시 경로 공존 — "왜 안 지웠나" 설명 필요.

**점수: 9/10** (2차 8 → 9. ADR·구현·테스트 밀도가 포트폴리오 강점.)
