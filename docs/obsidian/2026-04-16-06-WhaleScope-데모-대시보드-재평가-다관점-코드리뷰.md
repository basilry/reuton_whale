## 1. 화면 평가 요약

### 결론

Claude Code 수정 이후 데모 HTML과의 시각적 유사도는 크게 개선됐다. 특히 Admin 화면은 데모의 sticky top navigation, 12-column 운영 대시보드, 서비스 헬스 카드, 운영 체크리스트, floating assignment badge를 상당히 잘 따라왔다. User Insights도 데모의 fixed top nav, left sidebar, bento grid, market mood gauge, Telegram CTA, footer 구조를 대부분 반영했다.

다만 제출 품질 관점에서는 아직 “보이는 완성도”와 “실제 동작/접근성/운영 안전성” 사이에 몇 가지 간극이 남아 있다.

핵심 잔여 리스크:

- Admin 서비스 카드의 `지금 실행`, `로그 보기`, `설정` 버튼이 실제 동작하지 않는다.
- Admin 운영 알림 센터에 `telethon_listener` JSON payload가 그대로 노출된다.
- Material Symbols 아이콘 텍스트가 스크린리더/접근성 트리에 그대로 노출된다.
- Mobile Admin에서 상단 네비게이션 링크가 완전히 사라져 `/insights` 이동성이 낮다.
- User Insights의 `뉴스` 링크는 `href="#"`로 실제 목적지가 없다.
- UI 파일과 CSS 파일이 매우 커져 유지보수성이 떨어진다.

## 3. 다관점 리뷰

### 3.1 제품/과제 관점

현재 상태는 뤼튼 Product Engineer 과제 제출물로서 강점이 뚜렷하다.

- “실제 데이터 기반 AI 요약/큐레이션 서비스”라는 과제 의도가 화면에서 더 잘 드러난다.
- 데모 HTML의 시각 언어를 실제 Next.js/Google Sheets 데이터와 결합했다.
- 운영자용 `/`와 사용자용 `/insights`가 역할상 분리되어 제품 기획 의도가 명확하다.

보완 필요:

- 평가자가 버튼을 눌렀을 때 아무 일도 안 일어나면 완성도가 낮아 보인다.
- 운영 로그 raw JSON은 “사람이 읽는 언어로 변환한다”는 제품 핵심 가치와 충돌한다.
- 기능이 없는 토글/아이콘은 데모 장식인지 실제 기능인지 명확히 구분해야 한다.

### 3.2 UX/콘텐츠 관점

좋아진 점:

- Admin 첫 화면은 데모와 매우 유사하며, 운영 상태를 빠르게 파악할 수 있다.
- User Insights는 브리핑, 분위기, 시그널, 워치리스트, 스토리, CTA 순서가 자연스럽다.
- 실제 데이터의 숫자와 사람이 읽는 문장이 함께 보여 설득력이 높다.

아쉬운 점:

- Admin 운영 알림 센터에 JSON이 그대로 보여 UX 완성도가 낮아진다.
- Admin 모바일은 네비게이션이 사라져 사용자가 `/insights`로 이동하기 어렵다.
- User Insights의 일부 컨트롤은 실제 클릭 결과가 없어 “시각적 목업”에 가깝다.

### 3.3 디자인/시각 관점

좋아진 점:

- Material Design 계열 색상 토큰과 ocean blue 톤이 안정적으로 적용됐다.
- 데모의 넓은 여백, 큰 타이포그래피, rounded card, soft shadow가 잘 재현됐다.
- Floating assignment badge와 sidebar CTA는 과제 제출물의 맥락을 잘 강화한다.

아쉬운 점:

- Desktop에서 카드와 타이포그래피가 매우 커서 정보 밀도가 낮다. 데모 유사성은 높지만 실운영 도구로는 약간 과장되어 보일 수 있다.
- Admin mobile에서 service card가 길게 누적되어 주요 브리핑까지 도달하는 데 스크롤 비용이 크다.
- Material icon 텍스트가 접근성 트리에 그대로 노출되어 스크린리더 사용성에 문제가 있다.

### 3.4 프론트엔드 코드 관점

좋아진 점:

- `npm run dashboard:typecheck`, `lint`, `build`가 모두 통과한다.
- Server Component 기반 App Router 구조를 유지하고 있다.
- 실제 Google Sheets 데이터 렌더링을 유지했다.

아쉬운 점:

- `apps/dashboard/app/page.tsx`가 매우 커져 UI, normalization, formatting, view logic이 한 파일에 집중되어 있다.
- `apps/dashboard/app/globals.css`도 1,400줄 이상으로 커져 Admin 전용 스타일과 전역 스타일의 경계가 약하다.
- `/insights`도 page와 CSS module이 커져, 향후 작은 UI 수정도 회귀 위험이 커진다.

### 3.5 운영/보안 관점

좋아진 점:

- API 인증과 에러 sanitization 개선이 기존 변경에 포함되어 있다.
- listener health는 Sheets 연결 여부가 아니라 `telethon_listener` heartbeat와 `tg_whale_events` fallback 기반으로 판단한다.

위험:

- 운영 로그 raw JSON이 화면에 직접 노출된다.
- `지금 실행` 버튼이 실제 pipeline trigger로 오해될 수 있다.
- 프로필/알림/토글 UI가 실제 사용자 상태와 연결되어 있지 않아 운영 혼선을 만들 수 있다.

## 4. 코드리뷰 Findings

### Finding 1. Major - 운영 알림 센터에 raw JSON이 노출됨

Location:

- `apps/dashboard/lib/metrics.ts:337-349`
- `apps/dashboard/app/page.tsx:480-512`

문제:

`normalizeSystemLogRows`가 `row.details`를 그대로 `message`에 넣고, `humanizeLogMessage`가 JSON payload를 해석하지 못해 운영 알림 센터에 아래 같은 값이 그대로 표시된다.

```text
{"category": "telethon listener", "level": "info", "payload": {...}}
```

영향:

- “사람이 읽는 언어로 변환한다”는 WhaleScope의 핵심 UX와 충돌한다.
- 평가자 입장에서는 아직 개발 중인 내부 로그가 노출되는 것처럼 보인다.
- 향후 Google API 오류나 내부 payload가 들어오면 운영 정보 노출 위험이 커진다.

권장 수정:

- `normalizeSystemLogRows`에서 raw `details`를 그대로 message로 쓰지 않는다.
- `parseJsonSafe`로 `details.payload.event`, `category`, `level`을 해석해 문장화한다.
- 예: `listener_start` → `Telegram listener가 @whale_alert_io 채널 수신을 시작했습니다.`
- 해석 불가 JSON은 `운영 이벤트가 기록되었습니다. 자세한 내용은 서버 로그를 확인하세요.`로 축약한다.

### Finding 2. Major - 서비스 카드 버튼이 실제 동작하지 않는데 실행 버튼처럼 보임

Location:

- `apps/dashboard/app/page.tsx:882-890`
- `apps/dashboard/app/page.tsx:1004-1011`

문제:

Admin 서비스 카드에 `지금 실행`, `로그 보기`, `설정` 버튼이 표시되지만 `onClick`, `href`, disabled 상태, 설명이 없다.

영향:

- 사용자는 `지금 실행`을 누르면 pipeline이 실행될 것으로 기대한다.
- 이전 요구사항 중 “대시보드에서 python -m src.main 실행 버튼” 니즈와 연결되어 있어 오해 가능성이 크다.
- 평가자가 버튼을 눌렀을 때 아무 변화가 없으면 완성도가 낮아 보인다.

권장 수정:

- 실제 실행 기능이 아직 없으면 `<button disabled>`와 `준비 중` 설명을 붙인다.
- 더 나은 방향은 `/api/pipeline/run` 같은 명시적 trigger endpoint를 만들되, `DASHBOARD_PASSWORD` 인증과 confirmation을 반드시 둔다.
- 단순 이동 목적이면 `<a href="#operator-checklist">Runbook 보기</a>`처럼 라벨을 바꾼다.

### Finding 3. Major - Material Symbols 아이콘 텍스트가 접근성 트리에 노출됨

Location:

- `apps/dashboard/app/page.tsx:967-974`
- `apps/dashboard/app/page.tsx:993-1010`
- `apps/dashboard/app/insights/page.tsx:477-490`
- `apps/dashboard/app/insights/page.tsx:553-558`

문제:

Material Symbols를 텍스트 span으로 렌더링하면서 decorative icon에 `aria-hidden="true"`가 없다. 실제 Chrome accessibility snapshot에서도 `waves`, `send`, `dns`, `dashboard`, `auto_awesome` 등이 텍스트처럼 노출된다.

영향:

- 스크린리더 사용자는 “waves”, “send”, “dns” 같은 의미 없는 텍스트를 듣게 된다.
- 접근성 품질이 낮아지고, UX 평가에서 감점 요인이 될 수 있다.

권장 수정:

- 장식용 아이콘에는 `aria-hidden="true"`를 부여한다.
- 버튼 아이콘은 버튼 자체의 `aria-label`로 의미를 제공하고 내부 icon span은 숨긴다.
- 반복되는 패턴은 `<MaterialIcon name="..." decorative />` 같은 작은 컴포넌트로 추출한다.

### Finding 4. Minor - Admin 모바일에서 네비게이션이 완전히 사라짐

Location:

- `apps/dashboard/app/globals.css:1127-1129`

문제:

`max-width: 860px`에서 `.top-navbar__nav`가 `display: none` 처리된다. 모바일 Admin 화면에서는 `/insights`, `#signals`, `#transactions`로 이동할 상단 경로가 없다.

영향:

- 모바일 사용자는 화면 간 이동이 어렵다.
- 데모 재현성은 높지만 실제 제출물을 모바일로 보는 평가자에게는 탐색성이 떨어진다.

권장 수정:

- 모바일에서 nav를 완전히 숨기지 말고 horizontal pill nav로 축약한다.
- 또는 WhaleScope 로고 옆에 `Insights` pill 하나만이라도 유지한다.

### Finding 5. Minor - User Insights의 `뉴스` 링크가 실제 목적지 없이 `#`로 연결됨

Location:

- `apps/dashboard/app/insights/page.tsx:29-34`

문제:

상단 nav의 `뉴스` 항목이 `href="#"`다.

영향:

- 클릭 시 의미 있는 이동이 없다.
- 완성된 제품보다 데모 목업처럼 보인다.

권장 수정:

- 실제 뉴스 섹션을 만들거나 링크를 제거한다.
- 준비 중인 기능이라면 disabled-like label 또는 `aria-disabled`를 사용한다.

### Finding 6. Suggestion - Admin page와 CSS가 너무 커져 유지보수성이 떨어짐

Location:

- `apps/dashboard/app/page.tsx`
- `apps/dashboard/app/globals.css`
- `apps/dashboard/app/insights/page.tsx`
- `apps/dashboard/app/insights/insights.module.css`

문제:

Admin `page.tsx`는 normalization, formatting, status mapping, presentation이 한 파일에 섞여 있다. `globals.css`도 Admin 전용 스타일과 전역 스타일이 함께 커졌다.

영향:

- 작은 UI 수정에도 회귀 가능성이 커진다.
- 데모 제출 직전에는 괜찮지만, 장기 운영 로드맵에서는 컴포넌트 단위 검토가 어려워진다.

권장 수정:

- `components/admin/*`로 `TopNavbar`, `ServiceHealthGrid`, `BriefPanel`, `SignalsPanel`, `Timeline`, `OperationLog`를 분리한다.
- formatting/normalization은 `lib/dashboard-display.ts`로 이동한다.
- Admin CSS는 CSS module 또는 route-scoped class 체계로 분리한다.

## 6. 최종 평가

현재 화면은 “데모 HTML과 실제 서비스 구현의 간극”을 상당히 줄인 상태다. 뤼튼 과제 제출 관점에서 시각적 완성도는 이전보다 명확히 좋아졌고, 실제 데이터 연동이 살아 있어 단순 HTML 데모보다 제품성이 높다.

다만 최종 제출 전에는 “보이는 컨트롤이 실제로 무엇을 하는가”, “운영자가 읽어야 하는 정보가 JSON이 아닌 문장인가”, “스크린리더에 불필요한 아이콘 텍스트가 노출되지 않는가”를 정리해야 한다. 이 세 가지를 해결하면 데모 유사성과 실제 제품 품질이 함께 올라간다.
