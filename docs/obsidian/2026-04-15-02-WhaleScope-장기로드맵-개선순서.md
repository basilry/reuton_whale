# WhaleScope 뤼튼 Product Engineer 과제형 로드맵 및 개선안 순서

## 전제

이 프로젝트는 일반 장기 제품 개발 프로젝트가 아니라 **뤼튼테크놀로지스 Product Engineer 과제 전형 제출물**이다.

따라서 우선순위는 다음 순서로 재정렬한다.

1. 평가자가 문제정의와 제품 판단을 빠르게 이해할 수 있어야 한다.
2. 로컬에서 재현 가능한 동작 구현체가 있어야 한다.
3. README만 보고 실행, 구조, 기술 스택, AI 도구 활용 방식을 이해할 수 있어야 한다.
4. 기술적 확장성보다 7일 과제 안에서 완성도 있는 작은 제품 검증이 중요하다.
5. 장기 로드맵은 제출 이후 계획으로 두고, 제출 전에는 MVP 완성도와 설명력을 우선한다.

## 과제 요구사항 요약

과제 성격:

- Product Engineer로서의 역량을 종합적으로 확인하기 위한 과제
- 코딩 테스트가 아니라 문제 발견, 가설 설정, 작은 제품 구현과 검증 과정을 평가
- 기술적으로 화려한 구현보다 사용자에게 의미 있는 경험을 만드는 판단을 더 중요하게 봄
- AI 도구를 활용해 혼자서 얼마나 빠르고 효과적으로 동작하는 제품을 만들 수 있는지 확인

선택 도메인:

- A. AI 사주/운세 서비스
- B. AI 이미지 생성 서비스
- C. AI 요약/큐레이션 서비스

WhaleScope의 선택 도메인:

- **C. AI 요약/큐레이션 서비스**

제출물:

- Git 레포지토리 1개
- One Pager
- 동작하는 구현체
- 로컬 실행 가능 상태 또는 배포 URL
- README에 실행 방법, 기술 스택, 사용한 AI 도구와 활용 방식 포함

One Pager 포함 항목:

- 이 도메인을 선택한 이유
- 기존 서비스들이 해결하지 못하고 있는 타겟과 문제
- 이 문제를 해결하기 위한 핵심 기능과 그 근거
- 이 피처의 성공을 판단할 수 있는 지표

## WhaleScope 과제 포지셔닝

WhaleScope는 "크립토 고래 거래를 추적하는 봇"이 아니라, **정보 과부하가 큰 온체인 이벤트를 사용자에게 의미 있는 시그널로 요약/큐레이션하는 AI 서비스**로 설명해야 한다.

제품 한 줄 설명:

- WhaleScope는 공개 온체인 데이터와 공개 Telegram 고래 알림을 수집해, 규칙 기반으로 중요한 이벤트를 선별하고 LLM이 한국어 브리핑으로 요약해 주는 AI 큐레이션 서비스다.

선택 도메인 이유:

- 요약/큐레이션은 LLM의 강점이 사용자 가치로 직접 연결되는 영역이다.
- 크립토 시장은 공개 데이터가 많지만 개인 사용자가 매일 해석하기 어렵다.
- 단순 알림은 많지만 "왜 중요한가", "지금 무엇을 봐야 하는가"를 정리해 주는 서비스는 부족하다.
- WhaleScope는 Product Engineer 과제에서 요구하는 문제 발견, 가설, 구현, 검증 흐름을 보여주기 좋다.

타겟 사용자:

- 크립토 시장을 계속 모니터링하지만 온체인 데이터를 직접 분석할 시간은 부족한 개인 투자자/리서처
- Whale Alert, 거래소 공지, Telegram 알림을 보지만 노이즈가 많아 핵심만 받고 싶은 사용자
- 한국어로 빠르게 맥락을 파악하고 싶은 사용자

핵심 문제:

- 기존 알림 서비스는 이벤트를 많이 보내지만 중요도와 맥락을 충분히 설명하지 않는다.
- 온체인 데이터, Telegram 알림, 가격 데이터가 분리되어 있어 사용자가 직접 조합해야 한다.
- 사용자는 "거래 발생"보다 "이게 지금 볼 만한 신호인가"를 알고 싶다.

핵심 가설:

- 규칙 기반 선별과 LLM 요약을 결합하면, 사용자는 원천 이벤트를 직접 뒤지는 것보다 더 빠르게 핵심 whale signal을 이해할 수 있다.
- 모든 이벤트를 분석하는 것보다 signal-first 큐레이션이 비용과 품질 측면에서 더 실용적이다.
- Telegram 브리핑과 Streamlit 대시보드 조합은 과제 기간 안에 제품 경험을 보여주기 적합하다.

## 과제 제출 기준 로드맵

### Phase A. 과제 서사와 One Pager 고정

목표 기간: D0-D1

목표:

- "왜 이 서비스를 만들었는가"를 먼저 고정한다.
- 평가자가 README와 One Pager를 읽고 3분 안에 제품 의도를 이해하게 만든다.

핵심 작업:

- `docs/one-pager.md` 또는 README 상단에 과제용 One Pager 섹션 추가
- 선택 도메인 `C. AI 요약/큐레이션 서비스` 명시
- 타겟 사용자와 문제를 투자 조언이 아니라 정보 큐레이션 문제로 정의
- 핵심 기능을 `수집 -> 선별 -> 요약 -> 전달 -> 확인` 흐름으로 설명
- 성공 지표를 과제 수준에서 정의

완료 기준:

- One Pager에 과제 PDF의 4개 요구 항목이 모두 들어간다.
- WhaleScope가 왜 AI 요약/큐레이션 서비스인지 명확하다.
- README 첫 화면에서 과제 전형용 프로젝트임을 알 수 있다.

### Phase B. 실제 LLM API 기반 제출 Demo Path 정리

목표 기간: D1-D2

목표:

- 실제 개발 서비스와 같은 경로로 LLM 한국어 브리핑이 생성되는 모습을 보여준다.
- smoke demo는 주 데모가 아니라 환경 이슈 대비 fallback으로 둔다.
- 과제 평가자가 "이 서비스가 실제로 AI 요약/큐레이션을 수행한다"는 점을 확인할 수 있게 한다.

핵심 작업:

- `python -m src.main` 또는 별도 `scripts/demo_real_llm.py`를 과제용 주 데모 경로로 지정
- fixture 또는 제한된 실제 수집 이벤트를 사용하되, 브리핑 생성은 실제 LLM API를 통과하게 구성
- README에 "Real LLM Demo"와 "Smoke Fallback"을 분리
- 실제 LLM 호출 결과 sample output을 `docs/demo-output.md` 또는 README에 첨부
- LLM provider, model, prompt version, token/cost/latency가 `analysis_log`에 남는지 확인
- `.env.example`과 실행 순서를 과제 평가자 기준으로 정리
- Streamlit 또는 CLI 결과 중 하나에서 실제 LLM 브리핑 결과를 빠르게 확인 가능하게 만든다.

완료 기준:

- `.env`에 LLM/Sheets/Telegram 설정을 넣으면 실제 서비스 경로로 브리핑이 생성된다.
- README의 첫 데모 경로가 실제 LLM API 호출을 포함한다.
- API 키가 없는 환경에서도 smoke fallback과 sample output으로 구조를 확인할 수 있다.
- 실패 시 LLM key, Sheets credential, Telegram token 중 어떤 의존성이 필요한지 명확히 보인다.

### Phase C. 제출 전 필수 코드 정상화

목표 기간: D2-D4

## 개선안 실행 순서

### 1순위. 과제용 One Pager 작성

이유:

- 과제 제출물의 명시 요구사항이다.
- Product Engineer 평가는 문제정의와 제품 판단을 먼저 본다.
- 코드가 좋아도 One Pager가 없으면 과제 의도를 놓친 제출물이 된다.

작업:

- `docs/one-pager.md` 생성 또는 README에 One Pager 섹션 추가
- 선택 도메인: `C. AI 요약/큐레이션 서비스`
- 선택 이유, 타겟/문제, 핵심 기능/근거, 성공 지표 작성
- 투자 조언이 아니라 정보 큐레이션 서비스임을 명확히 표현

### 2순위. README를 과제 평가자 기준으로 재구성

이유:

- README는 동작 구현체의 사용설명서이자 제품 설명서다.
- 과제 PDF가 README에 실행 방법, 기술 스택, AI 도구 활용 방식을 요구한다.

작업:

- 상단에 과제 전형용 프로젝트임을 명시
- Real LLM Quick Demo 추가
- Smoke Fallback과 Production Mode 분리
- 기술 스택 표 유지
- 사용한 AI 도구와 활용 방식 추가
- 제품 판단과 trade-off 섹션 추가

### 3순위. 실제 LLM API 기반 데모 경로 강화

이유:

- 이 과제의 본질은 AI 요약/큐레이션 서비스이므로 실제 LLM 호출 결과를 보여줘야 한다.
- smoke-only 데모는 안정적이지만 제품의 핵심 경험을 충분히 증명하지 못한다.
- 평가자는 "규칙으로 고른 시그널이 LLM을 통해 어떻게 사용자 언어로 바뀌는가"를 봐야 한다.

작업:

- 과제용 `Real LLM Demo` 명령을 README에 최상단 배치
- fixture 또는 작은 샘플 이벤트를 입력으로 사용해 비용과 실행 시간을 통제
- 실제 LLM API로 생성된 한국어 브리핑 sample output 저장
- `analysis_log`에 provider/model/prompt_version/cost/latency가 남는지 검증
- `scripts/smoke_pipeline.py`는 API 키가 없는 평가 환경을 위한 fallback으로 유지

### 4순위. `address_activity` 누적 경로 복구

이유:

- README 아키텍처와 실제 runtime의 불일치를 해결한다.
- baseline 기반 시그널의 전제 조건이다.
- 코드리뷰 Major 1이며, 평가자가 구조를 보면 발견하기 쉬운 결함이다.

작업:

- `Event -> address_activity row` adapter 추가
- `run_pipeline()` Stage 9에 `append_address_activity(raw_events)` 호출 추가
- 중복 key 정의
- 통합 테스트 추가

### 5순위. Signal summary 저장 구조 개선

이유:

- 이 프로젝트는 transaction list가 아니라 signal curation 서비스다.
- aggregate/TG-only signal에서 빈 심볼/0 USD가 저장되면 제품 경험이 약해진다.

작업:

- `_signals_to_top5()`를 signal-first 구조로 개선
- `signal_id`, `rule`, `summary`, `severity`, `score`, `source`, `evidence_count`, `window` 저장
- 금액을 모르는 경우 0으로 왜곡하지 않도록 표현
- 대시보드/README 용어 정리

### 6순위. Telegram 브리핑 정합성 수정

이유:

- 사용자 경험의 핵심은 LLM 한국어 브리핑이다.
- 현재 개인화 경로가 LLM 본문을 대체할 수 있어 제품 가치 전달이 흔들린다.

작업:

- `brief_text`를 기본 본문으로 항상 유지
- 개인화는 관심 시그널 섹션 append 방식으로 변경
- 관심 시그널이 없어도 전체 브리핑은 전달
- distributor 테스트 추가

### 7순위. `analysis_log` telemetry 정상화

이유:

- LLMRouter의 장점은 provider, prompt, token, cost, latency를 계측하는 데 있다.
- 과제 관점에서도 "AI 도구를 어떻게 활용했는가"를 보여주는 근거가 된다.

작업:

- `ANALYSIS_LOG_HEADERS` 확장 또는 `llm_usage_log` 분리
- migration script 또는 README 수동 migration 절차 작성
- 실제 row에 `model_id`, `prompt_version`, `cost_usd`, `latency_ms`가 남는 테스트 추가

### 8순위. Product Decisions 문서화

이유:

- 과제는 결과물뿐 아니라 과정의 고민과 선택을 본다.
- 기술 구현보다 제품적 판단을 설명하는 것이 중요하다.

작업:

작업:

## 과제 제출 전 하지 말아야 할 일

- Postgres 전환부터 시작하지 않는다.
- 사용자별 LLM 브리핑 생성처럼 비용이 커지는 기능부터 하지 않는다.
- outcome tracking 없이 rule을 많이 추가하지 않는다.
- README보다 내부 아키텍처 리팩토링에 시간을 더 쓰지 않는다.
- smoke-only 데모로 끝내지 않는다.
- API 키가 없는 환경을 위해 fallback과 sample output은 유지하되, 주 데모는 실제 LLM 호출이어야 한다.
- 투자 추천 서비스처럼 보이게 포지셔닝하지 않는다.

## 제출용 성공 지표

제품 지표:

- 사용자가 1분 안에 오늘의 핵심 whale signal을 이해할 수 있는가
- 상위 시그널이 raw transaction보다 더 읽기 쉬운가
- 브리핑이 "무슨 일이 있었는가"와 "왜 볼 만한가"를 설명하는가

구현 지표:

- 실제 LLM API를 통해 한국어 브리핑이 생성되는가
- smoke fallback이 외부 API 없이 실행되는가
- 전체 테스트가 통과하는가
- README 명령어가 실제로 동작하는가
- 주요 데이터 흐름이 저장소에 남는가

과제 평가 지표:

- 선택 도메인과 이유가 명확한가
- 타겟과 문제가 구체적인가
- 핵심 기능과 근거가 연결되는가
- 성공 지표가 정의되어 있는가
- AI 도구 활용 방식이 설명되어 있는가
- 7일 안에 만들 수 있는 범위로 잘 잘랐는가
