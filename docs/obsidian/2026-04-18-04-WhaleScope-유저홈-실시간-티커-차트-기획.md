# 유저홈 상단 실시간 티커 & 차트 기획

## 0. 요약

유저홈 최상단에 **Upbit/Binance 오픈 API 기반 실시간 티커와 미니 차트**를 배치한다(이슈 #9). 이 문서는 [[2026-04-18-02-WhaleScope-Binance-Upbit-데이터소스-확장분석]] 의 백엔드 파이프라인 계획을 기반으로, **사용자에게 보이는 UI 레이어**의 설계를 추가한다.

- 대상: 운영자 페이지(`/admin`) 가 아닌 **유저홈(`/`)** 최상단.
- 사용자: 일반 사용자(고래 흐름을 읽는 사람). 운영 지표는 제외.
- 시장 패리티: Binance (글로벌 가격) + Upbit (원화 프리미엄) 동시 노출로 **김치 프리미엄** 도 한눈에 표현 가능.

## 1. 배치 컨텍스트

### 1.1 상단 영역 정리

현재 `app/page.tsx` 의 상단 영역:

```
[TopNavbar (고정)]
  ↓
[InsightsSidebar (좌)]  [hero + brief 카드 (우)]
                        "시장 인텔리전스" + 연결 칩
```

개선 후 상단:

```
[TopNavbar]
  ↓
[실시간 티커 스트립 — 가로 스크롤 또는 고정 4~6 심볼]  ← NEW
  ↓
[InsightsSidebar] [hero + brief 카드 + 실시간 미니 차트 위젯]
```

### 1.2 사용자 동선 원칙

- **티커 스트립은 절대 사용자의 시선을 독점하면 안 된다**. 고래 브리핑이 여전히 메인 컨텐츠.
- 스트립 높이는 최대 40~48px (mobile), 48~56px (desktop).
- 미니 차트는 유저가 특정 심볼을 클릭했을 때 확장되는 형태.

## 2. 데이터 소스 전략

### 2.1 분석 문서(Binance/Upbit 데이터소스 확장분석)와의 관계

위 문서는 **백엔드 파이프라인의 Stage 4.5**(거래소 마켓 스냅샷)를 정의한다. 해당 문서의 파이프라인은 **느린 주기(1~5분)** 로 SignalEngine에 데이터를 주입한다. 본 문서의 **유저 UI 티커/차트** 는 그와 **별도의 빠른 경로**가 필요하다.

| 경로 | 주체 | 주기 | 용도 |
|---|---|---|---|
| 백엔드 Stage 4.5 | Python 수집기 (src/ingestion/binance.py, upbit.py) | 1~5분 | SignalEngine/브리핑 참조 |
| 프론트 티커 경로 | Next.js 서버 또는 브라우저 | 2~5초 | 유저에게 실시간 느낌 전달 |

두 경로의 **데이터는 같은 소스(Binance/Upbit 공식 API)를 쓰지만, 계층이 다르다**. 저장소에 커밋하는 것은 백엔드 경로이고, 프론트 티커는 디스플레이 전용.

### 2.2 프론트 티커 경로 선택지

| 옵션 | 설명 | 장점 | 단점 | 추천 Phase |
|---|---|---|---|---|
| A. 브라우저에서 직접 WebSocket 구독 | `wss://stream.binance.com:9443/ws/btcusdt@ticker` 등 | 서버 부담 0, 저지연 | CORS/접속수 제한, 브라우저 연결 유지 이슈 | Phase 1 (글로벌 심볼) |
| B. 브라우저에서 REST 폴링 | `GET /api/v3/ticker/24hr` 2~5초 | 구현 단순 | 지연 있음, rate limit 압박 | 항상 fallback |
| C. Next.js API Route 프록시 + SSE | 서버가 WebSocket 받아 여러 클라에 분배 | 연결수 관리 유리 | 서버 인프라 필요(Node.js 인스턴스 상시) | Phase 2 |
| D. Edge Function + KV 캐시 | Vercel/Cloudflare Workers | 글로벌 저지연 | 쓰기 빈도 제약, 1~2s 신선도만 보장 | 대안 |

**결론**:
- **Phase 1**: A(WebSocket) + B(fallback) 하이브리드.
- **Phase 2**: 트래픽이 커지면 C(SSE 프록시) 로 전환해 연결 집중/보안 강화.

### 2.3 Binance 엔드포인트

- WebSocket: `wss://stream.binance.com:9443/ws/<symbol>@ticker` — 1초 간격 24hr 티커.
- WebSocket (다중): `wss://stream.binance.com:9443/stream?streams=btcusdt@ticker/ethusdt@ticker/...`
- REST: `GET https://api.binance.com/api/v3/ticker/24hr?symbols=["BTCUSDT","ETHUSDT",...]`
- REST (미니 차트용 klines): `GET /api/v3/klines?symbol=BTCUSDT&interval=1m&limit=60`

### 2.4 Upbit 엔드포인트

- WebSocket: `wss://api.upbit.com/websocket/v1` (요청 바디로 구독)
  ```json
  [{"ticket":"whalescope"},{"type":"ticker","codes":["KRW-BTC","KRW-ETH","USDT-BTC"]}]
  ```
- REST: `GET https://api.upbit.com/v1/ticker?markets=KRW-BTC,KRW-ETH`
- REST (분 캔들): `GET https://api.upbit.com/v1/candles/minutes/1?market=KRW-BTC&count=60`

### 2.5 심볼 매핑 (중요)

| 표시 라벨 | Binance 심볼 | Upbit 마켓코드 |
|---|---|---|
| BTC/USDT | BTCUSDT | (없음; USDT-BTC 있지만 주요 가격은 KRW-BTC) |
| BTC/KRW | (없음) | KRW-BTC |
| ETH/USDT | ETHUSDT | — |
| ETH/KRW | — | KRW-ETH |
| USDT/KRW | (없음) | KRW-USDT (김치 프리미엄 계산에 유용) |
| SOL/USDT | SOLUSDT | KRW-SOL (있음) |

**김치 프리미엄** = (Upbit KRW-BTC 가격 / 현재 환율) / Binance BTCUSDT - 1. 환율은 별도 API(Korea Bank or exchangerate-api)에서 취득. Phase 2 로 분리.

## 3. 티커 UI 설계

### 3.1 기본 요소(원자)

```
┌─────────────────────────────────────────────┐
│ BTC  $67,420  +1.23%  ▲                     │
│       ₩96,200,000                           │
└─────────────────────────────────────────────┘
```

- **1행**: 심볼 · 기준통화 가격 · 변화율 · 방향 아이콘
- **2행**: 보조 통화 (Binance 가 기준이면 Upbit KRW 를 보조로, 또는 반대)
- **색상**:
  - 상승: `--good` (타이포 색만, 배경은 중립)
  - 하락: `--bad`
  - 변화 0 근처: `--muted`
- **스파크라인**(옵션): 1분 x 60 klines 로 얇은 선.

### 3.2 레이아웃 변형

#### 3.2.1 데스크탑(≥1024px) — 고정 그리드

상단에 6개 티커 카드를 한 줄에 보여준다. 가로 스크롤 없음.

```
[ BTC ] [ ETH ] [ SOL ] [ XRP ] [ USDT·프리미엄 ] [ 원/달러 환율 ]
```

#### 3.2.2 태블릿(768~1023px) — 4 카드 + 횡스크롤

#### 3.2.3 모바일(≤767px) — 3 카드 + 횡스크롤

터치 스와이프로 추가 심볼 노출. `scroll-snap-type: x mandatory`.

### 3.3 심볼 선정 원칙

**데스크탑 기본 6개**:
1. BTC
2. ETH
3. SOL (또는 국내 관심 상위 1종)
4. XRP (국내 상시 상위)
5. **김치 프리미엄 지표 (USDT 기준)**
6. **USD/KRW 환율**

**변경 가능성**: 하드코딩 대신 운영 환경변수 또는 `config/home-ticker.json` 로 관리.

### 3.4 상호작용

- **클릭**: 해당 심볼의 미니 차트 모달 또는 확장 패널 열림 (§4).
- **롱프레스 / 우클릭(데스크탑)**: 심볼 교체 UI (Phase 2).

### 3.5 접근성

- 업데이트마다 DOM 이 갱신되므로, 변경 내용을 스크린리더가 **읽지 않도록** `aria-live` 는 **꺼둔다**. 읽기 편한 요약은 별도 요청 시만.
- 색상 외에도 **화살표 아이콘(`▲`/`▼`)** 으로 방향 전달 (색맹 대응).
- 반짝임(flash) 애니메이션은 `prefers-reduced-motion` 시 비활성.

## 4. 미니 차트 위젯

### 4.1 목적

유저홈에서 깊은 분석이 아니라 **"지금 방향성"** 과 **"직전 1시간 움직임"** 을 전달. 풀 차트는 별도 페이지(`/markets/<symbol>`) 로.

### 4.2 차트 라이브러리 후보

| 라이브러리 | 장점 | 단점 | 번들 |
|---|---|---|---|
| Recharts | React 친화, 간단 | 무거움, 커스터마이즈 한계 | ~120KB gzip |
| visx (airbnb) | 유연, D3 기반 | 러닝커브 | ~80KB (부분 import) |
| lightweight-charts (TradingView) | 금융 차트 전문, 성능 | 커스터마이즈 제한 | ~40KB |
| Chart.js + react-chartjs-2 | 대중적, 쉬움 | 캔버스 렌더 — 반응형 어색 | ~70KB |
| d3 raw | 완전 제어 | 구현 많음 | ~50KB (필요 부분만) |

**권장**: `lightweight-charts` (TradingView). 금융 미니차트에 최적화, 성능 우수. 디자인 토큰과의 커플링은 테마 API 로 해결.

### 4.3 차트 데이터

- 소스: Binance REST `klines` (또는 Upbit 분 캔들).
- 간격: 1분, 60개 = 직전 1시간.
- 라이브 업데이트: 티커 가격이 들어올 때마다 마지막 캔들의 close 값만 업데이트.
- 거래량 표시 여부: MVP 는 가격선만 (심플).

### 4.4 UI

```
┌─────────────────────────────────────────────┐
│ BTC/USDT                          ×         │
│ $67,420                                     │
│  ╭─────────────────────────────────────╮    │
│  │  ∠\__/\_/\__/‾‾\_∠‾‾/\__           │    │
│  ╰─────────────────────────────────────╯    │
│ 1시간 변동폭: $66,980 ~ $67,620             │
│ 전체 차트 보기 →                            │
└─────────────────────────────────────────────┘
```

- 헤더: 심볼 + 현재가.
- 차트: 1시간 라인.
- 바닥: min/max 텍스트 + `/markets/<symbol>` 링크.

### 4.5 배치

- Phase 1: 티커 카드 클릭 시 **하단에 슬라이드 확장**.
- Phase 2: 심볼별 상세 페이지 `/markets/<symbol>` 로 이동.

## 5. 실시간 연결 아키텍처

### 5.1 컴포넌트 분리

```
app/page.tsx
  ↓ (Server Component)
<HomePageShell />
  ↓
<MarketTickerStrip symbols={["BTC","ETH","SOL","XRP","USDT-PREMIUM","FX-KRW"]} />
  ↓ (Client Component, "use client")
└─ internally uses hooks:
   - useBinanceTickerStream(symbols)   // WebSocket 관리
   - useUpbitTickerStream(markets)     // WebSocket 관리
   - useFxRate()                       // 환율
   - useMiniChart(symbol, interval)    // 히스토리 + 라이브
```

### 5.2 `useBinanceTickerStream` 훅

```ts
import { useEffect, useRef, useState } from "react";

type TickerSnapshot = {
  symbol: string;
  last: number;
  changePct: number;
  updatedAt: number;
};

export function useBinanceTickerStream(symbols: string[]) {
  const [snapshots, setSnapshots] = useState<Record<string, TickerSnapshot>>({});
  const wsRef = useRef<WebSocket | null>(null);

  useEffect(() => {
    const streams = symbols.map((s) => `${s.toLowerCase()}@ticker`).join("/");
    const ws = new WebSocket(`wss://stream.binance.com:9443/stream?streams=${streams}`);
    wsRef.current = ws;

    ws.onmessage = (evt) => {
      try {
        const payload = JSON.parse(evt.data);
        const data = payload.data;  // 24hr ticker
        setSnapshots((prev) => ({
          ...prev,
          [data.s]: {
            symbol: data.s,
            last: parseFloat(data.c),
            changePct: parseFloat(data.P),
            updatedAt: data.E,
          },
        }));
      } catch {}
    };

    // 자동 재연결
    ws.onclose = () => {/* exponential backoff */};
    ws.onerror = () => {/* log */};

    return () => ws.close();
  }, [symbols.join(",")]);

  return snapshots;
}
```

### 5.3 Fallback — REST 폴링

WebSocket 연결 실패 시 3초 간격 REST 폴링으로 전환. 전환 조건:
- `ws.onerror` 발생 후 재시도 3회 실패
- 브라우저가 WebSocket 미지원(거의 없음, 모니터링만)
- 특정 기업 네트워크에서 `wss` 차단

### 5.4 Tab Visibility 최적화

- `document.visibilityState === "hidden"` 일 때 업데이트 주기를 낮추거나 WebSocket을 닫는다.
- 재활성화 시 서버로부터 최신 상태를 받아 즉시 갱신.

### 5.5 에러/연결 상태 노출

- 연결 상태 뱃지: "연결됨" / "재연결 중" / "오프라인" (티커 스트립 오른쪽에 작은 dot).
- 30초 이상 데이터 갱신이 없으면 "지연" 배지.

## 6. 성능/비용 가드

### 6.1 렌더 최적화

- 티커 업데이트는 초당 수십 회 발생할 수 있음. **React 렌더 폭주 방지** 가 핵심.
  - 가격 바뀌지 않으면 setState 생략 (`===` 체크).
  - `useMemo`, `memo` 로 카드 단위 재렌더 격리.
  - 대규모 트래픽 시 `useSyncExternalStore` 기반 외부 스토어로 전환.

### 6.2 Binance rate limit

- WebSocket은 REST 보다 rate limit 완화. 다만 **연결 수 제한** 있음. 다중 브라우저 탭에서 같은 사용자가 여러 연결을 만들지 않도록 **BroadcastChannel** 로 탭 간 공유도 고려 (Phase 2).

### 6.3 비용

- WebSocket 직접 구독 방식은 사실상 무료.
- SSE 프록시 방식으로 전환 시 Render Node.js 인스턴스 필요 → 월 $7 이상.

## 9. 위험 & 오픈 이슈

### 9.1 데이터 신뢰성

- Binance/Upbit API 의 스팟 가격이 곧바로 "시장가" 는 아님. 특히 얇은 오더북 심볼은 왜곡 가능.
- 사용자에게 혼란을 주지 않도록 **출처 노출**: "데이터: Binance, Upbit" 을 스트립 오른쪽 끝에 작게.

### 9.2 법적/약관

- Binance WebSocket 재배포 약관 확인 필요. **사용자의 브라우저가 직접 구독**하는 방식이면 대개 안전. 서버가 중계해 여러 사용자에게 재배포하는 방식은 약관 검토 필요.
- Upbit WebSocket 도 마찬가지.

### 9.3 페이지 성능 영향

- 실시간 티커가 유저홈의 Core Web Vitals(LCP, CLS, INP) 에 악영향을 주지 않도록:
  - 초기 렌더에는 서버에서 받은 스냅샷으로 티커 placeholder 표시 (SSR) → WebSocket 연결 후 대체.
  - CLS 방지: 카드 고정 폭/높이.
  - INP 방지: 티커 업데이트 시 event loop block 최소화.

### 9.4 김치 프리미엄의 정의

- "KRW-BTC / (Binance BTCUSDT × USD→KRW)" — 1 이상이면 프리미엄.
- 환율 소스 변동성 큼. 단기 변동은 환율 탓인지 프리미엄 탓인지 모호할 수 있음. 툴팁으로 산출식 노출 필요.

## 11. 검증 체크리스트

### 기능
- [ ] 6개 심볼이 2초 이내에 최초 렌더.
- [ ] 1초 이내 Binance 틱 반영.
- [ ] WebSocket 끊김 시 3초 내 REST fallback.
- [ ] Tab hide 시 연결 닫힘, show 시 재연결.
- [ ] 심볼 클릭 시 1초 이내 미니 차트 표시.

### 성능
- [ ] 티커 스트립으로 인한 INP 증가 < 50ms.
- [ ] 홈페이지 LCP ≤ 2.5s (국내 기준).
- [ ] CLS = 0 (고정 폭/높이).

### 접근성
- [ ] 키보드만으로 심볼 간 포커스 이동 가능 (좌우 화살표 또는 Tab).
- [ ] 색맹 모드에서도 ▲/▼ 아이콘으로 방향 식별 가능.
- [ ] `prefers-reduced-motion: reduce` 시 flash 제거.

### 디자인
- [ ] 토큰 외 하드코딩 hex/rgba 없음.
- [ ] 다크/라이트 테마 모두 색 대비 ≥ 4.5:1 (텍스트), 3:1 (UI).
- [ ] AI-slop 금지 패턴(gradient text, side-stripe, `transition: all`) 위반 0.

### 운영
- [ ] 연결 실패율 모니터링(Sentry tag `feature=home-ticker`).
- [ ] Rate limit 경고 발생 시 자동 백오프 로그.

## 12. 다른 문서와의 경계

- **백엔드 Binance/Upbit 수집기**([[2026-04-18-02-WhaleScope-Binance-Upbit-데이터소스-확장분석]]): 시그널 엔진을 위한 저빈도 경로. 본 문서(프론트 고빈도 경로)와 **저장소를 공유하지 않는다**.
- **브리핑 파이프라인**([[2026-04-18-06-WhaleScope-브리핑-시장분위기-데이터파이프라인-재설계]]): 시장 분위기의 `drivers` 중 하나로 Binance/Upbit 변동성을 참조 가능. 공유점은 그 수준만.
- **감시지갑/스토리 문서**: 무관(온체인 전용).
- **v2 종합개선안**: 본 문서는 §2(사이드바 스크롤 스파이) 의 `id="brief"` 앵커 위에 위치하므로, 스크롤 스파이의 상단 여백(rootMargin)이 티커 스트립 높이를 포함해야 한다. → §2의 CSS `scroll-padding-top` 값에 티커 높이 포함시켜야 함. 구현 시 명시할 것.
