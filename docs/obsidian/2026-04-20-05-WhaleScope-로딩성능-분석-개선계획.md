# WhaleScope 로딩 성능 분석 및 개선안 (v1)

## 배경

Redis L2 캐시 통합 직후에도 초기 진입 체감 로딩이 느리다. 이 문서는 구체적 병목을 코드/빌드/런타임 측정 기반으로 식별하고, 영향도 순 개선안을 정리한다.

## 현황 계측 (dev, localhost:3000)

| 경로 | cold total | warm total | payload |
|---|---|---|---|
| `/` | 10.5 s | 1.35 s | 192 KB HTML |
| `/admin` | 14.6 s | 1.61 s | 638 KB HTML |
| `/insights` | 1 ms (5줄 re-export) | 1 ms | 1 B |

- cold은 Next dev의 on-demand 컴파일이 섞여 과장된 수치. 그러나 warm에서도 1.3–1.6초면 체감상 느림.
- `/admin` HTML이 638 KB — 단일 `page.tsx`가 1,673 줄짜리 monolith에서 SSR됨을 시사.

## 빌드 산출물 분석 (`yarn build`)

```
Route (app)                    Size     First Load JS
/                             193 kB    312 kB
/admin                        6.32 kB   126 kB
/insights                     164 B     102 kB
/preview                      1.9 kB    113 kB
```

- `/` First Load JS **312 KB**(gzip). WhaleScope 같은 데이터 대시보드에 비해 무겁다(목표 ≤ 200 KB).
- `.next/static/chunks/app/page.js` = 8.4 MB(raw). `main-app.js` = 7.3 MB(raw). chunks 전체 16 MB.
- dependencies: `lightweight-charts` + `recharts` **양쪽 다 포함**. recharts는 `curated-wallet-detail-modal` 한 곳에서만 쓰임 → 통일 시 수십 KB 절감 가능.

## 병목 식별

### 1. Suspense 경계가 전무 — 스트리밍 0

```bash
$ grep -r "Suspense\|dynamic(\|loading.tsx" app/
app/error.tsx
```

- `error.tsx` 외에는 어떤 경계도 없음. 가장 느린 하위 컴포넌트가 TTFB를 혼자 결정.
- `app/loading.tsx`도 없음 → 사용자는 페이지가 완전히 준비될 때까지 빈 화면.

### 2. 루트 페이지의 직렬 await 4개

`apps/dashboard/app/page.tsx:478`:
```ts
export default async function InsightsPage() {
  const dictionary = await getCurrentDashboardDictionary();
  const language = await getCurrentDashboardLanguage();
  const state = await loadInsightState();   // ← 무거움 (~500ms–2s)
  const fearGreed = await getFearGreedData();
```

- dictionary/language는 쿠키 read라 가볍지만, 4개가 **순차 실행**. `Promise.all`로 묶으면 가장 느린 하나만큼만 걸림.

### 3. `/admin`은 1,673 줄짜리 단일 서버 컴포넌트

`apps/dashboard/app/admin/page.tsx:69`:
```ts
...(await getDashboardData()),
```

- 홈에서 쓰지 않는 admin-only 데이터(`render observability`, `brief_cost_ledger`, `llm_budget_log`, `broadcast_log` 등)까지 `getDashboardData`에서 **모두** 로드.
- 홈도 같은 함수를 쓰므로 위 무거운 데이터가 홈 로딩에도 끌려옴.

### 4. `getDashboardData` 한 번에 12+ Sheets tab + Render API

`apps/dashboard/lib/metrics.ts:2920`:
```ts
const [snapshotResult, curatedWalletBundle, serviceHealthRows,
       channelHealthRows, briefCostLedgerRows, broadcastLogRows,
       llmBudgetRows, watchedAddressRows, renderObservability]
  = await Promise.all([...]);
```

- `Promise.all` 병렬화 자체는 OK. 문제는 **필요 없는 데이터까지 매번 동시 호출** → Redis 미스 시 많은 Sheets API/외부 API를 동시 폭격.
- `loadRenderObservability`는 Render REST API. cold일 때 수초 걸릴 수 있음.

### 5. 클라이언트 모달·차트가 초기 번들에 포함

- 25개 client component 중 5개는 모달 (whale-story-detail, signal-detail, market-detail-chart, telegram-connect, curated-wallet-detail).
- `next/dynamic(() => import(...), { ssr: false })`로 분리 안 됨 → 열지도 않는 모달이 초기 JS에 포함.

### 6. `force-dynamic` 전면 적용

`app/page.tsx:23` `app/admin/page.tsx:44` 모두 `dynamic = "force-dynamic"`.

- 정적 최적화/ISR 봉쇄. 모든 요청이 서버 렌더 + Sheets/Redis 왕복 필연.
- 실제로는 홈도 60s ISR로 충분. 실시간성은 client fetch(`/api/dashboard`, `/api/stream`)로 해결 가능.

### 7. 캐시 warm-up 부재

- Redis L2 TTL 45s. 트래픽이 한산하면 거의 매 요청이 L2 miss → Sheets API 원격 호출.
- 스케줄러가 캐시를 갱신해주지 않음 → 체감상 "느릴 때 정말 느리다".

### 8. Upbit/Bitflyer 프록시는 `force-dynamic` + `cache: "no-store"` 가능성

- `/api/proxy/upbit/[...path]` `/api/proxy/bitflyer/[...path]` 모두 `force-dynamic`. 클라이언트에서 ticker 폴링 시마다 프록시 경유 → 서버 왕복 2회(브라우저 → Vercel → 외부).

## 개선안 (영향도 순)

### Phase A. 스트리밍 UI (가장 큰 체감 개선, 1–2 일)

1. **`app/loading.tsx` 추가**
   ```tsx
   export default function Loading() {
     return <DashboardSkeleton />;
   }
   ```
   - 페이지 shell + 스켈레톤을 즉시 표시. 체감 로딩 → 수십 ms.

2. **루트 페이지를 섹션별 Suspense로 분리**
   ```tsx
   <Suspense fallback={<BriefSkeleton />}><BriefSection /></Suspense>
   <Suspense fallback={<WatchlistSkeleton />}><WatchlistSection /></Suspense>
   <Suspense fallback={<SignalsSkeleton />}><SignalsSection /></Suspense>
   ```
   - 각 섹션이 독립적으로 resolve → TopNavbar/Hero는 즉시, 느린 섹션은 스트리밍.
   - LCP가 "페이지 전체 로딩"이 아니라 "Hero 렌더"로 이동.

3. **루트 4 awaits 병렬화**
   ```ts
   const [dictionary, language, state, fearGreed] = await Promise.all([
     getCurrentDashboardDictionary(),
     getCurrentDashboardLanguage(),
     loadInsightState(),
     getFearGreedData(),
   ]);
   ```
   - 구현 비용 5분. 직렬 → 병렬로 최대 50% 단축 가능.

### Phase B. 데이터 페치 다이어트 (0.5일)

4. **`getDashboardData`를 페이지 용도별 슬림 로더로 분리**
   - `getHomeData()` — transactions, signals, system_log, daily_brief, tg_whale_events, subscribers, curated watchlist, fear-greed. (대시보드 탭 배치 + curated만).
   - `getAdminData()` — 홈 데이터 + service_health, channel_health, brief_cost_ledger, broadcast_log, llm_budget_log, watched_addresses, render observability.
   - 홈의 외부 API 호출 수가 9 → 2 (Sheets batch, fear-greed)로 줄어든다.

5. **`loadRenderObservability`는 /admin 전용으로 이동**
   - 홈에서는 import 자체 제거.

6. **`getDashboardData` options에 `include: { adminExtras?: boolean }` 추가**
   - 기존 호출자 호환 유지.

### Phase C. 번들 사이즈 축소 (1일)

7. **`recharts` 제거, `lightweight-charts`로 통일**
   - `curated-wallet-detail-modal.tsx` 하나만 recharts 씀 → lightweight-charts로 치환 or modal 전용 lazy import.
   - 번들 추정 절감: 60–90 KB gzip.

8. **모달 컴포넌트 전부 `next/dynamic`로 lazy load**
   ```ts
   const WhaleStoryDetailModal = dynamic(
     () => import("@/components/whale-story-detail-modal"),
     { ssr: false, loading: () => null },
   );
   ```
   - 대상: whale-story-detail, signal-detail, market-detail-chart-modal, telegram-connect, curated-wallet-detail.
   - 열리지 않은 모달 JS는 초기 페이로드에서 제외 → First Load JS 100 KB 이상 절감 기대.

9. **`next.config.ts`에 `experimental.optimizePackageImports` 적용**
   ```ts
   experimental: {
     optimizePackageImports: ["lightweight-charts", "lucide-react"],
   },
   ```

10. **admin page 컴포넌트 분리**
    - 1,673 줄을 최소 6개 섹션 컴포넌트로 쪼갬. 각각을 Suspense로 감쌈.

### Phase D. 캐시 전략 고도화 (0.5일)

11. **탭별 TTL 차등화** (`lib/redis-cache.ts`)
    - Hot(빈번 업데이트): transactions, signals, system_log, daily_brief → 45 s
    - Warm(중간): tg_whale_events, subscribers → 2 min
    - Cold(안정): service_health, channel_health, brief_cost_ledger, broadcast_log, llm_budget_log, watched_addresses → 5–15 min

12. **warm-up 크론 추가** (Render cron + `/api/internal/cache-warm`)
    - 30초마다 `readDashboardSnapshot()` 1회 호출. 사용자는 항상 L2 hit로 응답.
    - 요청 레이턴시: cold 미스 500–1000 ms → L2 hit 30–50 ms로 상시 유지.

13. **ISR 복귀 검토**
    - 홈/`/admin`을 `revalidate = 30` + client hydrate(live updates)로 전환.
    - `force-dynamic` 제거 시 Vercel CDN edge cache 활용 가능.

14. **`/api/dashboard` `/api/signals` 등 REST 엔드포인트에 `Cache-Control`**
    ```ts
    return Response.json(data, {
      headers: { "Cache-Control": "s-maxage=30, stale-while-revalidate=60" },
    });
    ```

### Phase E. 프록시 & 외부 의존 (0.5일)

15. **Upbit/Bitflyer 프록시 응답에 단시간 캐시**
    - `Cache-Control: s-maxage=5, stale-while-revalidate=10`. 초단기지만 라이브 틱에서도 QPS 급감.

16. **Vercel 리전을 `icn1`(Seoul)로 고정**
    - `vercel.json` 또는 `next.config.ts`에 `regions: ["icn1"]`.
    - Upstash Redis 리전도 Seoul로 맞추면 L2 지연 < 10 ms 유지.

### Phase F. 측정 & 관측 (0.5일)

17. **Vercel Speed Insights 켜기** — LCP/FCP/INP/TTFB 실측.
18. **Sheets API 호출 카운터 로깅** — Redis hit/miss, quota 소비량 대시보드.
19. **페이지별 서버 컴포넌트 타이밍** (`performance.now()` 기반) — 어느 섹션이 느린지 파악.

## 기대 효과 (추정)

| Phase | 변경 | 체감 개선 | 구현 비용 |
|---|---|---|---|
| A | Suspense + loading.tsx + 4 awaits 병렬 | TTFB 체감 1.4s → 0.2s | 1–2 d |
| B | 데이터 페치 슬림화 | 홈 서버 작업 30–50% 절감 | 0.5 d |
| C | recharts 제거 + 모달 lazy | First Load JS 312 KB → ~200 KB | 1 d |
| D | 캐시 warm-up + 차등 TTL | L2 hit율 50% → 95% | 0.5 d |
| E | 프록시 캐시 + 리전 | 라이브 틱 QPS 감소, Redis RTT ↓ | 0.5 d |
| F | 관측 | 정량 검증 가능 | 0.5 d |

**합산**: 약 4–5 일 작업으로 체감 LCP를 데스크탑 기준 500 ms 이하로 낮출 수 있을 것으로 판단.
