# WhaleScope 유저홈 점검 및 개선안

## 1. 점검 대상

- URL: `https://whalescope.6esk.com/`
- 확인 시각: 2026-04-27 14:15 KST 전후
- 확인 방식:
  - 배포 유저홈 DOM snapshot 확인
  - `https://whalescope.6esk.com/api/dashboard` payload 확인
  - 관련 코드 확인

## 2. 결론

운영 파이프라인은 다시 돌고 있으나, 유저홈은 아직 “사용자가 읽고 판단하기 좋은 시장 브리핑” 단계까지 올라오지 못했다.

핵심 문제는 다음이다.

1. 최신 브리핑이 LLM signal 기반 브리핑이 아니라 `fallback_tx_based` 브리핑이다.
2. 최근 60분 거래 200건이 모두 `priced=0`, `unpriced=200`이라 USD 규모와 중요도 해석이 없다.
3. `highlights`가 Python list 문자열처럼 저장되어 UI에서 콤마 단위로 깨진다.
4. 유저홈의 규모 요약이 최신 브리핑 기준이 아니라 전체 누적 transaction count를 사용한다.
5. 4월 21~22일의 오래된 signals/stories가 “감지된 주요 시그널”처럼 현재형으로 노출된다.
6. Telegram CTA는 공개 채널 인원 대신 DM subscriber count를 보여 `현재 구독자 수는 0명`으로 보인다.

따라서 우선순위는 디자인 추가가 아니라 **브리핑 데이터 의미 정합성, fallback 설명, stale 데이터 라벨링, 사용자 문장화 품질 개선**이다.

## 3. 현재 배포 데이터 증거

최신 `latestBrief`:

```json
{
  "generatedAt": "2026-04-27T05:01:59.096Z",
  "summary": "최근 60분 온체인 이동 200건 감지. 대표 이동은 TRIA 210,592건 규모이며 성격은 지갑 간 이동입니다. USD 환산은 현재 가격 응답 지연으로 일부 보류되었습니다.",
  "alertCount": 0,
  "totalVolumeUsd": 0,
  "signalThemes": ["온체인 대형 이동", "시그널 부족 fallback"],
  "noteRaw": "fallback_tx_based|signals=0|window_min=60|transactions=200|priced=0|unpriced=200"
}
```

`highlights` 파싱 결과:

```json
[
  "['TRIA · 210",
  "592 TRIA · 지갑 간 이동'",
  "'ZKP · 202",
  "435 ZKP · 지갑 간 이동'",
  "'BARD · 110",
  "886 BARD · 지갑 간 이동'",
  "'BSB · 81",
  "423 BSB · 지갑 간 이동']"
]
```

최근 `signals`:

```text
latest signal = 2026-04-21T19:03:04.979Z
recent signals = 2026-04-21 / 2026-04-20
today signals = 0
```

Telegram:

```text
subscriberCount = 0
channelMemberCountLatest = 2
lastBroadcastStatus = sent
```

즉 유저홈에서 “현재 구독자 수 0명”은 공개 채널 기준이 아니라 DM subscriber 기준을 잘못 보여주는 것이다.

## 4. 화면별 문제

### 4.1 오늘의 고래 브리핑

현재 화면:

```text
최근 60분 온체인 이동 200건 감지.
대표 이동은 TRIA 210,592건 규모이며 성격은 지갑 간 이동입니다.
USD 환산은 현재 가격 응답 지연으로 일부 보류되었습니다.
```

문제:

- `TRIA 210,592건`은 사람이 읽기 애매하다. 토큰 수량인지 이벤트 수인지 즉시 알기 어렵다.
- `USD 환산 보류`가 핵심인데 risk banner는 “경고성 포인트는 제한적입니다”라고만 말한다.
- `fallback` 브리핑임을 badge로만 표시하고, 왜 fallback인지 설명하지 않는다.
- 사용자는 “시스템이 정상인가, 데이터가 부족한가, 시장이 조용한가”를 구분하기 어렵다.

원인 파일:

- `src/pipeline/brief.py`
  - `_build_transaction_fallback_brief`
  - `_fallback_summary`
  - `_fallback_highlights`
- `apps/dashboard/app/page.tsx`
  - `buildBriefCopy`
  - `buildBriefAnalysis`

### 4.2 주요 포인트 깨짐

현재 화면:

```text
['TRIA · 210 · 592 TRIA · 지갑 간 이동'
```

원인:

- `_fallback_highlights()`는 `list[str]`을 반환한다.
- `PostgresClient.save_daily_brief()`는 `highlights`를 `_text()`로 저장한다.
- 결과적으로 PostgreSQL에는 JSON array가 아니라 Python list 문자열이 들어간다.
- `normalizeStringList()`는 JSON parse 실패 후 `,`로 split한다.
- `210,592`의 콤마가 항목 구분자로 오해되어 하이라이트가 깨진다.

관련 코드:

- `src/pipeline/brief.py:_fallback_highlights`
- `src/storage/postgres_client.py:save_daily_brief`
- `apps/dashboard/lib/metrics.ts:normalizeStringList`

### 4.3 규모 요약 혼선

현재 화면:

```text
규모 요약
$0 · 5.9K건
```

문제:

- `$0`은 latest brief의 `totalVolumeUsd=0`이다.
- `5.9K건`은 latest brief의 60분 transaction 200건이 아니라 전체 `metrics.transactionCount=5923`이다.
- 사용자는 “이번 브리핑에서 5.9K건을 분석했는데 총액이 $0인가?”로 오해할 수 있다.

원인:

- `buildBriefAnalysis()`에서 `transactionCount = data?.metrics.transactionCount`를 사용한다.
- 최신 brief의 `noteRaw`에는 `transactions=200`이 있지만 UI 모델로 노출하지 않는다.

관련 코드:

- `apps/dashboard/app/page.tsx:buildBriefAnalysis`
- `apps/dashboard/lib/metrics.ts:parseBriefNote`

### 4.4 오래된 시그널/스토리 노출

현재 화면:

- “감지된 주요 시그널”에는 4월 21~22일 row가 현재형처럼 노출된다.
- “고래 스토리”도 4월 20~22일 signals 기반이다.

문제:

- 2026-04-27 현재 최근 signal은 없다.
- stale signal을 현재형 섹션에 그대로 보여주면 서비스가 실시간으로 신호를 잡는 것처럼 오해된다.

권장:

- 최근 24시간 내 signal이 없으면 section title을 “최근 감지된 주요 시그널 없음”으로 바꾼다.
- 오래된 signal은 “이전 사례” 또는 “최근 보관된 시그널” 섹션으로 분리한다.
- 각 signal card에 `마지막 감지: 4월 22일` stale badge를 표시한다.

관련 코드:

- `apps/dashboard/app/page.tsx`
- `apps/dashboard/lib/metrics.ts:recentSignals`
- `apps/dashboard/lib/whale-stories.ts`

### 4.5 Telegram CTA 구독자 수 오류

현재 화면:

```text
현재 구독자 수는 0명입니다.
```

실제 운영 데이터:

```text
telegram.channelMemberCountLatest = 2
metrics.subscriberCount = 0
```

문제:

- 유저홈은 공개 채널 중심 UX로 전환했는데, 여전히 DM subscriber count를 구독자 수로 보여준다.
- 공개 채널 관점에서는 `channelMemberCountLatest`를 써야 한다.

관련 코드:

- `apps/dashboard/app/page.tsx`
- `apps/dashboard/components/telegram-connect-modal.tsx`
- `apps/dashboard/lib/types.ts`

### 4.6 감시 지갑 섹션의 의미 부족

현재 화면:

- 다수 지갑이 `C등급 거래소`, `멀티체인`, generic note로 표시된다.
- 실제 related signal count는 대부분 0이다.

문제:

- 유저 관점에서 “왜 이 지갑을 봐야 하는가”가 아직 약하다.
- 오늘 활동이 없는 지갑과 오늘 의미 있는 지갑이 구분되지 않는다.

권장:

- “오늘 움직임 있음 / 관망 / 데이터 대기” badge를 추가한다.
- `C등급`만 보여주지 말고 “우선순위 낮음, 기준 지갑”처럼 해석 문구를 붙인다.
- 관련 signal이 없으면 “오늘은 참고 지갑으로만 유지”라고 명확히 말한다.

## 5. 개선 방향

### Phase 1 — 표현/파싱 즉시 수정

목표: 데이터 원천이 부족해도 유저가 오해하지 않게 만든다.

작업:

- `highlights`를 JSON array로 저장하거나 dashboard parser에서 Python list 문자열을 안전하게 파싱한다.
- `noteRaw`의 `transactions`, `priced`, `unpriced`, `signals`, `fallback_source`를 구조화해 `latestBrief.meta`로 노출한다.
- `buildBriefCopy.note`가 `brief.note`를 우선 사용하도록 바꾼다.
- `buildBriefAnalysis`의 규모 요약은 전체 누적 `metrics.transactionCount`가 아니라 latest brief meta의 `transactions=200`을 사용한다.
- `priced=0`이면 `$0`을 강조하지 말고 “USD 환산 대기”를 주요 상태로 표시한다.

수용 기준:

- 주요 포인트에 `['TRIA · 210` 같은 깨진 문자열이 나오지 않는다.
- 규모 요약이 `$0 · 5.9K건`으로 나오지 않는다.
- fallback 브리핑에서 “시그널 없음 / 가격 미확인 / 거래 200건 기준”이 사용자 문장으로 보인다.

### Phase 2 — fallback 브리핑 문장 품질 개선

목표: signal이 없어도 브리핑이 “부실한 오류 화면”처럼 보이지 않게 만든다.

작업:

- `_fallback_summary()`를 다음 구조로 바꾼다.
  - 한 줄 결론: “최근 60분은 가격 미확인 토큰 이동이 많아 방향성 판단을 보류합니다.”
  - 관측량: “온체인 이동 200건”
  - 가격 품질: “USD 환산 0건 / 미확인 200건”
  - 대표 이동: “TRIA 210,592 TRIA, 지갑 간 이동”
  - 해석: “거래소 유입/유출 신호는 아직 확인되지 않음”
- `highlights`는 숫자에 콤마가 들어가도 안전한 JSON array로 저장한다.
- `topTransactions`에 `amountToken`, `amountUsdKnown`, `movementLabel`, `interpretation`을 dashboard에 노출한다.

수용 기준:

- 유저가 “오늘은 조용한 시장인지, 데이터가 부족한지”를 5초 안에 구분할 수 있다.
- 가격 미확인 fallback이 숨겨지지 않는다.

### Phase 3 — signal/stories freshness 정리

목표: 오래된 signal을 현재 신호처럼 보이지 않게 한다.

작업:

- 최근 24시간 내 signal이 0이면 `감지된 주요 시그널` 섹션을 empty-state로 전환한다.
- stale signal은 `이전 감지 사례` 섹션으로 이동하거나 별도 badge를 붙인다.
- `whaleStories`도 최신 signal이 아니면 “보관된 스토리”로 라벨링한다.

수용 기준:

- 2026-04-21 signal이 2026-04-27 현재 신호처럼 보이지 않는다.
- 최근 signal 0건일 때도 서비스 상태가 자연스럽게 설명된다.

### Phase 4 — Telegram 공개 채널 지표 정합화

목표: 채널 중심 운영 UX와 숫자 표시를 맞춘다.

작업:

- 유저홈 구독자 수는 `adminObservability.telegram.channelMemberCountLatest`를 우선 사용한다.
- DM subscriber count는 유저홈에서 숨기거나 “개인 DM 구독자”로 별도 구분한다.
- channel member count를 못 읽으면 “채널 인원 확인 중”으로 표시한다.

수용 기준:

- 공개 채널에 2명이 있는데 유저홈에 0명으로 나오지 않는다.

### Phase 5 — 데이터 풍부화

목표: 브리핑 자체의 정보량을 높인다.

작업:

- 가격 resolver가 모르는 토큰은 “가격 미확인 토큰” bucket으로 분리한다.
- unknown token이 fallback 대표 이동을 독점하지 않도록 stable/major asset 우선순위를 둔다.
- `from/to owner_type=unknown` 비율을 줄이기 위해 watched address label과 exchange address registry를 확장한다.
- signal이 없어도 `large_unpriced_transfer`, `unknown_token_burst`, `exchange_contract_interaction` 같은 low severity observation을 별도 observation 테이블 또는 brief-only item으로 제공한다.

수용 기준:

- 가격 미확인 토큰이 많아도 브리핑이 “TRIA 210,592건” 한 줄로 끝나지 않는다.
- 사용자가 “오늘 관찰된 것은 가격 확정 신호가 아니라 미확인 토큰 이동 클러스터”라고 이해한다.

## 10. 2026-04-27 구현 반영 메모

이번 반영은 Phase 1~4의 “오해 방지” 범위를 우선 닫았다.

- `highlights` 저장 경로를 Postgres/Sheets 모두 JSON array 기반으로 맞췄고, 기존 Python list 문자열도 dashboard parser가 안전하게 복구하도록 했다.
- `noteRaw`의 `signals`, `transactions`, `priced`, `unpriced`, `window_min`, `fallbackMode`를 `latestBrief.meta/quality`로 구조화했다.
- 유저홈 규모 요약은 전체 누적 row 수가 아니라 최신 brief meta의 거래 수를 우선 사용한다.
- `priced=0` fallback에서는 `$0`을 주요 수치처럼 보여주지 않고 `USD 환산 대기` 상태로 표시한다.
- fallback summary는 “가격 미확인 거래 fallback”임을 명시하고, 대표 이동은 토큰 수량과 이동 성격을 분리해 설명한다.
- 최근 24시간 signal이 없으면 오래된 signal을 현재 감지처럼 보여주지 않고 “이전 감지 사례”로 분리한다.
- Telegram CTA는 DM subscriber count가 아니라 공개 채널 `channelMemberCountLatest`를 우선 사용한다. 홈은 운영 전체 집계를 켜지 않고 기본 `channel_health` 로딩 값만 사용하도록 경량화했다.

남은 작업은 데이터 풍부화와 시각적 상세 노출이다.

- top transaction 상세 문장을 유저홈에 직접 노출하는 UI는 아직 남아 있다.
- whaleStories stale 라벨링은 아직 남아 있다.
- 가격 미확인 토큰 bucket, major/stable 우선 fallback ranking, exchange address registry 확장, low severity observation 설계는 다음 페이즈로 남긴다.

## 9. 우선 결론

현재 유저홈의 문제는 운영 장애가 아니라 **fallback 상태를 사용자에게 제대로 설명하지 못하는 UX/데이터 모델 문제**다.

가장 먼저 고쳐야 할 것은 `highlights` 깨짐과 `$0 · 5.9K건` 혼선이다. 이 두 가지는 데이터가 부족하더라도 서비스 신뢰도를 크게 떨어뜨린다.

그 다음은 stale signal을 현재 신호처럼 보이지 않게 하는 것이다. 유저에게는 “오늘 유의미한 신호 없음”이 더 정직하고 신뢰도 높은 표현이다.
